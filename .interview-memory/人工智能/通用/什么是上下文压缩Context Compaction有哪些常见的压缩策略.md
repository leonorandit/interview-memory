## 速记卡片
`缩 | 摘要映射分层存，Token省下半边天 | 丢细节保结构，长上下文降本增效`

## 原始知识
- **动机**：长上下文窗口≠低成本，KV Cache 显存 O(n²)，推理速度随长度衰减
- **策略一：文本摘要**：历史对话做 LLM 摘要，保留关键信息丢弃冗余
- **策略二：语义检索/向量召回**：RAG 替代全量上下文，只召回相关 chunk
- **策略三：分层压缩**：近期原始文本+远期摘要+最远期只保留实体/意图
- **策略四：KV Cache 压缩**：H2O、StreamingLLM、SnapKV，只保留关键 token 的 KV
- **策略五：选择性丢弃**：基于注意力权重裁剪低贡献 token
- **策略六：结构化压缩**：将自然语言转为 JSON/表格/知识图谱，信息密度更高
- **策略七：外部记忆**：用数据库/向量库/知识图谱外挂记忆，模型只读摘要指针
- **关键指标**：压缩率、信息保留率、下游任务准确率衰减
- **2026趋势**：在线自适应压缩（根据任务难度动态选择压缩强度）

## 单字锚点 + 防崩推导
- **锚点字**：缩
- **扩展口诀**：缩字一把刀，摘要检索KV削（摘要、检索、KV Cache 压缩三把刀）
- **为什么选它**："缩"直接对应 compaction/compression，字形"纟"（缠绕/整理）+"宿"（住宿/留存），暗示把散乱信息整理后留存精华
- **防崩推导**：
  - 如果只记"缩"→ 想到"压缩什么"→ 文本长度 / KV Cache / 外部记忆
  - 想到"怎么缩"→ 摘要（丢冗余）、检索（只取相关）、KV削（算法层压缩）
  - 想到"缩完怕什么"→ 信息丢失→所以要分层，近期精、远期粗
  - 漏了"结构化"？→ "缩"衣节食，结构化就是给信息"节食"，用更少token存更多意思

## 一字一画面
**缩** —— 像**羽绒服真空压缩袋**：蓬松的羽绒（原始长文本）被抽气泵（摘要算法）抽成扁平方块，挂钩标签（检索索引）露在外面，需要时找到标签快速充气恢复。（本质：保核心体积，去冗余空气）

## 面试话术
1. "上下文压缩的核心矛盾是显存/延迟成本与信息完整性的 trade-off，我通常按三层来做：对话层做摘要、知识层做 RAG 检索、底层做 KV Cache 算法压缩。"
2. "最常被忽略的是压缩后的效果评估，不能只比 token 数，要看下游任务准确率衰减了多少。"
3. "2026年我认为自适应压缩会成主流，简单任务压缩狠一点，复杂推理任务保留更多原始上下文。"

## 防混补丁
- **vs RAG**：RAG 是换一条路径（不压缩，只召回相关），上下文压缩是全量信息的压缩表示，两者可叠加
- **vs 微调/长上下文模型**：长窗口模型（如1M token）是"扩容"，压缩是"节流"，成本敏感场景节流更有效
- **摘要≠翻译**：摘要是信息提炼，可能改变表述；结构化压缩（JSON/图谱）是信息重组，保留关系

## 追问预演
- Q：摘要过程中关键信息丢了怎么办？
  - A：分层存储（近几轮原始保留+历史摘要），关键实体单独维护实体库
- Q：KV Cache 压缩对实时推理影响多大？
  - A：StreamingLLM 等方案可以做到几乎无损，因为注意力本来就稀疏，但复杂推理任务压缩率要保守
- Q：压缩策略怎么选？
  - A：先profile实际context长度分布，>80%场景在4k以内就不需要激进压缩，长尾场景做分层兜底
