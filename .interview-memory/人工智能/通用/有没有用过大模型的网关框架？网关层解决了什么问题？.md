# 有没有用过大模型的网关框架？网关层解决了什么问题？

## 场景触发词
LLM Gateway、大模型网关、LiteLLM、模型路由、多厂商接入、LLM 限流、Token 计费、模型 Fallback

## 速记卡片
`网 | 网多选缓限成观 | LLM 网关七功能`

## 原始知识

- **大模型网关（LLM Gateway）**：专门面向 LLM API 调用的统一管理层，介于应用和多模型厂商之间
- **常见框架**：LiteLLM（最流行，开源）、Helicone（可观测性）、Langfuse（Tracing）、OpenRouter（统一接入）

- **多厂商统一接入（多）**
  - 一套接口对接 OpenAI、Anthropic、Azure、Google、本地模型等
  - 屏蔽各厂商 API 差异（参数名、返回格式、认证方式）
  - 新增模型只需改配置，不用改业务代码

- **智能路由与 Fallback（选）**
  - 按成本优先、质量优先、延迟优先自动选型
  - 主模型超时/报错时自动 Fallback 到备用模型
  - 支持权重路由（如 70% GPT-4 + 30% Claude）
  - **易混点**：不是负载均衡，是语义化选型

- **重复请求缓存（缓）**
  - 相同 Prompt 命中缓存直接返回，不调用模型
  - Embedding 请求缓存效果最好（输入固定）
  - 可配缓存 TTL 和相似度阈值

- **限流与配额管理（限）**
  - 按 Token 数限流、按请求数限流、按用户配额限流
  - Token 限流更精准，因为不同模型成本差异大
  - 防止单用户/单应用冲垮系统或耗尽预算

- **成本监控与计费（成）**
  - 精确到每次调用的 Input/Output Token 费用
  - 按项目/用户/模型维度统计成本
  - 设置预算告警，超阈值自动切换低价模型

- **可观测性（观）**
  - 统一日志：记录每次请求的 Prompt、响应、延迟、Token 数
  - Tracing：追踪请求在网关层的处理链路
  - 延迟分析：P99、模型对比、瓶颈定位

- **安全过滤（附）**
  - PII（个人敏感信息）检测与脱敏
  - Prompt 注入攻击检测
  - 内容审核（输出过滤）

## 单字锚点 + 防崩推导
- **锚点字**：网
- **扩展口诀**：网多选缓限成观
- **为什么选它**：“网”是题目核心词，所有功能都围绕网关展开
- **防崩推导**：
  1. 网 → **网**关是统一入口
  2. 多 → **多**厂商统一接入
  3. 选 → 智能**选**型路由 + Fallback
  4. 缓 → **缓**存重复请求
  5. 限 → **限**流配额管控
  6. 成 → **成**本监控计费
  7. 观 → 可**观**测性（日志/Tracing）

## 一字一画面
- **网** —— 像 **国际电话转接局**：[旋转拨号盘] 打到哪个国家都先经过这里，自动转接到对应线路。（网关统一入口）
- **多** —— 像 **万国插座面板**：[墙上面板] 美标、欧标、英标都能插。（多厂商统一接入）
- **选** —— 像 **智能导航**：[手机导航] 根据拥堵程度和费用自动选最优路线，主路堵了自动切换备用路。（智能路由 + Fallback）
- **缓** —— 像 **快递代收点**：[驿站货架] 同一个包裹多次查询，直接看货架标签不用再去仓库。（缓存重复请求）
- **限** —— 像 **高速公路收费站**：[ETC 栏杆] 车流量大时限流，防止堵死，超出额度不放行。（限流配额）
- **成** —— 像 **话费账单**：[每月账单] 精确到每次通话的分钟数和费用。（成本监控）
- **观** —— 像 **监控大屏**：[数据中心大屏] 实时看到每条线路的流量、延迟和异常。（可观测性）
