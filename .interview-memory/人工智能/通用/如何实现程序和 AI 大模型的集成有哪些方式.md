## 原始知识

- **直接API调用**：HTTP请求调用OpenAI/Claude/文心等API，最简单灵活
- **SDK集成**：使用官方SDK如openai-python/spring-ai/langchain封装调用，简化开发
- **本地模型部署**：Ollama/vLLM/Transformers本地运行，数据隐私好
- **Function Calling**：工具调用让模型调用程序函数，双向交互
- **Embedding集成**：向量检索增强，RAG架构的基础
- **Agent框架**：LangChain/Spring AI编排复杂流程

## 单字锚点 + 防崩推导

**锚点字**：接

**扩展口诀**：接框函，四方式

**为什么选它**：所有集成问题的核心都是"怎么接"——接。接什么、怎么接、接多深，全围绕这个字展开。

**推导链**：
1. 接 → 接入大模型 = 程序和AI建立连接
2. 直 → 直接API：HTTP调用，最原始最灵活，所有语言的万能钥匙
3. 框 → 框架/SDK：Spring AI、LangChain、OpenAI SDK，封装了重试、流式、错误处理
4. 函 → Function Calling：函数调用，让模型不仅能说，还能"做"——调程序接口
5. 本 → 本地部署：Ollama/vLLM，数据不出域，隐私安全
6. Agent → 复杂编排：多工具链、多轮对话、记忆管理，用Agent框架搭工作流

## 一字一画面

- **接** —— 像插电源插头：找准接口插进去，设备就能通电工作（本质：建立程序与模型的通信通道）
- **直** —— 像徒手拧螺丝：不用工具，直接上手，灵活但费力（本质：裸HTTP请求，无封装，全手动）
- **框** —— 像电动螺丝刀套装：不同批头对应不同螺丝，省力标准化（本质：SDK统一封装，开发效率高）
- **函** —— 像遥控器按按钮：你按"开关"，电视自己执行亮屏动作（本质：模型发指令，程序执行函数）
- **本** —— 像自家发电机：不依赖电网，自己供电，数据不外流（本质：本地部署，隐私可控）

## 面试话术

"程序和AI集成的路线很多， simplest 的就是直接调API，用HTTP发请求。实际工程里更常用SDK或框架，比如Spring AI或者LangChain，封装了流式输出、重试这些脏活累活。如果对隐私要求高，可以用Ollama或vLLM本地部署。再高级一点就是Function Calling，让模型能调用我们程序的函数，实现双向交互。复杂场景就用Agent框架编排整个工作流。"

## 防混补丁

- **容易和什么搞混**：容易把Function Calling和普通API调用混为一谈——Function Calling是模型主动调程序，不是程序调模型。
- **补丁口诀**：程序调模型是单向，模型调函数是双向，接的方式不一样。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| SDK和API调用什么区别？ | API是底层协议，SDK是基于API的封装库，提供了更友好的接口和额外功能 |
| 本地部署和云端API怎么选？ | 敏感数据选本地，追求性能和新模型选云端，也可以混合 |
| Function Calling怎么实现？ | 在Prompt里定义函数签名，模型判断需要时返回函数调用JSON，程序执行后把结果再喂给模型 |
| Agent框架解决什么问题？ | 解决多步骤、多工具、记忆持久化、流程编排等复杂场景 |
| Embedding集成属于哪种方式？ | 属于数据层集成，把文本向量化后存向量库，检索时给模型做RAG增强 |
