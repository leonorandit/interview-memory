## 原始知识

- **不是简单均分**：LLM推理有批处理Batching，吞吐不是线性均分
- **连续批处理**：Continuous Batching让新请求动态加入当前批次
- **PagedAttention**：vLLM的内存管理技术，大幅提高GPU利用率
- **流式输出**：Token是流式生成的，用户不是等全部生成完才看到
- **首Token延迟**：TTFT（Time To First Token）影响用户感知
- **每Token延迟**：TPOT（Time Per Output Token）决定流畅度
- **瓶颈分析**：GPU显存、模型大小、Batch Size限制、网络延迟、KV Cache管理、请求排队
- **优化方向**：增大Batch Size、模型量化、多卡并行、请求合并、缓存命中

## 单字锚点 + 防崩推导

**锚点字**：并

**扩展口诀**：并批缓，四分析

**为什么选它**：问题本身就是从"并发"引出的——1000个用户同时来。只要记住"并"，就能推导出批处理、流式、缓存、排队等一系列概念。

**推导链**：
1. 并 → 并发（Concurrency）：1000用户同时请求
2. 批 → Batching：GPU一次处理一批请求，不是一个个串行
3. 缓 → 流式（Stream）：Token像水流一样缓缓输出，用户边生成边看
4. 四分析 → 四大瓶颈方向：显存、Batch、网络、排队
5. TTFT → 首Token延迟：从请求到第一个字出来多久，影响"卡不卡"
6. TPOT → 每Token延迟：后面每个字间隔多久，影响"流畅不流畅"
7. PagedAttention → vLLM的秘诀：像操作系统的虚拟内存，让显存不浪费

## 一字一画面

- **并** —— 像1000人同时涌进餐厅：不是每人一个灶台，而是大厨一锅炒多份（本质：请求汇聚，批量处理）
- **批** —— 像机场大巴：不是一个人发一辆车，而是坐满一车统一走（本质：Batching提高吞吐）
- **缓** —— 像打开水龙头：不是等水箱满了才出水，而是一开就有，细水长流（本质：流式输出降低感知延迟）
- **存** —— 像冰箱容积：食材（KV Cache）太多，冰箱（显存）放不下，就没法同时做太多菜（本质：显存是硬瓶颈）
- **排** —— 像医院挂号：诊室（GPU）满了，后面的人只能在大厅排队等叫号（本质：请求排队导致延迟）

## 面试话术

"肯定不是简单均分到1 token/s。LLM推理有批处理机制，GPU会同时处理一批请求，而且Token是流式输出的，用户边生成边看到。真正要分析的是两个指标：TTFT首Token延迟和TPOT每Token延迟。瓶颈通常在显存、Batch Size、网络延迟和请求排队这几个地方，优化方向包括增大Batch、模型量化、多卡并行这些。"

## 防混补丁

- **容易和什么搞混**：容易把总吞吐除以用户数当成单用户体验——实际上批处理和流式输出让单用户体验远好于简单除法。
- **补丁口诀**：吞吐除人数是误区，批处理加流式才是真相，看TTFT和TPOT不慌张。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 什么是Continuous Batching？ | 动态批次，一个请求生成完立刻换新的进来，不像静态Batch要等整批结束 |
| PagedAttention为什么能提速？ | 把KV Cache分页管理，避免内存碎片和预留浪费，显存利用率大幅提升 |
| TTFT和TPOT哪个更重要？ | 聊天场景TTFT更重要（用户等第一个字），生成场景TPOT更重要（流畅度） |
| 模型量化怎么帮助性能？ | INT4/INT8量化减少显存占用，能放更大Batch，吞吐提升 |
| 多卡并行有哪些方式？ | 张量并行（TP，单请求跨多卡）、流水线并行（PP，层间分布）、数据并行（DP，多副本同时处理不同请求） |
