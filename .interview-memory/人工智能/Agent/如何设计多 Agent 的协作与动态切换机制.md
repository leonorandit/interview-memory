# 如何设计多 Agent 的协作与动态切换机制？

## 场景触发词
多 Agent 协作、动态切换、Agent 路由、状态同步、通信机制、协作模式、负载均衡、故障转移

## 速记卡片
`协 | 协通路态切 | 五维设计法`

## 原始知识

- **协作模式（协）**
  - 层级式：主从结构，一个总控 Agent 分配任务给子 Agent
  - 对等式：各 Agent 平等协商，达成共识后行动
  - 市场式：拍卖/竞价机制，Agent 自主竞标任务
  - 竞争式：多个 Agent 同时尝试，选最优结果
  - **易混点**：按业务复杂度、信任关系和决策效率选模式
  - **追问点**：四种模式各自的优缺点和适用场景？

- **通信机制（通）**
  - 同步通信：RPC/HTTP 直接调用，低延迟但耦合度高
  - 异步通信：消息队列（如 RabbitMQ、Kafka）、事件总线，解耦且可削峰
  - 共享内存：同一进程内 Agent 共享状态，速度快但隔离性差
  - **易混点**：多 Agent 推荐异步为主、同步为辅，避免级联故障
  - **追问点**：消息丢失怎么办？需要幂等和重试机制

- **路由调度（路）**
  - 中央协调器：统一入口，根据任务类型/负载/能力匹配分发
  - 分布式协商：Agent 自主发现彼此，通过协议选举或协商
  - 负载均衡：轮询、权重、最少连接、能力匹配
  - **易混点**：中央模式简单但单点风险，分布式灵活但复杂度高
  - **追问点**：协调器挂了怎么办？需要主备或去中心化降级

- **状态同步（态）**
  - 共享上下文：Redis/数据库作为共享状态层，切换时读取
  - 事件溯源：记录所有事件流，新 Agent 通过回放重建状态
  - 快照机制：定期保存完整状态，快速恢复
  - **易混点**：状态同步是动态切换的基础，没有同步就切不了
  - **追问点**：状态冲突怎么解决？需要乐观锁或向量时钟

- **动态切换（切）**
  - 能力匹配：根据任务需求标签，匹配具备对应能力的 Agent
  - 故障转移：Agent 超时/异常时，自动切换到备份 Agent
  - 热切换：运行时无缝切换，用户无感知
  - 上下文传递：切换时把历史对话、中间结果、记忆打包传递
  - **易混点**：切换不是简单的转发请求，要保证状态一致性
  - **追问点**：切换时正在进行的任务怎么办？需要事务或补偿机制

- **设计原则**
  - 解耦：Agent 之间通过标准协议通信，不直接依赖实现
  - 可观测：全程埋点，切换链路可追溯
  - 容错：超时、重试、熔断、降级四件套
  - 弹性：Agent 可动态注册/注销/扩缩容

- **锚点字**：协

  3. 路 → 路由**路**由与调度（中央/分布式/负载均衡）
  4. 态 → 状**态**同步（共享上下文/事件溯源/快照）

## 面试话术
- 问设计思路 → “设计多 Agent 协作与动态切换，口诀是**协通路态切**。第一，**协**——选协作模式，层级/对等/市场/竞争按场景定；第二，**通**——设计通信机制，推荐异步消息队列解耦；第三，**路**——设计路由调度，中央协调器或分布式协商分派任务；第四，**态**——状态同步，共享上下文+事件溯源保证切换不丢信息；第五，**切**——动态切换，按能力匹配选 Agent，故障时自动转移，上下文随切换传递。关键难点是切换时的状态一致性。”
- 问难点 → “最大的难点是**状态一致性和上下文不丢失**。切换时正在进行的任务需要事务或补偿机制，状态冲突需要乐观锁或向量时钟解决。”

## 防混补丁
- 容易和“单 Agent 多工具”搞混。多 Agent 是多个独立智能体协作，单 Agent 多工具是一个大脑用多个手脚。补丁：**多 Agent 是多个人协作，单 Agent 是一个人用多把工具**。

## 追问预演
1. **问**：动态切换时上下文怎么传递？
   **答**：通过共享状态存储（如 Redis）、消息总线广播上下文快照、或事件溯源重建状态。关键是序列化协议和版本兼容。
2. **问**：中央协调器和分布式协商怎么选？
   **答**：Agent 数量少、任务明确时选中央协调器；Agent 数量多、需要自组织时选分布式协商（如拍卖、投票机制）。
3. **问**：多 Agent 怎么避免死锁或循环依赖？
   **答**：设置超时机制、依赖图检测环、优先级抢占、或引入协调器仲裁。
4. **问**：状态冲突怎么解决？
   **答**：乐观锁（版本号）、向量时钟（分布式因果）、或集中式锁（协调器仲裁）。按一致性要求选。
