## 顺口溜

插拔换策略，裁剪压预算。

## 单字锚点 + 防崩推导

- **锚点字**：插
- **扩展口诀**：插换裁压，四策略
- **为什么选它**：Context Engine的核心设计理念是"可插拔"，"插"字直接点出架构灵魂
- **推导链**：
  1. 插 → 可插拔 → 解耦业务与Context管理
  2. 换 → 灵活切换 → 不同场景换不同引擎
  3. 裁 → 裁剪窗口 → 滑动窗口、Token预算
  4. 压 → 压缩压实 → 摘要压缩、Compaction
  5. 四策略 → 滑动窗口、Token预算、摘要压缩、Compaction、分层Context（合并理解）

## 一字一画面

- **插** —— 像USB接口：银色金属头对准插槽，咔哒一声插进去，灯亮起（本质：标准化接口，即插即用）
- **换** —— 像换手机卡：弹出旧SIM卡，新卡滑入卡槽，信号满格（本质：运行时替换，不影响整机）
- **裁** —— 像裁缝剪布：大卷布匹按尺寸裁剪，边角料掉落（本质：按规则取舍，控制长度）
- **压** —— 像真空压缩袋：蓬松棉被抽气后变扁，体积减半（本质：信息密度提升，空间释放）

## 面试话术

1. "做Context Engine这层抽象，主要是为了把Context的生命周期管理和业务逻辑彻底解耦，这样Agent核心不用关心Token怎么算、窗口怎么滑，只专注推理本身。"
2. "插拔设计让我们能根据场景换策略——比如客服场景用滑动窗口保证近期上下文，研报场景用摘要压缩保留历史精华，Token紧张时用Compaction做一次性的历史归档。"

## 防混补丁

- **容易和什么搞混**：容易和普通配置管理搞混，以为只是换个参数。
- **补丁口诀**：不是换参数，是换引擎；接口统一，内核不同。

## 追问预演

1. **Q：Compaction策略和摘要压缩有什么区别？**
   A：摘要是逐轮精简文本，Compaction是把多轮对话打包成结构化摘要，更像归档。
2. **Q：分层Context是怎么分的？**
   A：通常分短期工作记忆、中期摘要记忆、长期知识记忆三层，越远的层越压缩。
3. **Q：如果两个策略同时启用，优先级怎么定？**
   A：OpenClaw里是管道式组合，先触发预算检查，再触发压缩，最后Compaction，按配置顺序执行。
