## 速记卡片

调 | 调描函，三步动 | API描述→生成参数→动态执行→反馈结果

---

## 原始知识

- **API描述文档**：OpenAPI / Function Schema 定义工具
- **LLM生成函数调用参数**：模型根据意图输出结构化调用请求
- **动态绑定执行**：框架映射到实际函数并运行
- **结果反馈给LLM**：执行结果回传，模型继续推理
- **工具注册与发现机制**：新工具自动注册，LLM动态感知可用工具

## 单字锚点 + 防崩推导

**锚点字**：调

**扩展口诀**：调描函，三步动

**为什么选它**：问题核心是"动态API调用"，"调"字直接对应"调用"，一眼锁定主题。

**推导链**：
1. 调 → 调用 → API调用 → 怎么动态做？
2. 描 → 描述（API描述文档 / Function Schema）→ 先让LLM知道有什么工具
3. 函 → 函数（LLM生成函数调用参数）→ LLM根据用户意图决定调哪个函数、传什么参
4. 三步动 → 三个动态步骤
   - 第一步：工具描述注册（OpenAPI Schema / Function Schema）
   - 第二步：LLM决策生成参数（Function Calling）
   - 第三步：动态绑定执行（框架把名称和参数映射到实际代码）
5. 闭环补充：执行完结果要反馈给LLM，LLM再决定是返回给用户还是继续调别的工具
6. 若只记得"调描函"：
   - 调 = 动态API调用
   - 描 = 描述（Schema定义）
   - 函 = 函数（生成调用）
   - 三步动的第三步 = 执行（动态绑定+运行）
   - 然后自然补全：结果要返回给LLM（反馈），工具要注册和发现（机制）
7. 顺序保险：描→函→动，先描述、再生成、后执行，像点菜：看菜单→点菜品→厨房做。

## 面试话术

"LLM Agent做动态API调用，通常走这几步：首先把API用OpenAPI或者Function Schema描述清楚，注册到Agent的工具库里；然后用户提需求时，LLM判断该调哪个函数，生成具体的调用参数；接着框架动态绑定到实际的函数去执行，执行结果再回传给LLM；LLM看结果决定是否继续调别的工具或者直接回答用户。整个过程是动态的，新工具只要按Schema注册进来，LLM就能自动发现和使用。"

"简单来说就是'描述-生成-执行-反馈'的循环，核心在于LLM通过Function Calling把自然语言意图翻译成结构化API调用，动态执行后再把结果吃回去继续推理。"

## 追问预演

**Q1**：Function Schema具体长什么样？
**A**：通常是JSON格式，包含函数名、描述、参数列表（名称、类型、描述、是否必填），比如OpenAI的functions参数格式。

**Q2**：如果LLM生成的参数格式错了怎么办？
**A**：框架层可以做校验，比如JSON Schema校验，失败了让LLM重试或返回错误信息；也可以先做代码解析再执行，捕获异常。

**Q3**：工具注册与发现机制怎么实现？
**A**：新工具上线时把Schema注册到工具注册中心，Agent启动时拉取全量工具列表，或者运行时动态热加载，LLM的prompt里带上可用工具的描述。

**Q4**：动态调用和ReAct有什么关系？
**A**：ReAct是推理-行动的循环框架，动态API调用是ReAct中'Action'的具体落地方式之一，两者结合让Agent既能思考又能调用外部能力。
