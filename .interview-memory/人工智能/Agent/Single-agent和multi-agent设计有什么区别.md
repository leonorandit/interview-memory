# Single-agent 和 multi-agent 设计有什么区别

## 速记卡片
`单 | 单简多协，看规模 | 一人vs团队`

## 原始知识
- **Single-Agent**：一个 LLM + 工具 + 记忆，独立完成任务
- **Single 优势**：架构简单、低延迟、易调试、易部署、上下文一致
- **Single 劣势**：能力天花板有限、复杂任务易陷入局部最优、无法并行、单点故障
- **Multi-Agent**：多个 Agent 各司其职（规划、执行、评审、工具等），通过通信协议协作
- **Multi 优势**：分工明确、可并行、能力互补、可互相评审纠错、适合复杂任务
- **Multi 劣势**：延迟高、调试难、通信开销大、一致性难保证、复杂度指数增长
- **选型原则**：小规模简单任务 → Single；大规模复杂任务 → Multi
- **演进路径**：通常从 Single 开始，遇到瓶颈再拆 Multi
- **易混点**：Multi-Agent 不是 Agent 越多越好，拆分粒度是关键
- **追问点**：什么时候拆 Multi？通信方式有哪些？一致性怎么保证？

## 顺口溜
**单简多协，看规模**

## 单字锚点 + 防崩推导
- **锚点字**：**单**
- **扩展口诀**：**单简多协**
- **为什么选它**：Single vs Multi 的核心差异从"单不单"开始——一个大脑还是多个大脑。
- **防崩推导**：
  - 单 → Single-Agent（一个 LLM + 工具 + 记忆）
  - 简 → 简单（架构简单、低延迟、易调试）
  - 多 → Multi-Agent（多个 Agent 各司其职）
  - 协 → 协作（通信协议、分工、评审）
  - 看 → 看规模/看场景（小规模用单，大规模用多）

## 一字一画面
- **单** —— 像**单人自行车**：一个人骑，想怎么骑怎么骑，灵活但载重有限。（Single-Agent：一个 LLM + 工具 + 记忆，架构简单）
- **简** —— 像**单打独斗的侠客**：一把剑走天下，决策快但遇到大军就吃力。（Single-Agent：低延迟、易调试，但能力天花板有限）
- **多** —— 像**团队作战**：有人负责侦查、有人负责进攻、有人负责后勤。（Multi-Agent：多个 Agent 各司其职，能力互补）
- **协** —— 像**接力赛**：一棒接一棒，需要交接配合。（Multi-Agent：通过通信协议协作，分工明确，可互相评审）
- **看** —— 像**看菜单点菜**：一个人简单吃碗面，一群人聚餐要分工合作。（选型看规模和复杂度）

## 面试话术
- 问区别 → "Single-Agent 和 Multi-Agent 口诀是**单简多协，看规模**。Single-Agent 像单人自行车，一个大脑 + 工具，架构简单、低延迟、易调试，适合任务边界清晰的场景。Multi-Agent 像团队作战，多个 Agent 各司其职，通过通信协作，适合复杂任务分工，但延迟高、调试难、通信开销大。选型看规模和复杂度：小规模简单任务用 Single，大规模复杂任务用 Multi。"
- 问演进 → "实际项目中，很多系统会从 Single 开始，遇到瓶颈再拆成 Multi。关键是找到合理的拆分粒度——太细通信开销大，太粗失去多 Agent 意义。"

## 防混补丁
- 容易以为 Multi-Agent 就是多调几次 API。补丁：**Multi-Agent 是多个有独立角色和状态的 Agent 协作，不是同一个模型调多次**。
- 容易认为 Multi 一定比 Single 好。补丁：**Multi 有通信开销和一致性成本，简单任务用 Single 更高效**。

## 追问预演
1. **什么时候该从 Single 拆成 Multi？**
   → 任务复杂度超过单 Agent 能力边界、需要多领域专业知识、需要并行执行、需要互相评审纠错时。
2. **Multi-Agent 通信方式有哪些？**
   → 直接消息传递、共享内存/黑板、发布订阅、主从协调、A2A/MCP 协议。
3. **Multi-Agent 一致性怎么保证？**
   → 全局状态同步、事务性消息、幂等设计、超时重试、冲突仲裁机制。
4. **Single-Agent 怎么扩展能力？**
   → 增加工具数量、提升模型能力、优化 Prompt、引入记忆压缩和长期记忆。
5. **常见的 Multi-Agent 协作模式有哪些？**
   → 层级式（主从）、对等式（协商）、流水线式（串行）、评审式（互相检查）、竞争式（多方案投票）。
