## 速记卡片

适 | 适转统，三步骤 | 内部统一，适配器转换，校验兜底

## 原始知识

- **OpenAI** Function Calling格式
- **Claude** Tool Use格式
- **Google** Function Calling格式
- **字段命名**不同
- **类型系统**差异
- 统一**内部Schema**定义
- 编写**Provider适配器**
- **转换层**自动映射字段
- 支持**自定义扩展**
- **校验层**统一验证

## 顺口溜

适者转，转者统，统后验，验后通。

## 单字锚点 + 防崩推导

- **锚点字**：适
- **扩展口诀**：适转统，三步骤
- **为什么选它**：题目核心是"Schema适配"，"适"字就是题眼，适配差异，统一输出
- **推导链**：
  1. 适 → 适配 → 不同Provider格式差异需要适配
  2. 转 → 转换 → Provider适配器做字段映射
  3. 统 → 统一 → 内部统一定义Schema，核心无感
  4. 三步骤 → 统一内部Schema、编写适配器、转换层映射
  5. 校验层 → 统一验证，防止转换失真
  6. 自定义扩展 → 新Provider接入只需写适配器

## 一字一画面

- **适** —— 像电源变压器：美国110V电器插进中国220V插座，变压器自动变压，电器正常工作（本质：异构输入转兼容输出）
- **转** —— 像同声传译耳机：台上说英文，耳机里出中文，观众各取所需（本质：字段级实时映射）
- **统** —— 像国际通用货币：各国货币先换成美元结算，再换回本国货币，中间标准统一（本质：内部锚定一个标准，降低复杂度）

## 面试话术

1. "我的思路是内部先定义一套统一的Tool Schema标准，然后给每个LLM Provider写一个适配器，负责把自己的格式转内标、把内标转回自己格式，这样Agent核心只写一次工具定义。"
2. "OpenClaw就是在中间加了一层转换层，OpenAI叫functions、Claude叫tools这些字段差异自动映射，最后还有校验层保证转换没丢字段，新Provider接入就是加一个新适配器。"

## 防混补丁

- **容易和什么搞混**：容易和API网关搞混，以为做协议转发。
- **补丁口诀**：不是转协议，是转Schema；字段名和类型系统的翻译官。

## 追问预演

1. **Q：类型系统差异怎么处理？比如OpenAI和Claude对数组的定义不同？**
   A：在适配器里做类型映射表，内部用JSON Schema子集做统一表达，出和入各做一次类型转换。
2. **Q：如果Provider新增了字段，怎么快速支持？**
   A：适配器支持扩展字段透传，核心不认识的字段原样保留，只转换已知的标准字段。
3. **Q：校验层校验什么？**
   A：校验必填字段是否丢失、类型是否匹配、范围是否越界，确保转换后的Schema在目标Provider那里能过检。
