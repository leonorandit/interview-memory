## 单字锚点 + 防崩推导

**锚点字：等**

- **扩展口诀**：等重追，三处理
- **为什么选它**："等"=幂等，题目核心就是幂等性处理，"等"字直接对应幂等中的"等"。
- **推导链**：
  1. 等 → 幂等 → 同样的操作执行多次结果一致
  2. 为什么Agent特别需要 → 网络超时重试（重）
  3. 更多原因 → LLM重复生成相同工具调用、用户重复提交、Agent循环重新执行
  4. 副作用已产生怎么办 → 三处理
  5. 处理一 → 幂等Token去重，防止重复执行
  6. 处理二 → 操作日志记录+可回滚设计+补偿机制
  7. 处理三 → 副作用标记不可重复执行+人工介入确认
  8. 总结 → 等重追，日志回，补偿人

## 一字一画面

**等** —— 像**电梯防夹感应门**：银色金属门反复开合，同一只手伸过去三次，门只响应一次，如果已经夹到东西，紧急按钮亮起红灯，保安对讲机传来人工确认声（本质：幂等是防重复触发，副作用处理是事后补救和人工兜底）。

## 面试话术

"Agent系统特别需要幂等性，主要原因有四个：网络超时会导致自动重试，LLM可能重复生成相同的工具调用，用户可能重复提交请求，Agent自身也可能在循环中重新执行同样的操作。如果工具已经产生了副作用，我们会通过三种方式处理：第一，用幂等Token做去重，防止重复执行；第二，通过操作日志记录和可回滚设计，配合补偿机制撤销或修正结果；第三，对不可撤销的副作用做标记，防止再次执行，并在必要时引入人工介入确认。"

## 防混补丁

容易和**普通接口幂等**搞混？记住：**等有LLM，普只有用户**——Agent系统的幂等性挑战还来自LLM不可控的重复生成，而普通系统主要是用户重试。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 幂等Token怎么设计？ | 业务唯一键+操作类型哈希，Gateway层统一拦截校验。 |
| 不可回滚的操作怎么办？ | 提前标记风险等级，执行前二次确认，执行后锁定状态。 |
| 补偿机制是同步还是异步？ | 通常异步执行，通过 Saga 模式或事务消息保证最终一致。 |
