# 原始知识

- 超大结果**问题**：占用大量Context Window，**挤压对话空间**，导致Token超限，**影响推理质量**，成本激增
- 处理方案：**截断Truncation**，只保留前N行或关键部分
- 处理方案：**摘要Summarization**，LLM压缩为要点
- 处理方案：**丢弃Discard**，超出阈值拒绝接收
- OpenClaw做法：**Tool Result大小限制**
- **自动摘要**大结果
- **分段处理**
- **优先级排序**

# 单字锚点 + 防崩推导

## 锚点字：截

- **扩展口诀**：截（截断）丢（丢弃）要（摘要/要点）三处理
- **为什么选它**："截"是对付超大结果的第一反应，画面锋利好记
- **推导链**：
  1. 看到"截" → 想到**截断**，工具返回**超大结果**
  2. 问题是什么？**占Context Window** → 挤对话空间 → **Token超限** → 推理差 → **成本高**
  3. "截" → **Truncation截断**：只保留前N行或关键部分
  4. "要" → **Summarization摘要**：LLM压缩成要点（"要"=要点）
  5. "丢" → **Discard丢弃**：超出阈值直接拒绝
  6. OpenClaw怎么做？**大小限制** + **自动摘要** + **分段处理** + **优先级排序**

# 一字一画面

- **截** —— 像[大坝开闸截流]：灰色混凝土大坝闸门落下，湍急黄泥水被拦住只留顶部细流通过（本质：截断只让关键内容通过）
- **丢** —— 像[垃圾桶盖弹开]：不锈钢垃圾桶"哐"的一声，黑色垃圾袋Overflow被直接推进去，盖子合上（本质：超出阈值的结果直接丢弃）
- **要** —— 像[实习生做会议纪要]：会议室内录音笔转动，实习生奋笔疾书把三小时讨论浓缩成半页A4纸重点（本质：LLM把大结果摘要成要点）

# 面试话术

"工具返回超大结果会大量占用Context Window，挤压对话空间，导致Token超限甚至影响推理质量。通用做法有三种：截断只保留前N行、用LLM做摘要压缩、或者直接丢弃超出阈值的结果。OpenClaw的做法更系统，设置了Tool Result大小限制，对大结果自动摘要，还支持分段处理和优先级排序，确保上下文不被挤爆。"

# 防混补丁

# 追问预演

| 追问 | 简要答案 |
|------|----------|
| 截断会不会丢失关键信息？ | 会，所以OpenClaw会优先保留包含错误信息、关键决策点的部分，而不是简单砍头 |
| 自动摘要是用同一个LLM吗？ | 通常是的，但可以用更小更快的模型做摘要，节省成本和延迟 |
| 分段处理是什么意思？ | 把超大结果切成多段，只把当前最相关的一段塞进上下文，其余存档或延后处理 |
