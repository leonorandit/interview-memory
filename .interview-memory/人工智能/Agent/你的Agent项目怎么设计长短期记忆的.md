# 你的 Agent 项目怎么设计长短期记忆的？

## 速记卡片
`感 | 感存取用 | 记忆流水线`

## 原始知识
- **感**（感知层）：接收用户输入、工具返回结果、环境状态变化，作为记忆的原始素材
- **存**（存储层）：
  - 短期记忆：当前会话上下文（Context Window）+ Redis Session 状态
  - 长期记忆：向量数据库（pgvector / Milvus / Chroma）存储历史交互 Embedding
  - 情节记忆：时序数据库记录对话轨迹
  - 知识记忆：外部知识库通过 RAG 接入
- **取**（检索层）：
  - 语义检索：Embedding 相似度召回相关历史记忆
  - 时序检索：按时间线召回最近 N 条记录
  - 混合检索：关键词 + 向量双路召回，RRF 融合排序
- **用**（应用层）：
  - 将检索到的记忆注入 System Prompt 或用户消息前
  - 个性化回复（记住用户偏好、历史意图）
  - 辅助决策（参考过往成功经验或失败教训）
- **生命周期管理**：
  - 写：每次交互后 LLM-as-Judge 评估重要性，高价值才入长期记忆
  - 压：对话超长按 Sliding Window 滑动，或做摘要压缩
  - 更：定期去重、合并相似记忆、更新用户画像
  - 忘：TTL 过期、重要性衰减、低价值记忆定期清理
- **工程要点**：多租户隔离（user_id + session_id）、记忆权限分级、读写性能优化
- **易混点**：长期记忆 vs RAG 知识库（经历 vs 百科）
- **追问点**：窗口满了怎么办、记忆检索不准、记忆爆炸、多租户隔离

## 顺口溜
**感存取用**

## 单字锚点 + 防崩推导
- **锚点字**：**感**
- **扩展口诀**：**感存取用**
- **为什么选它**：Agent 记忆设计的起点是"感知"输入，"感"字启动整个记忆流水线。
- **防崩推导**：
  - 感 → 感知层，接收用户输入、工具返回、环境变化
  - 存 → 存储层，短期（上下文/Redis）+ 长期（向量库/pgvector）
  - 取 → 检索层，Embedding 语义检索 + 时序检索 + 混合检索
  - 用 → 应用层，注入 Prompt、个性化回复、辅助决策

## 一字一画面
- **感** —— 像**雷达扫描**：24 小时接收外界信号，什么都先收进来。（本质：感知层接收输入）
- **存** —— 像**分拣快递**：急的放桌面（短期），重要的存档案柜（长期），百科放图书馆（知识库）。（本质：分层存储）
- **取** —— 像**智能搜索引擎**：输入关键词，同时从档案柜和图书馆里找到最相关的资料。（本质：混合检索召回）
- **用** —— 像**秘书递材料**：开会前把相关档案摆在领导面前，领导据此做决策。（本质：记忆注入 Prompt 辅助决策）

## 面试话术
- 问设计 → "我项目里的 Agent 记忆设计是流水线式的，口诀是**感存取用**。**感**——感知层接收用户输入和工具返回；**存**——分层存储，短期放上下文和 Redis，长期放 pgvector 向量库；**取**——用 Embedding 语义检索 + 时序检索做混合召回；**用**——把召回的记忆注入 System Prompt，实现个性化回复。另外配套生命周期管理：写入时 LLM 打分过滤垃圾信息，长了做摘要压缩，定期去重更新，低价值记忆自动遗忘。"
- 问难点 → "最大难点是记忆检索不准和记忆爆炸。检索不准我们用混合检索 + Rerank 优化；记忆爆炸通过重要性评分 + TTL + 定期归档来控制。"

## 防混补丁
- 容易把长期记忆和 RAG 知识库搞混。长期记忆是 Agent 自己的经历，知识库是外部文档。补丁：**经历是日记，知识是百科**。
- 容易觉得所有对话都要存长期记忆。其实大部分闲聊没价值，要用 LLM-as-Judge 过滤。补丁：**不是每件事都值得记住**。

## 追问预演
1. **短期记忆窗口满了怎么办？**
   → ① Sliding Window 保留最近 N 轮；② 对历史做 LLM 摘要，只保留摘要；③ 关键信息提取后存入长期记忆，释放窗口。
2. **长期记忆检索不准怎么优化？**
   → ① 混合检索（向量 + 关键词）；② 引入 Rerank 模型精排；③ 按用户/会话做 metadata 过滤；④ 定期做记忆聚类和去重。
3. **怎么避免长期记忆爆炸？**
   → ① 写入门槛：LLM 评估重要性，低分丢弃；② TTL 自动过期；③ 定期合并相似记忆；④ 用户级配额限制。
4. **多租户怎么隔离记忆？**
   → 向量库索引按 user_id 分区，或 metadata 过滤；短期记忆用 Redis Key 带 user_id + session_id 前缀；严格权限校验防止越权查询。
5. **记忆和 RAG 知识库怎么配合？**
   → 记忆是"我经历过什么"，知识库是"世界上有什么"。检索时两路并行召回，记忆路负责个性化，知识路负责事实性，最后在 Prompt 里分段呈现。
6. **LLM-as-Judge 打分记忆靠谱吗？**
   → 有一定误差，但可以设低阈值保证召回率，高风险场景（如用户关键偏好）人工抽检。也可以用规则辅助（如包含用户明确指令自动高分）。
