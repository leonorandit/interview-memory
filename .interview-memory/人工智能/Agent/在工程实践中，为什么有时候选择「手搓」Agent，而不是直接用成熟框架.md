# 在工程实践中，为什么有时候选择「手搓」Agent，而不是直接用成熟框架？

## 场景触发词
手搓 Agent、自研 Agent、框架选型、LangChain 缺点、为什么不直接用框架、Agent 工程实践、框架 overhead

## 速记卡片
`搓 | 搓白轻灵稳 | 手搓四优势`

## 原始知识

- **白盒可控**
  - 自己写代码，全流程透明，每个环节都可控
  - **易混点**：出问题能快速定位，不被框架黑盒绑架
  - **追问点**：框架内部做了什么，出了 Bug 是框架问题还是业务问题？

- **轻量高效**
  - 无框架 overhead，只写业务需要的逻辑
  - 不背整套框架的依赖、初始化、兼容层
  - **易混点**：简单场景下代码量比用框架还少

- **灵活定制**
  - 业务场景特殊时，框架通用抽象反而碍事
  - 手搓可以自由调整 Prompt 模板、调用链、重试策略
  - **追问点**：框架的"最佳实践"不一定适合你的业务

- **稳定可靠**
  - 无外部依赖风险，框架版本升级、API 变动不影响业务
  - 不用担心框架弃用或 Breaking Change
  - **易混点**：框架升级导致线上故障的案例

- **场景驱动选型**
  - **简单场景**：直接调 LLM API + 几行代码，比学框架更快
  - **复杂场景**：需要工具链、记忆、协作时，框架更有优势
  - **核心原则**：没有银弹，按场景选

## 面试话术
- 问为什么手搓 → “工程实践中选择手搓 Agent，主要有四点原因，口诀是**搓白轻灵稳**。第一，**白**盒透明，自己写代码每个环节都可控，出了问题能快速定位；第二，**轻**量高效，没有框架的 overhead，只写业务需要的逻辑；第三，**灵**活定制，业务场景特殊时，框架的通用抽象反而碍事；第四，**稳**定可靠，不担心框架版本升级、API 变动或弃用风险。简单场景下，手搓往往比学框架更快上线。”
- 问选型建议 → “手搓和用框架不是对立的，核心是**场景驱动**：简单场景手搓快，复杂生态用框架。需要快速集成大量工具链或多人协作时，LangChain 这类框架更有优势。”

## 防混补丁
- 容易和“重复造轮子”搞混。手搓不是反对框架，而是**场景驱动选型**。补丁：**简单场景手搓快，复杂生态用框架**。

## 追问预演
1. **问**：手搓有什么缺点？
   **答**：重复造轮子、缺乏生态、多人协作难统一。团队规模大或需要快速集成多种工具时，框架更有优势。
2. **问**：什么时候应该用框架？
   **答**：需要快速原型、集成大量工具链、团队协作需要统一规范、或框架恰好匹配业务模型时。
3. **问**：LangChain 这类框架的核心价值是什么？
   **答**：抽象通用模式（Chain、Agent、Memory）、提供工具集成、降低入门门槛。代价是黑盒和 overhead。
4. **问**：怎么判断一个场景该手搓还是用框架？
   **答**：看三个指标：复杂度（工具链数量）、团队规模（是否需要统一规范）、迭代速度（框架能否加速）。简单、特殊、快的场景选手搓；复杂、通用、团队大的场景选框架。
