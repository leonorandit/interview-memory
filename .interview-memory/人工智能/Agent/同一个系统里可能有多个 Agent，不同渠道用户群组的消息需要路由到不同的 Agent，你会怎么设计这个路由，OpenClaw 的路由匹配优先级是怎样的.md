## 速记卡片

路 | 路精群，三匹配 | 精确优先，群组次之，渠道兜底

## 原始知识

- 基于**渠道**路由
- 基于**用户ID**路由
- 基于**群组**路由
- 基于**意图**路由
- **规则引擎**
- **精确匹配** > 群组匹配
- 群组匹配 > **渠道匹配**
- 渠道匹配 > **默认Agent**
- 支持**正则表达式**
- **多条件组合**匹配

## 顺口溜

先精后群再渠道，默认兜底错不了。

## 单字锚点 + 防崩推导

- **锚点字**：路
- **扩展口诀**：路精群，三匹配
- **为什么选它**：路由问题，"路"字就是题眼，一指定魂
- **推导链**：
  1. 路 → 路由 → 消息怎么找到对的Agent
  2. 精 → 精确匹配 → 用户ID一对一，优先级最高
  3. 群 → 群组匹配 → 群聊按群组ID路由
  4. 三匹配 → 精确 > 群组 > 渠道
  5. 兜底 → 默认Agent，什么都没匹配上就走它
  6. 正则 + 多条件 → 扩展能力，组合规则

## 一字一画面

- **路** —— 像高速收费站：ETC车道、人工车道、货车车道，车辆按类型分流（本质：消息按特征走不同通道）
- **精** —— 像VIP专属通道：人脸识别确认身份，直接放行到专属休息室（本质：最细粒度匹配，优先级最高）
- **群** —— 像旅游团集体登机：导游举旗，一团人走团队通道（本质：按群组归属批量路由）

## 面试话术

1. "路由设计我会采用规则引擎的方式，支持多维度条件组合，优先级上精确匹配最优先，比如某个用户指定走专属Agent；其次是群组匹配，整个群聊走同一个Agent；最后是渠道匹配，同一类渠道走默认Agent。"
2. "OpenClaw的实现是精确匹配大于群组匹配大于渠道匹配大于默认Agent，同时支持正则和多条件组合，这样既能满足特殊用户专属服务，又能批量管理群组行为。"

## 防混补丁

- **容易和什么搞混**：容易和负载均衡搞混，以为是随机分发。
- **补丁口诀**：路由是按规则定向，不是均匀打散；精确命中，不是轮询。

## 追问预演

1. **Q：如果精确匹配和群组匹配冲突了怎么办？**
   A：按优先级，精确匹配胜出，个人专属规则覆盖群组规则。
2. **Q：正则匹配性能会不会有问题？**
   A：OpenClaw会预编译正则，且优先检查精确匹配，正则只在同优先级内使用，避免全量遍历。
3. **Q：多条件组合是怎么组合的？**
   A：支持AND/OR/NOT逻辑，比如"渠道是Telegram AND 用户ID在VIP列表"才路由到Agent A。
