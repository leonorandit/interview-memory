# 如何赋予 LLM 规划能力？

## 场景触发词
LLM 规划、Agent 规划能力、任务拆解、CoT、ToT、Plan-and-Execute、思维链、思维树、推理规划、自主规划

## 速记卡片
`规 | 规步树反计 | 四法赋规划`

## 原始知识

- **CoT 思维链（步）**
  - 让 LLM 把推理过程一步一步显式写出来，不直接给答案
  - 实现：Prompt 里加 "Let's think step by step" 或示例演示中间步骤
  - **易混点**：CoT 提升的是推理能力，不是真正的任务规划，但它是规划的基础
  - **追问点**：Zero-shot CoT 和 Few-shot CoT 的区别？

- **ToT 思维树（树）**
  - 让模型探索多条解决路径，评估后选最优解
  - 实现：生成多个候选思考 → 用 LLM 或规则评估 → 选择最佳分支继续深入
  - **易混点**：CoT 是单一路径，ToT 是多路径搜索
  - **追问点**：ToT 的评估器怎么设计？搜索广度 vs 深度？

- **Reflection 反思（反）**
  - 执行完一轮后自我检查，发现错误修正重试
  - 实现：让 LLM 评估自己的输出（"刚才的答案对吗？"），或外部验证器触发重试
  - **易混点**：Reflection 是事后修正，不是事前规划
  - **追问点**：反思的触发条件是什么？无限循环怎么办？

- **Plan-and-Execute（计）**
  - 先把复杂任务拆解成完整计划，再按步骤执行
  - 实现：Planner 生成任务列表 → Executor 逐个执行 → 可嵌套子任务
  - **易混点**：和 ReAct 的区别在于，Plan-and-Execute 是先定全局计划，ReAct 是走一步看一步
  - **追问点**：计划失败时怎么回滚或重新规划？

- **组合使用**
  - 实际项目中常组合：CoT + Reflection（逐步推理+自我纠错）
  - Plan-and-Execute + ReAct（先定大计划，执行时用 ReAct 应对变化）
  - ToT + 评估器（多路径探索+自动评分）

## 单字锚点 + 防崩推导
- **锚点字**：规
- **扩展口诀**：规步树反计
- **为什么选它**：“规”= 规划，直接对应题目核心词
- **防崩推导**：
  1. 规 → 赋予 LLM **规**划能力
  2. 步 → 一**步**一步想（CoT 思维链，显式中间推理）
  3. 树 → **树**状分叉探索（ToT 多路径评估选最优）
  4. 反 → **反**思修正迭代（Reflection 执行后自我检查）
  5. 计 → 先**计**划再执行（Plan-and-Execute 任务拆解+编排）

## 面试话术
- 问如何赋予规划 → “赋予 LLM 规划能力主要有四种核心方法，口诀是**规步树反计**。第一，**步**——CoT 思维链，让模型一步一步显式推理，就像解数学题写步骤；第二，**树**——ToT 思维树，让模型同时探索多条路径，评估后选最优；第三，**反**——Reflection 反思，执行完自我检查，发现错误修正重试；第四，**计**——Plan-and-Execute，先把复杂任务拆解成完整计划再执行。实际项目中常组合使用，比如 CoT + Reflection 让模型既能逐步推理又能自我纠错。”
- 问选型 → “简单推理用 CoT，需要探索的复杂问题用 ToT，确定性任务用 Plan-and-Execute，需要环境反馈的动态任务用 ReAct。反思能力可以作为通用增强叠加在任何模式上。”

## 防混补丁
- 容易和“推理能力”搞混。**规划是推理的子集，但规划强调任务拆解和步骤编排**。补丁：**推理是想，规划是拆+排**。
- 容易混淆 Plan-and-Execute 和 ReAct。补丁：**Plan 是先定攻略再出发，ReAct 是走一步看一步**。

## 追问预演
1. **问**：CoT 和 ToT 的核心区别是什么？
   **答**：CoT 是单一路径逐步推理，ToT 是多路径并行探索+评估。CoT 适合简单问题，ToT 适合需要探索的复杂问题。
2. **问**：Plan-and-Execute 和 ReAct 怎么选？
   **答**：任务步骤明确、环境稳定时选 Plan-and-Execute；需要实时反馈、环境动态变化时选 ReAct。也可以结合：先 Plan 定大方向，执行时用 ReAct 应对细节变化。
3. **问**：Reflection 具体怎么实现？
   **答**：让 LLM 对自己的输出进行自我评估（如"我的答案有逻辑漏洞吗？"），或设计外部验证器（如代码编译、单元测试），发现错误后触发重试或修正。
4. **问**：怎么防止规划死循环或无限反思？
   **答**：设置最大迭代次数、反思深度限制、或引入外部终止条件（如时间上限、成本上限）。
