# 复杂任务怎么做的任务拆分？为什么要拆分？效果如何提升？

## 场景触发词
任务拆分、任务分解、复杂任务处理、Task Decomposition、Agent 任务规划、大模型复杂任务

## 速记卡片
`拆 | 拆分解并，四步法 | 任务拆分`

## 原始知识

### 为什么要拆分？

1. **降低认知负荷**：大模型单次处理的上下文有限，拆分后每步只聚焦一个子任务
2. **减少错误传播**：一步错不等于全盘错，可以局部修正
3. **提升可解释性**：每一步的输出可追溯，便于调试和优化
4. **复用子能力**：不同子任务可以用不同的工具或模型，按需组合
5. **提高成功率**：简单任务的准确率远高于复杂任务

### 怎么拆分？——四步法

**Step 1：识别任务类型**
- 判断任务是序列型（有依赖顺序）还是并行型（可独立执行）

**Step 2：定义子任务边界**
- 每个子任务有明确的输入、输出和完成标准
- 子任务之间通过接口传递数据

**Step 3：建立依赖关系**
- 画出任务 DAG（有向无环图）
- 明确哪些子任务可以并行，哪些必须串行

**Step 4：合并验证**
- 执行完所有子任务后，汇总结果
- 检查一致性，处理冲突

### 常见拆分策略

| 策略 | 适用场景 | 例子 |
|------|----------|------|
| **按步骤拆分** | 流程型任务 | 写报告：选题→收集资料→大纲→正文→校对 |
| **按模块拆分** | 系统型任务 | 开发功能：前端→后端→数据库→测试 |
| **按条件拆分** | 决策型任务 | 客服：判断问题类型→转对应处理模块 |
| **迭代拆分** | 优化型任务 | 代码生成：写代码→测试→修复→再测试 |

### 效果如何提升？

1. **Few-shot 示例**：给模型看拆分好的例子，教会它怎么拆
2. **明确接口定义**：子任务之间的输入输出格式标准化
3. **中间结果验证**：每步结束后检查输出质量，不合格重试
4. **反思迭代**：执行完后回顾整个流程，优化拆分策略
5. **人机协同**：关键节点让人类审核，复杂子任务交给更强的模型

## 单字锚点 + 防崩推导
- **锚点字**：拆
- **扩展口诀**：拆分解并
- **为什么选它**：“拆”= 拆分，直接对应题目核心词
- **防崩推导**：
  1. 拆 → 拆分任务
  2. 分 → **分**解为子任务（按步骤/模块/条件）
  3. 解 → **解**决每个子任务（降低复杂度）
  4. 并 → **并**合并验证结果（确保一致性）
  5. 四步法 → 识别类型→定义边界→建立依赖→合并验证

## 一字一画面
**拆** —— 像 **拆包裹**：[纸箱] 一个大箱子拆开，里面有几个小盒子，分别处理后再组装。
**分** —— 像 **流水线工位**：[工厂车间] 每个工人只做一道工序，最后产品组装完成。
**解** —— 像 **解魔方**：[彩色方块] 先拼底层，再拼中层，最后顶层，分步解决。
**并** —— 像 **拼图**：[散落的拼图片] 每块拼好后再组合成完整画面，检查是否严丝合缝。

## 面试话术
复杂任务拆分的核心思路是**拆分解并四步法**：先识别任务类型，再定义子任务边界，建立依赖关系，最后合并验证。拆分的目的是降低单次认知负荷、减少错误传播、提升可解释性。效果提升的关键在于：用 Few-shot 示例教会模型怎么拆，明确子任务间的接口定义，每步做中间结果验证，执行完后反思迭代。实际项目中，简单任务直接做，中等任务按步骤拆，复杂任务用 DAG 编排，关键节点加人类审核。

## 防混补丁
**别混**：不要把拆分和并行搞混。有些子任务有依赖关系，必须串行执行。口诀：**能并则并，有依则串**。
