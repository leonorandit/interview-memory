## 速记卡片

渠 | 渠转统，三抽象 | 统一消息格式，解耦渠道与Agent核心

## 原始知识

- 统一**消息格式**
- **Message Interface**
- 解耦渠道与Agent**核心**
- **适配器模式**
- 定义Message接口
- 发送和接收方法
- 渠道**配置注册**
- 事件**回调机制**
- 支持**Webhook**和**轮询**两种模式

## 顺口溜

渠水流转统一道，收发配置两不扰。

## 单字锚点 + 防崩推导

- **锚点字**：渠
- **扩展口诀**：渠转统，三抽象
- **为什么选它**：题目核心是"多渠道接入"，"渠"字一眼定题，水渠导流，万源归一
- **推导链**：
  1. 渠 → 渠道 → Telegram、飞书、钉钉等多渠道
  2. 转 → 转换 → 适配器模式，各渠道消息转统一格式
  3. 统 → 统一 → Message Interface，统一收发接口
  4. 三抽象 → 消息抽象、发送抽象、接收抽象
  5. 配置注册 + 回调机制 + Webhook/轮询 → 具体实现方式

## 一字一画面

- **渠** —— 像田间水渠：多条小溪从四面八方汇入一条主水渠，水流方向一致（本质：多渠道汇聚，统一流向Agent核心）
- **转** —— 像万能转换插头：英标、美标、欧标插进同一个转换头，输出都是国标接口（本质：异构输入转统一输出）
- **统** —— 像机场安检口：不同航站楼旅客汇入同一套安检流程，行李规格统一（本质：标准化处理，核心无感）

## 面试话术

1. "我会先定义一个Message Interface，把不同渠道的消息抽象成统一结构，这样Agent核心只认这一套格式，不用关心是Telegram还是钉钉。"
2. "每个渠道写一个Adapter实现Channel Plugin接口，负责把原始消息转成内部Message，同时支持Webhook推送和主动轮询两种接入方式，渠道注册到配置中心即可热插拔。"

## 防混补丁

- **容易和什么搞混**：容易和网关层搞混，以为要做协议转发。
- **补丁口诀**：渠道抽象在业务层，不做网络转发，只做消息翻译。

## 追问预演

1. **Q：Webhook和轮询怎么选？**
   A：有服务器暴露公网选Webhook，实时性好；内网或防火墙限制选轮询，兼容性强。
2. **Q：如果飞书消息格式升级了怎么办？**
   A：只改飞书Adapter，内部Message Interface不变，Agent核心零改动。
3. **Q：事件回调机制具体怎么设计？**
   A：定义ChannelEvent类型，Adapter收到消息后封装成Event投递到事件总线，Agent订阅处理。
