# 单字锚点 + 防崩推导

## 锚点字：裁

- **扩展口诀**：裁（裁剪）总（Token总数控制）滑（滑动窗口）三策略
- **为什么选它**："裁"是Context Window管理的核心动作，画面感强
- **推导链**：
  1. 看到"裁" → 想到**裁剪**，Context Window**有上限**
  2. 问题是什么？Token上限被**对话历史+工具结果**占满
  3. "滑" → **滑动窗口**：老消息像滑出去一样被丢弃
  4. "总" → **Token计数**：总数超标就截断
  5. 还有一个通用策略 → **Summarization摘要**：把历史压成摘要
  6. OpenClaw怎么做？**智能裁剪** + 保留System Prompt + **压缩对话** + **分阶段管理** + **Token预算分配**

# 一字一画面

- **裁** —— 像[裁缝剪布料]：银色大剪刀"咔嚓"剪去灰色旧布边，留下红色新布料和金色线头（本质：裁剪丢弃旧消息，保留关键内容）
- **总** —— 像[体重秤数字跳]：白色电子秤上数字从"0"一路跳到"100"，红灯亮起"滴滴"报警（本质：Token计数超限触发截断）
- **滑** —— 像[手机相册滑动删除]：手指左滑聊天消息，红色"删除"按钮露出，旧消息一条接一条飞走（本质：滑动窗口丢弃最早的消息）

# 面试话术

"LLM的Context Window有Token上限，长对话时历史消息和工具结果会把窗口占满。通用做法有滑动窗口、Token计数截断和摘要压缩。OpenClaw用的是智能裁剪策略，保留System Prompt和关键历史，对旧对话做压缩摘要，还会分阶段管理上下文和分配Token预算，保证Agent在长对话中也能正常工作。"

# 追问预演

| 追问 | 简要答案 |
|------|----------|
| Token预算分配是什么意思？ | 给System Prompt、历史消息、工具结果分别设Token配额，防止某一部分独占窗口 |
| 分阶段管理上下文怎么做？ | 不同阶段采用不同策略，比如早期保留完整历史，后期启用摘要+最近几轮 |
| 智能裁剪和简单截断有什么区别？ | 简单截断直接砍头，智能裁剪会识别并保留关键决策点和System Prompt，只压缩或丢弃低价值消息 |
