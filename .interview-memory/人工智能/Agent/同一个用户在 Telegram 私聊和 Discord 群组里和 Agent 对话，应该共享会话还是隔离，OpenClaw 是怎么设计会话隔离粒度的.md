## 速记卡片

隔 | 隔渠群，三粒度 | 默认三元组隔离，可配置跨渠道共享

## 原始知识

- 按**用户ID**隔离
- 按**渠道**隔离
- 按**群组**隔离
- 按**会话ID**隔离
- 默认按**用户+渠道+会话**三元组隔离
- **可配置**隔离粒度
- 支持**跨渠道共享**配置
- 私聊和群组**默认隔离**
- 同一用户**不同群组隔离**

## 顺口溜

人渠话，三元锁，私群分，可打破。

## 单字锚点 + 防崩推导

- **锚点字**：隔
- **扩展口诀**：隔渠群，三粒度
- **为什么选它**：题目核心是"隔离粒度"，"隔"字就是题眼，隔离还是共享，先定隔
- **推导链**：
  1. 隔 → 隔离 → 会话隔离问题
  2. 渠 → 渠道 → 不同渠道天然隔离
  3. 群 → 群组 → 同一渠道不同群组隔离
  4. 三粒度 → 用户、渠道、群组三个维度
  5. 默认三元组 → 用户ID + 渠道 + 会话ID
  6. 可配置 → 业务需要时可打破隔离，跨渠道共享

## 一字一画面

- **隔** —— 像酒店房间卡：同一个人在不同酒店刷不同的房卡，进不同的房间，房间里的物品互不干扰（本质：不同上下文空间，记忆不串台）
- **渠** —— 像两条平行铁轨：火车在A轨和在B轨行驶，轨道之间有道岔但默认不通（本质：渠道是天然边界）
- **群** —— 像微信群和QQ群：同一个手机号，微信群里聊的内容不会出现在QQ群里（本质：群组是第二道边界）

## 面试话术

1. "私聊和群组我觉得应该默认隔离，因为场景完全不同——私聊是个人助理，群组是社区管理，共享上下文会串戏。"
2. "OpenClaw默认用用户ID加渠道加会话三元组做隔离，但提供了配置化能力，比如如果用户明确要求跨渠道同步个人偏好，可以打通，不过默认还是隔离最安全。"

## 防混补丁

- **容易和什么搞混**：容易和用户认证隔离搞混，以为是权限隔离。
- **补丁口诀**：会话隔离是上下文隔离，不是权限隔离；同用户不同会话，记忆不共享。

## 追问预演

1. **Q：什么场景下需要跨渠道共享会话？**
   A：个人助理场景，用户希望在Telegram开始的问题，在飞书能接着聊，这时可配置共享。
2. **Q：三元组隔离会不会导致同一个群每句话都是新会话？**
   A：不会，会话ID在群聊里通常是群组ID本身，所以群内连续对话是同一个会话。
3. **Q：隔离粒度太细会不会影响用户体验？**
   A：会，所以OpenClaw做成可配置的，默认偏安全隔离，业务需要时可放宽。
