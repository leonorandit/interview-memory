## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 窗 | 窗历工，三约束 | 窗口历史工具全挤 |

## 原始知识

- **Context Window定义**：LLM能同时处理的最大Token数
- **包含内容**：System Prompt + 对话历史 + 工具调用 + 当前输入
- **历史太长须截断**：超上限丢失早期信息
- **工具调用占Token**：结果回传占用大量空间
- **影响任务全貌理解**：早期目标被截断后Agent忘初衷
- **多轮后空间骤减**：每轮都在消耗窗口
- **模型差异**：4K/8K/32K/128K/1M各不同

## 顺口溜

窗历工，三约束：
窗口小，历史长，工具挤，三样东西抢地方。

## 单字锚点 + 防崩推导

**锚点字**：窗

**扩展口诀**：窗历工，三约束

**为什么选它**：Context Window直译就是"上下文窗口"，"窗"字最形象——窗口大小固定，东西多了装不下。

**推导链**：
1. 窗 → 窗口里装什么？System Prompt + 历史 + 工具调用 + 当前输入
2. 历 → 历史（对话历史随轮次增长，越来越长）
3. 工 → 工具（工具调用结果回传占用大量Token）
4. 三约束 → 历史太长必须截断、工具结果占用多、多轮后可用空间急剧减少
5. 为什么核心？因为Agent是循环的，每轮都在吃窗口，比单次问答紧张得多

## 一字一画面

窗 —— 像**飞机舷窗**：椭圆形小窗，外面的云不断飘过，窗框大小固定不变（本质：容量上限是硬件属性，无法突破）

历 —— 像**卷轴展开**：泛黄的纸卷越拉越长，最后一段必须裁掉才能塞进窗框（本质：历史超限后截断，早期信息丢失）

工 —— 像**搬家纸箱塞书**：厚厚的工具结果像百科全书，一本塞进去占掉半箱空间（本质：工具调用结果是Token大户）

## 面试话术

"Context Window是LLM能同时处理的最大Token数，包括System Prompt、对话历史、工具调用和当前输入。它对Agent工程特别关键，因为Agent是循环工作的，每轮对话、每次工具结果回传都在吃窗口。历史太长必须截断，工具结果又特别占地方，多轮之后Agent可能就忘了最初的目标。不同模型窗口差异很大，从4K到1M都有，选型时必须考虑业务场景。"

## 防混补丁

容易和**输出长度限制**搞混？补丁口诀：窗口是输入+输出的总容量，输出限制只是其中一半。

## 追问预演

**Q：怎么缓解窗口不够的问题？**
A：滑动窗口截断历史、Summarization压缩、向量库存长期记忆按需检索。

**Q：工具结果太占Token怎么办？**
A：工具结果做摘要后再回传，或者分片按需返回。

**Q：不同模型窗口怎么选？**
A：简单问答4K够用，Agent场景建议32K起步，复杂工作流上128K或更高。

## 关联知识点

- Summarization压缩（缓解窗口压力的手段）
- Token计算与截断（窗口管理的技术实现）
