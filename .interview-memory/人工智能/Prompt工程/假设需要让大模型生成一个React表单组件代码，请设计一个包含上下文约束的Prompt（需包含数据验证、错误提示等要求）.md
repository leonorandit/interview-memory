## 原始知识

- **角色**定义（前端开发专家）
- **上下文**约束（React + TS + Tailwind 环境）
- **功能**需求（表单组件 + 字段定义）
- **验证**规则（必填 / 格式 / 长度）
- **错误**处理（即时报错 + 红色边框提示）
- **代码**规范（组件化 + 类型安全）
- **输出**格式（源码 + 使用说明）
- 表单验证库三选一：**react-hook-form**（配套好）、**yup**（生态成熟）、**zod**（类型推导强）

## 顺口溜

约功验，四约束（7 字）

## 单字锚点 + 防崩推导

- **锚点字**：约
- **扩展口诀**：约功验，四约束
- **为什么选它**："约" = 约束 = Prompt 工程的本质就是给大模型签订一份"约束合同"
- **推导链**：
  1. 约 → 约束
  2. 约束 → Prompt 设计是给模型设限
  3. 设限分四块：角色（你是谁）、上下文（在什么环境）、功能验证（做什么 + 怎么验）、代码输出（什么规范 + 什么格式）
  4. 口诀"约功验"覆盖功能 + 验证，前后补齐角色、上下文、错误、代码、输出
  5. React 表单场景下，验证库记 **zod**（类型推导）、**yup**（成熟）、**react-hook-form**（绑定快）

## 一字一画面

**约** —— 像【签合同按红手印的手】：甲方乙方白纸黑字画红线，每一条款都是不能越界的约束（本质：Prompt 就是人和模型之间的合同条款）。

## 面试话术

"设计 Prompt 就像拟合同，先定角色身份，再圈定技术栈上下文，然后列功能清单，最后把验证规则、错误提示、输出格式都写成条款，模型才不会乱来。React 表单我通常会指定 TS + react-hook-form + zod 这套组合，保证类型安全。"

## 防混补丁

容易和**调参**搞混？一句补丁：**Prompt 是喂话术，调参是改权重，约字记合同。**

## 追问预演

- **Q：为什么需要角色定义？**
  - A：让模型切换知识领域，输出更贴近专家水平，减少无关内容。
- **Q：验证库选 zod 还是 yup？**
  - A：追求类型推导和 TS 友好选 zod，团队熟悉度高选 yup，两者都能和 react-hook-form 无缝配合。
- **Q：错误提示怎么在 Prompt 里约束？**
  - A：明确指定"字段校验失败时显示中文错误文案，输入框边框变红，错误信息紧跟输入框下方"。
