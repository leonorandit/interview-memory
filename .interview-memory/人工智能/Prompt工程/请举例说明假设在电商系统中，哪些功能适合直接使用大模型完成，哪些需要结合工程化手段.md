## 原始知识

- **适合**直接用大模型：
  - 商品描述生成（创意文案）
  - 客服话术（多轮对话）
  - 评论情感分析（文本理解）
  - 智能推荐解释（自然语言生成）
- **需要**工程化手段：
  - 库存扣减（强一致性事务）
  - 价格计算（精准零差错）
  - 订单状态流转（状态机严格定义）
  - 支付链路（金融级安全合规）
  - 搜索排序（延迟低 + 可解释）
- **区分**标准：**创造性 vs 确定性**、**延迟容忍 vs 实时性**、**容错高 vs 零差错**
- **结合**方式：大模型生成 + 规则校验 + 事务保证 + 人工审核

## 单字锚点 + 防崩推导

- **锚点字**：合
- **扩展口诀**：合直规，两分类
- **为什么选它**："合" = 工程化结合 = 大模型不能单打独斗，必须和工程代码结合才能落地电商
- **推导链**：
  1. 合 → 结合 → 大模型 + 工程化结合
  2. 结合 → 两类任务划分：
     - 直（直接用）：创意类、理解类、生成类 → 容错高、延迟可接受
     - 规（工程化）：规则类、事务类、计算类 → 强一致性、零差错、低延迟
  3. 区分标准：看任务是否需要**严格数学正确性**（价格 / 库存）和**强一致性**（支付 / 订单）
  4. 落地方式：大模型做"面子"（文案、解释），工程做"里子"（库存、价格、支付）

## 一字一画面

**合** —— 像【双桨划船】：左桨是蓝紫色的创造力（大模型生成文案），右桨是银灰色的规则力（工程代码扣库存），只用左桨打转圈，双桨配合船才直往前冲（本质：大模型与工程化是互补双桨，缺一不可）。

## 面试话术

"电商系统里，商品文案、客服话术、评论分析这种创意和理解类任务适合直接上大模型；但库存扣减、价格计算、订单流转、支付链路这些需要强一致性、零差错的必须走工程化，大模型可以辅助生成但绝不能直接操作核心数据，这就是合直规两分类的思路。"

## 防混补丁

容易和**大模型万能论**搞混？一句补丁：**合字记双桨，创意归模型，数据归代码。**

## 追问预演

- **Q：搜索排序为什么不能用大模型直接做？**
  - A：延迟高（几百毫秒 vs 几十毫秒）、需要实时索引更新、排序逻辑要求可解释性强，用工程 + 算法（如 Learning to Rank）更可控。
- **Q：推荐解释怎么用大模型？**
  - A：召回和排序由工程算法做，推荐理由由大模型根据用户画像和商品属性生成自然语言解释。
- **Q：客服话术可以直接用模型输出吗？**
  - A：可以辅助生成，但涉及退款、赔偿等敏感操作必须经规则引擎校验或人工确认。
