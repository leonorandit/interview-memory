# 为什么有些特定的推理模型不支持 MCP 协议？

## 速记卡片
`锁 | 闭源生态锁，架构引擎早 | 锁定+时差`

## 原始知识
- **生态锁定**：闭源厂商维护自有工具标准（OpenAI Function Calling、Google A2A 等）
- **架构锁定**：模型内部工具调用机制与 MCP 协议不兼容
- **引擎锁定**：推理部署框架（vLLM、TGI、TensorRT-LLM 等）未集成 MCP
- **性能顾虑**：额外协议层增加通信延迟，极致性能场景倾向直连
- **时间差**：模型定型早于 MCP 诞生， retrofit 成本高
- **易混点**：不支持 ≠ 模型能力弱
- **追问点**：如何通过适配层让老模型支持 MCP？

## 顺口溜
**闭源生态锁，架构引擎早**

## 单字锚点 + 防崩推导
- **锚点字**：**锁**
- **扩展口诀**：**三把锁，一个早**
- **为什么选它**："锁"字既体现封闭性，又有画面感（被锁住无法接入）。
- **防崩推导**：
  - 锁 → 谁锁的？→ 厂商锁生态、模型锁架构、引擎锁集成
  - 早 → 出来得早 → 定型早于 MCP → 改造成本高
  - 额外：性能顾虑 → 锁的另一面是"怕慢"

## 一字一画面
- **闭** —— 像**私人庄园大门**：门口只认自家门禁卡，不认市政公交卡。（闭源厂商维护自有标准）
- **源** —— 像**源头被掐住**：水流只能走指定管道。（生态锁定）
- **生** —— 像**生物隔离**：不同物种不能通婚，基因库不兼容。（架构差异）
- **态** —— 像**态字本身站立**：各自站各自的队，标准分裂。（多标准并存：Function Calling / A2A / LangChain Tools）
- **锁** —— 像**三把连环锁**：生态一把、架构一把、引擎一把，缺一不可。（三重锁定）
- **架** —— 像**房屋框架已定**：盖完的房子再改户型，代价极大。（模型定型难改）
- **构** —— 像**构字本身搭木架**：木头已经搭好，换钢材要重来。（架构固化）
- **引** —— 像**引擎盖焊死**：汽车引擎盖打不开，没法加装新零件。（推理引擎未开放接口）
- **擎** —— 像**擎天柱本身**：庞大身躯转身慢，新协议接入需要全身体检。（推理框架复杂度高）
- **早** —— 像**早班车开走了**：MCP 是后来的班车，早班车上的乘客换不了票。（时间差、改造成本）

## 面试话术
- 问原因 → "核心原因是**三把锁加一个早**。第一，生态锁定，很多闭源厂商有自己的工具调用标准，比如 OpenAI 的 Function Calling；第二，架构锁定，模型内部机制没有按 MCP 设计；第三，引擎锁定，vLLM、TGI 这些推理框架没集成 MCP；最后，很多模型定型比 MCP 诞生早， retrofit 成本高。另外，极端性能场景可能也会顾虑 MCP 带来的额外延迟。"
- 问解决 → "可以通过中间件或适配层做转换，把 MCP 请求翻译成模型原生的工具调用格式。"

## 防混补丁
- 容易和"模型能力不够"搞混。**不支持 MCP 不是智商问题，是接口问题**。补丁：**不是不会，是不通**。

## 追问预演
1. **MCP 和 Function Calling 的区别？**  
   → Function Calling 是模型层面的能力（输出 JSON 调工具），MCP 是协议层面的标准（统一工具发现、调用、上下文传递）。两者互补，Function Calling 可以在 MCP 之下工作。

2. **怎么让不支持的模型用上 MCP？**  
   → 中间件适配层：MCP Client → Adapter → 模型原生接口。或者推理框架侧增加 MCP Server 插件。

3. **MCP 相比直接调用有什么优势？**  
   → 标准化（一次接入到处用）、动态发现（运行时感知工具）、安全隔离（模型不直接接触敏感资源）。

4. **开源模型一定支持 MCP 吗？**  
   → 不一定。支持与否取决于推理框架和部署方式，跟开源/闭源没有必然关系。
