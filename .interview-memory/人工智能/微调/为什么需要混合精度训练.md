## 原始知识

- **FP16加速计算**：矩阵运算吞吐翻倍
- **显存减半**：参数占存砍一半， batch 开大
- **Loss Scaling**：防梯度下溢，先放大再缩回
- **Tensor Core**：必须 FP16 才能喂满
- **FP32主权重**：累加用 FP32，精度不崩
- **自动混合**：PyTorch AMP 自动切换
- **三倍收益**：速度、省存、能效齐升

## 单字锚点 + 防崩推导

**锚点字：混**

- 扩展口诀：**混**合精度，**混**的是 FP16 和 FP32
- 为什么选它：题目开门见山就是"混合精度"，"混"字一眼命中
- 推导链：
  1. 混 → 混合了谁？→ FP16（计算）+ FP32（存储/累加）
  2. 混 → 为什么混？→ 快（FP16 算得快）+ 省（显存省一半）
  3. 混 → FP16 有什么问题？→ 精度低，梯度下溢 → Loss Scaling
  4. 混 → 硬件支持？→ Tensor Core 必须用 FP16
  5. 混 → 怎么混？→ 前向 FP16，梯度 FP16，权重更新 FP32

## 一字一画面

**混** —— 像鸳鸯火锅：红汤（FP16）翻滚沸腾算得快，白汤（FP32）温和兜底保鲜味，中间隔板互不串味，一锅两吃各尽其职。（本质：FP16 负责冲锋陷阵做矩阵乘法，FP32 负责幕后守精度做累加更新。）

**快** —— 像红色跑车在专用赛道（Tensor Core）上狂飙，同样的路比蓝色轿车（FP32）快一倍。（本质：FP16 位宽减半，硬件吞吐翻倍。）

**省** —— 像行李箱突然压缩了一半体积，同一辆车能多塞一倍的行李（batch size 翻倍）。（本质：FP16 占 2 字节 vs FP32 占 4 字节，显存直接减半。）

**溢** —— 像漏斗漏沙子，太细的沙（小梯度）直接漏光，垫张纸（Loss Scaling）先兜住再倒回去。（本质：FP16 表示范围小，梯度下溢成 0，先乘大数再除回来。）

## 面试话术

"混精训练说白了就是用 FP16 做前后向计算、FP32 存主权重和做梯度累加。好处特别直接：一是计算速度能翻倍，因为 Tensor Core 吃 FP16；二是显存直接砍半，batch 能开更大。当然 FP16 精度低，梯度容易下溢，所以得配个 Loss Scaling，先把梯度放大再缩回来，精度基本不丢。"

## 防混补丁

容易和"量化"搞混？一句补丁：

> **混是训练时混精度，量化是推理时砍位数。**

（混合精度训练：FP16+FP32，可恢复；INT8/INT4 量化：永久砍精度，推理专用。）

## 追问预演

1. **Q：Loss Scaling 具体怎么做？**
   A：前向算 loss 后乘一个 scale（比如 2^16），反向传播梯度也跟着放大，更新权重前再把梯度除以 scale，防止 FP16 梯度下溢成 0。

2. **Q：为什么 FP32 主权重不能也换成 FP16？**
   A：权重更新量通常很小，FP16 累加会丢精度，长期训练模型就崩了，所以权重和优化器状态必须 FP32。

3. **Q：什么场景不适合混合精度？**
   A：本身对精度极度敏感的模型（某些强化学习、科学计算），或者硬件不支持 Tensor Core 的老显卡，用了反而慢。
