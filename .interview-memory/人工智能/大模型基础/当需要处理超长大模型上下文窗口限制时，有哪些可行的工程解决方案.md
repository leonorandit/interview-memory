## 原始知识

- **切分**文本（滑动窗口 / RecursiveCharacterTextSplitter）
- **检索**增强（RAG：只召回相关片段，降低输入长度）
- **摘要**压缩（提炼关键信息，短文本替代长文本）
- **长窗**模型（Claude 100K / 200K、Gemini 1M Token）
- **Map**-Reduce（分而治之：分段处理再聚合结果）
- **窗口**重叠（切分时保留重叠区，防断句丢语义）

## 顺口溜

## 单字锚点 + 防崩推导

- **锚点字**：长
- **扩展口诀**：长切滑，四方案
- **为什么选它**："长" = 长上下文 = 核心矛盾是输入文本太长，模型装不下
- **推导链**：
  1. 长 → 超长文本 → 模型窗口装不下
  2. 装不下 → 四招工程化解法：
     - 切：文本切分 + 滑动窗口（把长变短，一段段喂）
     - 滑：RAG 检索增强（滑出关键块，只喂相关段）
     - 压：摘要压缩（压扁信息，保留核心）
     - 换：换长窗口模型（Claude 200K / Gemini 1M）或 Map-Reduce 分治
  3. 选型逻辑：对精度要求高选 RAG，对全局理解要求高选长窗模型或 Map-Reduce，对实时性要求高选摘要。

## 一字一画面

**长** —— 像【超长卷轴铺不开】：卷轴太长桌子放不下，先切成段（切分 + 滑动窗口），再抽出关键章节（RAG 检索），再写个目录（摘要压缩），或者直接换张大桌子（长窗口模型）（本质：长度限制是物理瓶颈，工程方案是空间腾挪术）。

## 面试话术

"超长上下文主要有四个解法，一是切分加滑动窗口，保留重叠区防断句；二是 RAG 只召回相关片段；三是先做摘要压缩提炼核心；四是直接上 Claude 200K 或 Gemini 1M 这种长窗口模型，或者 Map-Reduce 分治处理后再聚合。"

## 防混补丁

容易和**直接 truncation 截断**搞混？一句补丁：**长字记四招，截断丢信息，工程方案要记牢。**

## 追问预演

- **Q：滑动窗口怎么设计？**
  - A：按 Token 长度切，保留 10%~20% 重叠区，防止句子在切口处断裂丢语义。
- **Q：Map-Reduce 适合什么场景？**
  - A：批量处理长文档的同类任务，如批量总结、批量提取关键词，先 Map 分段处理再 Reduce 聚合。
- **Q：RAG 和长窗口模型怎么选？**
  - A：知识库动态更新选 RAG，单次超长文档全局理解选长窗口模型，两者也可组合使用。
