# RAG 中如何计算文本相似度？常见算法有哪些？

## 速记卡片
`似 | 似向量，点余欧 | 语义转向量再比距`

## 原始知识
- **核心思路**：文本→Embedding向量→算向量距离/相似度
- **点积（Dot Product）**：简单快速，受向量模长影响
- **余弦相似度（Cosine）**：只看方向不看长度，最常用
- **欧氏距离（Euclidean）**：绝对距离，适合同模长场景
- **曼哈顿距离**：网格路径距离，较少用
- **稠密检索**：语义Embedding+余弦，捕近义
- **稀疏检索**：BM25/TF-IDF，捕关键词
- **混合检索**：稠密+稀疏，RRF融合
- **易混点**：余弦和欧氏不是永远等价，仅当向量单位化时才等价

## 顺口溜
**似向量，点余欧**

## 单字锚点 + 防崩推导
- **锚点字**：**似**
- **扩展口诀**：**似距点余欧**
- **为什么选它**："似"=相似度，直接点题，信息密度最高
- **防崩推导**：
  - 似 → 相似度 → 先把文本变成向量（Embedding）
  - 距 → 距离越近越相似
  - 点 → 点积：快，但受长度影响
  - 余 → 余弦：.normalize后比方向，最常用
  - 欧 → 欧氏：比绝对距离，KNN常用
  - RAG首选 → 余弦相似度，语义对齐好

## 一字一画面
- **似** —— 像**两滴水靠得多近**：距离近就相似。（本质：空间 proximity）
- **点** —— 像**两人同时举手**：举得越高（模长越大）点积越大，但可能方向不同。（本质：投影乘积）
- **余** —— 像**两把弓箭指向**：只看箭尖朝向是否一致，不看弓大小。（本质：方向一致性）
- **欧** —— 像**量两点间直线**：尺子直接量绝对远近。（本质：绝对距离）

## 面试话术
- "RAG里算文本相似度，标准流程是**似向量，点余欧**。先把查询和文档都 embedding 成向量，然后算相似度。最常用的是余弦相似度，因为它只看语义方向、不受文本长度影响；点积快但受模长干扰；欧氏距离在KNN场景也用。"
- "实际工程中，很多团队用混合检索，稠密向量捕语义，BM25捕关键词，再用RRF融合结果。"

## 防混补丁
- 容易认为"距离小=相似度高就是直接画等号"。补丁：**余弦相似度=1-余弦距离，欧氏距离和余弦距离只有在单位向量时才等价**。

## 追问预演
- **问**：为什么Embedding后不用Jaccard相似度？
  - **答**：Jaccard基于集合交集，是词袋时代的做法，无法捕捉语义相似（如"国王"和"君主"）。
- **问**：向量维度对相似度计算有什么影响？
  - **答**：维度越高表达能力越强，但计算量越大，高维还会出现"维度灾难"使距离趋于均匀。
