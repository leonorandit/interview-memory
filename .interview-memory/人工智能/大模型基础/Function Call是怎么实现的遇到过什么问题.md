## 速记卡片

调 | 调定执反，四步走 | Function Call 实现 + 踩坑

## 原始知识

- **Function Call 实现四步流程**
  - **Step 1：定义（Define）**
    - 把工具/函数写成 JSON Schema，包含函数名、参数类型、描述、required 字段
    - 注入到 LLM 的 system prompt 或 tool 字段中
  - **Step 2：识别（Detect）**
    - LLM 收到用户 query 后，判断是否该调工具
    - 输出特殊格式：如 `<tool_call>`、`<function>`、JSON 块
  - **Step 3：执行（Execute）**
    - 解析 LLM 返回的函数名和参数，校验参数合法性
    - 调用真实函数/服务，获取返回结果
  - **Step 4：反馈（Feedback）**
    - 把工具返回结果拼回对话上下文，再次发给 LLM
    - LLM 基于结果生成最终回答
- **实际踩过的四个坑**
  - **格式坑**：LLM 输出的 JSON 不合法（缺少引号、字段拼写错误、多余注释）
    - 解决：JSON Schema 校验 + 正则清洗 + 失败重试（Auto-fix Loop）
  - **逾越坑**：LLM 调用了一个不存在的工具，或传了非法参数值
    - 解决：严格白名单校验 + 参数范围约束 + 默认值兜底
  - **循环坑**：LLM 陷入反复调用同一工具的循环（如不断查天气但参数不变）
    - 解决：最大调用次数限制（max_iterations）+ 调用历史去重检测
  - **超时坑**：工具执行时间太长（如数据库查询、网络请求），导致整体链路卡死
    - 解决：工具调用加超时熔断（circuit breaker）+ 异步降级 + 超时后返回友好提示
- **额外工程细节**
  - **并行调用**：GPT-4 支持 parallel function calling，一次返回多个 tool_call
  - **流式处理**：SSE 流式输出时，工具调用可能跨多个 chunk，需要缓冲组装完整 JSON
  - **安全沙箱**：执行外部命令/代码时需要沙箱隔离（如 Docker、Wasm）
  - **幂等性**：工具调用失败后重试，必须保证不会产生副作用（如重复扣款）

## 单字锚点 + 防崩推导

- **锚点字**：**调**
- **扩展口诀**：调定执反，四步走 | 格逾循超，四坑踩
- **为什么选它**："调"是工具调用最核心的动作，也是最容易出问题的地方。
- **防崩推导**：
  - 调 → 调用 → 四步流程
  - 调 → 定义工具（JSON Schema）→ LLM 识别要调什么 → 执行真实函数 → 反馈结果给 LLM
  - 四坑 → 格式（格）→ 逾越（越界）→ 循环（循）→ 超时（超）

## 一字一画面

- **调** —— 像**打电话叫人帮忙**：你先告诉对方都有谁可以帮忙（定义工具），对方判断要不要找人（识别），找到人后吩咐任务（执行），对方干完活回来汇报（反馈）。（本质：四步流程）
- **格** —— 像**收到的短信乱码**："明天3点"写成"明天3奌"，系统解析失败。（本质：LLM 输出 JSON 格式不合法）
- **逾** —— 像**叫了一个不存在的外卖员**：你点了肯德基，但接单的是"麦当劳骑手小李"，系统查无此人。（本质：LLM 调用了未定义的工具）
- **循** —— 像**自动回复机器人死循环**：你问客服"快递到哪了"，机器人反复说"请提供订单号"，你提供了它还是重复问。（本质：LLM 反复调用同一工具参数不变）
- **超** —— 像**外卖骑手卡在电梯里**：你饿了一个小时，订单状态一直显示"配送中"。（本质：工具调用超时无响应）

## 面试话术

"Function Call 的实现口诀是**调定执反，四步走**。第一步**定义**，把工具写成 JSON Schema 注入给 LLM；第二步**识别**，LLM 判断用户意图后输出要调用的函数和参数；第三步**执行**，后端解析 JSON 调用真实函数并拿到结果；第四步**反馈**，把结果塞回上下文让 LLM 生成最终回答。实际落地踩过四个坑：**格式坑**——LLM 输出 JSON 不合法，我们用 Schema 校验 + 正则清洗 + 失败重试解决；**逾越坑**——LLM 调了不存在的工具或传了非法参数，用白名单 + 参数范围约束兜底；**循环坑**——LLM 反复调用同一工具，用 max_iterations 限制 + 去重检测解决；**超时坑**——工具执行慢导致链路卡死，用超时熔断 + 异步降级解决。"

## 防混补丁

容易和 "MCP" 搞混。Function Call 是**大模型能力层面的工具调用协议**，MCP 是**系统层面的标准化工具接口规范**。补丁：**Function Call 是 LLM 的技能，MCP 是工具提供商的接口标准，两者可以结合使用**。

## 追问预演

- **Q：LLM 输出 JSON 不合法怎么办？**
  → 三层防护：① JSON Schema 预校验参数类型；② 正则提取 JSON 块，过滤多余字符；③ 失败时进入 Auto-fix Loop，让 LLM 自己修正。
- **Q：怎么防止 LLM 调用危险工具？**
  → 白名单机制：只暴露允许的工具；敏感操作（写库、执行命令）需要二次确认或人工审核；沙箱隔离执行环境。
- **Q：流式输出时怎么处理 Function Call？**
  → SSE 流式场景下，工具调用 JSON 可能跨多个 chunk，需要在网关层做缓冲组装，等收到完整 JSON 后再执行。
- **Q：工具调用失败怎么给用户友好的反馈？**
  → 区分失败类型：参数错误 → 提示用户补全；工具超时 → 提示"服务繁忙请重试"；工具不可用 → 提示"该功能暂时无法使用"，同时让 LLM 基于已有信息尽量回答。
