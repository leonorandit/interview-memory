# RAG 中的查询重写 Query Rewriting 是什么？如何优化检索效果？

## 速记卡片
`重 | 重写扩转同，召回更精准 | 改写→扩召回`

## 原始知识
- 用户原始查询常存在歧义、省略、口语化问题
- **重写**：用LLM把原始问题改写成更规范、更明确的查询
- **扩展**：一题拆多题，从不同角度生成子查询（HyDE、Multi-Query）
- **转译**：把自然语言转成结构化查询（如SQL、过滤器条件）
- **同义**：注入同义词、近义词，扩大语义覆盖
- **加粗易混点**：Query Rewriting ≠ Prompt Engineering，前者改的是检索侧的查询，后者改的是生成侧的提示

## 顺口溜
重写扩转同

## 单字锚点 + 防崩推导
- **锚点字**：**重**
- **扩展口诀**：**重写扩转同**
- **为什么选它**："重"字一出现，立刻联想到"重新写一遍"，是Query Rewriting最直观的动作。
- **防崩推导**：
  - 重 → 重新写 → 把口语化/歧义 query 改规范
  - 重 → 重写后还能扩 → 一题变多题（Multi-Query）
  - 重 → 转 → 转成结构化查询（如Text-to-SQL）
  - 重 → 同 → 注入同义词扩大覆盖
  - 最终目的：召回更精准

## 一字一画面
- **重** —— 像**橡皮擦改作文**：学生交上来的草稿太乱，老师用红笔圈出歧义处，重新写一遍通顺的句子。（本质：把模糊查询规范化）
- **扩** —— 像**一束光变三束探照灯**：一个问题从三个角度打光，暗处的文档也能被照到。（本质：Multi-Query扩展召回面）
- **转** —— 像**翻译官把方言转成普通话**：方言（自然语言）→ 普通话（结构化查询），让数据库听得懂。（本质：Text-to-SQL/Filter转换）
- **同** —— 像**撒网时多绑几个钩**：主钩是"汽车"，副钩挂"车辆/轿车/机动车"，捞上来的更多。（本质：同义词扩展）

## 面试话术
- "Query Rewriting的核心是**重写扩转同**。比如用户问'那个红色的多少钱'，系统会先重写成'红色iPhone 16的价格是多少'，再扩展出'红色iPhone 16官方售价'、'红色iPhone 16各渠道价格'等子查询，召回率能提升30%以上。"
- "我们线上用的是LLM做改写+HyDE生成伪文档做向量检索，这样既能消歧义又能扩展语义边界。"

## 防混补丁
- 容易和"提示词工程"搞混。Query Rewriting改的是**检索侧的输入**，Prompt Engineering改的是**生成侧的输入**。补丁：**检索前重写，生成前Prompt**。

## 追问预演
- **追问1**：HyDE是什么？→ Hypothetical Document Embeddings，让LLM先生成假设答案再向量化，用答案的向量去检索。
- **追问2**：Multi-Query会增加延迟怎么办？→ 并行发多个查询，或者先用轻量模型生成子查询。
- **追问3**：怎么评估改写效果？→ 看改写前后的Recall@K和NDCG有没有提升。
