# RAG 中如何选择 Embedding Model

## 速记卡片
`选 | 语维上，速域证 | 六因素`

## 原始知识
- 选择Embedding模型需考虑6个因素：
  1. **语**（语言）- 中文优先BGE，英文优先OpenAI/E5，多语言用M3E
  2. **维**（维度）- 维度越高表达能力越强，但存储和计算成本越高（768/1024/1536/4096）
  3. **上**（上下文长度）- 文档长短决定需要的上下文支持（512/2048/8192+）
  4. **速**（速度/资源）- 小模型快但效果略差，大模型慢但效果好
  5. **域**（领域适配）- 通用模型 vs 领域微调模型（医学、法律、代码）
  6. **证**（许可证/成本）- 开源免费 vs 商用API付费
- **易混点**：只看效果不看成本、只看维度不看速度
- **追问点**：怎么权衡维度？什么时候微调？

## 顺口溜
**语维上，速域证**

## 单字锚点 + 防崩推导
- **锚点字**：**选**
- **扩展口诀**：**六因素，选模型**
- **为什么选它**：面试题的核心就是"如何选择"，"选"字直接回应问题。
- **防崩推导**：
  - 选（选择）→ 根据六个因素选Embedding模型
  - 语（语言）→ 中文？英文？多语言？
  - 维（维度）→ 768/1024/1536，维度影响效果和成本
  - 上（上下文长度）→ 512/2048/8192，文档有多长？
  - 速（速度）→ 推理速度快不快？资源够吗？
  - 域（领域）→ 通用场景还是垂直领域？
  - 证（许可证/成本）→ 开源免费还是API付费？

## 一字一画面
- **语** —— 像**选翻译**：找中文翻译还是英文翻译？（本质：语言匹配）
- **维** —— 像**选相机像素**：像素越高照片越清，但文件越大。（本质：维度权衡）
- **上** —— 像**选车续航**：跑长途要续航长的。（本质：上下文长度匹配）
- **速** —— 像**选车排量**：排量小省油但慢，排量大费油但快。（本质：速度资源权衡）
- **域** —— 像**选医生**：普通体检找全科医生，心脏病找心内科。（本质：领域适配）
- **证** —— 像**选餐厅**：路边摊免费但环境一般，米其林贵但体验好。（本质：许可证和成本）

## 面试话术
- 问怎么选 → "选择Embedding模型口诀是**语维上，速域证**。**语**——语言，中文用BGE，英文用OpenAI/E5，多语言用M3E；**维**——维度，768-1024通用，1536+效果更强但成本高；**上**——上下文长度，短文档用512，长文档用2048+；**速**——速度和资源，小模型快，大模型好；**域**——领域适配，通用场景用预训练模型，垂直场景建议微调；**证**——许可证和成本，开源免费还是API付费。实际选型要综合这六个因素，没有绝对最好的，只有最适合的。"
- 问维度怎么权衡 → "维度越高表达能力越强，但向量数据库的存储和检索成本也越高。一般768-1024是甜点区，性价比最高。"

## 防混补丁
- 容易只看排行榜选模型，忽略实际场景。补丁：**MTEB是参考，场景是标准**。

## 追问预演
- **问**：什么时候微调Embedding模型？→ 垂直领域（医学、法律、代码）或特定业务场景，通用模型检索效果不佳时。
- **问**：维度和效果成正比吗？→ 不是线性关系。768→1024提升明显，1024→4096提升边际递减。
- **问**：可以换模型吗？→ 可以，但要重新生成所有向量并重建索引，迁移成本较高。
