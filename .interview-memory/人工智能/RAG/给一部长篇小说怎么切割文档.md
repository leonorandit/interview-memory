## 速记卡片

章 | 章回段句，层切留 | 长篇小说分块策略

## 原始知识

- **小说 vs 普通文档的差异**
  - 强叙事结构：章 → 回 → 段 → 句，层层递进
  - 上下文依赖极强：人物关系、情节铺垫、前后呼应，切散就丢语义
  - 对话密集：对话是推进情节的核心，不能把一句对话劈成两半
  - 角色众多：同一个人物可能有多个称呼（绰号、别名），需要上下文才能识别
- **四层切割策略（从大到小）**
  - **第一层：章（Chapter）**
    - 按自然章节切，一章作为一个独立语义单元
    - 适用：章内情节相对完整，跨章检索时以章为粒度召回
    - 上限：如果一章过长（如《红楼梦》某些回目），进入第二层
  - **第二层：回/节（Section）**
    - 按回目、小节、场景切换切
    - 标志：时间跳转、地点切换、POV 视角切换（如《冰与火之歌》）
    - 一个回目通常 500~2000 token，是 RAG 的"黄金粒度"
  - **第三层：段（Paragraph）**
    - 按自然段落切，保留对话完整性
    - 关键规则：对话不能切断——"他说："和后面的引号内容必须在同一块
    - 如果一个段落超过 chunk_size，进入第四层
  - **第四层：句（Sentence）**
    - 按句子切，作为兜底策略
    - 只用于超长段落（如大段环境描写、心理独白）
    - 必须保留相邻句子的 overlap
- **防切散的关键手段**
  - **Overlap 加倍**：小说 overlap 建议 20%~30%（普通文档 10%），因为叙事连贯性要求更高
  - **对话保护**：切割点不能落在引号内，必须等对话结束再切
  - **元数据标注**：每个块标注章节号、回目名、主要人物、时间线，召回后能做上下文拼接
  - **角色图谱辅助**：把人物关系、别名映射预先建好，RAG 召回时做实体链接增强
- **Chunk Size 建议**
  - 章级：2000~4000 token（适合"某章讲了什么"类查询）
  - 回/节级：500~1500 token（适合情节细节查询，黄金粒度）
  - 段/句级：256~512 token（适合金句、对话引用查询）
  - **混合策略**：三层索引同时建——章级粗召回 + 回级精召回 + 段级补充，Query Routing 根据问题类型路由到不同粒度

## 单字锚点 + 防崩推导

- **锚点字**：**章**
- **扩展口诀**：章回段句，层切留
- **为什么选它**：长篇小说的最基本结构单位就是"章"，"章"字一出立刻锚定小说场景。
- **防崩推导**：
  - 章 → 长篇小说 → 怎么切？
  - 章 → 第一层：按章切（最自然）
  - 回 → 第二层：按回目/节切（场景切换）
  - 段 → 第三层：按段落切（保对话完整）
  - 句 → 第四层：按句子切（兜底）
  - 层切 → 从大到小逐层切，不是一刀切
  - 留 → 留 overlap，防断句防断情节

## 一字一画面

- **章** —— 像**撕日历**：每天一张，撕下来还能独立看，但前后连起来是完整的一年。（本质：章是独立语义单元）
- **回** —— 像**电视剧一集**：每集有完整起承转合，但季与季之间有主线串联。（本质：回目是黄金检索粒度）
- **段** —— 像**剪纸不能剪到人脸**：剪刀必须沿着轮廓走，不能把眼睛鼻子剪两半。（本质：段落是保对话完整的最小单位）
- **句** —— 像**拼图的最后一片**：实在装不下就按片切，但每片边缘要留咬合齿。（本质：句子级切割 + overlap）

## 面试话术

"长篇小说的切割和普通文档完全不同，口诀是**章回段句，层切留**。小说有强叙事结构，上下文依赖极强，不能一刀切。第一层按**章**切，一章一个语义单元；章太长的按**回目/节**切，以场景切换为边界；再细的按**段落**切，但必须保护对话完整——不能把一句对话劈成两半；超长段落才按**句子**兜底切。防切散的关键是**overlap 加倍**到 20%~30%，因为小说叙事连贯性要求比普通文档高得多。另外必须打**元数据标签**——章节号、主要人物、时间线，召回后能把碎片拼回完整情节。我们项目上对百万字小说建了三层索引：章级粗召回 + 回级精召回 + 段级补充，Query Routing 根据问题是问整体情节还是问某句台词，路由到不同粒度检索。"

## 防混补丁

容易和 "普通文档切割" 搞混。普通文档（论文、手册）可以按固定 token 切，小说不行。补丁：**论文切丢了是丢知识点，小说切丢了是丢命——人物命运和情节命脉**。

## 追问预演

- **Q：如果小说没有明确章节怎么办？（如意识流小说）**
  → 用 LLM 做语义分块——让模型识别场景切换、时间跳转、POV 变化作为切割点，比按 token 数切更保语义。
- **Q：对话特别多的小说（如剧本）怎么切？**
  → 以"一场戏"为一个块，或把连续对话整体保留。如果单段对话超长，按角色轮次切，而不是按 token 数切。
- **Q：怎么保证召回后能把碎片拼回完整情节？**
  → 元数据标注章节号 + 时间线 + 人物，召回后按顺序拼接；或者用 Graph RAG 把人物关系、事件因果建成图谱，碎片通过图谱关联。
