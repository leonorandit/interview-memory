# RAG 中文档切割的 chunk_size 和 overlap 应该如何设置？

## 速记卡片
`切 | 切大块保语义，留重叠防断肢 | size vs overlap`

## 原始知识
- **chunk_size**：单块长度，大→语义完整但噪音多，小→精准但上下文断裂
- **overlap**：相邻块重叠字数，防断句断意
- **常规值**：size 200~500 tokens，overlap 10%~20%
- **自适应**：按文档结构（段落、句子）切，而非固定字数
- **加粗易混点**：overlap不是越大越好，太大会导致重复检索、成本上升

## 顺口溜
大块保语义

## 单字锚点 + 防崩推导
- **锚点字**：**切**
- **扩展口诀**：**大块保语义**
- **为什么选它**："切"直接对应"文档切割"，是RAG ingest阶段最关键的环节。
- **防崩推导**：
  - 切 → 切菜 → 块太大（size大）→ 语义完整但不易消化
  - 切 → 块太小（size小）→ 精准但断了上下文
  - 切 → 留overlap → 像切肉留点筋连着 → 防断肢
  - 切 → 一般200-500 tokens → 10%-20% overlap
  - 切 → 按结构切优于按字数切

## 一字一画面
- **切** —— 像**切生日蛋糕**：切太大（size=1000）一块吃不完还掉渣；切太小（size=50）奶油和蛋糕分离，吃不出味道。（本质：chunk_size平衡完整性与精准性）
- **大** —— 像**切西瓜大块**：瓜瓤完整，籽都在里面，但一口咬下去可能吃到不甜的部分。（本质：大块语义完整但可能混噪音）
- **小** —— 像**切土豆丝**：每根均匀精准，但炒完分不清哪根是哪根，断了来源。（本质：小块精准但上下文断裂）
- **叠** —— 像**贴创可贴重叠贴**：两张创可贴边缘重叠，伤口才不会露出来。（本质：overlap防断句断意）

## 面试话术
- "文档切割我们总结为**切大块保语义，留重叠防断肢**。chunk_size一般设200-500 tokens，太大检索不精准，太小上下文断裂；overlap设10%-20%，防止句子被切断。但更好的做法不是固定字数，而是按段落、句子语义切分，比如用RecursiveCharacterTextSplitter。"
- "我们项目里会对不同文档类型设不同策略，技术文档按标题层级切，对话记录按轮次切。"

## 防混补丁
- 容易认为"overlap越大越好"。太大导致重复检索、向量库膨胀、LLM Prompt浪费。补丁：**叠一点是桥，叠太多是堵**。

## 追问预演
- **追问1**：怎么评估切分效果？→ 看检索后的答案完整性，如果经常"前半句在A块、后半句在B块导致答不全"，就是切分问题。
- **追问2**：有图片的文档怎么切？→ 图文分离，图片OCR后文本块与附近文本做关联。
- **追问3**：代码文档怎么切？→ 按函数/类切，保留函数签名和注释，overlap放在函数间的导入语句上。
