## 原始知识

- **知识库路由**：按领域意图路由到对应知识库，避免无关检索拖慢速度
- **混合检索**：向量检索+关键词检索双管齐下，提高召回率
- **Rerank精排**：交叉编码器重排序，把最相关的往前放
- **上下文压缩**：去除冗余信息，保留关键上下文，减少幻觉
- **多路召回**：同时查询多个库，取Top-K合并去重
- **一致性校验**：检索结果与问题相关性打分，低置信度拒绝回答

## 顺口溜

路查排，五策略，
多压混，一气成。

（路由查路，精排先后，多路召回，压缩去冗，混合检索，一气校验成。）

## 单字锚点 + 防崩推导

**锚点字**：路

**扩展口诀**：路查排，五策略，多压混，一气成

**为什么选它**：RAG面对多知识库，第一步就是决定"走哪条路"——路由是整个流程的起点，丢了"路"后面全乱。

**推导链**：
1. 路 → 知识库路由（Query Routing）：根据问题意图选择对应知识库，不走冤枉路
2. 查 → 混合检索（Hybrid Retrieve）：向量+关键词，查得更全
3. 排 → Rerank精排（Re-rank）：交叉编码器，排得更准
4. 多 → 多路召回（Multi-recall）：多个库同时查，合并去重
5. 压 → 上下文压缩（Compress）：压掉垃圾信息，减少幻觉
6. 混 → 混合检索的"混"，也对应多路混合
7. 一气成 → 一致性校验（Consistency Check）：最后一道关卡，低置信度直接拒答

## 一字一画面

- **路** —— 像高速公路分叉口：绿色路牌指向不同城市，导航自动选最优车道（本质：意图识别+库选择）
- **查** —— 像侦探同时用指纹和监控找人：两种证据互补，缺一不可（本质：语义+关键词双通道）
- **排** —— 像裁判打分排名：10个选手再比一轮，冠军才站C位（本质：二次精排提升准度）
- **多** —— 像多路水管汇入主道：各支流同时放水，过滤杂质后合并（本质：并行检索+去重）
- **压** —— 像榨汁机去渣留汁：果皮纤维扔掉，只留浓缩果汁（本质：冗余剔除，保核心上下文）
- **气** —— 像安检门：红色警报响，可疑人员不准进（本质：相关性阈值，低分拒答防幻觉）

## 面试话术

"多知识库RAG，我通常会从六个维度来保证效率和准确性。首先是**路由**，根据Query意图分到对应的知识库，避免全库扫描。然后是**混合检索**加**多路召回**保证召回率，再用**Rerank精排**把最相关的文档挑出来。最后通过**上下文压缩**减少Token消耗和幻觉，再加一道**一致性校验**的保险。"

## 防混补丁

- **容易和什么搞混**：容易只提向量检索，忽略关键词检索和Rerank的"查-排"组合拳。
- **补丁口诀**：先路后查再精排，多压混检六步来。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 路由怎么实现？ | 用意图分类器或向量相似度匹配，把Query映射到领域标签，再定向到对应知识库 |
| Rerank和普通排序区别？ | 普通排序是向量相似度，Rerank用交叉编码器把Query和文档一起编码，交互更深，准度更高 |
| 上下文压缩具体怎么做？ | 可以用LLM总结文档摘要，或提取关键句子，去掉重复和无关段落 |
| 一致性校验怎么打分？ | 用LLM或BERT模型判断检索结果是否回答了问题，低于阈值就返回"无法回答" |
| 多路召回合并策略？ | 取各路Top-K，按分数加权去重，或直接用RRF（Reciprocal Rank Fusion） |
