## 速记卡片

## 原始知识

- **Query重写/扩展**：把用户query改写成多个同义表达
- **Embedding模型微调**：用领域数据精调向量模型，让语义更准
- **混合检索**：向量+关键词+BM25多路召回互补
- **重排序Rerank**：召回后精排，把最匹配的顶上去
- **反馈闭环**：用户点击反馈调优，越用越准
- **意图识别模块前置**：先分意图，再定向检索

## 顺口溜

意不对，五路追：
改写扩，微调陪，
混合搜，重排位，
反馈闭环意图归。

## 单字锚点 + 防崩推导

- **锚点字**：意
- **扩展口诀**：意改写，五改进
- **为什么选它**：问题核心是"意图不匹配"，记住"意"就记住了方向
- **推导链**：
  1. 意 → 意图不匹配
  2. 先改query → **Query重写/扩展**
  3. 改完还错 → **Embedding模型微调**
  4. 向量不够 → **混合检索**补位
  5. 召回太粗 → **重排序Rerank**精修
  6. 长期迭代 → **反馈闭环**
  7. 源头根治 → **意图识别模块前置**

## 一字一画面

## 面试话术

"当RAG召回和query意图不匹配时，我会从五个方向改进：首先是Query重写扩展，把用户问题换一种说法再去搜；其次是微调Embedding模型，让它更懂我们领域的语义；然后是混合检索，向量加关键词双管齐下；接着上Rerank做精排；最后搭一个反馈闭环，根据用户点击持续优化。"

## 防混补丁

容易和"检索速度慢"搞混？一句补丁：

## 追问预演

- **Q：Query重写具体怎么做？**
  A：可以用LLM生成同义query，或者做HyDE假设文档嵌入。
- **Q：混合检索怎么融合结果？**
  A：RRF倒数排序融合，或者加权线性融合。
- **Q：反馈闭环数据少怎么办？**
  A：冷启动用人工标注，积累到一定量再上线模型。
