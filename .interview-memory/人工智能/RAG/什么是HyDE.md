# 什么是 HyDE？

## 速记卡片
`假 | 假答代查，反召补沟 | HyDE原理`

## 原始知识
- **HyDE** = Hypothetical Document Embeddings（假设文档嵌入）
- **核心流程**：用户查询 → LLM 生成假设答案 → Embedding → 向量检索
- **本质**：用**假设答案的 embedding** 代替**原始查询的 embedding** 去检索
- **解决的问题**：查询与文档之间的**语义鸿沟**（query-doc mismatch），尤其是查询短、文档长、表达不对齐时
- **优点**： bridging 语义鸿沟，召回更准，尤其适合领域知识密集、用户表达不专业的场景
- **缺点**：多一次 LLM 调用，增加延迟和成本；假设答案若出错，错误会传播给检索结果
- **易混点**：HyDE 不是查询重写，也不是 RAG-Fusion
- **追问点**：和 Query Rewriting 的区别？什么时候用最好？缺点怎么缓解？

## 顺口溜
**假答代查，反召补沟**

## 单字锚点 + 防崩推导
- **锚点字**：**假**
- **扩展口诀**：**假答代查**
- **为什么选它**：HyDE 的灵魂是"假设答案"（Hypothetical Document），先想到"假"字就能串起全部。
- **防崩推导**：
  - 假 → 假设 → LLM 根据查询生成假设答案
  - 假 → 代替 → 不用原查询，用假答的 embedding 去检索
  - 反 → 反向 → 先答后查，反着来（传统是 query→doc，HyDE 是 query→answer→doc）
  - 召 → 召回 → 假答的语义更丰富，召回更准
  - 补 → 补鸿沟 → 查询短、文档长，语义不对齐，假答 bridging

## 一字一画面
- **假** —— 像**先写参考答案再查标准答案**：考试时看不懂题，先按自己理解写个草稿（假设答案），再拿草稿去对标准答案。（假设文档生成）
- **答** —— 像**拿着草稿去图书馆**：不是带着问题去问管理员，而是带着"我认为答案应该长这样"的草稿去比对书架。（用答案代替查询）
- **反** —— 像**反向画像**：警察先根据目击者描述画出嫌疑人画像（假设答案），再拿着画像去通缉库比对。（先答后查的反向逻辑）
- **沟** —— 像**架桥过河**：用户问"怎么修"（短查询），文档写"故障排查三步走"（长文档），中间隔着河，HyDE 用假答搭桥。（ bridging 语义鸿沟）

## 面试话术
- 问 HyDE → "HyDE 是 Hypothetical Document Embeddings，口诀是**假答代查，反召补沟**。传统 RAG 拿用户查询直接检索，但查询往往很短，和文档有语义鸿沟。HyDE 的做法是先用 LLM 生成一个假设答案，再用这个假设答案的 embedding 去检索，相当于拿着'我认为答案长这样'的草稿去对标准答案。优点是 bridging 语义鸿沟，召回更准；缺点是多一次 LLM 调用，有延迟和成本，而且假设答案错了会带偏检索。"
- 问区别 → "HyDE 和 Query Rewriting 不一样：**重写是换种问法（query→query），HyDE 是替换成答案（query→doc）**。"

## 防混补丁
- 容易和"查询重写 Query Rewriting""RAG-Fusion"搞混。
- 补丁：**重写是换种问法，Fusion是多问几遍，HyDE是先写答案再反查。**

## 追问预演
1. **HyDE 和 Query Rewriting 的核心区别？**
   → Query Rewriting 是把 query 改写成更清晰的 query（还是问题）；HyDE 是把 query 生成假设答案（变成文档），用文档 embedding 去检索。

2. **HyDE 的缺点是什么？怎么缓解？**
   → 多一次 LLM 调用，增加延迟和成本；假设答案若有幻觉，错误会传播。缓解：控制生成长度；结合 Rerank 二次筛选；对简单/明确查询 fallback 到原始查询检索。

3. **什么时候用 HyDE 效果最好？**
   → 用户查询短、模糊、和文档语义鸿沟大时；领域知识密集、用户表达不专业时（如口语化表达查医学/法律文献）。

4. **HyDE 和 RAG-Fusion 的区别？**
   → HyDE 是生成一个假设答案去检索；RAG-Fusion 是生成多个查询变体，多路检索后融合。

5. **实现 HyDE 时需要注意什么？**
   → 生成假设答案的 prompt 设计很关键（要引导生成"理想文档"而非直接答案）；假设答案和文档必须共用同一个 embedding 模型；生成长度要控制，避免噪音。
