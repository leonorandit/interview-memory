# RAG 中的提示压缩

## 速记卡片
`压 | 长压短，省窗钱 | 提示压缩`

## 原始知识
- 提示压缩（Prompt Compression）：把检索到的长上下文压缩成更短的提示再喂给LLM
- 为什么需要：
  - 检索结果太多，超过LLM上下文窗口限制
  - 长上下文增加推理成本（按token计费）
  - 无关信息会稀释注意力，降低生成质量
- 压缩方法：提取摘要、删除冗余、保留关键句、选择性编码
- **易混点**：提示压缩 vs 上下文截断
- **追问点**：压缩会不会丢失信息？怎么平衡压缩率和信息保留？

## 顺口溜
**长压短，省窗钱**

## 单字锚点 + 防崩推导
- **锚点字**：**压**
- **扩展口诀**：**压长文，省窗力**
- **为什么选它**：提示压缩的核心动作就是"压"，把长文压短。
- **防崩推导**：
  - 压（压缩）→ 把检索到的长上下文压短
  - 长（长上下文）→ 检索结果可能很长
  - 短（短提示）→ 压缩后的提示更短
  - 省（省窗口）→ 不超出LLM上下文限制
  - 钱（省钱）→ 减少token消耗，降低成本
  - 力（省力）→ 减少注意力稀释，提高生成质量

## 一字一画面
- **压** —— 像**真空压缩袋**：把蓬松的棉被压成薄饼，不占地方。（本质：压缩上下文长度）
- **长** —— 像**冗长的会议记录**：几十页纸，大部分废话。（本质：检索结果可能包含冗余信息）
- **短** —— 像**会议纪要**：一页纸把重点说完。（本质：保留关键信息，去除冗余）
- **省** —— 像**省房租**：房间小了，租金也少了。（本质：节省上下文窗口和推理成本）

## 面试话术
- 问什么是提示压缩 → "提示压缩是RAG中把检索到的长上下文压短再喂给LLM的技术，口诀是**长压短，省窗钱**。**压**——把检索结果压缩；**短**——生成更短的提示；**省窗**——节省LLM上下文窗口；**省钱**——减少token消耗。比如检索回10篇文档，先压缩成一段摘要再喂给模型，比直接塞原文效果好。"
- 问和截断区别 → "截断是简单粗暴地砍掉后面部分，可能丢掉关键信息；压缩是智能提炼，保留核心内容。补丁：**截断是剁手，压缩是瘦身**。"

## 防混补丁
- 容易和"上下文截断"搞混。截断是硬砍，压缩是智能提炼。补丁：**截断是剁手，压缩是瘦身**。

## 追问预演
- **问**：压缩会不会丢失信息？→ 会，所以要在压缩率和信息保留之间平衡。可以用重要性评分，保留关键句，删除冗余。
- **问**：有哪些压缩方法？→ 提取摘要、句子选择（Selective Context）、基于信息熵的压缩、LLM-based压缩（让模型自己提炼）。

## 手撕代码（提示压缩伪代码）

```python
# 1. 检索到长上下文
retrieved_docs = vector_db.search(query, top_k=10)
long_context = "\n".join([doc.text for doc in retrieved_docs])

# 2. 压缩：提取关键句或生成摘要
compressed = llm.compress(
    prompt=f"从以下上下文提取与问题'{query}'最相关的关键信息，控制在500字内：\n{long_context}"
)

# 3. 生成回答
answer = llm.generate(f"基于以下信息回答问题：\n{compressed}\n问题：{query}")
```
