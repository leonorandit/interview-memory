# RAG 中的分块

## 速记卡片
`块 | 长切短，限精噪 | 文档切分`

## 原始知识
- 分块（Chunking）：把长文档切分成小块（Chunk）
- 为什么需要：
  - **限**（长度限制）- Embedding模型有最大输入长度限制（如512/1024/8192 tokens）
  - **精**（检索精度）- 小块语义更集中，检索匹配更精准
  - **噪**（减少噪声）- 大块包含太多无关信息，Embedding会被稀释
- 分块大小影响：太小→丢失上下文；太大→噪声多、匹配差
- **易混点**：分块 vs 分片（Sharding）
- **追问点**：分块大小怎么选？重叠（Overlap）有什么作用？

## 顺口溜
**长切短，限精噪**

## 单字锚点 + 防崩推导
- **锚点字**：**块**
- **扩展口诀**：**长文档，切短块**
- **为什么选它**：分块的核心产出就是"块"，面试官问分块，答"块"字就能展开。
- **防崩推导**：
  - 块（Chunk）→ 文档被切成的小块
  - 长（长文档）→ 原始文档很长
  - 切（切分）→ 把长文档切成块
  - 限（长度限制）→ Embedding模型有输入长度限制
  - 精（精度）→ 小块语义集中，检索更精准
  - 噪（噪声）→ 大块噪声多，Embedding被稀释

## 一字一画面
- **块** —— 像**切豆腐**：把一大块豆腐切成小方块，方便下锅。（本质：文档切分）
- **长** —— 像**长长的面条**：一整根没法吃，必须切段。（本质：长文档超出模型处理能力）
- **切** —— 像**刀工**：切得大小均匀，菜才好吃。（本质：分块策略要合理）
- **限** —— 像**碗的大小**：碗只有这么大，必须切小才能装下。（本质：Embedding模型输入长度限制）
- **精** —— 像**精准夹菜**：小块容易夹到想吃的，大块一夹一堆。（本质：小块检索更精准）

## 面试话术
- 问为什么分块 → "RAG中的分块口诀是**长切短，限精噪**。**长**——原始文档很长；**切**——切成小块；**短**——每块长度适中；**限**——Embedding模型有输入长度限制；**精**——小块语义集中，检索更精准；**噪**——大块包含太多无关信息，Embedding会被噪声稀释。分块是RAG的基础，块的大小直接影响检索效果。"
- 问分块大小怎么选 → "一般100-500 tokens，看文档类型。技术文档可以小一点（100-200），小说散文可以大一点（300-500）。关键是有重叠（Overlap，比如20%），防止上下文断裂。"

## 防混补丁
- 容易和"数据库分片（Sharding）"搞混。分块是切文档，分片是切数据库。补丁：**分块切文档，分片切数据库**。

## 追问预演
- **问**：分块大小怎么选？→ 100-500 tokens常见，技术文档小（100-200），叙述文本大（300-500）。需要实验调优。
- **问**：重叠（Overlap）有什么作用？→ 防止上下文断裂，保证语义连贯。一般设10%-20%重叠。
- **问**：分块太小会怎样？→ 丢失上下文，语义不完整。比如"他"指代谁，小块里可能找不到。

## 手撕代码（分块伪代码）

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 递归分块：先按段落，再按句子，再按字符
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 每块500字符
    chunk_overlap=50,    # 重叠50字符（10%）
    separators=["\n\n", "\n", "。", "！", "？", " "]
)

chunks = text_splitter.split_text(long_document)
```
