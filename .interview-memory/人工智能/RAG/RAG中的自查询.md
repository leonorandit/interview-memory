# RAG 中的自查询

## 速记卡片
`滤 | 自解滤，查更准 | 筛子`

## 原始知识
- 自查询（Self-Query）：让LLM分析用户查询，提取结构化过滤条件，生成带过滤条件的检索请求
- 为什么需要：
  - 用户查询中包含隐式过滤条件（"最近一年的报告"、"价格>1000的商品"）
  - 纯向量检索无法处理这些结构化条件
  - 通过元数据过滤提高检索**精度**（Precision）
- 与查询扩展的区别：扩展提高**召回**，自查询提高**精度**
- **易混点**：自查询 vs 查询扩展、自查询 vs Rerank
- **追问点**：和扩展能一起用吗？需要LLM参与吗？支持哪些过滤条件？

## 顺口溜
**自解滤，查更准**

## 单字锚点 + 防崩推导
- **锚点字**：**滤**
- **扩展口诀**：**自解滤，查更准**
- **为什么选它**：自查询的核心动作是从查询中**提取过滤条件**，"滤"是灵魂。
- **防崩推导**：
  - 滤（过滤）→ 自查询的核心是从查询中提取过滤条件
  - 解（理解）→ LLM先理解用户查询中的结构化意图
  - 查（查询）→ 生成带过滤条件的结构化查询
  - 更准 → 通过元数据过滤提高检索精度

## 一字一画面
- **滤** —— 像**筛子**：把查询里的"最近一年"、"价格>1000"这些条件筛出来。（本质：提取结构化过滤条件）
- **解** —— 像**翻译官**：把用户的自然语言翻译成数据库能懂的过滤语句。（本质：LLM理解查询意图）
- **查** —— 像**带条件的搜索**：在搜索框里不仅输入关键词，还勾选了筛选条件。（本质：生成结构化查询）

## 面试话术
- 问什么是自查询 → "自查询是RAG中的一种查询优化技术，口诀是**自解滤，查更准**。**解**——LLM先理解用户查询中的结构化意图；**滤**——从查询中提取过滤条件，比如'最近一年的报告'里提取出时间过滤条件；**查**——生成带过滤条件的结构化查询去检索向量数据库；**更准**——通过元数据过滤提高检索精度。自查询和查询扩展不同：扩展是'撒网'提高召回，自查询是'筛子'提高精度。"
- 问和扩展能一起用吗 → "能。先用自查询提取过滤条件缩小范围，再用查询扩展生成多个查询提高召回，最后用Rerank精排。口诀：**自查询筛子，扩展撒网，Rerank挑鱼**。"

## 防混补丁
- 容易和"查询扩展"搞混。查询扩展是增加候选集（提高召回），自查询是精确过滤（提高精度）。补丁：**扩展是撒网，自查询是筛子**。
- 容易和"Rerank"搞混。Rerank是检索后精排，自查询是检索前加过滤条件。补丁：**自查询是进门先检票，Rerank是进门后再排队**。

## 追问预演
- **问**：自查询和查询扩展能一起用吗？→ 能。先用自查询提取过滤条件缩小范围，再用查询扩展生成多个查询提高召回。
- **问**：自查询需要LLM参与吗？→ 需要。通常用一个专门的LLM Prompt来提取查询中的过滤条件、排序要求等。
- **问**：自查询支持哪些过滤条件？→ 元数据过滤（时间、类别、作者）、范围过滤（价格区间、数值比较）、布尔过滤等。

## 手撕代码（自查询伪代码）

```python
# 1. LLM理解查询并提取过滤条件
structured_query = llm.parse(
    prompt=f"从查询'{query}'中提取过滤条件，输出JSON格式："
           f"{{'keywords': '...', 'filters': {{'year': '>', 'category': '='}}}}"
)

# 2. 生成向量查询
query_embedding = embedding_model.encode(structured_query.keywords)

# 3. 带过滤条件的向量检索
results = vector_db.search(
    query_embedding, 
    top_k=10,
    filters=structured_query.filters  # 元数据过滤：year>2024 AND category='报告'
)
```
