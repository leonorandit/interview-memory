## 速记卡片

切 | 固递语智，重叠留 | 文档分块策略 + 防语义切散

## 原始知识

- **为什么需要分块**
  - Embedding 模型有输入长度限制（通常 512~8192 token）
  - 检索精度要求：块太大 → 噪声多、召回粗；块太小 → 上下文丢失、语义破碎
- **四大分块策略**
  - **固定长度分块（Fixed-size）**：按固定 token 数切，简单快速，但容易切断句子
  - **段落/语义分块（Semantic）**：按段落、章节、标题切，保留自然语义边界
  - **递归分块（Recursive）**：先按大粒度（段落）切，超长的再按小粒度（句子）二次切
  - **智能分块（Agentic/LLM-based）**：让 LLM 判断切割点，动态决定块大小和边界
- **防语义切散的核心手段**
  - **Overlap（重叠）**：相邻块之间保留 10%~20% 的重叠内容，确保上下文衔接
  - **大块保语义**：优先按段落/章节切，不把同一段内容劈成两半
  - **边界感知**：切割点选在句号、换行、标题处，避免从句中切断
  - **元数据标注**：给每个块打上来源文档、章节、页码标签，召回后能做上下文拼接
- **Chunk Size 选型**
  - 256 token：适合 QA、短答案场景，精度高但可能丢上下文
  - 512 token：通用甜点，兼顾精度和语义完整性
  - 1024+ token：适合长文档摘要、代码块，但噪声会增加
  - 动态 chunk：根据文档类型（论文 vs 邮件 vs 代码）自适应调整

## 单字锚点 + 防崩推导

- **锚点字**：**切**
- **扩展口诀**：固递语智，重叠留
- **为什么选它**："切" 是文档切割最直观的动作，切不好就散。
- **防崩推导**：
  - 切 → 怎么切？→ 四种策略
  - 固 → 固定长度切（最简单）
  - 递 → 递归分块（大切不动再细切）
  - 语 → 语义/段落切（保边界）
  - 智 → 智能分块（LLM 动态切）
  - 重叠留 → 防切散的关键：overlap + 边界感知

## 一字一画面

- **固** —— 像**菜刀切黄瓜**：每刀间隔固定，快但可能把籽切飞。（本质：固定长度，简单高效但边界乱）
- **递** —— 像**拆俄罗斯套娃**：先拆大盒（段落），装不下的再拆小盒（句子）。（本质：递归粒度递减）
- **语** —— 像**按页撕书**：每章一页一页撕，不会把一句话撕成两半。（本质：按语义边界切）
- **智** —— 像**裁缝量体裁衣**：不是统一尺寸，而是根据身材（内容）动态裁剪。（本质：LLM 判断最佳切割点）
- **重叠留** —— 像**铺地砖留缝**：两块砖之间留缝隙，热胀冷缩不开裂。（本质：overlap 保证块间上下文衔接）

## 面试话术

"文档切割的核心口诀是**固递语智，重叠留**。四种策略：第一是**固定长度**，按 token 数切，简单快速但容易切断语义；第二是**递归分块**，先按段落切，超长的再按句子二次切；第三是**语义分块**，按自然段落、标题边界切，最大程度保留语义完整性；第四是**智能分块**，用 LLM 判断最佳切割点。防切散的关键有三点：**大块保语义**——优先按段落切，不把一句话劈开；**留重叠**——相邻块保留 10%~20% 重叠，确保上下文衔接；**边界感知**——切割点选在句号、换行处。我们项目上用的是递归分块 + 512 token + 64 token overlap，在 C-MTEB 评测集上召回率比固定 256 token 提升了 12%。"

## 防混补丁

容易和 "Embedding 模型选型" 搞混。分块是**检索前的预处理**，Embedding 是**检索时的编码**，两者独立但互相影响。补丁：**分块决定喂什么，Embedding 决定怎么比**。

## 追问预演

- **Q：代码文档怎么切？**
  → 按函数/类边界切，保留函数签名和注释；或者按 AST 语法树节点切，确保逻辑块完整。
- **Q：Overlap 设置多少合适？**
  → 一般 10%~20%，512 token 的块留 50~100 token overlap。太大浪费存储，太小接不上上下文。
- **Q：表格数据怎么切？**
  → 整张表作为一个块，或者按行切但保留表头元数据。不能把一行数据劈成两块。
- **Q：有没有不分块的方案？**
  → 长上下文模型（如 Gemini 1M、Claude 200K）可以整篇丢进去，但检索时还是需要定位到相关片段，通常配合 "sentence-window retrieval" 做精确定位。
