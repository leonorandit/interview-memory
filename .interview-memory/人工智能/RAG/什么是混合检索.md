# 什么是混合检索

## 速记卡片
`混 | 稀稠混，补短长 | 稀稠互补`

## 原始知识
- 混合检索 = 稀疏检索（Sparse/BM25）+ 稠密检索（Dense/向量）的结合
- 稀疏检索：基于关键词精确匹配，擅长产品ID、人名、专业术语
- 稠密检索：基于语义向量相似度，擅长理解意图、同义词、上下文
- 融合方式：RRF（倒数排名融合）、加权分数融合、Rerank重排
- **易混点**：混合检索 vs 多路召回
- **追问点**：RRF原理是什么？怎么调权重？什么时候不需要混合检索？

## 顺口溜
**稀稠混，补短长**

## 单字锚点 + 防崩推导
- **锚点字**：**混**
- **扩展口诀**：**混稀稠，补短长**
- **为什么选它**：混合检索的核心就是"混"——把稀疏和稠密混在一起，取长补短。
- **防崩推导**：
  - 混 → 混什么？稀疏检索 + 稠密检索
  - 稀（稀疏/BM25）→ 擅长精确关键词匹配，不擅长语义理解
  - 稠（稠密/向量）→ 擅长语义匹配，不擅长精确关键词
  - 短 → 各自的短板被对方补上
  - 长 → 两者的长处结合起来，召回更全面

## 一字一画面
- **混** —— 像**鸳鸯火锅**：一边是清汤（稀疏检索），一边是红汤（稠密检索），两种味道各取所长。（本质：混合检索结合多种检索策略）
- **稀** —— 像**查字典**：精确匹配每个字，找得快但不理解上下文。（本质：稀疏检索/BM25，基于关键词精确匹配）
- **稠** —— 像**读小说理解情节**：能理解语义和上下文，但可能漏掉具体人名或产品ID。（本质：稠密检索/向量检索，基于语义相似度）
- **短** —— 像**只用一种工具修东西**：字典查不到近义词，小说读不懂专业术语，都有盲区。（本质：单一检索方式的局限性）
- **长** —— 像**工具箱里有各种工具**：字典+语义理解 combined，既能精确匹配又能理解意图。（本质：混合检索取长补短，覆盖更多场景）

## 面试话术
- 问什么是混合检索 → "混合检索，口诀是**稀稠混，补短长**。它把**稀疏检索**（如BM25关键词匹配）和**稠密检索**（如向量语义检索）结合起来。稀疏检索擅长精确关键词匹配，但不理解语义；稠密检索擅长语义理解，但对精确关键词（如产品ID、人名）容易漏掉。混合检索取长补短，让召回结果更全面、更准确。"
- 问解决什么问题 → "主要解决单一检索方式的局限性。纯向量检索对精确术语匹配差，纯关键词检索对语义理解差。混合检索通过多路召回+融合排序，显著提升RAG系统的检索质量，减少幻觉。"

## 防混补丁
- 容易和"多路召回"搞混。多路召回是同时用多种策略检索；混合检索特指**稀疏+稠密**的结合，是多路召回的一种常见形式。补丁：**多路是策略，混合是稀稠**。

## 追问预演
- **问**：混合检索怎么融合两种结果？→ 常见的有RRF（Reciprocal Rank Fusion，倒数排名融合）、加权分数融合、或者把两者结果都送给Rerank模型重排。
- **问**：RRF的原理是什么？→ RRF不需要调参，直接对两个列表的排名取倒数求和，排名越靠前的分数越高，简单有效。公式：score = Σ(1/(k+rank))。
- **问**：什么时候不需要混合检索？→ 如果数据量小、查询简单、或者已经有很好的向量嵌入，单一检索可能就够了。

## 手撕代码（RRF伪代码）

```python
def reciprocal_rank_fusion(sparse_results, dense_results, k=60):
    scores = {}
    # 稀疏检索排名
    for rank, doc_id in enumerate(sparse_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    # 稠密检索排名
    for rank, doc_id in enumerate(dense_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    # 按融合分数排序
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```
