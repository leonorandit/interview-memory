## 原始知识

- **Re-Reading定义**：让LLM重新阅读检索结果和原始问题，生成更准确的回答
- **解决问题**：LLM忽略关键信息、看漏文档细节、答非所问
- **实现原理**：第一次生成初步回答→第二次检查是否遗漏→第三次基于检查修正回答
- **Spring AI实现**：自定义Advisor实现ReReadingAdvisor
- **插入点**：在retrieve和generate之间插入重读逻辑
- **可配置项**：重读次数、Prompt模板

## 顺口溜

复重理，三步走，
初检正，漏不留。

（先复现初稿，再重检遗漏，最后理正答案。）

## 单字锚点 + 防崩推导

**锚点字**：复

**扩展口诀**：复重理，三步走

**为什么选它**：Re-Reading的核心就是"重新"——复。只要记住"复"，就记住了重新阅读、重新检查、重新整理的全过程。

**推导链**：
1. 复 → Re-Reading = 重新阅读（Re-read）
2. 重 → 重新检查（Re-check）：对照原文，检查有无遗漏关键信息
3. 理 → 重新整理（Re-organize）：基于检查结果修正回答
4. 三步走 → 初稿 → 检查 → 正稿，三个阶段
5. Spring AI → Advisor机制：在retrieve后、generate前插入重读逻辑，像过滤器一样层层加工

## 一字一画面

- **复** —— 像学生考前复看错题本：红笔圈出的重点再看一遍，防止重蹈覆辙（本质：二次审视，降低遗漏率）
- **重** —— 像质检员打着手电复检零件：第一次流水线下来的产品，逐个再查一遍（本质：交叉验证，发现盲区）
- **理** —— 像理发师最后修边：剪完整体再看，修掉不整齐的碎发（本质：基于反馈精修输出）
- **步** —— 像上台阶：左脚初稿，右脚检查，最后一步稳稳落地（本质：分阶段递进，不可跳步）

## 面试话术

"Re-Reading是一种让LLM'再看一遍'的策略，主要是解决大模型在生成回答时容易忽略检索结果中关键信息的问题。实现上就是三步：先让模型生成初步回答，然后把原始问题和检索结果再喂给它，让它自我检查有没有漏掉什么，最后基于检查意见修正输出。在Spring AI里，可以通过自定义Advisor，在retrieve和generate之间插入这个重读逻辑。"

## 防混补丁

- **容易和什么搞混**：容易和RAG本身的检索步骤混淆——Re-Reading不是重新去检索，而是重新阅读已有的检索结果。
- **补丁口诀**：不是重新查，而是重新看，三遍读透再交卷。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Re-Reading和Self-RAG区别？ | Self-RAG会判断是否需要检索并可能多次检索；Re-Reading是固定重读已有结果，不额外检索 |
| 为什么LLM会忽略关键信息？ | 注意力分散、上下文太长、关键信息被淹没在冗余文本中 |
| Spring AI的Advisor机制是什么？ | 一种AOP式扩展点，可以在ChatClient调用链的不同阶段插入自定义逻辑 |
| 重读几次合适？ | 通常2-3次，太多会增加延迟和成本， diminishing returns |
| Prompt模板怎么设计？ | 让模型扮演检查员角色，明确列出检查清单：是否回答原问题？有无遗漏事实？有无矛盾？ |
