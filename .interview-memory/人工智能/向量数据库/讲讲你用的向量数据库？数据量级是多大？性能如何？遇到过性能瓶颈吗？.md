# 讲讲你用的向量数据库？数据量级是多大？性能如何？遇到过性能瓶颈吗？

## 原始知识

- **数据量级与选型（量+选）**
  - 十万级以下：Chroma、FAISS（内存索引），开发测试首选
  - 百万级：Qdrant、pgvector，单机足够，支持过滤
  - 千万级：Milvus 单机、Weaviate，需关注内存和索引构建
  - 亿级以上：Milvus 分布式、Pinecone、Zilliz，必须分片
  - **易混点**：不是越大越好，小数据用重型库是浪费
  - **追问点**：你们实际数据量多大？怎么扩容的？

- **索引与召回性能（索）**
  - Flat（暴力搜索）：100% 召回，数据量小（<10万）时可用
  - HNSW：图索引，查询快（毫秒级），召回率 95%+，但内存占用高
  - IVF：倒排文件索引，省内存，构建快，查询稍慢，适合大数据
  - PQ（乘积量化）：压缩向量，省内存，召回率略降，适合高维大数据
  - **易混点**：HNSW 不是万能的，数据量极大时内存会成为瓶颈
  - **追问点**：你们用的什么索引？召回率要求多少？

- **查询延迟与 QPS（延）**
  - P99 延迟：HNSW 通常 <50ms，IVF 通常 <200ms
  - QPS：单机 HNSW 可达 1000~5000，分布式可水平扩展
  - 召回率 vs 延迟 trade-off：ef_search 越大召回越高但越慢
  - **易混点**：只看平均延迟不看 P99，峰值时抖动严重
  - **追问点**：延迟抖动怎么排查？怎么保证 P99 稳定？

- **写入吞吐与索引构建（写）**
  - 批量写入比单条快 10 倍以上
  - HNSW 增量构建有锁竞争，高并发写入会阻塞
  - 大数据量初始导入时，先建 Flat 再转 HNSW 更快
  - 异步索引构建：先写数据，后台慢慢建索引
  - **易混点**：写入后立即查询可能查不到（索引未构建完）
  - **追问点**：日增量多少？怎么保证写入不丢数据？

- **混合查询瓶颈（混）**
  - 向量+标量过滤（如 metadata.category='tech'）是常见需求
  - 先过滤再向量搜索：过滤后数据太少，向量搜索不准
  - 先向量搜索再过滤：召回很多再筛，浪费计算
  - 解决方案：多索引组合、预过滤分区、混合路由
  - **易混点**：不是所有向量库都支持高效混合查询
  - **追问点**：你们怎么做的混合查询？过滤条件多吗？

- **常见性能瓶颈与优化**
  - 内存瓶颈：HNSW 需要全量内存，数据量大时单机扛不住 → 换 IVF 或分片
  - CPU 瓶颈：高并发查询时 CPU 打满 → 水平扩展、缓存热点向量
  - 网络瓶颈：跨机房调用延迟高 → 就近部署、客户端缓存
  - GC 抖动：Java/Go 实现的向量库 GC 时延迟 spike → 预热、调 GC 参数
