## 原始知识

- TCP **流式传输**无消息边界
- **粘包**：多个小包粘成一个大包
- **拆包**：一个大包拆成多个小包
- FixedLengthFrameDecoder **定长**解码
- LineBasedFrameDecoder **换行符**分隔
- DelimiterBasedFrameDecoder **自定义分隔符**
- LengthFieldBasedFrameDecoder **长度字段**解码
- 自定义协议解决

## 顺口溜

拆定长，三解码；
换行符，长度域。

## 单字锚点 + 防崩推导

- **锚点字**：拆
- **扩展口诀**：拆定长，三解码
- **为什么选它**：问题核心是"拆"包与粘包，拆字直接对应 TCP 流式数据需要被拆分
- **推导链**：
  1. 拆 → 拆包 / 粘包
  2. 拆包/粘包 → TCP 流式无边界
  3. Netty 解决 → 四种解码器
  4. 四种 → FixedLength（定长）、LineBased（换行）、Delimiter（分隔符）、LengthField（长度域）
  5. 终极方案 → 自定义协议

## 一字一画面

- **拆** —— 像快递分拣员：传送带上包裹连在一起，他用刀按固定间距切开（本质：按规则把流式数据切成独立消息）
- **定** —— 像食堂标准餐盒：每盒装 250 克，多了切掉少了补零（本质：FixedLengthFrameDecoder 固定长度切分）
- **长** —— 像卷尺量布：每 1 米剪一刀，不管花色怎么变（本质：以长度为唯一标准，不关心内容）
- **解** —— 像摩斯电码翻译员：听到滴滴答答按约定规则翻译成字母（本质：解码器按协议还原消息）

## 面试话术

- "粘包拆包本质是 TCP 流式传输没有消息边界，Netty 提供了一堆解码器来解决，我记成'拆定长，三解码'。"
- "固定长度、换行符分隔、自定义分隔符，还有最常用的是 LengthFieldBasedFrameDecoder，在消息头里放长度字段，配合自定义协议使用。"

## 防混补丁

容易和 UDP 混淆？UDP 天然有边界不会粘包。  
**补丁口诀**：TCP 流，UDP 包，粘拆只在流传输。

## 追问预演

- **Q：为什么 TCP 会粘包？**  
  A：TCP 是流式协议，发送端攒够数据或接收端缓冲区限制会导致多包合并，内核不维护消息边界。
- **Q：生产环境用哪种最多？**  
  A：LengthFieldBasedFrameDecoder 最灵活，配合魔数+版本+长度+payload+校验和的自定义协议。
- **Q：自定义协议怎么设计？**  
  A：魔数 + 版本号 + 序列化方式 + 长度字段 + payload + 校验和。
