# 如何合理地设置 Java 线程池的线程数？

## 速记卡片
`核 | 核IO，两公式 | CPU加一，IO翻倍`

## 原始知识
- **CPU密集型**：线程数 = CPU核数 + 1
- **IO密集型**：线程数 = CPU核数 × 2（或更大）
- **混合型**：拆分为两个线程池
- **压测调整**：实际值根据压测结果调优
- **上下文切换**：线程过多反而降低性能
- **内存限制**：线程栈默认1M，线程多耗内存
- **网络/DB限制**：连接池、带宽也是瓶颈
- **公式细化**：N × (1 + WT/ST)，WT等待时间，ST计算时间

## 顺口溜
**核IO，两公式，压测定**

## 单字锚点 + 防崩推导
- **锚点字**：**核**
- **扩展口诀**：**核IO，两公式**
- **为什么选它**：线程数设置的核心依据是"CPU核数"，无论哪种类型都从核数出发。
- **防崩推导**：
  - 核 → CPU核心数 → 线程数设置的基准
  - 两种类型 → CPU密集型（计算多）+ IO密集型（等待多）
  - CPU密集型公式 → 核数 + 1（减少上下文切换）
  - IO密集型公式 → 核数 × 2 或更大（利用等待时间）
  - 更细公式 → N × (1 + WT/ST)
  - 混合任务 → 拆两个池，各配各的
  - 最终 → 压测调整，考虑内存/网络/数据库限制

## 一字一画面
- **核** —— 像**厨房灶台**：4个灶眼（4核），最多同时炒4道菜。（本质：CPU并行上限）
- **加一** —— 像**4个灶+1个备菜台**：4个厨师炒菜，1个在旁边切配，不抢灶台。（本质：CPU密集型+1，应对偶发线程停顿）
- **翻倍** —— 像**4个灶配8个厨师**：炒菜5分钟，等汤10分钟，8人轮流转，灶台不闲。（本质：IO密集型利用等待时间）
- **拆** —— 像**中餐厨房分冷热档**：热菜组4人，凉菜组2人，互不耽误。（本质：混合型任务拆分线程池）
- **压** —— 像**试营业调人手**：先按公式招人，开业后看排队长度再增减。（本质：压测调整）

## 面试话术
- "线程数设置先看任务类型，口诀是**核IO，两公式**。CPU密集型，线程数一般设成CPU核数加1，减少上下文切换；IO密集型，设成核数乘以2或更大，因为线程等IO时CPU空着。更精确的可以用公式N × (1 + 等待时间/计算时间)。混合任务建议拆成两个线程池。最后一定要压测，还要看内存、数据库连接池这些瓶颈。"

## 防混补丁
- 容易把CPU密集型和IO密集型公式搞反。补丁：**CPU忙就少人，IO等就多人**。
- 容易只看公式不看实际瓶颈。补丁：**公式是起点，压测是终点**。

## 追问预演
- **问：为什么CPU密集型是核数+1而不是核数？**
  - 答：+1是为了应对某个线程因缺页中断、GC等短暂停顿时，多一个线程能顶上，不浪费CPU周期。
- **问：IO密集型为什么可以设很大？**
  - 答：IO等待时不占CPU，大线程数能让CPU总有可执行的线程，提高利用率。但太大上下文切换开销会吃光收益。
- **问：公式N × (1 + WT/ST)怎么理解？**
  - 答：N是核数，WT是IO等待时间，ST是CPU计算时间。比例越大说明越IO密集，线程数应越多。
- **问：线上怎么动态调整线程池参数？**
  - 答：可以封装动态线程池，通过配置中心或管理后台实时修改corePoolSize、maximumPoolSize等，结合监控指标如队列堆积量、活跃线程数来判断。
