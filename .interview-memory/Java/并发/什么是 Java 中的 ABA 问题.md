# 什么是 Java 中的 ABA 问题？

## 速记卡片
`版 | 版戳改，三解法 | CAS漏洞`

## 原始知识
- ABA是**CAS的漏洞**
- A→B→A，**值同实际已改**
- **AtomicStampedReference**，版本号解决
- **AtomicMarkableReference**，布尔标记解决
- 链表/栈操作易中招
- 解决靠**版本戳或标记**
- 业务层避免对象复用

## 顺口溜
**版戳改，三解法**

## 单字锚点 + 防崩推导
- **锚点字**：**版**
- **扩展口诀**：**版戳改，三解法**
- **为什么选它**："版"即版本号，ABA问题的根因是"值一样但中间被改过"，解药就是给值加版本号。
- **防崩推导**：
  - 版 → 版本号 → AtomicStampedReference → 戳记（stamp）→ 每次修改+1
  - A(1)→B(2)→A(3)，值同版不同 → CAS比较值+版本 → 发现被改 → 更新失败
  - 改 → ABA本质是值被改了两遍 → 中间态丢失
  - 三解法 → ①版本号（Stamped）②布尔标记（Markable）③业务规避（不复用对象）

## 一字一画面
- **版** —— 像**图书版本号**：同一本书第一版和第三版封面一样（都是A），但内容被改过两次，不是同一本书了。（本质：值相同≠状态相同）
- **戳** —— 像**红色邮戳日期**：信封上盖邮戳，每次经过邮局盖一个新日期，哪怕回到同一个邮局，日期也不同了。（本质：版本号标记变更次数，戳不同即状态不同）
- **改** —— 像**变色龙变绿变红再变绿**：颜色回到绿色，但中间经历过红色，不是原来的那条龙了。（本质：ABA中间态丢失，CAS只比较终态）

## 面试话术
- "ABA问题是CAS的经典漏洞。口诀是**版戳改，三解法**。ABA就是A变成B又变回A，CAS比较值发现没变就更新，但实际上中间被篡改过。解决方法是加版本号，比如AtomicStampedReference，每次修改版本号+1，值一样版本不一样也能发现。"
- "在实际业务里，链表或栈的CAS操作最容易中招，比如节点被删了又插回来，引用相同但对象不同了，会导致空指针或者数据不一致。"

## 防混补丁
- 容易和"乐观锁/悲观锁"搞混。ABA是CAS的特有问题，不是锁的问题。
- 补丁：**CAS的漏洞，锁不背锅**

## 追问预演
1. **AtomicStampedReference和AtomicMarkableReference有什么区别？**
   - Stamped是整数版本号，可以记录修改次数，适合完整ABA检测；Markable是布尔标记，只能记录"是否被修改过一次"，适合只需要知道"干净/脏"的场景。

2. **实际业务中ABA有什么影响？**
   - 链表或栈的CAS操作，如果节点被删除又重新插入，值相同但引用对象不同，会导致空指针或数据不一致。比如栈顶节点被另一个线程替换了内容，但引用没变，当前线程继续操作就会崩溃。

3. **除了版本号还有其他解决方式吗？**
   - 业务上避免复用对象（比如每次new新节点）、使用更强一致性保证如synchronized/Lock，或者Java 9引入的VarHandle配合内存屏障语义。

4. **为什么叫ABA而不是ABC？**
   - 因为首尾状态相同（都是A），中间经历了B，CAS只比较首尾值发现没变就更新了，所以叫ABA问题。如果是ABC，CAS本来就会失败，不是问题。
