## 原始知识

- **可见性**：MESI缓存一致性协议，内存屏障保证修改立即可见
- **禁止指令重排序**：happens-before规则，内存屏障
- **不保证原子性**：i++仍需synchronized或AtomicInteger
- 适用场景：**状态标志位**、双重检查锁DCL中的单例
- volatile vs synchronized：一个轻量可见，一个重量互斥

## 单字锚点 + 防崩推导

**锚点字：见**

- **扩展口诀**：见（可见性）禁（禁止重排）序（顺序），三作用
- **为什么选它**：volatile最核心的作用就是"可见"——一个线程改了，其他线程立刻"看见"，这是它存在的根本原因
- **推导链**：
  - 见 → **可见性** → 线程A修改volatile变量，线程B立即可见（MESI协议 + 内存屏障刷主存）
  - 禁 → **禁止指令重排序** → happens-before规则保证代码执行顺序不被编译器/CPU打乱
  - 序 → 顺序 → 注意：只保证**单线程内**顺序，不保证多线程操作的原子组合
  - 三作用 → 可见、禁排、但不原子
  - 不原子 → **i++不行**（三步：读、改、写），需要 **synchronized/AtomicInteger**
  - 场景 → **状态标志位**（如 boolean running = true/false）、**DCL单例**（配合synchronized用）
  - vs synchronized → volatile轻量（只保证可见+有序），synchronized重量（保证可见+有序+**原子+互斥**）

## 一字一画面

**见** —— 像[灯塔光芒海上可见]：[线程A在黑夜里修改了灯塔颜色，volatile就像一道强光瞬间扫过海面，所有线程B/C/D的望远镜里立刻看到新颜色，没有延迟，没有盲区]（本质：缓存一致性+内存屏障保证修改对所有线程立即可见）

**禁** —— 像[交警指挥禁止变道]：[CPU和编译器像跃跃欲试的司机，总想变道超车（指令重排），volatile的内存屏障像路口的交警，举着禁止牌：此路必须按顺序走]（本质：内存屏障阻止编译器和CPU对volatile前后指令进行重排序）

**序** —— 像[排队买票不插队]：[代码指令像排队的人，volatile确保前面的人买完票，后面的人才能上前。但注意：每个人都只买一张票（单操作），不能包下整叠票（复合操作不原子）]（本质：保证有序性，但不保证复合操作的原子性）

## 面试话术

> "volatile有三个作用，我用三个字概括：见、禁、序。第一是可见性，一个线程修改了volatile变量，其他线程能立刻看到，底层靠MESI缓存一致性协议和内存屏障。第二是禁止指令重排序，保证代码执行顺序符合happens-before规则。第三是它不提供原子性，所以i++这种复合操作不行，得用synchronized或者AtomicInteger。常见适用场景有两个：一个是状态标志位，比如控制循环退出的boolean变量；另一个是双重检查锁单例模式里，给实例加volatile防止半初始化。"

## 防混补丁

**容易和synchronized搞混以为能替代锁？**  
补丁口诀：**见禁序，不锁门** —— volatile只管看得见、不乱排，不管互斥不互斥，复合操作必翻车。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 为什么i++不是原子的？ | i++分三步：读值、+1、写回。三步之间可能被其他线程打断 |
| DCL为什么要加volatile？ | 防止指令重排序导致的半初始化对象（对象引用先赋值，构造方法后执行） |
| MESI协议是什么？ | CPU缓存一致性协议：Modified/Exclusive/Shared/Invalid四种状态 |
| volatile的内存屏障具体怎么插？ | 写操作前StoreStore屏障，后StoreLoad屏障；读操作后LoadLoad+LoadStore屏障 |
| volatile能保证64位long的原子性吗？ | 在64位JVM上可以，因为64位系统对long/double的读写天然原子；但32位JVM分两次读写，volatile保证原子可见 |
