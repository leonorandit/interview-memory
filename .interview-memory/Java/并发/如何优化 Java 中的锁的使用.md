## 原始知识

- **减小锁粒度**（分段锁 / ConcurrentHashMap）
- **降低持有时间**（缩小 synchronized 块）
- **读写分离**（ReadWriteLock / StampedLock）
- 利用 JDK6 **锁升级** 优化
- **无锁编程**（CAS / Atomic 类 / Volatile）
- **避免死锁**（顺序加锁 / 超时释放）
- **锁消除**（JVM 编译时优化）

## 单字锚点 + 防崩推导

- **锚点字**：细
- **扩展口诀**：细分读，五优化
- **为什么选它**：优化锁的第一直觉就是减小粒度，从"粗"到"细"是核心思维
- **推导链**：细 → 细粒度锁 → 分段（ConcurrentHashMap） → 分解（缩小同步块） → 读（读写分离） → 五优化（粒度 / 时间 / 读写 / 无锁 / 死锁）

## 一字一画面

**细** —— 像[手术刀切豆腐]：[白瓷盘里嫩豆腐被银色手术刀切成整齐小块，每块独立不粘连，筷子夹哪块都行]（本质：细粒度，把大锁切成小锁）

## 面试话术

"锁优化第一条就是减小粒度，像 ConcurrentHashMap 分段锁那样。第二条是缩短持有时间，别把 IO 操作放同步块里。读多写少用 ReadWriteLock 或 StampedLock，竞争不激烈直接用 CAS 无锁。还要注意加锁顺序一致，防止死锁。"

## 防混补丁

容易和 "锁升级" 搞混？一句补丁：**锁升级是 JVM 自动变，锁优化是程序员主动降。**

## 追问预演

**Q: StampedLock 和 ReadWriteLock 区别？**  
A: StampedLock 支持乐观读，性能更好但使用复杂。

**Q: 什么时候用无锁 CAS？**  
A: 低竞争、简单计数或状态切换场景。

**Q: 锁粗化是什么场景？**  
A: 连续多次加锁解锁，JVM 合并成一个大锁，减少开销。
