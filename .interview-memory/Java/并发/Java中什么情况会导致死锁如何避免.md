# Java 中什么情况会导致死锁？如何避免？

## 速记卡片
`环 | 环互请，四条件 | 四缺一不死锁`

## 原始知识
- **互斥**：资源独占，一次一个线程
- **请求保持**：占着碗里的看着锅里的
- **不可剥夺**：抢不走，只能自己放
- **循环等待**：A等B，B等C，C等A
- **破坏互斥**：用共享资源（难）
- **破坏请求保持**：一次性申请全部资源
- **破坏不可剥夺**：设置超时，tryLock
- **破坏循环等待**：按固定顺序加锁
- **jstack**：排查死锁利器
- **银行家算法**：预分配检查安全序列

## 顺口溜
**环互请，四条件，破环序超时**

## 单字锚点 + 防崩推导
- **锚点字**：**环**
- **扩展口诀**：**环互请，四条件**
- **为什么选它**：死锁的终极形态是"循环等待"，像一条咬尾蛇，画面最强。
- **防崩推导**：
  - 环 → 循环等待 → 死锁四条件之一
  - 四条件还有谁？环（循环等待）→ 互（互斥）→ 请（请求保持）→ 不可剥夺
  - 怎么避免？破坏四条件 → 最常用的是破"环"（顺序加锁）+ 破"不可剥夺"（tryLock超时）
  - 排查工具 → jstack 看线程状态 BLOCKED
  - 理论算法 → 银行家算法（预分配检查安全序列）

## 一字一画面
- **环** —— 像**三条蛇咬尾**：红蛇咬住蓝蛇尾巴，蓝蛇咬住绿蛇，绿蛇又咬住红蛇，谁都不能松口。（本质：循环等待）
- **互** —— 像**独享马桶**：一次只能一个人用，不能两人挤。（本质：互斥条件）
- **请** —— 像**端着盘子要汤**：左手端着盛好的菜（占着资源），右手伸出去要汤（请求新资源）。（本质：请求并保持）
- **破** —— 像**统一编号叫号**：大家都按1号柜→2号柜→3号柜的顺序排队，永远形成不了环。（本质：顺序加锁破坏循环等待）
- **超** —— 像**限时占座**：座位上放手机定时，到点没回来就清走。（本质：tryLock超时释放）

## 面试话术
- "死锁产生需要**四个条件同时满足**，口诀是**环互请**——循环等待、互斥、请求保持、不可剥夺。避免的话，最实际的是**破环序超时**：按固定顺序加锁破坏循环等待，或者用tryLock设置超时破坏不可剥夺。排查用jstack看BLOCKED线程，理论上还有银行家算法。"

## 防混补丁
- 容易和"活锁"搞混。死锁是大家都卡着不动，活锁是大家互相礼让却永远让不完。补丁：**死锁卡死，活锁乱动**。
- 容易和"饥饿"搞混。饥饿是某个线程一直得不到执行机会。补丁：**死锁互卡，饥饿被冷落**。

## 追问预演
- **问：死锁和活锁的区别？**
  - 答：死锁是互相等待资源，状态不变；活锁是线程不断改变状态响应对方，但永远无法继续。
- **问：线上如何排查死锁？**
  - 答：jstack -l pid 打印线程堆栈，搜索"Found one Java-level deadlock"；或者Arthas的thread -b。
- **问：tryLock怎么避免死锁？**
  - 答：设置超时时间，获取不到就释放已持有的锁，过段时间再重试，打破"不可剥夺"条件。
- **问：银行家算法了解吗？**
  - 答：模拟预分配资源，检查是否还存在安全序列，如果存在才分配，属于死锁避免策略。
