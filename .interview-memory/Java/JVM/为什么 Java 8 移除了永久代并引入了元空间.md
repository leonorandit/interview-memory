# 为什么 Java 8 移除了永久代（PermGen）并引入了元空间（Metaspace）？

## 速记卡片
`元 | 元本限，三原因 | OOM、自动扩容、GC优化`

## 原始知识
- **永久代固定大小**：`-XX:MaxPermSize` 设死上限，**容易 OOM**
- **元空间用本地内存**：不在 JVM 堆里，**自动扩容**，默认只受物理内存限制
- **类信息迁移**：类元数据、常量池、方法元数据从**堆内**移到**本地内存**
- **减少 Full GC 频率**：永久代 GC 效率低，元空间独立后**降低 Full GC 压力**
- **字符串常量池移回堆**：JDK 7 已先移回堆，JDK 8 彻底清理 PermGen
- **易混点**：元空间不是无限大，`-XX:MaxMetaspaceSize` 仍可设上限
- **追问点**：元空间也会 OOM 吗？类加载器泄漏怎么排查？

## 顺口溜
**元本限，三原因**

## 单字锚点 + 防崩推导
- **锚点字**：**元**
- **扩展口诀**：**元本限，三原因**
- **为什么选它**：问题是"为什么引入元空间"，"元"字直接对应答案主体 Metaspace，信息密度最高。
- **防崩推导**：
  - 元 → 元空间（Metaspace）→ 为什么引入？→ 三个原因
  - 本 → **本地内存** → 不在 JVM 堆，用 Native Memory，可自动扩容
  - 限 → **永久代受限** → PermGen 固定大小 `-XX:MaxPermSize`，类多就 OOM
  - 三原因 → 限（固定大小易 OOM）+ 本（本地内存自动扩容）+ GC 优化（减少 Full GC）
  - 额外点 → 字符串常量池已在 JDK 7 移回堆；类元数据、方法元数据随 JDK 8 进元空间

## 一字一画面
- **元** —— 像**小区拆掉固定车位改露天停车场**：原来地下车库（PermGen）只有 100 个固定车位，车多就停不下；现在露天（Metaspace）随地停，只要地面够大就能扩。（本地内存自动扩容）
- **本** —— 像**把仓库从楼里搬到楼外空地**：楼里（JVM 堆）面积紧张，楼外空地（Native Memory）宽敞得多。（脱离堆内存）
- **限** —— 像**100 人的电梯限重报警**：上来第 101 个人就哔哔响（OOM），只能硬挤或赶走一个。（永久代固定上限）
- **GC** —— 像**小区保洁只扫楼里不扫停车场**：停车场（元空间）自己管理，楼内（堆）保洁工作量少了，垃圾车（Full GC）不用老来。（减少 Full GC 频率）

## 面试话术
- "Java 8 移除永久代引入元空间，核心原因可以总结为**元本限，三原因**。第一，**限**——永久代大小是固定的，设 `-XX:MaxPermSize`，类加载多或者动态代理多的时候非常容易 OOM；第二，**本**——元空间使用本地内存（Native Memory），脱离 JVM 堆，可以自动扩容；第三，元空间独立后**减少了 Full GC 的频率**，因为永久代的 GC 效率本来就比较低。"
- "另外，类元数据、方法元数据都移到了元空间，字符串常量池在 JDK 7 就已经先移回堆了。当然元空间也不是绝对无限大，可以通过 `-XX:MaxMetaspaceSize` 设上限，一般线上建议配一个合理值防止无限增长。"

## 防混补丁
- 容易误以为元空间完全不会 OOM。补丁：**元空间默认受物理内存限制，但配了 MaxMetaspaceSize 或发生类加载器泄漏，照样会 OOM**。

## 追问预演
1. **问**：元空间也会 OOM 吗？怎么排查？
   **答**：会。主要是类加载器泄漏（如 OSGi、热部署场景），用 `jstat -class` 看加载类数量，结合 MAT/VisualVM 分析 ClassLoader 引用链。
2. **问**：字符串常量池为什么移回堆？
   **答**：PermGen 大小固定，字符串常量池如果太大（比如 `intern` 滥用）会导致 OOM；移回堆后由常规 GC 管理，且能享受堆的自动扩容。
3. **问**：-XX:MaxMetaspaceSize 不设置会怎样？
   **答**：默认不限制，理论上用光物理内存才会 OOM；但线上建议设置，防止异常场景下无限增长拖垮机器。
4. **问**：JVM 内存结构 JDK 8 还有哪些变化？
   **答**：永久代彻底移除；元空间登场；字符串常量池已在 JDK 7 移入堆；堆内存划分（新生代/老年代）逻辑不变。
