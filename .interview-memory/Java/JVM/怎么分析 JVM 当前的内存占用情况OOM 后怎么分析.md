## 单字锚点 + 防崩推导

**锚点字：映**

- **扩展口诀**：映（内存映像）堆（堆转储）线（引用链），四分析
- **为什么选它**：OOM分析的核心就是"内存映像"（heap dump），像拍照定格案发现场，一切分析都围绕这张快照
- **推导链**：
  - 映 → 内存映像 → **实时分析**先铺垫：jstat -gcutil 看Eden/Survivor/Old各区使用率
  - 映细查 → jmap -histo 看对象**直方图**，谁占最多内存一目了然
  - 映定格 → jmap -dump 生成**堆转储文件**，或用 `-XX:+HeapDumpOnOutOfMemoryError` 让OOM时**自动生成**
  - 堆 → 用 **MAT/VisualVM** 打开dump文件
  - 线 → MAT里看 **Dominator Tree** 找最大对象 → **引用链**追溯谁持有它 → 找到**GC Roots**确认泄漏路径
  - 四分析 → 实时分析、工具分析、OOM后分析、原因归类
  - 原因 → **内存泄漏**（长生命对象持有短生命对象）、**大对象**（一次性分配过大）、**集合无限增长**（List/Map无界添加）

## 一字一画面

**映** —— 像[相机快门定格车祸现场]：[OOM发生瞬间，相机咔嚓一声拍下整个堆内存的照片，每个对象都像车辆停在原地，凶手车型的车牌号（对象类型和大小）清清楚楚]（本质：heap dump是内存快照，MAT分析定格现场）

**堆** —— 像[仓库盘点货架]：[你把堆内存的货架一层层拉开，jmap -histo像扫码枪，哔哔哔扫出每种货物的数量体积；MAT的Dominator Tree像按体积排序的榜单，最大的箱子排在最上面]（本质：通过直方图和支配树定位大对象）

**线** —— 像[抽丝剥茧找线头]：[一根线从泄漏对象出发，穿过一个个引用节点，最后缠在GC Roots上，就像毛线团找到线头，一拉就知道谁是始作俑者]（本质：引用链追溯从对象到GC Roots的持有路径）

## 面试话术

> "JVM内存分析分实时和事后两种。实时的话，我用 jstat -gcutil 看各代使用率，jmap -histo 看对象分布。如果需要深入，就用 jmap -dump 生成堆转储，然后MAT打开分析。OOM后的分析，关键是提前配置 -XX:+HeapDumpOnOutOfMemoryError，让JVM自动留现场。拿到dump后，我用MAT的Dominator Tree找占用最大的对象，再通过引用链追溯到GC Roots，定位泄漏源。常见OOM原因就三类：内存泄漏、大对象、集合无限增长。"

## 防混补丁

**容易和CPU飙高排查搞混？**  
补丁口诀：**CPU盯线程，内存盯对象** —— CPU问题看线程栈和热点方法，内存问题看堆dump和对象引用链。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| jmap -dump 对线上有影响吗？ | 有，会STW，大堆可能卡顿数秒甚至分钟，建议用gcore + jhsdb，或arthas的heapdump |
| Dominator Tree是什么？ | 支配树，展示对象的保留集（retained heap），找出如果一个对象被回收能释放多少内存 |
| 怎么判断是内存泄漏还是内存溢出？ | 泄漏：对象不再使用却被持有，持续增长；溢出：正常使用的对象确实太多 |
| GC Roots有哪些？ | 虚拟机栈局部变量、方法区静态属性、常量、JNI引用、Clazz对象等 |

## 关联知识点

- GC算法与垃圾回收器（CMS、G1、ZGC）
- 内存泄漏常见场景（ThreadLocal、静态集合、监听器未注销）
- MAT高级功能（Path to GC Roots、Leak Suspects）
