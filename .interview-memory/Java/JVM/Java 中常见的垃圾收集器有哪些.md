## 原始知识

- **Serial**：单线程、STW、客户端模式
- **Parallel**：多线程、STW、吞吐量优先、Server默认
- **CMS**：低延迟、并发标记清除、碎片多、**已废弃**
- **G1**：分区回收、可预测停顿、平衡吞吐和延迟、**JDK9+默认**
- **ZGC**：超低延迟、STW<10ms、并发整理、**JDK11+**
- **Shenandoah**：与ZGC类似、OpenJDK
- **Epsilon**：测试用、不回收

## 单字锚点 + 防崩推导

**锚点字：串**

- **扩展口诀**：串并CMSG，五收集
- **为什么选它**：Serial是JDK最早、最经典的垃圾收集器，"串"字自带"一个接一个"的画面感，天然对应单线程STW。
- **推导链**：
  1. 串 = Serial = 单线程 + STW（串行排队做垃圾清理）
  2. 并 = Parallel = 多线程 + STW（并行一起干，吞吐量优先）
  3. C = CMS = Concurrent Mark Sweep = 并发低延迟（老年代，已废弃）
  4. M = 没有M，这里把G和Z拆开：G = G1 = 分区回收，Z = ZGC = 超低延迟
  5. 五收集 = Serial、Parallel、CMS、G1、ZGC（主流五个）
  6. 补充：Shenandoah（类似ZGC）、Epsilon（测试不回收）

## 一字一画面

**串** —— 像[一串红灯笼逐个亮]：一个人拿着扫帚逐个清理房间，其他人都被拦在门外（STW），清理完一个再下一个（本质：单线程串行回收）。

**并** —— 像[五把扫帚同时扫大街]：多个清洁工一起上阵，虽然 still 封路（STW），但扫得更快，适合车多的主干道（本质：多线程并行追求吞吐量）。

**C** —— 像[边通车边捡垃圾]：路上不封，清洁工和车辆并行，但捡完路面坑坑洼洼（碎片多），且这个方案已被叫停（本质：并发低延迟，标记-清除有碎片）。

**G** —— 像[把城市分成网格块]：每块区域独立打分，优先清理垃圾最多的块，可预测多久扫完一条街（本质：Region分区，可预测停顿时间）。

**Z** —— 像[吸尘器边吸边整理]：几乎不封路，停顿时间极短（<10ms），且整理碎片（本质：并发整理，超低延迟）。

## 面试话术

"Java常见的垃圾收集器可以记成'串并CMSG'。Serial是单线程STW，Parallel是多线程STW吞吐量优先，CMS是并发低延迟但碎片多已废弃，G1是JDK9以后默认的分区回收器，ZGC是JDK11+的超低延迟收集器，STW能控制在10毫秒以内。"

## 追问预演

1. **G1和CMS的区别？**
   G1是分区回收（Region），CMS是整个老年代；G1可预测停顿时间，CMS不能；G1是标记-整理+复制，CMS是标记-清除有碎片。

2. **ZGC和Shenandoah的区别？**
   两者目标都是超低延迟，ZGC是Oracle JDK，Shenandoah是OpenJDK，实现细节不同但思路类似。

3. **JDK8默认收集器是什么？**
   Parallel Scavenge（新生代）+ Parallel Old（老年代）。

4. **JDK9+默认收集器是什么？**
   G1。
