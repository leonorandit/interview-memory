## 原始知识

- **低延迟**（减少**GC停顿时间STW**，适合交互型应用/**CMS**/**G1**/**ZGC**）
- **高吞吐量**（提高应用运行时间占比，适合批处理/**Parallel GC**）
- **内存占用**（降低**Footprint**）
- **调优手段**（调整堆大小/选择合适的GC收集器/调整GC参数/分析GC日志）

## 单字锚点 + 防崩推导

**锚点字**：停

**扩展口诀**：停吞吐，两目标

**为什么选它**：GC调优最痛的感受就是"停顿"（Stop-The-World），用户直观体验到的是卡顿，"停"字直接指向STW，是调优的核心痛点。

**推导链**：

```
停（Stop-The-World 停顿）
  ↓
吞（吞吐量）—— 吞吐 = 应用运行时间 / 总时间
  ├─ 吞吐高 = GC时间短 = Parallel GC擅长
  └─ 吞吐低 = GC拖后腿
  ↓
吐（延迟）—— 停顿长则延迟高
  ├─ 低延迟 = STW短 = CMS/G1/ZGC擅长
  └─ 高延迟 = 用户卡顿明显
  ↓
两目标（核心矛盾）：
  ├─ ① 低延迟（交互型/金融/游戏）
  └─ ② 高吞吐（批处理/大数据/离线计算）
  └─ ③ 内存占用（降低Footprint，云原生场景）
```

## 一字一画面

- **停** —— 像红灯亮起：整条马路瞬间静止，车辆行人全部冻住（本质：STW期间所有业务线程暂停）
- **吞** —— 像狼吞虎咽：吃得快，消化也快，整体效率高（本质：吞吐高意味着花在GC上的时间比例小）
- **吐** —— 像细嚼慢咽：每一口都短，但顿数多，单次不卡（本质：低延迟追求单次GC停顿极短）
- **Z** —— 像幽灵般透明：清理垃圾时人还在正常走动，感觉不到（本质：ZGC目标毫秒级停顿，近乎无感）

## 面试话术

> "JVM GC调优主要看两个目标：低延迟和高吞吐量，它们通常是互斥的。如果做的是Web应用或者交易系统，优先考虑低延迟，选G1或者ZGC；如果是批处理或大数据计算，优先考虑吞吐量，选Parallel GC。"

> "调优步骤一般是：先设合理的堆大小，然后选合适的收集器，再通过GC日志分析实际停顿时间和频率，最后调整比如新生代老年代比例、并发线程数这些参数。"

## 防混补丁

**容易和什么搞混**：低延迟 vs 高吞吐。低延迟是单次GC短，高吞吐是GC总占比低。低延迟可能GC次数多反而总时间多。

**补丁口诀**：停顿短像闪电快，吞吐高像马拉松；闪电频繁也耗电，马拉松慢但跑得远。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| G1和CMS有什么区别？ | G1是Region化+标记整理，可预测停顿；CMS是标记清除，碎片多。JDK9后G1成为默认 |
| ZGC为什么能做到低延迟？ | 染色指针+读屏障+并发转移，几乎全并发，目标<10ms停顿 |
| 怎么判断该调哪个参数？ | 先看GC日志，如果Young GC频繁就调大新生代，如果Full GC频繁就排查内存泄漏或调大老年代 |
| 堆内存越大越好吗？ | 不是。堆越大GC单次停顿越长，需要在吞吐和延迟间找平衡 |
| G1的-XX:MaxGCPauseMillis靠谱吗？ | 是目标值不是保证值，G1会尽量调整Region回收数量来满足，但极端情况可能不达标 |
