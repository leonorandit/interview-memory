## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 混 | 混编解，JIT热 | 解释执行启动快运行慢，编译执行启动慢运行快，JVM先解释后JIT编译热点代码 |

## 原始知识

- **解释执行**：逐行解释字节码，**启动快**，**执行慢**
- **编译执行**：JIT将热点代码编译为**本地机器码**，**启动慢**，**执行快**
- JVM采用**混合模式**：先解释，热点代码JIT编译
- HotSpot VM：**C1**（客户端编译器，快编译次优码）和**C2**（服务端编译器，慢编译最优码）
- **分层编译**：0层解释 → 1层C1 → 2层C1 +  profiling → 3层C1全速 → 4层C2

## 顺口溜

混编解，JIT热，
解释快起慢跑别纠结。
热点代码C1C2烤，
分层编译越跑越快绝。

## 单字锚点 + 防崩推导

**锚点字：混**

- **扩展口诀**：混编解，JIT热
- **为什么选它**：JVM不是单一模式，而是"混合"两种执行方式
- **推导链**：
  1. 混 → 混合模式（Hybrid）
  2. 混了哪两种 → 解释执行 + 编译执行
  3. 解释啥特点 → 逐行翻译字节码，启动快、执行慢
  4. 编译啥特点 → JIT编译器把热点代码转成机器码，启动慢、执行快
  5. JVM怎么选 → 先解释执行，跑多了的热点代码交给JIT
  6. JIT还有分层 → C1快编译、C2深优化，0~4层逐级晋升

## 一字一画面

- **混** —— 像[鸳鸯火锅]：左边清汤（解释）咕嘟咕嘟马上能涮，右边红油（编译）慢热但越煮越香，中间隔板可切换（本质：两种模式共存，按需选择）
- **编** —— 像[预制菜打包]：厨师把热门菜品提前切好腌好封进真空袋，客人点单直接下锅，比现切快三倍（本质：提前编译，运行时直接执行机器码）
- **解** —— 像[同声传译]：译者戴着耳机逐句翻译，外宾说一句翻一句，不用等全文但语速受限（本质：逐行解释，即时响应但效率受限）
- **热** —— 像[烧红的铁块]：暗红→橘红→白炽，温度越高锻造效率越高，冷铁回炉重新加热（本质：热点代码执行次数越多，JIT优化程度越深）

## 面试话术

"解释执行是逐行翻译字节码，好处是启动快，坏处是跑得慢。编译执行是JIT把热点代码编译成本地机器码，启动有编译开销，但执行速度接近原生。JVM用的是混合模式，一开始解释执行，等代码跑热了再用JIT编译优化。HotSpot里还有C1和C2两层编译器，分层编译让代码越跑越快。"

## 防混补丁

容易和"Java是解释型语言"搞混？一句补丁：
> Java先解释后编译，跑热了就是编译型，冷启动才是解释型。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| JIT是什么的缩写？ | Just-In-Time，即时编译 |
| 热点代码怎么判定？ | 方法调用计数器 + 回边计数器，超过阈值触发编译 |
| C1和C2有什么区别？ | C1编译快优化浅，C2编译慢优化深（激进优化） |
| 分层编译有几层？ | 5层：0解释，1~3 C1逐级加 profiling，4 C2 |
| 为什么不一上来就编译？ | 编译有开销，不是所有代码都值得编译，冷代码解释更省 |
| AOT和JIT什么关系？ | AOT是运行前编译（如GraalVM native-image），JIT是运行时编译 |

## 关联知识点

- JVM执行引擎（Interpreter + JIT Compiler）
- HotSpot JIT编译器（C1 / C2 / Graal）
- 分层编译（Tiered Compilation）
- 热点探测（方法调用计数器、回边计数器）
- 逃逸分析、栈上分配、标量替换
- AOT编译（Ahead-Of-Time）
