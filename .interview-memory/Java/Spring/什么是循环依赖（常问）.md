# 什么是循环依赖（常问）？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 圈 | 圈构三，两解法 | 构造器抛异常，Setter三级缓 |

## 原始知识

- **定义**：两个或多个Bean相互引用形成闭环（A依赖B，B依赖A）
- **构造器注入**：无法解决循环依赖，抛出BeanCurrentlyInCreationException
- **Setter注入**：可通过三级缓存解决
- **三级缓存**：singletonObjects/earlySingletonObjects/singletonFactories
- **@Lazy**：延迟加载打破循环
- **@DependsOn**：控制加载顺序

## 顺口溜

圈构三，两解法，
构造抛异Setter化。
三级缓存提前曝，
Lazy懒加载破它。

## 单字锚点 + 防崩推导

**锚点字**：圈

**扩展口诀**：圈构三，两解法

**为什么选它**：循环依赖的本质是"循环"，一个圈把Bean们套在一起，"圈"字最形象。

**推导链**：
1. 圈 → 循环依赖，A→B→A形成一个圈
2. 构 → 构造器注入，创建对象时就要传参，必然死循环
3. 三 → 三级缓存解决Setter循环依赖
4. 两解法 → ①三级缓存（Setter注入） ②@Lazy延迟加载
5. 三级缓存详解 → singletonObjects（成品池）→ earlySingletonObjects（半成品池）→ singletonFactories（工厂池）
6. 为什么Setter能解决 → 先实例化（反射new出来，不管属性），再注入属性。实例化完就能放进三级缓存，B创建时从缓存拿A的引用。
7. 为什么构造器不能 → 构造器参数在实例化时必须传入，此时A还没new完，B又要A，死锁。

## 一字一画面

**圈** —— 像**两条红色贪吃蛇首尾互咬**：蛇A咬蛇B尾巴，蛇B咬蛇A尾巴，谁也不松口，谁也动不了（本质：对象创建时的相互等待导致死锁）

**三** —— 像**三层透明抽屉从上到下**：最上层抽屉（singletonObjects）放组装好的成品玩具；中间抽屉（earlySingletonObjects）放缺胳膊的半成品；最下层抽屉（singletonFactories）放玩具模具（ObjectFactory）。B需要A时，先从最上层找，没有就拉开中间层拿半成品（本质：通过提前暴露半成品对象的引用来打破循环）

## 面试话术

"循环依赖就是两个Bean互相引用，比如A依赖B，B又依赖A，形成一个圈。Spring里构造器注入是解决不了循环依赖的，因为创建A的时候需要B，创建B的时候又需要A，直接抛异常。但Setter注入可以解决，原理是三级缓存：对象先实例化出来放进缓存，再去填充属性，这样B创建的时候能从缓存拿到A的半成品引用。另外也可以用@Lazy延迟加载或者@DependsOn来控制顺序。"

## 防混补丁

容易和死锁搞混？一句补丁："死锁是线程互等，循环依赖是对象互等。"

## 追问预演

**Q：三级缓存分别是什么？**
A：singletonObjects（一级，成品）、earlySingletonObjects（二级，半成品）、singletonFactories（三级，工厂）。

**Q：为什么用三级而不是两级？**
A：三级缓存主要是为了生成代理对象，如果不需要AOP，两级也能工作。

**Q：原型模式Prototype有循环依赖吗？**
A：Spring不处理Prototype的循环依赖，直接抛异常。

**Q：@Lazy怎么解决循环依赖？**
A：注入的是代理对象，真正使用时才初始化目标Bean，从而打破循环。

**Q：SpringBoot怎么检测循环依赖？**
A：默认允许Setter循环依赖，但可通过allowCircularReferences=false禁用。
