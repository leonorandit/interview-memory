## 速记卡片

注 | 注构字，三方式 | 构造器注入/Setter注入/字段注入，@Autowired按类型，@Resource按名称

---

## 原始知识

- **DI定义**：Dependency Injection，依赖注入，IoC的具体实现方式
- **构造器注入**：推荐方式，依赖明确，不可变，必填
- **Setter注入**：可选依赖，可重新设置，灵活
- **字段注入**：@Autowired直接注字段，简单但难以测试
- **@Resource**：按名称注入（JSR-250）
- **@Autowired**：按类型注入（Spring）

## 顺口溜

注构字，三方式，构造推荐字简单，Setter灵活可重填。

## 单字锚点 + 防崩推导

- **锚点字**：注
- **扩展口诀**：注构字，三方式
- **为什么选它**：DI=依赖注入，"注"是题目和概念的核心动作，所有方式都是"注入"的具体手段
- **推导链**：
  1. 注 → 注入 = DI = Dependency Injection
  2. DI 是 IoC（控制反转）的具体实现方式 → 原本对象自己创建依赖，现在由容器注入
  3. 构 → 构造器注入（Constructor Injection）→ Spring官方推荐
  4. 字 → 字段注入（Field Injection）→ @Autowired注在字段上
  5. Setter注入夹在中间 → 通过setter方法注入
  6. 三方式 = 构造器 + Setter + 字段
  7. 注解差异：@Autowired（Spring，按类型） vs @Resource（Java标准，按名称）

## 一字一画面

- **注** —— 像医生给病人打针输液：护士拿着针筒（容器），将药水（依赖对象）精准推入病人血管（目标对象），病人自己不用去找药（本质：依赖注入，由外部容器把依赖推送给对象，而非对象自己创建）
- **构** —— 像盖楼时钢筋水泥一起浇筑：塔吊在楼房骨架还没封顶前，就把钢筋（依赖）和混凝土一起浇筑进承重柱（构造函数），封顶后柱子里的钢筋再也拔不出来（本质：构造器注入，对象创建时一次性注入，依赖不可变，强烈推荐）
- **字** —— 像便利贴直接贴在电脑屏幕上：员工懒得整理文件，直接把黄色便利贴（@Autowired）啪地贴在显示器边框（字段）上，虽然方便但显得凌乱，别人想借用这台电脑时不知所措（本质：字段注入，写法简单但隐藏了依赖关系，不利于单元测试）
- **Setter** —— 像酒店房间按需更换用品：住客入住后，按铃叫服务员，今天送牙刷，明天加被子，后天换枕头，需求随时可以改（本质：Setter注入，对象创建后再注入，依赖可选，可重新设置）

## 面试话术

1. "DI就是依赖注入，是IoC控制反转的一种具体实现方式，意思是对象的依赖不是自己去创建，而是由Spring容器在适当的时候注入进来。"
2. "Spring里实现DI主要有三种方式：构造器注入、Setter注入和字段注入。Spring官方推荐构造器注入，因为依赖明确、对象不可变，也方便测试。"
3. "另外要注意@Autowired是按类型注入的，如果有多个同类型Bean会报错，需要配合@Qualifier；而@Resource默认是按名称注入的，这是JSR-250的标准注解。"

## 追问预演

1. **问**：为什么Spring推荐构造器注入？
   **答**：依赖明确写在构造器参数里，一目了然；对象创建后依赖不可变，保证线程安全；避免循环依赖在启动期就能暴露；不需要反射注入，对测试友好。

2. **问**：字段注入有什么缺点？
   **答**：依赖关系隐藏在字段上，外部看不到类需要什么；必须依赖Spring容器才能注入，无法直接new出来做单元测试；可能注入null如果容器没管好。

3. **问**：@Autowired和@Resource有什么区别？
   **答**：@Autowired是Spring的注解，默认按类型注入，可以用@Qualifier指定名称；@Resource是JSR-250标准注解，默认按名称注入，找不到再按类型。

4. **问**：构造器注入能解决循环依赖吗？
   **答**：构造器注入在启动时如果存在循环依赖会直接报错；字段注入和Setter注入可以通过三级缓存解决单例Bean的循环依赖。

5. **问**：一个类同时用了三种注入方式，优先级是什么？
   **答**：构造器注入优先执行，然后是Setter注入，最后是字段注入。但实际项目中建议统一用一种，避免混乱。
