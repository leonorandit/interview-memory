## 原始知识

- **数据库自增**（分库分表ID冲突/单点瓶颈）
- **UUID**（无序/太长/**不适合索引**）
- **雪花算法Snowflake**（时间戳+机器ID+序列号/**64位**/long类型/**趋势递增**）
- **Leaf**（美团方案/**双号段缓冲**/高可用）
- **数据库号段**（批量取号段/**减少DB压力**）
- **Redis自增**（原子性/**性能高**/持久化风险）

## 顺口溜

雪自增，四方案：
Snowflake最常用。

## 单字锚点 + 防崩推导

**锚点字**：雪

**扩展口诀**：雪自增，四方案

**为什么选它**：雪花算法（Snowflake）是分布式ID的行业标准方案，"雪"字既代表Snowflake，又像雪花一样每片唯一不重复，天然关联ID唯一性。

**推导链**：

```
雪（雪花算法 Snowflake）
  ↓
自（自增）—— 想到自增类方案
  ├─ 数据库自增（单点/分库冲突）
  ├─ Redis自增（原子性高/持久化风险）
  └─ 数据库号段（批量取号/减少DB压力）
  ↓
增（对比后选优）—— 雪花算法胜出
  ├─ 64位long型
  ├─ 时间戳+机器ID+序列号
  └─ 趋势递增，适合索引
  ↓
四方案（完整枚举）：
  ├─ UUID（全局唯一但无序）
  ├─ 数据库自增/号段（依赖DB）
  ├─ Redis自增（依赖缓存）
  └─ 雪花算法（推荐/无依赖/高性能）
```

## 一字一画面

- **雪** —— 像冬日飘落的六角雪花：每片形状相似却绝不重复，落地有序排列（本质：趋势递增的全局唯一ID）
- **自** —— 像自动售票机：按顺序吐票，但多台机器可能吐出同号（本质：自增方案的分库冲突隐患）
- **增** —— 像楼梯逐级上升：一步比一步高，有序可循（本质：趋势递增利于B+树索引）
- **U** —— 像撒了一把碎纸片：满地都是，找不到规律，收拾费劲（本质：UUID无序且过长）

## 面试话术

> "分布式ID我首选雪花算法，它是64位long型，由时间戳、机器ID和序列号组成，趋势递增适合数据库索引，单机每秒能生成几十万个ID，性能很高。"

> "如果公司已有美团的Leaf中间件，也可以用Leaf的双号段缓冲方案，依赖数据库但做了高可用优化。UUID虽然唯一但不适合做主键索引，数据库自增在分库分表下会冲突。"

## 防混补丁

**容易和什么搞混**：雪花算法 vs 数据库自增。雪花是分布式无依赖，数据库自增是单点有瓶颈。

**补丁口诀**：雪花飘落不依赖天，自增发芽全靠根；分家之后根难共享，雪花遍地各自飞。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 雪花算法时钟回拨怎么办？ | 等待时钟追上/抛出异常/备用的序列号位扩展/美团Leaf采用Zookeeper校准 |
| 机器ID怎么分配？ | 手动配置/Zookeeper/etcd动态分配，最多1024个节点（10位） |
| 为什么趋势递增比完全递增好？ | 完全递增需要全局协调，趋势递增允许微小乱序，性能更高且满足索引友好 |
| Leaf的双号段是什么？ | 当前号段用完时异步加载下一号段，避免DB取号阻塞发号 |
| Redis自增有什么风险？ | 持久化失败可能丢号，需要RDB+AOF或集群模式保证可靠性 |

## 关联知识点

- Snowflake算法位分配（41位时间+10位机器+12位序列）
- 数据库分库分表ID冲突
- B+树索引与有序ID
- 分布式一致性（时钟同步）
- Leaf/滴滴TinyID等开源方案
- UUID版本差异（v1/v4）
