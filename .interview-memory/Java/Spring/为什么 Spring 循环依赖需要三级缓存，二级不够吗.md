## 原始知识

- **一级缓存**：成品对象（完全初始化好）
- **二级缓存**：提前暴露的**原始对象**
- **三级缓存**：ObjectFactory，可生成**代理对象**
- 三级缓存本质是处理**AOP代理对象**的循环依赖
- 只有二级缓存：AOP代理对象提前暴露时还没被代理，注入的是**原始对象**而非代理对象
- 第三级缓存的**lambda延迟创建**代理对象

## 顺口溜

代曝原，三原因，
一级成品二级裸，
三级工厂造代理，
没它注入就报错。

## 单字锚点 + 防崩推导

**锚点字：代**

- **扩展口诀**：代曝原，三原因
- **为什么选它**：循环依赖问题的核心卡点在于"AOP代理"——代理对象（代）和原始对象不是同一个，提前暴露时必须用工厂延迟创建代理。
- **推导链**：
  1. 代 = AOP代理 = Spring循环依赖的难点所在
  2. 曝 = 提前暴露 = 循环依赖时必须提前把半成品暴露出去
  3. 原 = 原始对象 = 二级缓存里放的是还没代理的原始对象
  4. 三原因 = 为什么需要三级？因为代理对象不能提前放进二级！
  5. 如果只有二级：B注入A时，A还没走完生命周期（没创建代理），B拿到的是原始A，后期A变成代理A，导致B里的A不是代理 → 不一致 → 报错
  6. 三级缓存（ObjectFactory）存的是一个lambda，调用时才创建代理，保证B拿到的是代理后的A

## 一字一画面

**代** —— 像[替身演员]：导演喊开机，替身（代理对象）先上场，真身（原始对象）在后台化妆。如果直接把真身派出去，观众看到的是素颜，后期再换脸就穿帮了（本质：代理对象和原始对象必须一致，提前暴露时得用工厂造替身）。

**曝** —— 像[半成品先上桌]：厨师A缺盐去找厨师B，把自己炒了一半的菜先端出去（提前暴露），否则两人互相等对方就 deadlock 了（本质：循环依赖必须提前暴露半成品打破死锁）。

**原** —— 像[素颜照]：二级缓存里放的是没化妆的原始对象，直接发出去会吓到观众（本质：AOP代理前和代理后是不同的对象，不能混用）。

## 面试话术

"Spring用三级缓存解决循环依赖，核心原因是AOP代理。一级缓存放成品对象，二级缓存放提前暴露的原始对象，三级缓存放ObjectFactory。如果只有二级缓存，循环依赖时注入的是原始对象而不是代理对象，会导致代理不一致的问题。三级缓存通过lambda延迟创建代理，保证注入的是代理后的对象。"

## 防混补丁

容易和Spring Bean生命周期搞混？
**补丁口诀**：实例化在先，初始化在后，代理在初始化里做，提前暴露靠工厂兜。

## 追问预演

1. **三级缓存分别是什么Map？**
   一级：singletonObjects（ConcurrentHashMap）；二级：earlySingletonObjects（ConcurrentHashMap）；三级：singletonFactories（HashMap）。

2. **如果不用AOP，二级缓存够吗？**
   够。没有AOP代理时，原始对象和最终对象是同一个，二级缓存就能解决循环依赖。

3. **Spring为什么不用setter解决循环依赖？**
   Spring支持setter循环依赖，但不支持构造器循环依赖（构造器里还没实例化完，无法提前暴露）。

4. **@Lazy能打破循环依赖吗？**
   能。@Lazy会生成代理对象延迟注入，从而打破循环依赖。
