# Spring 如何解决循环依赖

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 三 | 三缓曝，两前提 | 提前暴露半成品，构造器注入不行 |

## 原始知识

- **三** 级缓存（**singletonObjects** / **earlySingletonObjects** / **singletonFactories**）
- **曝** 光 ObjectFactory（**lambda 表达式**提前暴露）
- **AOP** 代理在第三级缓存生成（**SmartInstantiationAware**）
- **构** 造器注入无法解决（**对象未实例化**）
- **Set**ter / Field 注入可解决（**实例化后**注入）
- **@Lazy** 延迟加载绕过（**先注入代理**）

## 顺口溜

三缓曝，两前提  
（三级缓存提前曝，Setter 可解构造器恼）

## 单字锚点 + 防崩推导

> **锚点字：三**
>
> **扩展口诀**：三缓曝，两前提
>
> **为什么选它**：Spring 解决循环依赖的核心机制就是"三级缓存"，记住"三"就记住了骨架。
>
> **推导链**：
> - 三 → **三**级缓存（一级 singletonObjects 存成品，二级 earlySingletonObjects 存半成品，三级 singletonFactories 存 lambda 工厂）
> - 缓 → 三级都是**缓**存池
> - 曝 → 实例化后立刻**曝**光（把 ObjectFactory 放进第三级缓存）
> - 两前提 → 第一前提是 **Setter/Field 注入**（实例化之后才能暴露半成品）；第二前提是 **单例**（原型作用域每次新建，缓存无效）
> - 如果满足两前提 → 循环依赖时 B 可以从第三级缓存拿到 A 的 ObjectFactory，生成早期引用注入自己
> - 如果 A 需要 AOP → 第三级缓存的 lambda 会在 getEarlyBeanReference 时生成**代理对象**
> - 构造器注入不行 → 因为**对象还没实例化**，无法提前暴露
> - 绕过方案 → **@Lazy** 注入一个代理对象，延迟真正依赖的解析

## 一字一画面

**三** —— 像[三层抽屉柜]：最上层抽屉锁着成品娃娃（一级缓存），中层抽屉放着没穿衣服的半成品娃娃（二级缓存），下层抽屉藏着一个裁缝 lambda，谁要就现场做衣服（三级缓存）。A 和 B 互相要对方，A 先把自己塞进下层抽屉（**曝**光），B 打开下层抽屉拿到 A 的半成品穿上衣服（注入），然后 A 再回头拿成品 B。（本质：实例化后提前暴露半成品引用，打破循环等待）

## 面试话术

"Spring 解决循环依赖靠的是'三缓曝，两前提'：**三**级缓存里，一级存成品，二级存半成品，三级存 lambda 工厂；实例化后立刻**曝**光到三级缓存。两个前提是必须是单例，而且得是 **Setter 或 Field 注入**。**构造器注入**因为对象还没实例化，没法提前暴露，所以解决不了，这种情况可以用 **@Lazy** 注入代理来绕过。"

## 防混补丁

容易和"依赖注入的方式"搞混？一句补丁：  
**构造器注入是装配阶段，Setter 注入是初始化阶段，循环依赖只有在实例化完成后才能暴露半成品，所以构造器注入无解。**

## 追问预演

1. **问：三级缓存为什么一定要三级，两级行不行？**  
   答：两级在普通场景可以，但涉及 AOP 代理时会重复创建代理对象。第三级缓存存 lambda，保证代理只生成一次，且提前暴露的是代理引用。

2. **问：@Lazy 是怎么绕过循环依赖的？**  
   答：注入的是一个 CGLIB 代理对象，真正使用时才解析目标 bean，把循环依赖的强引用变成延迟引用。

3. **问：原型作用域（prototype）有循环依赖怎么办？**  
   答：Spring 不解决原型 bean 的循环依赖，每次 getBean 都新建实例，缓存无效，会直接抛 BeanCurrentlyInCreationException。

4. **问：第三级缓存的 lambda 里做了什么？**  
   答：getEarlyBeanReference()，如果 bean 需要 AOP，这里会调用 AbstractAutoProxyCreator 生成代理对象并返回。
