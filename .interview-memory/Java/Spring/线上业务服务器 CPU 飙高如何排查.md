## 原始知识

- **top** 定位高CPU进程
- **top -Hp pid** 定位高CPU线程
- **jstack** 打印线程栈，线程ID转16进制匹配
- 分析线程状态：**RUNNABLE**/死循环/频繁GC
- 查看**GC日志**，判断是否Full GC频繁
- **arthas/profile** 分析热点方法
- 检查是否有**死循环**、正则回溯、大对象创建

## 单字锚点 + 防崩推导

**锚点字：顶**

- **扩展口诀**：顶（top）堆（jstack）日（日志），五排查
- **为什么选它**：CPU飙高第一反应就是 `top` 命令，像登顶俯瞰全局，一眼锁定最耗CPU的进程
- **推导链**：
  - 顶 → top → 先看整体，找**进程**PID
  - 顶深入 → top -Hp pid → 再看线程，找**线程**TID
  - 线程ID转16进制 → jstack找nid=0x... → 看**线程栈**
  - 栈里看状态 → RUNNABLE且卡在自家代码 → **死循环/热点方法**
  - 若状态是GC → 看**GC日志** → Full GC频繁？
  - 仍定位不清 → **arthas/profile** 采样热点 → 揪出具体方法
  - 最后灵魂三问：**死循环**？正则回溯？大对象创建？

## 一字一画面

**顶** —— 像[登上山顶举望远镜]：[你站在服务器山顶，望远镜一扫，CPU占用最高的进程像红旗一样飘在最上面，PID就是红旗编号]（本质：top命令全局定位高CPU进程）

**堆** —— 像[翻垃圾堆找线索]：[你把线程栈像废纸一样摊在地上，用16进制放大镜（nid=0x...）逐行翻找，找到那根在疯狂运转的线程]（本质：jstack打印线程栈，十六进制线程ID匹配）

**日** —— 像[看日出日志光芒]：[GC日志像地平线上的光线，Full GC频繁就像太阳反复升起落下，每次升起都暂停整个世界]（本质：GC日志判断是否频繁Full GC导致CPU飙高）

## 面试话术

> "CPU飙高排查，我的思路是六步走。第一步用 top 定位进程，第二步 top -Hp 定位具体线程，把线程ID转成十六进制，第三步用 jstack 打印线程栈去匹配，看线程卡在哪个方法。如果是RUNNABLE状态而且是我们自己的业务代码，大概率是死循环或者热点方法。如果线程状态是GC，那就去看GC日志是不是Full GC太频繁。定位不准的时候，我会用arthas的profile做热点采样，直接看到哪个方法占用CPU最多。"

## 防混补丁

**容易和内存OOM排查搞混？**  
补丁口诀：**CPU看栈，OOM看堆** —— CPU飙高盯线程栈，OOM才盯堆内存。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 线程ID怎么和jstack对应？ | top -Hp拿到的线程ID是十进制，jstack里nid是十六进制，需要 printf "%x" TID 转换 |
| arthas的trace和profile区别？ | trace是方法级耗时监控，profile是CPU火焰图采样看热点 |
| 正则回溯怎么导致CPU飙高？ | 灾难性回溯（catastrophic backtracking），如 `(a+)+` 匹配超长字符串 |
| 大对象创建为什么CPU高？ | 频繁分配大对象触发频繁GC，或者分配逻辑本身计算复杂 |
