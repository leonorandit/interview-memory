# 如何设计一个点赞系统

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 赞 | 赞异缓，五设计 | 异步写 Redis，批量落库，幂等去重 |

## 原始知识

- **异** 步写入（**MQ 削峰**，避免直接写 DB）
- **缓** 存计数（**Redis incr** 原子操作）
- **批** 量落库（**定时任务**降低 DB 压力）
- **幂** 等处理（**同一用户**重复点赞去重）
- **展** 示优化（**推拉结合**，计数与用户列表分离）
- **冷** 热分离（**热数据**放 Redis，冷数据归档）

## 顺口溜

赞异缓，五设计  
（赞完丢 MQ，Redis 先记数，定时批量写库）

## 单字锚点 + 防崩推导

> **锚点字：赞**
>
> **扩展口诀**：赞异缓，五设计
>
> **为什么选它**：点赞系统的核心动作就是"赞"，所有设计都围绕这一个动作展开。
>
> **推导链**：
> - 赞 → 用户点了**赞**
> - 赞后不能直写 DB → 走**异**步（MQ 削峰，防突发流量冲垮数据库）
> - 异步前先写 → **缓**存（Redis incr 原子加计数，读的时候直接走 Redis）
> - 缓存不能总飘着 → **批**量落库（定时任务把 Redis 计数同步到 DB）
> - 用户可能狂点 → 要**幂**等（同一用户对同一内容只能点一次赞，去重防脏数据）
> - 前端展示要高效 → **展**示优化（推拉结合，计数和用户列表分开查）
> - 数据量大了 → **冷**热分离（近期热数据放 Redis，历史冷数据归档或降频）

## 一字一画面

**赞** —— 像[点赞手势变彩虹]：用户竖起大拇指点**赞**（触发），大拇指射出一道金光先飞进天上云朵（**异**步 MQ），云朵瞬间变成数字气球飘在头顶（**缓**存 Redis），每小时气球把数字倒进大河（**批**量落库），重复点赞的手势被镜面弹回（**幂**等去重），围观者先看气球数字再看详细名单（**展**示分离），旧气球收进仓库（**冷**热分离）。（本质：高并发写场景的标准削峰+聚合模式）

## 面试话术

"点赞系统是典型的**写多读多**场景，我的设计核心是'赞异缓，五设计'：用户点**赞**后先走 MQ**异**步削峰，Redis **缓**存做原子计数，定时任务**批**量同步到 DB，同时做好**幂**等去重防止重复点赞，展示层把计数和用户列表做分离，最后按时间做**冷**热数据分离。这样既能扛突发流量，也能保证最终一致性。"

## 防混补丁

容易和"秒杀系统"搞混？一句补丁：  
**秒杀是抢一份库存，点赞是累加计数，前者怕超卖用悲观锁，后者防重复用幂等。**

## 追问预演

1. **问：Redis 挂了怎么办？**  
   答：可以降级到本地缓存或限流，同时 Redis 做主从+哨兵保证高可用，计数丢失可通过 DB 对账修复。

2. **问：幂等怎么实现？**  
   答：用用户ID+内容ID 作为唯一键，Redis setnx 或者 DB 唯一索引都可以，推荐 Redis 原子判断更快。

3. **问：计数和用户列表不一致怎么办？**  
   答：计数走 Redis 保证性能，用户列表走 DB 保证准确，展示时容忍秒级延迟，定时对账兜底。

4. **问：如果取消点赞呢？**  
   答：同样走异步流程，Redis decr 减计数，DB 软删除或更新状态，保持和点赞同一套 pipeline。
