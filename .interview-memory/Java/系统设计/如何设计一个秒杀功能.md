## 单字锚点 + 防崩推导

**锚点字**：秒

**扩展口诀**：秒异缓，五设计

**为什么选它**：秒杀场景的核心诉求就是"秒级"扛住高并发，"秒"字本身就是场景名称，看到"秒"立刻锁定秒杀话题。

**推导链**：

```
秒（秒杀场景）
  ↓
异（异步下单）—— 同步扛不住 → MQ削峰
  ↓
缓（多级缓存）—— 数据库扛不住 → Redis预扣 + CDN静态化
  ↓
五设计展开：
  ├─ 限（限流）—— 入口闸机
  ├─ 异（异步）—— MQ排队
  ├─ 缓（缓存）—— Redis预扣 + CDN
  ├─ 防（防刷）—— 验证码 + 黑名单
  └─ 分（分片）—— 热点库存拆分
```

## 一字一画面

- **秒** —— 像红色发令枪：枪响瞬间万人冲出起跑线，只有最前面的能抢到（本质：瞬时流量洪峰）
- **异** —— 像医院挂号机：病人不挤窗口，先取票排队，窗口逐个处理（本质：异步解耦削峰）
- **缓** —— 像超市储物柜：顾客先放包再购物，不用每次都回家拿（本质：减少后端直接压力）
- **限** —— 像地铁闸机：绿灯才放行，红灯挡住，控制进站速度（本质：流量入口控制）
- **防** —— 像小区门禁：刷卡/刷脸才能进，陌生人挡外面（本质：识别并拦截恶意请求）

## 面试话术

> "秒杀的核心思路是层层设防：入口层用令牌桶限流，应用层用Redis原子递减做库存预扣，异步MQ处理订单落库，页面走CDN静态化，最后加上验证码防刷。关键是别让请求直接打到数据库。"

> "库存预扣用Redis的decr原子操作，扣成功才发MQ，扣失败直接返回已售罄，这样数据库只处理真正有库存的订单。"

## 防混补丁

**容易和什么搞混**：普通高并发 vs 秒杀。普通高并发是均匀流量，秒杀是瞬时脉冲。

**补丁口诀**：秒杀脉冲像洪水，普通并发像河流；洪水要闸要池要分流，河流只需扩宽道。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Redis预扣后挂了怎么办？ | 用分布式锁或Lua脚本保证原子性，预扣后发送MQ，消费者幂等处理 |
| 怎么防止超卖？ | Redis原子递减 + 数据库乐观锁（version字段）双重校验 |
| 热点Key怎么解决？ | 本地缓存 + Redis热点分片（一个商品拆成多份库存），或本地缓存兜底 |
| MQ堆积了怎么办？ | 消费者扩容 + 设置MQ过期时间 + 死信队列处理 |
| 限流算法选哪个？ | 令牌桶允许突发，漏桶更平滑；秒杀选令牌桶，允许合理突发 |
