## 单字锚点 + 防崩推导

**锚点字**：短

**扩展口诀**：短哈库，四步骤

**为什么选它**：短链系统的名称自带"短"字，看到"短"立刻联想到URL缩短，是整个系统的核心特征。

**推导链**：

```
短（短链系统）
  ↓
哈（哈希/编码）—— 长URL太长 → 哈希或自增ID → Base62编码
  ├─ Base62（0-9a-zA-Z = 62个字符）
  └─ 6位短码 = 62^6 ≈ 568亿，够用
  ↓
库（存储）—— 需要存映射关系 → MySQL + Redis缓存
  ├─ MySQL：短码为主键，长URL为值
  └─ Redis：热点短码缓存，加速查询
  ↓
四步骤展开：
  ├─ ① 生成（哈希/自增 + Base62）
  ├─ ② 存储（MySQL + Redis）
  ├─ ③ 跳转（302/301重定向）
  └─ ④ 监控（点击统计 + 布隆防重）
```

## 一字一画面

- **短** —— 像把一米长的面条剪成拇指长：方便入口，本质没变（本质：URL长度压缩，映射关系不变）
- **哈** —— 像榨汁机：一个大苹果进去，榨出一小杯浓缩汁，但能还原（本质：长URL通过哈希/编码压缩为短码）
- **库** —— 像快递柜：小格子贴编号，里面放大包裹，扫码即取（本质：短码为Key，长URL为Value存储）
- **跳** —— 像机场中转：票上写A口，实际飞B地，中转帮你换（本质：HTTP重定向，浏览器自动跳转目标地址）
- **布** —— 像小区黑名单：进门先查名单，坏人在外挡，好人快速过（本质：布隆过滤器快速判断短码是否已存在）

## 面试话术

> "短链系统的核心是映射表：长URL通过自增ID或哈希生成唯一标识，再用Base62编码成6位短码，存到MySQL并缓存到Redis。用户访问短链时返回302重定向到原始URL，同时用布隆过滤器防止重复生成。"

> "选型上我偏好自增ID + Base62，因为它没有哈希冲突问题，而且天然有序。如果用哈希算法比如MurmurHash，需要处理冲突，优点是相同URL得到相同短码，有幂等性。"

## 防混补丁

**容易和什么搞混**：短链跳转用301还是302。301会被浏览器永久缓存，无法再变更；302每次都会回源，可以修改映射。

**补丁口诀**：三零一像钉子户，一次记住不再问；三零二像传话筒，每次都要重新问。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 自增ID和哈希怎么选？ | 自增ID无冲突、有序；哈希有冲突但相同URL幂等。生产环境推荐自增ID |
| 短码用完了怎么办？ | 6位不够升到7位/8位，62^8 ≈ 218万亿，足够用 |
| 布隆过滤器误判怎么办？ | 误判只影响生成效率（认为已存在而跳过），不会导致数据错误；可配合数据库二次确认 |
| 怎么统计点击量？ | 发MQ异步统计/Redis incr计数，避免阻塞跳转主链路 |
| 短链安全问题怎么防？ | 链接鉴权（过期时间/访问密码）、内容安全审核、防爬虫限流 |
