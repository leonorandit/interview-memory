# Java 中 ConcurrentHashMap 1.7 和 1.8 之间有哪些区别

## 速记卡片

| 锚点字 | 口诀       | 核心差异               |
|--------|------------|------------------------|
| 段     | 段表CAS，三区别 | 1.7 分段锁，1.8 CAS+sync |

## 原始知识

- **1.7 结构**：Segment 数组 + HashEntry 数组，**分段锁**
- **1.8 结构**：Node 数组 + 链表/红黑树，**CAS + synchronized**
- **锁粒度**：1.7 锁 Segment（一段多桶），1.8 锁单个 Node（头节点）
- **size 计算**：1.7 分段累加+多次重试，1.8 CounterCell 数组+baseCount
- **并发度**：1.7 默认 16 个 Segment，不可扩容；1.8 理论上无上限
- **哈希算法**：1.8 简化为高位异或，1.7 多次移位异或
- **树化支持**：1.8 支持红黑树，1.7 不支持

## 顺口溜

> 1.7 分段锁，1.8 CAS 走；锁段变锁头，红黑树也有。

## 单字锚点 + 防崩推导

- **锚点字**：段
- **扩展口诀**：段表CAS，三区别
- **为什么选它**：1.7 最核心的特征就是**分段锁（Segment）**，一眼区分版本
- **推导链**：
  1. 段 → 1.7 分段锁（Segment 数组）→ 一段里面多个桶，锁粒度较粗
  2. 段 → 1.8 不用段了 → 用什么？→ CAS + synchronized（锁头节点）
  3. 表 → 数据结构变化 → 1.7 Segment+HashEntry，1.8 Node+链表/红黑树
  4. CAS → 无锁乐观锁 → put 时 CAS 尝试，失败再 synchronized
  5. 三区别 → 锁粒度、size 计算、数据结构

## 一字一画面

- **段** —— 像一截截橙色香肠串：整根香肠切成 16 段，吃哪段只切哪段的绳子（锁），其他段照样能吃。（本质：Segment 分段，每段独立加锁，提升并发度）
- **表** —— 像一张大桌子被分成 16 个小桌：1.7 每个小桌配一把锁，1.8 直接一张大桌，每盘菜（Node）各配一把小锁。（本质：1.8 去掉 Segment 中间层，结构更扁平）
- **C** —— 像两个人同时抢最后一个红包：谁先点到谁拿走（CAS 成功），另一个重试或排队。（本质：Compare-And-Swap 乐观锁，无锁竞争时零开销）
- **A** —— 像自动售票机：没人时直接买票（CAS），冲突时排队窗口（synchronized）。（本质：先 CAS 尝试，失败再降级为悲观锁）
- **S** —— 像保安只锁住一个抽屉：只锁发生冲突的那个桶的头节点，其他桶照常访问。（本质：synchronized 锁头节点，粒度极细）

## 面试话术

1. "1.7 是 Segment 数组加分段锁，默认 16 个 Segment，每个 Segment 继承 ReentrantLock；1.8 抛弃了 Segment，直接用 Node 数组，锁的粒度从一段降到了单个桶的头节点。"
2. "1.8 用 CAS 加 synchronized 来保证线程安全，put 时先用 CAS 尝试插入，如果冲突再用 synchronized 锁住头节点，同时引入了红黑树优化链表过长的问题。"
3. "size 计算上，1.7 是对每个 Segment 的 count 累加，可能要重试几次；1.8 用 CounterCell 数组分散热点，最后求和，并发更高。"

## 防混补丁

- **容易和 HashMap 搞混？** HashMap 完全无锁，CHM 线程安全；HashMap 允许 null，CHM 不允许。
- **容易和 Hashtable 搞混？** Hashtable 全局一把锁，CHM 1.7 分段锁、1.8 锁头节点，并发度天壤之别。
- **补丁口诀**：HashTable 全局锁，CHM 分段再变细，HashMap 无锁跑单骑。

## 追问预演

1. **Q：1.8 为什么去掉 Segment？**
   A：Segment 粒度还是太粗，一段包含多个桶，并发度受 Segment 数量限制；1.8 锁头节点粒度更细，支持更高并发，且 CAS 无锁场景性能更好。
2. **Q：1.8 的 size 怎么保证准确？**
   A：用 CounterCell 数组分散计数，每个线程操作不同 Cell 避免竞争，最后 baseCount + sum(CounterCell) 得到 size，高并发下是弱一致性。
3. **Q：1.8 什么情况下用 synchronized 不用 CAS？**
   A：首次插入桶时 CAS 设置头节点；如果桶已有节点（hash 冲突），则 synchronized 锁住头节点再操作链表或红黑树。
4. **Q：CHM 1.8 的 get 需要加锁吗？**
   A：get 全程无锁，因为 Node 的 val 和 next 用 volatile 保证可见性，读取时看到的就是最新值。
