## 原始知识

- 扩容阈值 = capacity × **load factor**
- 默认 **16** 扩容到 **32**（**2 倍**）
- JDK 1.8 **尾插法**（避免并发死循环）
- **rehash**：hash & oldCap 判断位置
- 树化阈值 **8** / 退化阈值 **6**
- JDK 1.7 **头插法**并发扩容导致死循环

## 顺口溜

扩倍迁，e尾插；
阈十六，八树化。

## 单字锚点 + 防崩推导

- **锚点字**：扩
- **扩展口诀**：扩倍迁，e尾插
- **为什么选它**：问题问的就是扩容机制，扩字直接对应容量扩大
- **推导链**：
  1. 扩 → 扩容触发
  2. 扩容 → 容量 **2 倍**
  3. 2 倍 → 元素重新迁移（rehash）
  4. 迁移位置 → hash & oldCap == 0 留原处，否则去原索引 + oldCap
  5. 插入方式 → JDK 1.8 **尾插法**
  6. 尾插 → 避免 1.7 头插法并发死循环
  7. 链表过长 → 长度 > 8 **树化**，< 6 **退化**

## 一字一画面

- **扩** —— 像气球吹大：直径翻倍，里面的豆子要重新分布到新位置（本质：容量翻倍，元素 rehash 迁移）
- **倍** —— 像细胞分裂：1 变 2，2 变 4，永远对称不零碎（本质：2 的幂次方，方便位运算定位）
- **迁** —— 像搬家分楼层：奇数层住原楼，偶数层搬新楼，看门牌号第一位决定（本质：hash & oldCap == 0 留原处，否则去新索引）
- **尾** —— 像排队买奶茶：新来的永远站最后，不插队就不打架（本质：尾插法避免并发时链表成环）

## 面试话术

- "HashMap 扩容我记成'扩倍迁，e尾插'，就是容量翻倍，元素迁移，JDK 1.8 用尾插法。"
- "迁移时用 hash & oldCap 判断位置，要么留在原索引，要么去原索引 + oldCap 的位置；链表长度超过 8 会树化成红黑树，低于 6 退化成链表。"

## 防混补丁

容易和 ConcurrentHashMap 混淆？HashMap 线程不安全，扩容时并发会死循环。  
**补丁口诀**：Hash 单，Concurrent 安，扩容别并发。

## 追问预演

- **Q：为什么 JDK 1.8 改尾插？**  
  A：1.7 头插法并发扩容会导致链表成环，get 时死循环；1.8 尾插法保持链表顺序，不会成环。
- **Q：为什么容量必须是 2 的幂？**  
  A：方便位运算，(n - 1) & hash 等价于 hash % n，且 rehash 时只需看 hash 新增的那一位。
- **Q：树化阈值为什么是 8？退化为什么是 6？**  
  A：泊松分布，正常情况下链表长度到 8 的概率极低（0.00000606），此时才值得树化开销；6 和 8 之间有缓冲避免频繁树化/退化。
