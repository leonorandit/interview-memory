# 为什么 JDK 1.8 对 HashMap 进行了红黑树的改动

## 速记卡片

| 锚点字 | 口诀       | 核心差异               |
|--------|------------|------------------------|
| 长     | 长链慢，红黑快 | 链表 O(n)，红黑树 O(log n) |

## 原始知识

- **链表退化**：哈希冲突严重时，链表过长，查询退化为 O(n)
- **红黑树优势**：自平衡二叉搜索树，查询/插入/删除均为 O(log n)
- **阈值 8 转树**：链表长度 ≥8 且数组长度 ≥64 时，链表转红黑树
- **阈值 6 退化**：链表长度 ≤6 时，红黑树退化回链表
- **空间权衡**：树节点（TreeNode）是链表节点（Node）的 2 倍大小
- **泊松分布**：hash 碰撞达到 8 的概率极低（0.00000606），大多数情况仍是链表
- **为什么不是 AVL**：红黑树插入删除旋转更少，统计性能更优

## 顺口溜

> 链表一长慢如龟，红黑树快似飞；八上树，六下链，空间换时间是精髓。

## 单字锚点 + 防崩推导

- **锚点字**：长
- **扩展口诀**：长链慢，红黑快
- **为什么选它**：改动的**根本原因**就是链表太长导致性能退化，"长"是起点
- **推导链**：
  1. 长 → 链表太长 → 查询变慢 O(n) → 用户体感卡顿
  2. 长 → 怎么解决？→ 换成更快的结构 → 红黑树 O(log n)
  3. 链 → 为什么不一直用树？→ 树节点占内存是链表 2 倍 → 大部分场景链表足够
  4. 慢 → 什么时候才慢？→ 极端 hash 冲突时 → 阈值 8 才转，概率极低
  5. 红黑快 → 红黑树特性 → 自平衡、颜色约束、旋转调整

## 一字一画面

- **长** —— 像一根红色的辣条越拉越长：要从头找到中间那根，得一根一根数过去，越吃越急。（本质：链表线性遍历，长度 n 则比较 n 次）
- **链** —— 像一串冰糖葫芦横着排：找中间那颗山楂，必须从左边第一颗挨个咬过去。（本质：链表无随机访问能力，只能顺序遍历）
- **黑** —— 像一棵黑皮甘蔗分节而立：粗壮的黑色主干，左右分枝整齐，砍哪节直接跳过去。（本质：红黑树有序分层，log n 次比较定位）
- **红** —— 像红绿灯约束交通：红色节点不能相连，黑高必须相等，违规就旋转调整。（本质：红黑树五大性质，通过颜色和旋转维持平衡）
- **八** —— 像八个人排队才开新窗口：队伍排满 8 人才升级 VIP 通道（红黑树），人少了又撤回去。（本质：阈值 8 是泊松分布的统计结果，避免频繁转换）

## 面试话术

1. "核心原因是当哈希冲突严重时，链表会变得越来越长，查询时间从理想的 O(1) 退化到 O(n)，这在极端场景下是不可接受的。"
2. "红黑树是自平衡二叉搜索树，能把查询复杂度稳定控制在 O(log n)，虽然树节点占用的空间是普通节点的两倍，但只在链表长度达到 8 才转换，而 hash 冲突达到 8 的概率已经非常低了。"
3. "设置 6 退化是为了留一个缓冲区间，避免在 8 附近频繁地在链表和红黑树之间来回转换，造成性能抖动。"

## 防混补丁

- **容易和 TreeMap 搞混？** TreeMap 全程红黑树，按 key 排序；HashMap 只在冲突严重时局部用红黑树，不保证有序。
- **容易以为所有桶都变树？** 只有冲突严重的桶才会变树，绝大多数桶仍是链表。
- **补丁口诀**：HashMap 局部树，TreeMap 全局树；八上六下链表驻。

## 追问预演

1. **Q：为什么是 8 和 6，而不是 7 和 7？**
   A：留一个缓冲带（6~8），避免在临界值附近频繁树化和退化；7 和 7 没有缓冲，一个删除就退化，一个插入又树化，抖动严重。
2. **Q：为什么树节点是链表节点的 2 倍？**
   A：TreeNode 继承 Node，额外维护了 parent、left、right、prev、red 五个字段，对象头和引用开销加起来约为普通 Node 的两倍。
3. **Q：为什么不用 AVL 树或跳表？**
   A：AVL 树查询略快但旋转更频繁，HashMap 插入场景更多，红黑树综合更优；跳表空间开销更大且不适合这个场景。
4. **Q：数组长度不足 64 时，链表超过 8 怎么办？**
   A：优先扩容（resize），而不是转树。因为数组太小是冲突的主因，扩容后链表自然变短。
