# 说说 Java 中 HashMap 的原理

## 速记卡片

| 锚点字 | 口诀       | 核心差异               |
|--------|------------|------------------------|
| 散     | 散链红黑，三原理 | 数组散、链表链、红黑树黑 |

## 原始知识

- **数组**：底层 Node[] table，初始容量 16
- **链表**：哈希冲突时拉链法，头插（1.7）→尾插（1.8）
- **红黑树**：链表长度 ≥8 且数组长度 ≥64 时转树
- **扰动函数**：`(h = key.hashCode()) ^ (h >>> 16)`，高 16 位与低 16 位异或，减少冲突
- **扩容机制**：2 倍扩容，重新散列（1.8 优化为按位与判断高低位）
- **JDK1.7 vs 1.8**：1.7 纯数组+链表，1.8 数组+链表+红黑树
- **线程不安全**：多线程下 1.7 头插死循环，1.8 尾插数据丢失
- **默认负载因子**：0.75，平衡时间与空间

## 顺口溜

> 数组散，链表连，红黑树来保平安。扰动异或十六位，二倍扩容再相见。

## 单字锚点 + 防崩推导

- **锚点字**：散
- **扩展口诀**：散链红黑，三原理
- **为什么选它**：HashMap 本质是**散**列表（Hash Table），"散"是核心数据结构
- **推导链**：
  1. 散 → 散列表 → 数组+链表/红黑树解决冲突
  2. 散 → 怎么散得匀？→ 扰动函数（高 16 位异或低 16 位）
  3. 散 → 散满了怎么办？→ 2 倍扩容 + rehash
  4. 链 → 链表太长怎么办？→ 红黑树（阈值 8）
  5. 红黑 → 红黑树是什么？→ 自平衡二叉搜索树，O(log n) 查询

## 一字一画面

- **散** —— 像一把绿豆撒进 16 个绿碗：豆子（Entry）落在不同碗里（数组下标），有的碗豆子太多就串成串（链表）。（本质：通过 hash 将 key 均匀分散到数组槽位）
- **链** —— 像一串红辣椒挂在绳上：红彤彤的辣椒（Node）一个连一个，找某个辣椒要顺着摸过去。（本质：拉链法解决哈希冲突）
- **黑** —— 像一棵黑松树干分叉：粗壮的黑色树干（根节点），左右分枝有序排列，找叶子不用走完全树。（本质：红黑树自平衡，查询效率 O(log n)）
- **扰** —— 像厨师颠勺翻匀炒饭：hashCode 高低位混合翻炒，让每粒米（key）分布更均匀。（本质：异或运算让 hash 更分散，减少碰撞）

## 面试话术

1. "HashMap 底层是一个数组加链表/红黑树的结构，1.8 之前是纯链表，1.8 之后引入了红黑树来解决链表过长导致的性能退化问题。"
2. "当 put 一个 key 时，先通过 hashCode 经过扰动函数计算下标，如果发生冲突就用拉链法，链表长度超过 8 并且数组长度超过 64 就会转成红黑树。"
3. "扩容是 2 倍扩容，1.8 做了优化，判断 hash 的高位是 0 还是 1 来决定新位置，不需要重新计算 hash。"

## 防混补丁

- **容易和 Hashtable 搞混？** Hashtable 全程 synchronized，HashMap 线程不安全；Hashtable 不允许 null key/value，HashMap 允许一个 null key。
- **容易和 ConcurrentHashMap 搞混？** CHM 线程安全，HashMap 不保证线程安全。
- **补丁口诀**：HashMap 无锁有 null，Concurrent 分段保平安，HashTable 全局一把锁。

## 追问预演

1. **Q：为什么负载因子是 0.75？**
   A：0.75 是时间和空间的平衡值，太高冲突多、查询慢，太低空间浪费大。数学上基于泊松分布，链表平均长度约 0.5。
2. **Q：为什么阈值是 8 转树、6 退化？**
   A：8 是基于泊松分布计算的，冲突达到 8 的概率已经极低（0.00000606），避免频繁在树和链表之间转换；6 和 8 之间留缓冲，防止频繁变来变去。
3. **Q：为什么用红黑树不用 AVL 树？**
   A：红黑树插入删除旋转次数更少，统计性能更优；AVL 树查询略快但旋转成本高，HashMap 插入更频繁，红黑树更均衡。
4. **Q：1.7 死循环怎么产生的？**
   A：并发扩容时头插法导致链表逆序，两个线程互相指向对方的节点，形成环形链表，get 时死循环。
