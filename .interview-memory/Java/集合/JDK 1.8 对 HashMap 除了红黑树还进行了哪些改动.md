# JDK 1.8 对 HashMap 除了红黑树还进行了哪些改动

## 速记卡片

| 锚点字 | 口诀       | 核心差异               |
|--------|------------|------------------------|
| 头     | 头改尾，扰变与 | 头插改尾插，hash 简化，扩容优化 |

## 原始知识

- **头插改尾插**：1.7 头插法（新节点插头部），1.8 尾插法（新节点插尾部）
- **避免死循环**：尾插法在并发扩容时不会形成环形链表
- **hash 算法简化**：1.7 四次移位异或，1.8 简化为 `h ^ (h >>> 16)`
- **扩容优化**：1.8 无需重新计算 hash，判断 `hash & oldCap == 0` 决定新位置
- **Entry 改 Node**：1.7 用 Entry，1.8 改名为 Node，TreeNode 继承 Node
- **插入时机**：1.7 先扩容再插入，1.8 先插入再扩容
- **扰动函数**：1.8 在 hashCode 基础上只做一次高位异或

## 顺口溜

> 头插改尾插，死循环不怕；hash 简化为异或，扩容位与定高下。

## 单字锚点 + 防崩推导

- **锚点字**：头
- **扩展口诀**：头改尾，扰变与
- **为什么选它**：1.7 并发死循环是面试高频考点，**头插法**是罪魁祸首，"头"是最醒目的改动
- **推导链**：
  1. 头 → 头插法 → 1.7 新节点插头部 → 并发扩容逆序 → 环形链表 → 死循环
  2. 头 → 改成尾插 → 保持原有顺序 → 不会环形 → 虽仍线程不安全，但无死循环
  3. 改 → 还有哪些改动？→ hash 算法简化、扩容优化、Entry 改名
  4. 扰 → 扰动函数简化 → 高位异或低位，减少碰撞
  5. 变 → 扩容位置判断变化 → `hash & oldCap` 代替重新取模
  6. 与 → 位与运算 → 0 留原位置，1 搬原位置+旧容量

## 一字一画面

- **头** —— 像插队的人挤到队伍最前面：一挤一推，队伍顺序全乱，两个人同时挤就变成人咬尾巴的圆圈。（本质：头插法导致链表逆序，并发扩容时形成环）
- **尾** —— 像排队买票自觉站最后：新来的永远站末尾，队伍顺序不变，再多人来也只是队伍变长。（本质：尾插法保持链表顺序，扩容时不会成环）
- **扰** —— 像把糖炒栗子上下翻匀：hashCode 的高 16 位和低 16 位炒一炒，让每颗栗子分布更均匀。（本质：异或运算混合高位信息，减少低位相同导致的碰撞）
- **与** —— 像开关决定走左门还是右门：`hash & oldCap` 的结果是 0 走左边（原位），是 1 走右边（原位+旧容量）。（本质：因为容量是 2 的幂，新下标只可能原位或原位+旧容量）
- **扩** —— 像搬宿舍只需跨半栋楼：知道自己在前半栋还是后半栋，直接平移过去，不用重新算门牌号。（本质：2 倍扩容时，新索引只取决于 hash 对应 oldCap 那一位是 0 还是 1）

## 面试话术

1. "最大的改动之一是头插法改成了尾插法，因为 1.7 并发扩容时头插会导致链表逆序，形成环形链表从而死循环，1.8 尾插虽然还是线程不安全，但避免了死循环。"
2. "hash 算法也简化了，1.7 做了四次移位和异或，1.8 只保留了一次高位异或，因为实际测试发现这么干已经足够分散了。"
3. "扩容时 1.8 不需要重新计算 hash，只需要判断 hash 值对应旧容量的那一位是 0 还是 1，0 就留在原位，1 就搬到原位加旧容量的位置，效率更高。"

## 防混补丁

- **容易以为 1.8 线程安全？** 尾插只是避免了死循环，HashMap 仍然线程不安全，并发场景要用 ConcurrentHashMap。
- **容易把 hash 算法和扰动函数搞混？** hash 算法是 `hashCode()` 的实现（Object 里），扰动函数是 HashMap 内部对 hashCode 再加工的方法。
- **补丁口诀**：HashMap 无锁心，尾插不死环有因；线程安全别靠它，Concurrent 才是真。

## 追问预演

1. **Q：1.7 死循环具体怎么形成的？**
   A：并发扩容时，线程 A 和 B 同时迁移同一个链表，头插法导致 B 把 A 迁移后的链表再次逆序，next 指针互相指向形成环，后续 get 遍历到此链表就死循环。
2. **Q：扩容优化具体省了什么计算？**
   A：省了对新容量取模（`hash % newCap`），改为一次位与判断（`hash & oldCap`），CPU 指令更少。
3. **Q：Entry 改 Node 只是改名吗？**
   A：主要是改名统一语义，同时 TreeNode 继承 Node，为红黑树化做铺垫，结构上更合理。
4. **Q：1.8 先插入再扩容有什么好处？**
   A：避免无效扩容，如果插入后没有冲突，可能不需要立即扩容；1.7 先扩容可能白扩一次。
