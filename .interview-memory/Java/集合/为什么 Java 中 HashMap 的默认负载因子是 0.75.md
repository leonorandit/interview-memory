## 原始知识

- 负载因子 = **元素个数/容量**
- 0.75是 **时间与空间的平衡点**
- 泊松分布 **链表平均长度<1**
- 过高 **空间省但查询慢**
- 过低 **查询快但空间浪费**
- 扩容代价 **rehash消耗性能**
- 默认16容量 **16×0.75=12阈值**

## 单字锚点 + 防崩推导

- **锚点字**：七
- **扩展口诀**：七时空，两平衡
- **为什么选它**：0.75就是"四分之三"，俗称"七成满"，"七"字直接映射0.75这个数字
- **推导链**：
  1. 七 → 0.75（七成满）
  2. 时 → 时间（查询效率，哈希冲突少）
  3. 空 → 空间（容量占用，不频繁扩容）
  4. 两平衡 → 时间和空间的权衡，由泊松分布统计学支撑

## 一字一画面

- **七** —— 像七分满的水杯：水倒太满（负载因子高）容易洒出来（哈希冲突多）；水倒太少（负载因子低）杯子利用率低（空间浪费）；七分刚刚好（0.75平衡点）。
- **时** —— 像高速公路收费站：车道多（容量大）但车少（元素少），通行极快（O(1)查询）；车太多堵成一锅粥（冲突成链表），查询变慢（O(n)甚至转红黑树O(log n)）。
- **空** —— 像租房：房间太大（容量过大）租金贵（内存浪费）；房间太小频繁搬家（频繁扩容rehash），0.75就像刚好住到七成时换大房，性价比最优。
- **两** —— 像天平两端：左边放时间效率（查询速度），右边放空间成本（内存占用），0.75恰好让天平水平（平衡态），这是JDK团队用泊松分布算出来的最优解。

## 面试话术

"负载因子就是元素个数除以容量。0.75其实是JDK团队在时间和空间之间找的一个平衡点。从统计学上看，符合泊松分布，这时候链表平均长度小于1，冲突概率很低。如果设太高比如0.9，空间是省了但冲突多查询慢；设太低比如0.5，查询快但扩容太频繁，空间浪费大。所以0.75是个经验加理论的折中值。"

## 防混补丁

**容易和什么搞混？** 有人以为0.75是固定扩容比例，其实扩容是容量翻倍，0.75只是触发扩容的阈值比例。

**补丁口诀**：七五是闹钟点，到时翻倍不是按比例涨。

## 追问预演

1. **Q：能不能把负载因子改成1？**
   A：可以但冲突概率大幅上升，链表变长甚至转红黑树，查询从O(1)退化到O(log n)，得不偿失。

2. **Q：泊松分布在HashMap里怎么体现的？**
   A：JDK8注释里明确提到，当负载因子0.75时，桶中元素个数服从λ=0.5的泊松分布，平均每个桶0.5个元素，冲突极少。

3. **Q：扩容时具体做了什么？**
   A：容量翻倍，把旧数组所有Entry重新计算hash位置（rehash）放入新数组，是个耗性能的操作，所以要避免频繁扩容。
