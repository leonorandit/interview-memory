## 原始知识

- **读未提交（RU）**：能看到**未提交**数据
- **读已提交（RC）**：只读**已提交**数据
- **可重复读（RR）**：事务内读**结果一致**
- **串行化（Serializable）**：事务**串行**执行
- **RU 问题**：脏读、不可重复读、幻读
- **RC 解决**：**脏读**解决，仍有不可重复读和幻读
- **RR 解决**：**脏读+不可重复读**解决，幻读部分解决
- **Serializable 解决**：**全部**解决，但性能最差
- **MySQL 默认**：**RR** 级别
- **Oracle 默认**：**RC** 级别

## 顺口溜

## 单字锚点 + 防崩推导

**锚点字**：级

**扩展口诀**：级读提交，幻可串

**为什么选它**："级"就是级别，四级隔离级别像四级台阶，越往上越严格，解决的问题越多。

**推导链**：
1. 锚点字"级" → 事务隔离级别
2. 级别 → 四级台阶 → RU、RC、RR、Serializable
3. 口诀"级读提交" → 读已提交（RC）+ 可重复读（RR）是中间两级
4. 口诀"幻可串" → 幻读在 RR 部分解决，串行化完全解决
5. RU（读未提交）→ 最宽松，连脏读都防不住
6. RC（读已提交）→ 解决脏读，但同一事务两次读可能不同（不可重复读）
7. RR（可重复读）→ MySQL 默认，解决不可重复读，快照读解决幻读
8. Serializable（串行化）→ 所有事务串行，解决一切并发问题，性能最差
9. 各级别实现 → RC/RR 靠 MVCC + 锁，Serializable 靠加锁或串行
10. 实际选择 → 默认 RR，追求性能且业务允许时可用 RC

## 一字一画面

**级** —— 像**四级红色台阶**：第一级地面最脏（RU），第二级擦了地板（RC），第三级铺了固定地毯（RR），第四级只准一个人走（Serializable）（本质：隔离级别从低到高，解决的问题越来越多）

**脏** —— 像**偷看别人没保存的草稿**：对方还在写，你已经看到了，他可能还会改（本质：脏读读到未提交的数据，可能被回滚）

**幻** —— 像**翻相册时照片突然变多**：同一本相册翻两次，第二次莫名其妙多了几张（本质：幻读指同一查询条件下，两次读取行数不同）

## 面试话术

"MySQL 有四个事务隔离级别，从低到高是读未提交、读已提交、可重复读和串行化。读未提交性能最好但会脏读；读已提交解决了脏读，Oracle 默认用这个；MySQL 默认是可重复读，通过 MVCC 解决了不可重复读，快照读也能避免幻读；串行化最严格，但性能太差，一般不用。"

## 防混补丁

容易把**不可重复读和幻读**搞混？补丁口诀：**不可重复是内容变，幻读是行数变，一个改旧行，一个插新行**。

## 追问预演

- **Q**：MySQL 的 RR 级别下真的完全没幻读吗？
  **A**：快照读通过 MVCC 和 Read View 可以避免，但当前读如 select for update 仍可能遇到幻读，需要靠间隙锁或临键锁。
- **Q**：为什么 Oracle 默认用 RC，MySQL 默认用 RR？
  **A**：Oracle 的 RC 配合多版本读已经能满足大部分业务；MySQL 选 RR 主要是为了兼容 statement 格式的 binlog，避免主从不一致。
- **Q**：实际项目中你会选哪个级别？
  **A**：默认 RR，如果业务允许读到最新提交的数据且对不可重复读不敏感，可以选 RC 提升并发性能。
