## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 下 | 下先滤，少回表 | 条件下推到存储引擎层先过滤，减少回表次数 |

## 原始知识

- **ICP**：Index Condition Pushdown
- MySQL **5.6 引入**
- 在**存储引擎层**先过滤索引条件
- **减少回表**次数
- Extra 显示 **Using index condition**
- 默认**开启**
- 只对**二级索引**有效
- 条件下推给引擎，引擎先判断再回表
- 减少 Server 层和引擎层的数据传输

## 顺口溜

下先滤，少回表，
索引条件往下推，
引擎层里先过滤，
不符条件不回表。

## 单字锚点 + 防崩推导

**锚点字**：下

**扩展口诀**：下先滤，少回表

**为什么选它**："下"就是下推，索引下推的核心是把过滤条件"下推"到存储引擎层，而不是在 Server 层过滤。

**推导链**：
1. 锚点字"下" → 索引下推（Index Condition Pushdown）
2. 下推 → 把条件推下去 → 从 Server 层推到存储引擎层
3. 推到存储引擎层 → 引擎在查索引时就能过滤
4. 先过滤 → 不符合条件的直接跳过
5. 跳过 → 不需要回表查数据行
6. 少回表 → 减少 IO，提升性能
7. 标志 → Extra 显示 Using index condition
8. 版本 → MySQL 5.6 引入，默认开启

## 一字一画面

**下** —— 像**快递分拣站**：包裹到了分拨中心，先在地面上按区域筛一遍，不是这个区域的直接踢走，不用送到楼上（本质：把 where 条件下推到存储引擎层，在索引扫描时就过滤掉不符合条件的记录）

**滤** —— 像**漏斗筛沙子**：沙子倒在漏斗里，细的先漏下去，大石子直接挡在上面（本质：存储引擎在遍历索引时，先用条件过滤，不符合的直接丢弃）

**回** —— 像**点名后回教室**：老师在走廊点名，叫到的人回教室，没叫到的不用动（本质：只有满足索引条件的记录才需要回表查完整数据）

## 面试话术

"索引下推是 MySQL 5.6 引入的一个优化，它会把 where 条件里能用索引过滤的部分下推到存储引擎层，让引擎在遍历索引的时候就先过滤一遍。这样不符合条件的记录就直接跳过了，不需要回表，能减少很多 IO。你看执行计划里 Extra 会显示 Using index condition。"

## 防混补丁

容易和**覆盖索引**搞混？补丁口诀：**覆盖是不回表，下推是少回表，覆盖靠字段全，下推靠条件筛**。

## 追问预演

- **Q**：索引下推对主键索引有效吗？
  **A**：主键索引是聚簇索引，数据就在索引里，不存在回表，所以 ICP 主要针对二级索引。
- **Q**：索引下推和覆盖索引能同时出现吗？
  **A**：不能，因为覆盖索引不需要回表，ICP 是为了减少回表次数，两者是互斥的优化场景。
- **Q**：什么情况下 ICP 不会生效？
  **A**：如果查询使用覆盖索引、或者使用聚簇索引、或者查询条件无法使用索引时，ICP 不会生效。
