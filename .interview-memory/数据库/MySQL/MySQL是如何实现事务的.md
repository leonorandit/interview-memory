## 原始知识

- **redo log**：崩溃恢复，保证**持久性**
- **undo log**：回滚操作，保证**原子性**，支撑MVCC
- **锁机制**：读写锁、行锁、间隙锁，保证**隔离性**
- 三者配合实现ACID
- redo是物理日志，undo是逻辑日志
- redo**顺序写**，性能高
- undo**多版本链**，读不阻塞写

## 单字锚点 + 防崩推导

**锚点字：redo**

- **扩展口诀**：redo回锁，三机制
- **为什么选它**：事务实现的核心三角，redo是崩溃后数据不丢的最后防线，最硬核
- **推导链**：
  1. redo → 事务提交后数据不能丢 → 对应**持久性（D）**
  2. redo是物理日志 → 顺序写磁盘 → 崩溃后**重放恢复**
  3. 回 → undo → 事务失败要回滚 → 对应**原子性（A）**
  4. undo也是多版本链 → 支撑**MVCC**，读不阻塞写
  5. 锁 → 多个事务同时跑 → 不能互相踩 → 对应**隔离性（I）**
  6. redo + undo + 锁 → 三者配合 → 实现完整ACID

## 一字一画面

**redo —— 像一台红色录音机**：事务每做一步，红色录音机就“咔哒”录下物理操作。突然断电，重启后按下播放键，所有已提交的操作重新演一遍，数据一点没丢。旁边还有一个蓝色倒带机（undo），出错了就倒带撤销。门口站着保安（锁），拦住不该进的人，保证屋里事务互不干扰。（本质：redo是崩溃恢复的保险，undo是回滚的后悔药，锁是并发的秩序维护）

## 面试话术

"MySQL实现事务主要靠三个机制：redo log保证持久性，数据库崩溃后能恢复；undo log保证原子性，失败能回滚，同时也支撑MVCC让读不阻塞写；锁机制保证隔离性，避免事务之间互相干扰。这三个机制配合起来，就实现了ACID特性。"

## 防混补丁

容易和binlog搞混？binlog是逻辑日志，用于主从复制和备份，redo log是物理日志，用于崩溃恢复，两阶段提交就是让它俩保持一致。  
**补丁口诀**：redo物理保恢复，binlog逻辑做复制，两者别混淆，二阶段来对齐。

## 追问预演

1. **Q：redo log和undo log的区别？**  
   A：redo是物理日志，记录页修改，用于崩溃恢复；undo是逻辑日志，记录反向操作，用于回滚和MVCC。

2. **Q：MVCC怎么实现的？**  
   A：每行数据有隐藏的事务ID和回滚指针，undo log形成版本链，读时根据Read View判断可见版本。

3. **Q：锁有哪些类型？**  
   A：行锁（Record Lock）、间隙锁（Gap Lock）、临键锁（Next-Key Lock），还有共享锁和排他锁。
