# MySQL 中如果我 select * from 一个有 1000 万行的表，内存会飙升么？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 流 | 流缓升，三因素 | 流式不飙升，缓存会飙升，看客户端和服务器 |

## 原始知识

- MySQL Server 层会缓存结果（net_buffer_size，默认 16KB，满则发客户端）
- InnoDB 层不会全表加载到内存（按页读取）
- 客户端如果一次性接收结果（如 JDBC 默认会缓存 ResultSet）则客户端内存飙升
- 使用流式查询（Statement.setFetchSize(Integer.MIN_VALUE)）或分页避免

## 顺口溜

```
Server层有缓冲，InnoDB按页读；客户端不设流，内存肯定飙
```

## 单字锚点 + 防崩推导

| 锚点字 | 扩展口诀 | 为什么选它 | 推导链 |
|--------|----------|------------|--------|
| 流 | 流式读取不飙内存 | "流" = 流式读取，像水管一样边读边流，不用全装下 | 流 → 流式查询（不缓存）/ 缓冲查询（缓存）→ Server 层缓冲（net_buffer）→ InnoDB 层按页读 → 客户端是否缓存 → 三因素决定内存是否飙升 |

**防崩推导**：只记得"流"字时，推导链如下：
1. "流"代表流式读取——像水管一样，水（数据）流过，不用全部装桶里
2. MySQL 服务端会不会飙升？——Server 层有 net_buffer（默认 16KB），满了就发客户端，不会堆积
3. InnoDB 会不会全表加载？——不会，按页读取，读一页处理一页
4. 那谁可能飙升？——**客户端**！如果客户端不用流式查询，一次性把 1000 万行全塞进内存，肯定爆
5. 解决方式？——**流式查询**（setFetchSize(Integer.MIN_VALUE)）或者**分页**

## 一字一画面

- **流** —— 像一根透明水管从数据库连到客户端：数据像水流一样源源不断通过，不用先全部装进一个大水缸（本质：边读边处理，不累积在内存）
- **缓** —— 像快递中转站的小货架：包裹（数据）到了先放货架上，满了就装车发走，不会无限堆积（本质：net_buffer_size 缓冲，默认 16KB，满则发送）
- **页** —— 像翻书：读一页翻一页，不会为了查一句话把整本书复印一遍（本质：InnoDB 按 16KB 页为单位读取）
- **升** —— 像气球被不断吹气：客户端不设流式查询，ResultSet 像气球越胀越大，直到爆掉（本质：客户端缓存全量结果导致内存飙升）

## 面试话术

> 这个问题要看三层：首先 MySQL Server 层有 net_buffer，默认 16KB，数据装满就发给客户端，不会自己爆；然后 InnoDB 存储引擎是按页读取的，也不会一次性把整个表加载到内存；但关键是客户端，如果你用的是 JDBC 默认方式，它会一次性把所有结果缓存到 ResultSet 里，那客户端内存肯定会飙升。解决方式要么用流式查询，要么分页。

## 防混补丁

- **容易和什么搞混？** 以为 MySQL Server 会因为 select * 大表而内存飙升，实际上 Server 和 InnoDB 都有保护机制，真正容易爆的是客户端。
- **补丁口诀**：服务端有缓冲，InnoDB 按页走；内存飙不飙，看客户端有没有开流。

## 追问预演

| 面试官追问 | 简要答案 |
|-----------|----------|
| net_buffer_size 是多大？ | 默认 16KB，可以配置 |
| InnoDB 是怎么读数据的？ | 按页读取，每页默认 16KB，不会全表加载 |
| JDBC 怎么开启流式查询？ | 设置 Statement.setFetchSize(Integer.MIN_VALUE) |
| 分页和流式查询的区别？ | 分页是分批次取，每次取固定条数；流式查询是服务端边生成边发送，客户端边读边处理 |
| 除了内存，还有什么风险？ | 大查询会占锁、占连接、刷脏页，影响其他请求 |
