## 原始知识

- **等**：wait-for graph（等待图）检测循环
- **超**：innodb_lock_wait_timeout 超时机制
- **顺**：资源排序，**顺序加锁**
- **减粒度**：减少事务粒度，尽快释放锁
- **降隔离**：降低隔离级别（RR→RC）
- **索优化**：索引优化，减少锁范围
- **自动检测**：InnoDB自动检测并回滚小事务
- **死锁日志**：show engine innodb status 查看
- **循环等待**：死锁四条件之一
- **持有等待**：死锁四条件之一

## 单字锚点 + 防崩推导

**锚点字：等**

- **扩展口诀**：等超顺，四解法；检测等，释放超，预防顺，根治减。
- **为什么选它**：死锁的本质是"循环等待"，"等"字直接点破核心；wait-for graph 是数据库内置的死锁检测手段，面试高频考点。
- **推导链**：
  1. 等 → 等待图（wait-for graph）→ 检测死锁 → 发现循环依赖
  2. 超 → 超时（timeout）→ innodb_lock_wait_timeout → 超时强制释放
  3. 顺 → 顺序（顺序加锁）→ 资源全局排序 → 破坏循环等待条件
  4. 四解法：
     - 检测：等图自动检测，回滚小事务
     - 释放：超时机制，时间到强制释放
     - 预防：顺序加锁，所有事务按同一顺序申请资源
     - 根治：减少事务粒度 + 索引优化 + 降低隔离级别
  5. 补充：show engine innodb status 看死锁日志

## 面试话术

1. "死锁就是事务之间形成了循环等待，InnoDB会用wait-for graph自动检测，发现死锁就回滚代价最小的事务。"
2. "除了自动检测，还可以设置innodb_lock_wait_timeout超时释放；从根本上预防的话，让业务层按固定顺序加锁，或者减少事务粒度、优化索引缩小锁范围。"
3. "有时候把隔离级别从RR降到RC，能减少间隙锁的使用，从而降低死锁概率。"

## 防混补丁

容易和**锁等待**搞混？一句补丁：

> 锁等待是排队等一个人，死锁是四个人互相等；锁等待能解开，死锁解不开必须杀一个。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| InnoDB怎么检测死锁？ | 维护wait-for graph，发现循环依赖就判定死锁 |
| 死锁了回滚哪个事务？ | 回滚undo log最小的事务（代价最小的） |
| 怎么查看死锁信息？ | show engine innodb status，看LATEST DETECTED DEADLOCK |
| 顺序加锁具体怎么做？ | 业务层对所有资源按固定规则排序，比如按主键升序加锁 |
| 为什么RR比RC更容易死锁？ | RR有间隙锁和临键锁，锁范围更大，冲突概率更高 |
