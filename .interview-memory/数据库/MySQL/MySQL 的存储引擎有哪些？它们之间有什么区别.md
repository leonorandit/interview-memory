## 原始知识

- **InnoDB**：支持**事务**（ACID）
- InnoDB：**行级锁**，并发高
- InnoDB：**聚簇索引**，数据与索引在一起
- InnoDB：**MVCC** 实现高并发
- InnoDB：**崩溃恢复**（redo log）
- **MyISAM**：**表级锁**，并发低
- MyISAM：**非聚簇索引**，索引和数据分离
- MyISAM：**全文索引**（早期版本）
- MyISAM：**压缩存储**，节省空间
- MyISAM：不支持事务
- **Memory**：数据存**内存**，重启丢失
- **Archive**：只支持**插入和查询**，压缩存储
- MySQL 5.5+ **默认 InnoDB**

## 顺口溜

引My聚，两主引，
事务行锁 InnoDB，
表锁全文 MyISAM，
内存 Archive 各不同。

## 单字锚点 + 防崩推导

**锚点字**：引

**扩展口诀**：引My聚，两主引

**为什么选它**："引"是存储引擎的"引"，也是引出对比的核心。记住"引"就想到引擎，想到引擎就想到最主要的两个：InnoDB 和 MyISAM。

**推导链**：
1. 锚点字"引" → 存储引擎
2. 存储引擎 → 主流两个：InnoDB、MyISAM
3. "引My聚" → InnoDB + MyISAM + 聚簇索引（InnoDB 是聚簇）
4. "两主引" → 两个主流引擎的对比
5. InnoDB → 事务（transaction）、行锁（row lock）、聚簇索引（clustered）、MVCC、崩溃恢复
6. MyISAM → 表锁（table lock）、非聚簇索引、全文索引、压缩存储、无事务
7. 其他 → Memory（内存）、Archive（归档）
8. 默认 → 5.5 之后默认 InnoDB

## 一字一画面

**引** —— 像**汽车引擎盖**：打开看到里面两大核心引擎，一个 InnoDB 涡轮增压，一个 MyISAM 自然吸气（本质：存储引擎是数据库的核心动力装置，决定性能和特性）

**聚** —— 像**书架上的书和目录放在一起**：目录（索引）后面直接跟着书（数据），找目录就等于拿到书（本质：聚簇索引的叶子节点直接存储数据行，减少一次 IO）

**表** —— 像**教室门锁**：锁了整个教室（表），一个人进去其他人全等（本质：表锁粒度大，并发低，但开销小）

## 面试话术

"MySQL 最主要的存储引擎是 InnoDB 和 MyISAM。InnoDB 支持事务，有行级锁，用的是聚簇索引，数据直接存在索引叶子节点上，还支持 MVCC 和崩溃恢复。MyISAM 不支持事务，只有表级锁，索引和数据是分开存储的，不过它支持全文索引和压缩存储。现在 MySQL 默认用的是 InnoDB。"

## 防混补丁

容易把**聚簇索引和非聚簇索引的存储位置**搞混？补丁口诀：**InnoDB聚一起，MyISAM分两家，聚簇索引即数据，非聚索引存地址**。

## 追问预演

- **Q**：为什么 InnoDB 要设计聚簇索引？
  **A**：聚簇索引把数据和索引放一起，根据主键查询时只需要一次 IO 就能拿到数据，效率高；同时保证了数据按主键有序存储。
- **Q**：MyISAM 什么场景下还能用？
  **A**：读多写少、不需要事务、需要全文索引（早期版本）或者需要压缩存储节省空间的场景。
- **Q**：Memory 引擎适合什么场景？
  **A**：临时数据、缓存表，但要注意重启数据会丢失，且不支持 text/blob 大字段。
