## 原始知识

- 索引**不一定有效**
- **失效场景**：函数操作、类型转换、like左模糊、or条件、不等于、is null
- **Explain四看**：**type**、**key**、**rows**、**Extra**
- **索引选择性**：区分度低则效果差
- **强制索引**：FORCE INDEX hint

## 顺口溜

函数转换左模糊，or不等is null；
Explain四列盯紧，type最要命。

## 单字锚点 + 防崩推导

**锚点字：查**

- 扩展口诀：**查 Explain，四看效**
- 为什么选它："查"既是排查的"查"，也是查询的"查"，一语双关
- 推导链：
  1. 查 → 排查索引效果
  2. 排查 → **Explain** 命令
  3. Explain → **四看**：type、key、rows、Extra
  4. type → 访问类型（system > const > eq_ref > ref > range > index > ALL）
  5. key → 实际使用的索引
  6. rows → 扫描行数，越小越好
  7. Extra → "Using index" 好，"Using filesort""Using temporary" 差
  8. 查 → 索引也有**失效场景**：函数、转换、左模糊、or、不等于、is null
  9. 查 → 索引**选择性**（区分度）低时效果差

## 一字一画面

- **查** —— 像安检员拿着放大镜看四张证件照：type是身份级别，key是门卡号，rows是排队人数，Extra是特殊备注（本质：Explain四列对应索引效果的四个维度）
- **转** —— 像把中文名字音译成乱码再找人，系统认不出来（本质：类型转换导致索引失效）

## 面试话术

1. "索引不一定有效，比如对索引列做函数操作、隐式类型转换、like '%xx'左模糊，or条件组合不当，都会导致索引失效。"
2. "排查的话我用 Explain，重点看四列：type是不是ALL全表扫描，key有没有用到索引，rows扫描了多少行，Extra有没有出现Using index或者Using filesort这些信号。"

## 防混补丁

容易和"有索引就走索引"搞混？记住：**优化器会选错，FORCE INDEX 可纠偏**。

## 追问预演

1. **like 'xx%' 会失效吗？** 不会，右模糊可以走索引；左模糊 '%xx' 会失效，因为B+树无法按前缀定位。
2. **is null 一定不走索引吗？** 不一定，MySQL 8.0 优化后部分场景可以走，但早期版本和数据分布极端时容易失效。
3. **FORCE INDEX 有什么风险？** 强制指定索引后优化器失去选择权，如果数据分布变化，指定索引可能不再是最佳，导致性能下降。
