# 在 MySQL 中建索引时需要注意哪些事项？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 选 | 选短小，六注意 | 高选择性、小字段、短前缀 |

## 原始知识

- **选择性高** 区分度大
- **最左前缀** 联合索引顺序
- **避免冗余** 重复索引浪费
- **控制数量** 增删改变慢
- **小字段** 节约存储空间
- **长字符串** 用前缀索引
- **频繁更新列** 不适合建索引
- **覆盖索引** 减少回表查询

## 顺口溜

选高区分，短前缀；
小字段，左顺序；
控数量，免冗余。

## 单字锚点 + 防崩推导

- **锚点字**：选
- **扩展口诀**：选短小，六注意
- **为什么选它**：建索引首先要“选”对列，选不对等于白建。
- **推导链**：
  1. 选 → 选择性：选区分度高的列，性别这种低选择性不选
  2. 短 → 短前缀：长字符串用前缀索引，字段要短小
  3. 小 → 小字段：字段小节约空间，一页存更多行
  4. 六注意：
     - ① 最左前缀原则（联合索引顺序）
     - ② 避免冗余索引
     - ③ 控制索引数量（写性能）
     - ④ 小字段/前缀索引
     - ⑤ 避免频繁更新列建索引
     - ⑥ 覆盖索引减少回表

## 一字一画面

- **选** —— 像[筛子]：一个大筛子，高选择性的豆子（大颗粒）漏不下去被选中，低选择性的沙子（小颗粒）漏走了（本质：区分度高的列才能有效过滤数据）
- **短** —— 像[短跑赛道]：索引叶子节点是一页页短跑道，短字段占的跑道窄，一页能并排跑更多行（本质：小字段/前缀索引让 B+ 树一页存更多数据）
- **小** —— 像[小盒子]：B+ 树节点是小盒子，盒子小才能一层层多装，树矮查询快（本质：控制字段大小和索引数量，降低树高度）

## 面试话术

- “建索引首先看选择性，选区分度高的列，比如用户 ID 比性别好；联合索引要遵循最左前缀原则，把最常查询的列放左边。”
- “还要控制索引数量，因为每次增删改都要维护索引；长字符串用前缀索引，小字段更省空间；尽量用覆盖索引减少回表。”

## 防混补丁

容易和“索引越多越好”搞混？**索引是读的加速器、写的减速带**，写多的表要慎用。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 前缀索引怎么定长度？ | 看区分度，比如 SELECT COUNT(DISTINCT LEFT(email,6))/COUNT(DISTINCT email)。 |
| 覆盖索引是什么？ | 查询字段都在索引里，不需要回表查数据行。 |
| 联合索引(a,b,c)，查 b 能用吗？ | 不能，缺少最左列 a，索引失效。 |
