## 速记卡片

| 锚点字 | 口诀 | 核心差异 |

| 查 | 查根叶，三步骤 | 根二分、层下沉、叶子取/回 |

## 原始知识

- **从根节点开始**：二分查找
- **逐层向下**：直到叶子节点
- **叶子节点双向链表**：支持顺序遍历
- **聚簇索引**：叶子直接取数据
- **非聚簇索引**：拿到主键再回表

## 顺口溜

根节点二分找，逐层下沉到叶梢；
叶子链表左右绕，聚簇直接非聚跑。

## 单字锚点 + 防崩推导

**锚点字：查**

- 扩展口诀：**查根叶，三步骤**
- 为什么选它："查"是查询动作的核心，贯穿整个 B+ 树检索流程
- 推导链：
  1. 查 → 查询 B+ 树
  2. 查 → 第一步：**根节点**开始二分查找
  3. 根节点 → 找到对应**子节点指针** → 下沉到下一层
  4. 第二步：**逐层向下**，每层都二分
  5. 第三步：到达**叶子节点**
  6. 叶子节点 → **双向链表** → 支持范围遍历
  7. 聚簇索引 → 叶子存整行 → **直接取数据**
  8. 非聚簇索引 → 叶子存主键 → **回表**再查

## 一字一画面

- **根** —— 像大楼导览台的红衣接待员，你报房间号，她一指电梯方向（本质：根节点只存键值范围，引导向下）
- **叶** —— 像走廊尽头的储物柜，柜门一打开，要么直接拿到文件（聚簇），要么只拿到取件码还得下楼再取（非聚簇）（本质：叶子节点才是数据或指针的终点）

## 面试话术

1. "查询从 B+ 树的根节点开始，在每一层做二分查找，确定下一层该走哪个子节点，一直下沉到叶子节点。"
2. "叶子节点之间用双向链表连接，所以范围查询可以在叶子层顺序遍历。如果是聚簇索引，叶子直接就是数据；非聚簇索引的话，叶子存的是主键，需要再回表查一次。"

## 防混补丁

容易和"B树查询"搞混？记住：**B+树数据全在叶，B树数据每层搁**。

## 追问预演

1. **B+ 树一般多高？** 通常 3-4 层就能存几千万行，因为 InnoDB 页大小 16KB，非叶子节点只存键值，扇出很大。
2. **为什么叶子节点用双向链表？** 支持范围查询和排序遍历，比如 BETWEEN、ORDER BY，可以顺序扫描而不用回退到上层。
3. **根节点在内存还是磁盘？** 根节点通常会被缓存在 Buffer Pool 里，热数据页基本在内存，所以实际 IO 往往只有 1-2 次。
