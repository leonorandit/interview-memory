## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 执 | 执解优执，四步走 | 连接器→解析器→优化器→执行器 |

## 原始知识

- **连接器**：验证身份，管理连接
- **查询缓存**：命中直接返回（8.0已移除）
- **解析器**：词法分析+语法分析
- **预处理器**：检查表、列是否存在
- **优化器**：生成**执行计划**，选最优索引
- **执行器**：调用存储引擎API
- **存储引擎**：实际**读写数据**

## 顺口溜

执解优执，四步走，  
连解优化再执行。  
词法语法先解析，  
执行计划优化定，  
引擎API真实干。

## 单字锚点 + 防崩推导

**锚点字：执**

- **扩展口诀**：执解优执，四步走
- **为什么选它**：SQL执行的完整流程，最终落脚在“执行”，但中间必经解析和优化，“执”是起点也是终点
- **推导链**：
  1. 执 → SQL要执行 → 先连上数据库 → **连接器**验证账号权限
  2. 连上后 → 有没有缓存？→ **查询缓存**（8.0前，现版本已废弃）
  3. 没缓存 → 得看懂SQL → **解析器**做词法+语法分析
  4. 看懂后 → 表和列对不对？→ **预处理器**检查语义
  5. 确认无误 → 怎么执行最快？→ **优化器**生成执行计划、选索引
  6. 计划有了 → 开始干活 → **执行器**一行行调用
  7. 执行器不碰磁盘 → 调**存储引擎API**真正读写数据

## 一字一画面

**执 —— 像一位穿黑袍的指挥官**：他站在城堡门口（连接器）检查通行证，然后走进大厅，先让翻译官（解析器）把密信翻译成母语，再让参谋（优化器）在沙盘上画出最佳进攻路线，最后挥动令旗，命令骑兵（执行器）冲锋，而真正的战马和刀枪都在城外的军营（存储引擎）里。（本质：SQL执行是分层协作，上层决策，下层执行）

## 面试话术

"一条SQL进来，首先是连接器验证身份，然后到解析器做词法和语法分析，把它转成MySQL能懂的结构。接着优化器会根据表结构、索引这些信息，生成一个最优的执行计划。最后执行器按照计划去调用存储引擎的API，真正读写数据。整个流程就是连、解、优、执四步走。"

## 防混补丁

容易和Redis执行流程搞混？Redis是单线程直接执行，没有解析器和优化器生成执行计划这一步。  
**补丁口诀**：MySQL分层走，Redis单线干，解析优化是关系型，键值直接无计划。

## 追问预演

1. **Q：优化器怎么选索引？**  
   A：基于成本模型，考虑扫描行数、回表成本、排序代价等，有时会用trace分析。

2. **Q：执行器怎么调用存储引擎？**  
   A：执行器按执行计划逐行调用引擎接口，比如InnoDB的handler接口，引擎负责具体的页读取、加锁等。

3. **Q：查询缓存为什么被移除？**  
   A：命中率低，任何对表的更新都会清空该表缓存，开销大，8.0版本彻底移除。
