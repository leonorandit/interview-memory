## 原始知识

- **MVCC**：多版本**并发控制**
- **核心目标**：读**不加锁**，读写不阻塞
- **undo log 版本链**：记录旧版本数据，形成**历史链条**
- **Read View**：快照读时的**可见性判断**
- **trx_id**：事务 ID，判断**可见性**
- **roll_pointer**：回滚指针，指向**上一版本**
- **RC 级别**：每次查询**新建** Read View
- **RR 级别**：事务开始时**固定** Read View
- **隐藏字段**：每行记录自带 DB_TRX_ID、DB_ROLL_PTR

## 顺口溜

版读不加，三组件，
undo 链条串旧版，
Read View 判可见，
RC 新视 RR 固。

## 单字锚点 + 防崩推导

**锚点字**：版

**扩展口诀**：版读不加，三组件

**为什么选它**："版"就是版本，MVCC 的核心就是多版本，版本链、版本可见性都是围绕"版"展开。

**推导链**：
1. 锚点字"版" → 多版本并发控制 MVCC
2. MVCC → 版读不加 → 读操作不加锁，读的是历史版本
3. 不加锁 → 靠什么实现？→ 三组件
4. 三组件 → undo log 版本链 + Read View + 隐藏字段
5. undo log 版本链 → 修改时旧数据入 undo，通过 roll_pointer 串联
6. 隐藏字段 → DB_TRX_ID（事务 ID）+ DB_ROLL_PTR（回滚指针）
7. Read View → 事务活跃 ID 列表 + 最大最小事务 ID，判断某版本是否可见
8. 快照读 → 普通 select 走 MVCC
9. 当前读 → select for update 不走 MVCC，要加锁
10. RC vs RR → RC 每次查新建 Read View，RR 事务内只建一次

## 一字一画面

**版** —— 像**档案室的文件修订版**：每次修改都留一份旧稿，按时间码编号叠成一摞，想看旧版就往上翻（本质：undo log 版本链通过 roll_pointer 串联历史版本）

**读** —— 像**免打扰模式看手机**：别人在改文件，你不用拦他，直接读自己打开时的那份快照（本质：快照读通过 Read View 隔离并发修改）

**链** —— 像**红色细绳串档案**：每份旧档案右下角都穿个红绳孔，一拉就牵出上一版（本质：roll_pointer 指向 undo log 中上一个版本）

## 面试话术

"MVCC 就是多版本并发控制，核心是让读操作不用加锁。它的实现主要靠三个东西：undo log 版本链、隐藏字段和 Read View。每次修改数据时，旧版本会写入 undo log，通过 roll_pointer 串成一条链。查询时根据 Read View 判断哪个版本对自己可见。RC 下每次查询都会新建 Read View，所以能读到已提交的最新数据；RR 下事务一开始就固定 Read View，所以能防止不可重复读。"

## 防混补丁

容易把**当前读和快照读**搞混？补丁口诀：**普通 select 是快照，for update 是当前读，快照走 MVCC，当前读要加锁**。

## 追问预演

- **Q**：MVCC 能解决幻读吗？
  **A**：快照读通过 MVCC 能避免幻读，但当前读（for update）需要靠间隙锁或临键锁来解决。
- **Q**：undo log 版本链会一直增长吗？
  **A**：不会，有 purge 线程定期清理不再需要的旧版本，避免无限膨胀。
- **Q**：为什么 RR 级别下还能出现幻读？
  **A**：快照读不会，但当前读会，因为当前读加的是临键锁，特殊场景下可能锁不住新插入的间隙。
