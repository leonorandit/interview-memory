## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 叶 | 叶矮顺，散红黑高 | 叶子有序链表、层高矮、范围快；哈希散、红黑高 |

## 原始知识

- **B+树叶子有序且链表连接**：范围查询高效
- **非叶子只存键值**：扇出大，**层高更低**
- **B+树 vs B树**：B树数据存全部节点，层高更高
- **vs 红黑树**：层高更高，磁盘 IO 更多
- **vs 哈希**：哈希**不支持范围查询**

## 顺口溜

叶子有序链表连，范围查询飞一般；
非叶只存键值轻，三层千万不简单；
哈希只懂等号算，红黑层高 IO 惨。

## 单字锚点 + 防崩推导

**锚点字：叶**

- 扩展口诀：**叶矮顺，散红黑高**
- 为什么选它："叶"是 B+ 树区别于其他结构最核心的特征——数据集中在叶子，且叶子有序+链表
- 推导链：
  1. 叶 → B+ 树**叶子节点**
  2. 叶子 → **有序** + **双向链表** → 范围查询高效（ORDER BY、BETWEEN）
  3. 叶 → 数据只存在叶子 → **非叶子只存键值** → 每页能存更多键 → **扇出大**
  4. 扇出大 → **层高矮** → 磁盘 IO 少（通常 3 层存千万级）
  5. 散 → **哈希索引** → 只支持等值查询 → **不支持范围**
  6. 红黑高 → **红黑树** → 二叉树 → 层高 O(logN) 且底数为 2 → 比 B+ 树高得多 → **磁盘 IO 多**
  7. B树 → 数据存在**所有节点** → 非叶子也存数据 → 扇出小 → **层高更高**

## 一字一画面

- **叶** —— 像秋天银杏叶排成金色长链铺在地上，你顺着叶子走就能从 1 号逛到 100 号（本质：叶子有序+链表，范围查询连续遍历）
- **矮** —— 像只有三层的大商场，电梯按两次就到所有柜台；不像摩天大楼要换乘好几次（本质：B+ 树扇出大，层高矮，IO 少）
- **散** —— 像把快递随机扔进不同编号的筐，找指定单号很快，但让你按日期顺序取出？得把所有筐翻一遍（本质：哈希离散存储，不支持顺序访问）

## 面试话术

1. "MySQL 选 B+ 树主要有三个原因：第一，叶子节点有序而且用链表连起来，范围查询和排序特别高效；第二，非叶子节点只存键值，不存数据，所以每页能存更多指针，树很矮，磁盘 IO 少；第三，相比哈希，B+ 树支持范围查询，相比红黑树，B+ 树扇出大、层高更低。"
2. "B 树的数据存在所有节点里，非叶子也带数据，导致扇出变小，同样数据量下树更高，IO 更多，所以 InnoDB 没有选 B 树。"

## 防混补丁

容易和"B树更好"搞混？记住：**B树每层都载货，B+树轻装把扇开**。

## 追问预演

1. **B+ 树一页多大？为什么是 16KB？** InnoDB 默认 16KB，和磁盘块大小对齐，减少 IO 次数；也可以调，但 16KB 是兼顾读写性能的平衡点。
2. **如果数据量极小，B+ 树还有优势吗？** 数据量极小的情况下，全表扫描可能更快，B+ 树优势不明显；但数据库要应对的是海量数据场景。
3. **LSM 树和 B+ 树怎么选？** LSM 树写优化，适合写多读少；B+ 树读写均衡，适合关系型数据库的混合负载。
