## 原始知识

- InnoDB **页大小16KB**
- 非叶子节点存**键值+指针**
- 单个指针约**14字节**
- 单页约存**1170个指针**
- 叶子节点存**整行数据**
- 假设单行**1KB**约16条
- **三层**约2000万条
- **四层**约200亿条

## 单字锚点 + 防崩推导

**锚点字：亿**

- **扩展口诀**：亿键叶，三层存
- **为什么选它**：三层B+树就能存“亿级以下”的两千万，四层直接跳到两百亿，“亿”是量级跳跃的分水岭
- **推导链**：
  1. 亿 → B+树容量大
  2. B+树大 → 问页多大？→ **16KB**
  3. 页16KB → 非叶子怎么存？→ 只存**键+指针**
  4. 键+指针 → 一个指针约14B → 一页约**1170个指针**
  5. 1170路 → 叶子存整行 → 假设1KB/行 → 一页约**16条**
  6. 三层 = 1170 × 1170 × 16 ≈ **2000万**
  7. 四层 = 再乘1170 ≈ **200亿**

## 一字一画面

**亿 —— 像一座金色金字塔**：塔底是一片绿油油的叶子草原，每片叶子都沉甸甸地载着一整行数据；塔身两层由无数银色钥匙和箭头搭成，钥匙闪闪发光，箭头指向下方。塔尖只有一个入口，却通向千万级数据的深处。（本质：B+树的分层结构，上层稀疏索引，下层密集数据）

## 面试话术

"InnoDB默认页大小是16KB，B+树的非叶子节点只存键值和指针，一个指针大约14字节，所以一页能存大概1170个指针。叶子节点存整行数据，假设一行1KB，那一页就是16条。这样算下来三层B+树就是1170乘1170乘16，大概是两千万条；如果是四层，那就是两百亿级别了。"

## 防混补丁

容易和B树搞混？B树每个节点都存数据，B+树只有叶子存数据，非叶子更瘦，所以路数更多，容量更大。  
**补丁口诀**：B树胖节点，B+瘦索引，叶子才存行，容量翻几倍。

## 追问预演

1. **Q：为什么不用B树而用B+树？**  
   A：B+树非叶子不存数据，同样页大小能存更多指针，树更矮，IO次数更少；且叶子节点链表连接，范围查询更快。

2. **Q：如果行大小不是1KB呢？**  
   A：行越小，每页存越多，容量越大；行越大则相反。极端情况下行超过半页可能产生页溢出。

3. **Q：怎么计算n层的容量？**  
   A：第n层的叶子数 = 1170^(n-1)，总容量 = 1170^(n-1) × 每页行数。
