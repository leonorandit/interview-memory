## 原始知识

- **慢**：开启**慢查询日志**（slow_query_log）定位问题SQL
- **索**：索引优化，**最左前缀**，覆盖索引
- **执**：Explain分析**执行计划**
- **四步调**：SQL改写、表结构优化、分区、分库分表
- **避免select ***：只查需要的列
- **子查询转join**：减少嵌套，提升效率
- **深度分页**：延迟关联、覆盖索引翻页
- **表结构**：合适字段类型、避免NULL、适度反范化
- **分区表**：按范围/哈希分区，减少扫描量
- **分库分表**：水平/垂直拆分，终极方案

## 顺口溜

Explain一看知进退。

## 单字锚点 + 防崩推导

**锚点字：慢**

- **扩展口诀**：慢索执，四步调；慢查抓，Explain看，索引优，SQL改，结构调，拆库表。
- **为什么选它**：SQL调优的起点就是"慢"，慢查询日志是调优的第一入口；"慢"字情绪强烈，面试场景里用户投诉的第一句话就是"这SQL好慢"。
- **推导链**：
  1. 慢 → 慢查询日志（slow_query_log / long_query_time）→ 定位慢SQL
  2. 索 → 索引优化 → 加索引 / 联合索引 / **最左前缀** / 覆盖索引
  3. 执 → Explain执行计划 → 看type、key、rows、Extra
  4. 四步调（调优层次由浅到深）：
     - 第一步：SQL改写 → 避免select * / 子查询转join / 深度分页优化（延迟关联）
     - 第二步：表结构优化 → 字段类型 / 索引设计 / 适度反范化 / 避免NULL
     - 第三步：分区表 → 范围分区 / 哈希分区，减少单表数据量
     - 第四步：分库分表 → 垂直拆分（业务解耦）/ 水平拆分（数据分散）

## 一字一画面

**慢** —— 像[黄色蜗牛爬黑板]：[一只黄色蜗牛在黑板上爬SQL语句，所过之处留下银色黏液痕迹，DBA拿放大镜追着找慢点]（本质：慢是信号，是调优的起点）

**索** —— 像[蓝色书签插书页]：[一本厚书里插满蓝色书签，要找哪页直接翻书签，不用从头翻到尾]（本质：索引是数据的目录，避免全表扫描）

## 面试话术

1. "SQL调优我一般是四步走：先用慢查询日志抓到慢SQL，然后用Explain看执行计划，接着从索引和SQL写法两方面优化，最后才考虑改表结构或者分库分表。"
2. "索引优化要注意最左前缀原则，联合索引(a,b,c)只能用于a、ab、abc的查询条件；另外覆盖索引能避免回表，性能提升很明显。"
3. "深度分页比如limit 100000,10会很慢，可以用延迟关联或者覆盖索引分页来优化；如果单表数据量实在太大，再考虑分区或者分库分表。"

## 防混补丁

容易和**加机器/加缓存**搞混？一句补丁：

> SQL调优是治根，加机器是治标；先Explain改索引，再考虑堆硬件。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 怎么开启慢查询日志？ | set global slow_query_log=on; set global long_query_time=1; |
| 什么情况下索引会失效？ | 函数操作、类型隐转、like '%x'左模糊、or条件、!=/<>
| 覆盖索引是什么？ | 查询字段都在索引里，不需要回表查数据行 |
| 深度分页怎么优化？ | 延迟关联：先查id再join；或者上次最后一条id作为条件 |
| 分库分表后怎么保证唯一ID？ | Snowflake、号段模式、UUID（不推荐） |
