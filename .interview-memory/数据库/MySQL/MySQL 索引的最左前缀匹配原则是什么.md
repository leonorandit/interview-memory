## 原始知识

- 联合索引按**最左列**开始匹配
- 查询条件必须从**索引最左边**字段起
- **跳过左边**字段，右边字段索引失效
- `like '%xx'` **左模糊**导致索引失效
- **范围查询**（`>`、`<`、`between`）后停止匹配
- 索引列顺序**不能中断**
- MySQL 优化器会**调整顺序**，但仅限于等值条件
- 遵循**B+树**有序排列特性

## 顺口溜

左到右，不断点，
跳左边，全白干，
范围来，停后面，
百分号，站左边，索引完。

## 单字锚点 + 防崩推导

**锚点字**：左

**扩展口诀**：左到右，不断点

**为什么选它**：最左前缀匹配原则的精髓就是一个"左"字，所有规则都围绕"从最左边开始"展开。

**推导链**：
1. 锚点字"左" → 联合索引从左到右匹配
2. 从左到右 → 查询条件必须从索引第一列开始
3. 第一列开始 → 跳过第一列，后面列都不走索引
4. 范围查询 → B+树无法确定下一节点，范围后停止匹配
5. like '%xx' → 百分号在左，不知道从哪里开始找，索引失效

## 一字一画面

**左** —— 像**排队检票**：从左到右依次检票，跳过第一个人后面全堵死（本质：B+树按最左列有序组织，必须从入口进入）

**断** —— 像**断桥**：桥中间断了，后面的车全过不去（本质：索引列顺序中断，后续列无法利用索引有序性）

**范** —— 像**红绿灯路口**：过了绿灯可以直行，但范围太大后面没法确定走哪条（本质：范围查询后字段值不确定，无法继续走索引）

## 面试话术

"最左前缀匹配原则就是说，联合索引它会从左到右依次匹配查询条件。如果查询跳过了最左边的字段，那后面的字段就用不上索引了。还有一个要注意的点是，如果中间有范围查询，那范围查询后面的字段也会停止匹配。"

## 防混补丁

容易和**索引列顺序可以任意写**搞混？补丁口诀：**顺序可调是优化器，最左匹配是铁律**。

## 追问预演

- **Q**：`where b=1 and a=2`，索引是`(a,b)`，能用索引吗？
  **A**：能，优化器会自动调整顺序为`a=2 and b=1`，满足最左前缀。
- **Q**：`where a>1 and b=2`，索引是`(a,b)`，`b`能走索引吗？
  **A**：不能，`a`是范围查询，后面`b`停止匹配。
- **Q**：`like 'abc%'`能用索引吗？`like '%abc'`呢？
  **A**：前者能，前缀确定可以走索引；后者不能，百分号在左不知道起点。
