## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 默 | 默可防，幻主从 | 默认 RR，防不可重复读和幻读，兼顾一致性与性能，兼容 statement binlog |

## 原始知识

- **默认级别**：可重复读（**RR**）
- **解决问题**：**不可重复读** + **幻读**（快照读）
- **实现方式**：MVCC + **间隙锁**
- **一致性**：事务内查询**结果稳定**
- **性能**：比 Serializable **好**，比 RC **略差**
- **binlog 兼容**：statement 格式下 RR **更安全**
- **主从一致**：避免 RC + statement binlog 导致**不一致**
- **历史原因**：早期 MySQL **主从复制**依赖 statement 格式
- **RC 风险**：RC + statement 可能记录**不一致顺序**
- **平衡选择**：一致性、性能、主从兼容三者**兼顾**

## 顺口溜

默可防，幻主从，
RR 默认非 RC，
MVCC 加间隙锁，
快照重复读不差，
statement 格式靠它押。

## 单字锚点 + 防崩推导

**锚点字**：默

**扩展口诀**：默可防，幻主从

**为什么选它**："默"就是默认，MySQL 默认隔离级别是 RR，记住"默"就能锁定 RR，再展开为什么选它。

**推导链**：
1. 锚点字"默" → 默认隔离级别
2. 默认 → MySQL 默认 RR（可重复读）
3. 口诀"默可防" → RR 能防不可重复读，也能防幻读（快照读）
4. 怎么防？→ MVCC + 间隙锁
5. MVCC → undo log 版本链 + Read View，快照读不加锁
6. 间隙锁 → 锁定范围，防止幻读插入
7. 口诀"幻主从" → 防幻读 + 兼容主从复制
8. 主从复制 → statement 格式下 RC 有风险
9. RC 风险 → 同一事务内多次读可能读到不同数据，binlog 记录顺序可能不一致
10. 为什么不是 Serializable？→ 性能太差；为什么不是 RC？→ 一致性不足 + binlog 兼容问题

## 一字一画面

**默** —— 像**默认上了三道保险的门**：第一道猫眼（MVCC 看历史），第二道插销（间隙锁挡插入），第三道门牌"statement 兼容"（本质：RR 默认综合了多版本读和范围锁，并兼容传统主从复制）

**防** —— 像**给数据穿了固定颜色的衣服**：事务开始后，所有人穿的都是同色衣服，中途别人换色你也看不见（本质：RR 下 Read View 固定，事务内读取的版本不变）

**从** —— 像**主唱和跟唱的歌词本必须一样**：主库（binlog）和从库执行顺序要一致，RR 减少歧义（本质：statement 格式 binlog 在 RR 下记录顺序更稳定）

## 面试话术

"MySQL 默认的事务隔离级别是可重复读。选择它有几个原因：首先 RR 能解决不可重复读和幻读，通过 MVCC 实现快照读，事务内数据一致；其次早期 MySQL 主从复制多用 statement 格式的 binlog，配合 RC 可能出现主从不一致，RR 更安全。虽然现在 row 格式 binlog 普及了，但 RR 依然是兼顾一致性和性能的平衡选择。"

## 防混补丁

容易把**为什么默认是 RR 和 RC 的区别**搞混？补丁口诀：**默是 RR 防重复，RC 新视读最新，Oracle 用 RC，MySQL 默 RR**。

## 追问预演

- **Q**：现在 binlog 都用 row 格式了，默认级别会改吗？
  **A**：8.0 之后依然默认 RR，但 RC 在 row 格式下确实更安全了，实际项目中根据业务场景可以选 RC。
- **Q**：RR 下用 MVCC 还是锁来防幻读？
  **A**：快照读靠 MVCC，当前读靠间隙锁或临键锁。
- **Q**：如果业务场景对不可重复读不敏感，用 RR 有什么代价？
  **A**：RR 下锁持有时间更长，间隙锁范围更大，并发性能比 RC 略差，可能增加死锁概率。
