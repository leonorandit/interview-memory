# MySQL索引为什么用B+树

## 速记卡片
`矮 | 叶矮顺，散红黑高 | 矮胖顺`

## 原始知识
- B+树：数据只存叶子节点，非叶子节点只存索引键
- 叶子节点通过指针相连，支持高效范围查询和排序
- 多叉树，树高低，磁盘IO次数少
- **易混点**：B树 vs B+树、聚簇 vs 非聚簇
- **追问点**：为什么不用Hash？为什么不用红黑树？页大小16KB怎么影响树高？

## 顺口溜
**叶矮顺，散红黑高**

## 单字锚点 + 防崩推导
- **锚点字**：**矮**
- **扩展口诀**：**矮胖叶相连**
- **为什么选它**：面试官问"为什么用B+树"，本质问的是"怎么减少磁盘IO"，答案就是一个字——**矮**。
- **防崩推导**：
  - 矮 → 多叉树 → 每层存更多键 → 树高低 → 磁盘IO少
  - 矮 → 只有叶子存数据 → 非叶子节点更"瘦"，能存更多键 → 更矮
  - 顺 → 叶子节点链表相连 → 范围查询顺着走，不用回溯
  - 反过来：B树数据散在各层 → 节点臃肿 → 不够矮；红黑树二叉 → 太高；Hash只支持单点 → 不顺

## 一字一画面
- **矮** —— 像**矮胖的仓库货架**：一层能放很多东西，总共就两三层高，伸手就够到。（本质：多叉树，树高低，磁盘IO少）
- **叶** —— 像**只有最底层抽屉有货**：上面几层只贴分类标签，不占地方，所以每层能贴更多标签。（本质：非叶子节点只存键，数据全在叶子，节点存更多键）
- **顺** —— 像**底层抽屉之间有滑道**：从A抽屉直接滑到B抽屉，不用爬回上面重新找。（本质：叶子节点链表相连，范围查询高效）
- **散** —— 像**每个抽屉里既放货又贴标签**：货占了标签的位置，每层标签少了，货架就得搭更高。（本质：B树非叶子节点也存数据，节点能存键更少，树更高）
- **红** —— 像**瘦高的二叉塔**：只有两个分叉，要搭几十层才能装下所有东西，每次拿货要爬很久。（本质：红黑树是二叉树，高度高，磁盘IO多）

## 面试话术
- 问为什么用B+树 → "核心原因是三个字：**矮、叶、顺**。第一，B+树是**矮胖**的多叉树，树高低，查数据时磁盘IO次数少；第二，数据只存在**叶子节点**，非叶子节点只存索引键，这样一层能存更多键，树就更矮；第三，叶子节点之间用指针**相连**，范围查询和排序时顺着链表走，不用回溯。相比之下，B树把数据散在各层，节点能存的键变少，树更高；红黑树是二叉的，高度太高；Hash虽然单查快，但不支持范围和排序。"
- 问B+树 vs B树 → "B+树非叶子节点只存键，数据全在叶子，叶子之间有链表；B树非叶子节点也存数据，没有叶子链表。结果是B+树更矮，范围查询更快。"

## 防混补丁
- 容易和"聚簇索引 vs 非聚簇索引"搞混。B+树是**索引的数据结构**，聚簇/非聚簇是**数据存储方式**。补丁：**B+树是车型，聚簇是停车方式**。
- 容易和"InnoDB vs MyISAM"搞混。InnoDB聚簇索引（数据在叶子），MyISAM非聚簇（叶子存地址）。补丁：**InnoDB货在抽屉，MyISAM抽屉里放地址**。

## 追问预演
- **问**：为什么不用Hash索引？→ Hash只支持等值查询，不支持范围查询和排序，且哈希冲突时性能不稳定。
- **问**：为什么不用红黑树？→ 红黑树是二叉树，高度太高，磁盘IO次数多；B+树是多叉树，更矮胖。
- **问**：16KB页大小有什么影响？→ 一页能存更多键，树更矮；如果存数据，一页存得少，树变高。
