## 原始知识

- **表锁**：锁定**整张表**，MyISAM 默认
- **元数据锁（MDL）**：保护**表结构**
- **行锁**：锁定**单行**，InnoDB 支持
- **记录锁（Record Lock）**：锁定**具体记录**
- **间隙锁（Gap Lock）**：锁定**范围间隙**
- **临键锁（Next-Key Lock）**：记录锁 + **间隙锁**
- **共享锁（S 锁）**：读锁，**可共享**
- **排他锁（X 锁）**：写锁，**独占**
- **意向锁**：表级**意向标记**，避免遍历
- **自增锁**：AUTO_INCREMENT **特殊锁**

## 单字锚点 + 防崩推导

**锚点字**：锁

**扩展口诀**：锁表行，共排两

**为什么选它**："锁"就是题目本身，表锁、行锁、共享锁、排他锁，两大维度（粒度 + 模式）全部涵盖。

**推导链**：
1. 锚点字"锁" → MySQL 锁类型
2. 口诀"锁表行" → 按粒度分：表锁、行锁
3. 表锁 → 表锁（MyISAM）+ 元数据锁 MDL（保护结构）
4. 行锁 → InnoDB 支持，更细粒度
5. 行锁细分 → 记录锁（单行）+ 间隙锁（范围）+ 临键锁（记录+间隙）
6. 口诀"共排两" → 按模式分：共享锁 S、排他锁 X
7. S 锁 → 读锁，多个事务可同时持 S 锁
8. X 锁 → 写锁，独占，与 S/X 都冲突
9. 意向锁 → 表级锁的"预告"，IS/IX，避免加表锁时遍历所有行
10. 自增锁 → 特殊场景，AUTO_INCREMENT 字段专用

## 一字一画面

**表** —— 像**锁了整个会议室**：一个人进去锁门，其他人全在外面等，简单粗暴（本质：表锁一次性锁定整张表，粒度大冲突高）

**行** —— 像**只锁自己坐的椅子**：别人可以坐其他椅子，只有你离开后才释放（本质：行锁只锁定单行记录，粒度小并发高）

**共** —— 像**多人共看一本图书馆的书**：大家都能看，但谁也不能在上面涂改（本质：共享锁 S 允许多事务同时读，互不阻塞）

**排** —— 像**独占试衣间**：里面有人时，别人既不能进也不能看（本质：排他锁 X 独占资源，与其他任何锁都互斥）

## 面试话术

"MySQL 的锁从粒度上分为表锁和行锁。表锁比如 MyISAM 用的表锁和元数据锁 MDL；行锁是 InnoDB 的特点，又分记录锁、间隙锁和临键锁。从模式上分为共享锁 S 和排他锁 X，S 锁用于读可以共享，X 锁用于写必须独占。另外还有意向锁作为表级预告，和自增锁这种特殊场景用的锁。"

## 防混补丁

容易把**记录锁和临键锁**搞混？补丁口诀：**记录锁单行，临键锁前后，防幻读靠临键，改单行用记录**。

## 追问预演

- **Q**：InnoDB 什么时候会升表锁？
  **A**：通常不会主动升级，但全表更新、不走索引导致全表扫描时，可能实际上锁定大量行，效果接近表锁。
- **Q**：意向锁有什么用？
  **A**：加表锁时不需要遍历所有行判断是否有行锁，看意向锁就能快速知道是否有冲突。
- **Q**：间隙锁在 RC 级别下生效吗？
  **A**：RC 级别下一般不用间隙锁，只有 RR 和 Serializable 下才会使用间隙锁来防幻读。
