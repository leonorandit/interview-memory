## 原始知识

- 查询字段**都在索引中**
- **无需回表**查询数据行
- 减少**磁盘 IO**
- Extra 显示 **Using index**
- 索引"覆盖"了查询需求
- 通常用于** select 字段较少**的场景
- 联合索引更容易实现覆盖
- 能显著提升查询性能

## 顺口溜

不回表，少 IO，
Using index 性能优。

## 单字锚点 + 防崩推导

**锚点字**：盖

**扩展口诀**：盖不回，查索引

**为什么选它**："盖"就是覆盖，覆盖索引的核心就是"盖住了查询需求"，不需要再回去查表。

**推导链**：
1. 锚点字"盖" → 覆盖索引
2. 覆盖 → 全部盖住 → 查询需要的所有字段都在索引里
3. 都在索引里 → 不需要回表 → 省一次 IO
4. 不回表 → 查索引树直接返回结果 → 性能高
5. 性能高 → Extra 显示 Using index
6. 怎么实现 → 联合索引包含查询字段，或者 select 只选索引列

## 一字一画面

**盖** —— 像**雨伞盖住全身**：一把伞把从头到脚都遮住，不用再跑回屋拿雨衣（本质：索引包含所有查询字段，直接返回结果，无需回表）

**回** —— 像**外卖送到门口又折返**：快递到了楼下，发现忘了带手机，又跑回楼上拿（本质：回表是根据索引地址再去数据页查一次，多一次 IO）

**树** —— 像**果树上摘苹果**：你站在树下，伸手就能摘到所有要的苹果，不用再去仓库拿（本质：B+树索引的叶子节点直接包含所需字段值）

## 面试话术

"覆盖索引就是说查询语句里要的所有字段，都在索引里面能找到，这样就不需要再回表去查数据行了。你看执行计划的时候，Extra 那一栏会显示 Using index，这就是覆盖索引的标志。它能减少一次磁盘 IO，查询性能会好很多。"

## 防混补丁

容易和**索引下推（ICP）**搞混？补丁口诀：**覆盖是不回表，下推是先过滤，一个省 IO，一个少回表**。

## 追问预演

- **Q**：覆盖索引一定能提升性能吗？
  **A**：大部分情况下能，但如果索引本身很大，或者 select * 导致索引无法覆盖，反而可能不如直接查主键。
- **Q**：怎么设计覆盖索引？
  **A**：把查询条件字段和 select 字段都放到联合索引里，让查询所需列都能从索引中获取。
- **Q**：覆盖索引和聚簇索引的关系？
  **A**：聚簇索引天然覆盖主键查询，因为叶子节点就是数据；二级索引需要包含查询所有列才能实现覆盖。
