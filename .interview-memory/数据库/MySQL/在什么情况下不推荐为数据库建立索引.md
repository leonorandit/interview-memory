# 在什么情况下，不推荐为数据库建立索引？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 少 | 少改重 NULL，五不建 | 数据少/更新多/大字段/低选择/NULL 多 |

## 原始知识

- **数据量太小** 全表扫描更快
- **频繁更新列** 维护开销大
- **低选择性** 性别/状态
- **大文本/BLOB** 索引太大
- **查询很少用** 浪费资源
- **NULL 值过多** 索引效果差
- **函数计算列** 索引失效

## 顺口溜

数据少，更新多；
大字段，NULL 重；
低选择，函数中——
这五样，不建宗。

## 单字锚点 + 防崩推导

- **锚点字**：少
- **扩展口诀**：少改重 NULL，五不建
- **为什么选它**：不建索引的第一反应就是数据“少”。
- **推导链**：
  1. 少 → 数据量少：全表扫描比走索引还快
  2. 改 → 频繁更新：索引维护成本高，写操作变慢
  3. 重 → 大字段重：TEXT/BLOB 太重，索引占用巨大空间
  4. NULL → NULL 值过多：索引区分度低，效果差
  5. 五不建：
     - ① 数据量太小
     - ② 频繁更新的列
     - ③ 大文本/BLOB
     - ④ 低选择性列
     - ⑤ NULL 值过多的列 / 参与函数计算的列

## 一字一画面

- **少** —— 像[小水坑]：小水坑只有几滴水，不需要架桥（索引），直接蹚水过去更快（本质：数据量少时全表扫描成本低于索引查找）
- **改** —— 像[不断翻修的桥]：桥（索引）刚修好，路（数据）就改道，工人不断拆桥重建，反而堵死交通（本质：频繁更新导致索引频繁重建，写性能暴跌）
- **重** —— 像[大象走钢丝]：一头 TEXT 大象试图站在 B+ 树钢丝上，钢丝被压弯，整棵树摇摇欲坠（本质：大字段索引占用空间巨大，降低树效率）

## 面试话术

- “有几种情况我不推荐建索引：一是数据量太小，全表扫描反而更快；二是频繁更新的列，维护索引开销大；三是大文本和 BLOB 类型，索引会占很大空间。”
- “另外，选择性低的列比如性别、NULL 值过多的列，以及参与函数计算的列，建了索引效果也不好。”

## 防混补丁

容易和“所有列都该建索引”搞混？**索引是空间换时间**，但写操作要维护，大字段和低区分度是性价比黑洞。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 数据量少是多少？ | 通常几百条以下，具体看表宽度和查询条件。 |
| 函数计算为什么不行？ | 因为对列用了函数后，优化器无法使用普通 B+ 树索引，要用函数索引（MySQL 8.0+）。 |
| NULL 值多怎么办？ | 考虑默认值替代 NULL，或看业务是否必须索引该列。 |
