## 速记卡片

| 久 | 久锁胀，三问题 | 锁阻塞、undo膨胀、主从延迟 |

## 原始知识

- **长时间持锁**：其他事务阻塞，**并发下降**
- **undo log膨胀**：历史版本积累，**空间暴涨**
- **主从延迟**：长事务未提交，**binlog不发送**
- **连接池耗尽**：长事务占连接不释放
- purge线程清理慢 → 老版本**无法回收**
- 查询可能**越来越慢**

## 顺口溜

## 单字锚点 + 防崩推导

**锚点字：久**

- **扩展口诀**：久锁胀，三问题
- **为什么选它**：长事务的核心特征就是“久”，而“久”必然带来锁、胀、延迟三大恶果
- **推导链**：
  1. 久 → 事务跑很久 → 锁一直不释放 → **锁阻塞** → 其他事务排队 → **并发下降**
  2. 久 → 数据被改了又改 → 每个旧版本都要存 → **undo log膨胀** → 磁盘暴涨 → purge清理不过来
  3. 久 → 事务不提交 → binlog不发送到从库 → **主从延迟** → 从库数据滞后
  4. 久 → 连接一直被占用 → **连接池耗尽** → 新请求拿不到连接
  5. 久 + 胀 → 历史版本链过长 → 连**查询都变慢** → 一致性读要遍历版本链

## 一字一画面

**久 —— 像一头霸占茅坑的犀牛**：它蹲在一扇金门（行锁）前半天不走，后面排队的动物越积越多（并发下降）；它身后堆满了发臭的旧草料（undo膨胀），清洁工（purge线程）怎么扫都扫不完；远处的信鸽（binlog）因为它没办完事，一直不能飞出去报信（主从延迟）；连门口等位的椅子（连接池）也被它的人占光了。（本质：长事务是资源占用的黑洞，越久危害越大）

## 面试话术

"长事务主要有三个问题：第一是锁长时间不释放，导致其他事务阻塞，并发能力急剧下降；第二是undo log不断累积，历史版本purge不掉，磁盘空间暴涨，查询也会变慢；第三是主从延迟，因为事务不提交，binlog就不会发送到从库。另外连接池也可能被占满。所以线上一定要监控和避免长事务。"

## 防混补丁

容易和大事务搞混？大事务是操作数据量大，长事务是持续时间长。大事务不一定久，长事务不一定大，但两者都该避免。  
**补丁口诀**：大事务量重，长事务时久，量大可能快，时久一定毒。

## 追问预演

1. **Q：怎么发现长事务？**  
   A：查information_schema.INNODB_TRX表，看trx_started时间，超过阈值就告警。

2. **Q：怎么避免或解决长事务？**  
   A：业务上拆小事务，设置innodb_lock_wait_timeout，用pt-kill自动杀长时间事务。

3. **Q：undo膨胀后怎么恢复？**  
   A：等长事务提交或回滚后，purge线程会逐渐清理。不能手动删，只能等自动回收。
