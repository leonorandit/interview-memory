## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 悲 | 悲先加，乐后比 | 悲观先上锁，乐观后比对 |

## 原始知识

- **悲观锁**：先加锁再操作，select for update
- **乐观锁**：后比对再提交，version/CAS机制
- **悲观适用**：写多读少场景
- **乐观适用**：读多写少场景
- **乐观隐患**：**ABA问题**
- **悲观代价**：锁等待、并发低
- **乐观代价**：重试开销、CPU忙
- **锁粒度**：行锁/表锁均可悲观
- **实现方式**：version字段或时间戳
- **数据库支持**：InnoDB行锁即悲观锁

## 顺口溜

悲先加锁再办事，
乐后比对才提交。
写多读少用悲观，
读多写少用乐观。

## 单字锚点 + 防崩推导

**锚点字：悲**

- **扩展口诀**：悲先加，乐后比；写多悲，读多乐。
- **为什么选它**：悲观锁是数据库原生支持的锁机制，面试常考基础概念，"悲"字情绪色彩鲜明，容易与"乐观"形成对比记忆。
- **推导链**：
  1. 悲 → 悲观锁 → 心态悲观（认为冲突必然发生）
  2. 悲观 → 先加锁 → select for update / 行锁 / 表锁
  3. 先加锁 → 写多读少场景（高冲突）
  4. 乐 → 乐观锁 → 心态乐观（认为冲突很少）
  5. 乐观 → 后比对 → version字段 / CAS / 提交前检查
  6. 后比对 → 读多写少场景（低冲突）
  7. 乐观隐患 → ABA问题（version可解）

## 一字一画面

**悲** —— 像[灰衣保安堵大门]：[灰衣保安先拦住所有人，查完身份证才放行，门一关谁也进不来]（本质：先阻塞后操作，安全第一）

**乐** —— 像[绿衣快递放门口]：[绿衣快递员直接把包裹放门口拍照就走，回头比对照片看有没有人动过]（本质：先操作后验证，效率优先）

## 面试话术

1. "悲观锁就像去银行取钱，先排队拿号锁柜台，办完才走；乐观锁就像自助存款，先存进去，系统后台对账发现冲突再重试。"
2. "写多读少用悲观锁，因为冲突概率高，先加锁反而省掉重试开销；读多写少用乐观锁，因为大部分操作不会冲突，省去锁等待。"
3. "乐观锁要注意ABA问题，比如version从1变2又变回1，CAS检测不到中间变化，可以加时间戳或状态机解决。"

## 防混补丁

容易和**共享锁/排他锁**搞混？一句补丁：

> 悲观乐观是策略，共享排他是实现；悲观可用排他锁，乐观不用数据库锁。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 乐观锁怎么实现？ | version字段或CAS，更新时比对version，一致才提交 |
| ABA问题怎么解决？ | 加时间戳、状态机，或用AtomicStampedReference |
| MySQL怎么实现悲观锁？ | select ... for update（排他锁）或 select ... lock in share mode（共享锁） |
| 乐观锁适合什么业务？ | 库存扣减、订单状态流转等读多写少且冲突概率低的场景 |
| 两种锁各有什么缺点？ | 悲观锁并发低、易死锁；乐观锁冲突高时重试多、CPU浪费 |
