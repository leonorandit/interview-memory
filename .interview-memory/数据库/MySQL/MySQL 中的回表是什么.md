## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 回 | 回主查，二步走 | 非聚查到主键，再回聚簇查数据 |

## 原始知识

- **回表**：非聚簇索引查到**主键值**
- 再拿主键值去**聚簇索引**查整行数据
- 产生**额外 IO**
- **覆盖索引**可避免回表
- 回表是"**二次查询**"的过程

## 顺口溜

非聚查到主键号，转身再回聚簇找；
覆盖索引能省掉，一次拿够不绕道。

## 单字锚点 + 防崩推导

**锚点字：回**

- 扩展口诀：**回主查，二步走**
- 为什么选它："回"本身就是回表的字眼，像"回家"、"返回"，天然带"折返"的动作感
- 推导链：
  1. 回 → 回表
  2. 回表 → **返回**到主键索引 → 所以是"**回主**"
  3. 回主查 → **两步走**：第一步在非聚簇索引查主键，第二步回聚簇索引查数据
  4. 回表 → 产生**额外 IO** → 性能下降
  5. 避免回表 → **覆盖索引** → 索引字段覆盖查询所需全部列

## 一字一画面

- **回** —— 像外卖员到了小区门口，发现只拿到楼栋号，还得折返回大门重新登记上楼（本质：多跑了一趟，产生额外 IO）
- **覆** —— 像外卖柜一次取出餐盒、饮料、筷子全齐，不用再跑第二趟（本质：覆盖索引让非聚簇索引直接满足查询）

## 面试话术

1. "回表就是先走非聚簇索引拿到主键ID，然后再拿着这个ID去聚簇索引里查整行数据，相当于查了两次B+树。"
2. "覆盖索引可以避免回表，因为查询的字段都在非聚簇索引里，叶子节点上的信息已经够用了，不需要再回主键索引。"

## 防混补丁

容易和"索引下推"搞混？记住：**回表是多跑一趟，下推是提前过滤**。

## 追问预演

1. **回表一定慢吗？** 不一定，如果主键查的是缓存页，或者数据量很小，开销不明显；但高并发下回表次数多会放大 IO 压力。
2. **怎么判断 SQL 有没有回表？** 看 Explain 的 Extra 列，出现 "Using index" 说明没回表；出现 "Using index condition" 或没有 Using index 则可能有回表。
3. **覆盖索引的代价是什么？** 索引列变多，索引体积变大，维护成本和空间占用上升，需要权衡。
