## 速记卡片

深 | 深子延，三解法 | 子查询+延迟关联+覆盖索引/游标

## 原始知识

- limit 1000000,10 **慢原因**（先扫描 1000010 行再丢弃）
- **延迟关联**（先查 id 再 join）
- **子查询**定位 id 范围
- **覆盖索引**避免回表
- 记录上次查询位置（**游标/书签**）
- **ES/缓存**辅助

## 顺口溜

深度分页像挖坑，  
limit 越后越慢腾。  
子查询缩范围，  
延迟关联省回表。  
书签标记下一页，  
ES 缓存来助攻。

## 单字锚点 + 防崩推导

**锚点字**：深

**扩展口诀**：深子延，三解法

**为什么选它**：深字直接点出深度分页的本质——offset 越大性能越差。

**推导链**：
深 → 深度分页 → limit offset,size → offset 很大时，MySQL 要先扫描 offset+size 行再丢弃前面 offset 行 → 主键索引扫描大量数据还要回表 → 解法一：子查询先定位 id 范围，再取数据，减少扫描 → 解法二：延迟关联，先查 id 列表再 join 原表，利用覆盖索引避免回表 → 解法三：记录上次查询位置（游标/书签），下次用 where id>last_id limit size，直接定位 → 辅助方案：ES/缓存

## 一字一画面

**深** —— 像从深井里打水：limit 1000000,10 就像摇辘轳先空转 100 万圈才接 10 桶水，手臂都酸了（CPU/IO 高）。子查询就像先问井底水位计（id 索引）现在第 100 万桶在哪，直接跳到那个深度；延迟关联是两个人配合，一个人在井口看水位计记下位置（查 id），另一个人直接去那个位置打水（join 原表）；书签法是每次在桶上贴标签，下次直接从标签位置继续打。（本质：避免大 offset 导致的大量无效扫描和回表）

## 面试话术

MySQL 深度分页主要问题是 limit offset,size 在执行时，引擎要先扫描 offset+size 行然后丢弃 offset 行，offset 越大越慢，而且还要回表查数据。常见的解决方法有三个：一是子查询，先定位到目标 id 范围再取数据；二是延迟关联，先通过覆盖索引查 id 列表，再 join 原表取完整数据，避免回表；三是用游标或者书签，记录上一次查询的最后一条记录 id，下次用 where id>last_id 直接定位。另外也可以考虑用 ES 或者缓存来做分页查询。

## 防混补丁

容易和普通分页搞混？普通分页 offset 小没问题，深度分页是 offset 极大，记住"浅分页直接 limit，深分页要拐弯"。

## 追问预演

1. **延迟关联具体怎么写 SQL？**
   答：`SELECT * FROM t JOIN (SELECT id FROM t ORDER BY id LIMIT 1000000,10) tmp ON t.id=tmp.id`
2. **为什么覆盖索引能避免回表？**
   答：索引树包含查询所需字段，不需要回聚簇索引查数据行。
3. **游标分页有什么缺点？**
   答：不支持跳页，只能上一页/下一页，且对排序字段有要求。
