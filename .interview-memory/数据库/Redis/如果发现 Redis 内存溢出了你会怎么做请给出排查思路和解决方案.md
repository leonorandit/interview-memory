# 如果发现 Redis 内存溢出了？你会怎么做？请给出排查思路和解决方案

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| **查** | 查大删，五解法 | 先查原因再大 key 删除，扩容、淘汰、优化、拆分、关持久化 |

## 原始知识

- **排查**：
  - `info memory` 看内存使用详情
  - `--bigkeys` / `redis-cli --bigkeys` 找**大 key**
  - 分析 **key 过期策略**，看有没有大量未过期 key
  - 查看**客户端连接数**，排查连接泄露
- **解决方案**：
  - **扩容**：加内存或加节点
  - **设置 maxmemory + 淘汰策略**：volatile-lru / allkeys-lru 等
  - **删除大 key 用 unlink**：非阻塞删除，避免 del 阻塞主线程
  - **优化数据结构**：hash / zset 拆分，string 改 hash
  - **拆分 hash 和 zset**：避免单 key 过大
  - **关闭不必要的持久化**：减少 fork 时的内存翻倍压力

## 顺口溜

**查大删，五解法**
> 查内存、找大 key，unlink 删除不堵塞，扩容淘汰优拆分

## 单字锚点 + 防崩推导

- **锚点字**：**查**
- **扩展口诀**：查大删，五解法
- **为什么选它**：遇到内存溢出，第一反应必须是"查"——不查清楚原因就动手，等于盲医乱开刀
- **推导链**：
  1. 查 → `info memory` → 先看 used_memory / used_memory_rss / maxmemory
  2. 查 → `--bigkeys` → 找出大 key，比如一个 string 100MB、一个 zset 百万成员
  3. 查 → 过期策略 → 是不是没有设置 TTL，或大量 key 同时过期引发雪崩
  4. 删 → 找到大 key 用 **unlink** 删（不是 del，del 会阻塞）
  5. 五解法 → 扩容（加机器）、淘汰（maxmemory-policy）、优化（改数据结构）、拆分（大 hash 拆小）、关持久化（减少 COW 内存压力）

## 一字一画面

- **查** —— 像**医生先拍 CT 再动刀**：病人喊头疼，医生不先拍片就开刀，那是草菅人命（本质：排查优先，info memory 和 bigkeys 是 CT 片）
- **大** —— 像**仓库里藏着一根巨大的钢管**：明明仓库没满，但一根管子占了半个空间，其他货进不来（本质：大 key 导致内存碎片化或集中占用）
- **删** —— 像**用叉车慢慢搬走钢管**：不用手搬（del 阻塞），用叉车（unlink 异步线程）慢慢挪（本质：unlink 后台线程释放内存，不阻塞主线程）

## 面试话术

> "遇到 Redis 内存溢出，我的思路是'查大删'——先查 info memory 看使用详情，再用 bigkeys 找大 key，然后用 unlink 非阻塞删除。解决方案上，五步走：扩容、设置 maxmemory 和淘汰策略、优化数据结构、拆分大 hash/zset、必要时关闭持久化减少 fork 内存压力。"

## 防混补丁

- **容易和什么搞混**：用 del 删大 key，导致 Redis 主线程阻塞，服务卡顿
- **补丁口诀**：大 key 要删用 unlink，del 一按全卡死

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| info memory 里哪个指标最关键？ | used_memory（Redis 实际使用）和 used_memory_rss（操作系统看到的），rss 远大于 used 说明碎片严重 |
| 有哪些淘汰策略？ | noeviction（直接报错）、allkeys-lru、volatile-lru、allkeys-random、volatile-ttl 等 |
| 为什么 fork 会占更多内存？ | 写时复制（COW），fork 后父进程修改页会复制一份，内存峰值可能翻倍 |
| unlink 和 del 的区别？ | del 在主线程同步释放内存，unlink 把 key 从字典摘掉，内存由后台线程异步释放 |
