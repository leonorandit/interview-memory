## 单字锚点 + 防崩推导

**锚点字**：锁

**扩展口诀**：锁红看，Lua保

**为什么选它**：锁是问题核心，所有实现都围绕"锁"字展开。

**推导链**：
锁 → Redis 分布式锁 → 加锁：SET key value NX EX（原子设置+过期） → 为什么要过期？防止客户端崩溃导致死锁 → 但业务执行时间可能超过过期时间 → Redisson 看门狗自动续期 → 解锁：必须判断 value 是否是自己的再删除 → 用 Lua 脚本保证判断+删除的原子性 → 主从架构下主节点宕机可能丢锁 → RedLock 方案：向多个独立 Redis 节点申请锁，过半成功才算获取

## 一字一画面

**锁** —— 像智能门锁配定时炸弹：租客（客户端）用密码（SET NX EX）开门，炸弹 10 分钟后爆炸（过期释放）。但如果租客还没搬完家，Redisson 看门狗就像一只机械犬，每隔几秒就按一下延迟按钮（续期）。退租时必须核对身份证再拆炸弹，Lua 脚本就是把"核对+拆除"写进同一个不可拆分的操作手册。（本质：利用 Redis 单线程特性实现互斥，过期机制防止死锁）

## 面试话术

Redis 实现分布式锁主要有这么几个要点。加锁用 SET 命令带上 NX 和 EX 参数，这样设置 key 和过期时间是原子操作，避免旧版 SETNX+EXPIRE 的非原子问题。然后 Redisson 的看门狗可以自动续期，防止业务没执行完锁就过期了。解锁的时候必须用 Lua 脚本，先判断 value 是不是自己设置的，再删除，保证原子性。另外主从架构下如果主节点挂了还没同步到从节点，锁会丢失，所以有人提出了 RedLock 方案，不过争议比较大。

## 防混补丁

容易和本地锁搞混？本地锁只防同进程线程竞争，分布式锁防跨进程竞争，记住"本地锁看门内，分布式锁看门外"。

## 追问预演

1. **为什么不用 SETNX+EXPIRE？**
   答：两个命令非原子，中间崩溃会导致死锁。
2. **Redisson 看门狗续期周期是多少？**
   答：默认锁过期时间的 1/3，即看门狗每 10 秒续期一次（默认 30 秒过期）。
3. **RedLock 有什么问题？**
   答：需要多个独立 Redis 实例，时钟漂移、网络延迟都可能导致误判，实现复杂。
