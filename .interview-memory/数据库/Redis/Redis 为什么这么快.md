## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 单 | 单内模，五原因 | Redis 快 = 单线程 + 内存 + IO 多路复用 + 高效数据结构 + 简单协议 |

## 原始知识

- **纯内存操作**：数据全在内存，读写接近内存速度
- **单线程**：避免上下文切换和锁竞争，命令顺序执行
- **IO 多路复用**：epoll 处理海量连接，单线程同时服务多个客户端
- **高效数据结构**：SDS、跳表、压缩列表、整数数组等专为性能设计
- **RESP 协议简单**：文本协议解析开销极低
- **C 语言实现**：贴近硬件，执行效率高

## 顺口溜

> 内存读飞快，单线程无锁，epoll 多路来。

## 单字锚点 + 防崩推导

- **锚点字**：单
- **扩展口诀**：单内模，五原因
- **为什么选它**：「单」是 Redis 最具辨识度的特征——单线程模型
- **推导链**：
  1. 「单」→ 单线程 → 无锁、无上下文切换
  2. 「内」→ 纯内存 → 不用读磁盘
  3. 「模」→ IO 多路复用模型（epoll）→ 单线程也能扛万连接
  4. 还有啥？→ 高效数据结构 + 简单协议 + C 语言
  5. 五个原因凑齐：单线程、内存、IO 多路复用、数据结构、协议简单

## 一字一画面

**单** —— 像一条单行的高速公路：只有一条车道（单线程），没有并道抢行的车祸（无锁竞争），所有车按顺序飞驰；公路建在内存芯片上（纯内存），收费站是自动化闸机（epoll），不用人工找零（RESP 简单协议），所以车再多也能快速通过。（本质：用单线程消灭并发开销，用 IO 多路复用弥补连接数）

## 面试话术

「Redis 快主要有五个原因：第一纯内存操作；第二是单线程模型，避免了上下文切换和锁竞争；第三是用 epoll 做 IO 多路复用，单线程也能处理大量连接；第四是底层数据结构是专门优化过的，比如 SDS、跳表；第五是 RESP 协议很简单，解析开销小。」

## 防混补丁

- **容易和 Memcached 搞混**：Memcached 也是内存缓存，但只支持简单 KV，多线程模型；Redis 单线程支持丰富数据结构。
- **补丁口诀**：Redis 单线数据多，Memcached 多线 KV 薄。

## 追问预演

1. **Q：单线程会不会成为瓶颈？**
   A：对于纯内存操作，CPU 通常不是瓶颈；但如果某个命令耗时太长（如 keys *），会阻塞整个实例。
2. **Q：Redis 6.0 为什么引入多线程？**
   A：多线程只用于网络 IO 读写，命令执行仍是单线程，目的是进一步提升网络吞吐量。
3. **Q：epoll 和 select 的区别？**
   A：epoll 没有文件描述符数量限制，采用事件驱动而非轮询，时间复杂度 O(1) 对比 select 的 O(n)。
