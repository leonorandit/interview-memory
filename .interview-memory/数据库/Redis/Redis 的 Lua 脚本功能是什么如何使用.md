## 速记卡片

原 | 原评调，三用途 | Lua 原子执行，网络往返省

## 原始知识

- Lua 脚本在 Redis 中**原子执行**
- **不会被其他命令打断**
- 用途：**原子操作**（库存扣减）
- 用途：**分布式锁释放**
- 用途：**复杂逻辑合并**
- **减少网络往返**（RTT）
- 命令：**EVAL** / **EVALSHA**
- 脚本缓存：**SHA1 摘要复用**
- 注意：**避免耗时操作阻塞 Redis**
- Redis 是**单线程**，脚本执行期间不处理其他命令

## 顺口溜

EVAL 进去原子跑，库存扣减锁释放好。
SHA1 缓存省带宽，耗时太长 Redis 恼。

## 一字一画面

**原** —— 像一扇旋转门一次只进一人：前面的人（Lua 脚本）不转完，后面的人（其他命令）卡在外面干瞪眼（本质：单线程原子执行，中间不插任何命令）

**评** —— 像厨师一次炒完三盘菜端上桌：不用炒一盘端一盘跑三趟（本质：多条命令合并执行，减少网络往返）

**调** —— 像一把钥匙插进锁孔拧开：判断锁是不是自己的 + 拧开删除，两个动作必须一气呵成（本质：复杂逻辑的原子组合，如分布式锁释放）

## 面试话术

"Redis 的 Lua 脚本最核心的价值是原子性，因为 Redis 是单线程的，一个 Lua 脚本执行期间不会处理其他命令。典型场景就是库存扣减和分布式锁释放，比如先判断锁是不是自己加的，再删锁，这两步必须原子做。"

"调用方式有两种：EVAL 直接传脚本字符串，EVALSHA 传 SHA1 摘要。生产里第一次用 EVAL，Redis 会缓存脚本，后面就直接用 EVALSHA 走缓存了，省带宽。但要注意脚本里不能写耗时逻辑，否则会阻塞整个 Redis。"

## 防混补丁

**容易和 MULTI-EXEC 事务搞混**？

补丁口诀：**"Lua 是预制菜一锅出，事务是排队买单逐个刷"**

- Lua：脚本在 Redis 服务端原子执行，中间不插命令
- MULTI-EXEC：客户端把命令排队，最后一次性提交，中间可能被别人插队（除非加 WATCH）

## 追问预演

**Q：Lua 脚本执行失败会回滚吗？**
A：不会。Redis 不提供事务回滚，脚本里的命令要么都执行，但出错不会撤销已执行的。

**Q：EVALSHA 缓存被清了怎么办？**
A：返回 NOSCRIPT 错误，客户端需要降级用 EVAL 重新传脚本，Redis 会重新缓存。

**Q：Lua 脚本里能调用 Redis 所有命令吗？**
A：大部分可以，但有一些限制，比如不能执行 ADMIN 类命令（SHUTDOWN、CONFIG 等）。

**Q：怎么避免 Lua 脚本阻塞 Redis？**
A：脚本逻辑要精简，避免循环遍历大数据集；复杂计算放客户端做；可以用 Redis 函数（Redis 7.0+）替代部分场景。
