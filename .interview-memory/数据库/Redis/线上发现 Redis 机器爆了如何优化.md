# 线上发现 Redis 机器爆了，如何优化？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 慢 | 慢命分，五优化 | 先查慢查询和内存，再分五路优化 |

## 原始知识

- **slowlog** 查慢查询
- **info memory** 看内存使用
- **bigkeys** 找内存大户
- **淘汰策略** LRU/LFU
- **过期策略** 定时+惰性
- **拆分大key** 化整为零
- **碎片整理** active defrag
- **横向扩容** 加机器/集群
- **读写分离** 降主库压力
- **hash存小对象** 省内存
- **监控告警** 提前发现

## 顺口溜

慢查询，查 bigkey；
命淘汰，分碎片；
加机器，读分离。

## 单字锚点 + 防崩推导

- **锚点字**：慢
- **扩展口诀**：慢命分，五优化
- **为什么选它**：机器爆了，第一反应是“慢”查询拖垮的，也可能是内存满了变“慢”。
- **推导链**：
  1. 慢 → 先排查：slowlog、info memory、bigkeys
  2. 命 → 内存优化：淘汰策略、过期策略
  3. 分 → 拆分+分散：拆分大key、碎片整理、读写分离、集群扩容
  4. 五优化 → 淘汰、过期、拆大key、碎片整理、扩容/读写分离

## 一字一画面

- **慢** —— 像[蜗牛背着重壳]：一只红色蜗牛背着巨大的 key 壳，爬得越来越慢，壳里掉出 bigkey（本质：内存爆满导致响应变慢，先抓慢查询和 bigkey）
- **命** —— 像[命悬一线]：Redis 内存条被拉到满格，红色警报闪烁，LRU 淘汰器像剪刀一样剪断不常用的线（本质：通过淘汰和过期策略给内存续命）
- **分** —— 像[分蛋糕]：一个巨大的 key 蛋糕被切成小块，碎片被吸尘器吸走，分别装进不同的 Redis 服务器盒子（本质：拆分大key、整理碎片、分布式分摊）

## 面试话术

- “如果发现 Redis 机器爆了，我会先排查，用 slowlog 和 info memory 定位问题，再用 redis-cli --bigkeys 找大 key。”
- “优化上，我会调整淘汰策略和过期策略，拆分大 key，开启 active defrag 整理内存碎片，必要时扩容集群或做读写分离。”
- “另外，小对象尽量用 hash 而不是 string 存储，能显著节省内存。”

## 防混补丁

容易和 MySQL 爆库搞混？**Redis 是内存问题**，先看 memory 和 bigkeys，不是看磁盘和慢 SQL。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| LRU 和 LFU 有什么区别？ | LRU 看最近一次访问时间，LFU 看访问频率。 |
| active defrag 是什么原理？ | jemalloc 分配导致碎片，Redis 后台线程整理连续内存。 |
| 大 key 怎么拆？ | 按业务维度拆分，比如 hash 拆成多个小 hash，list 分页。 |
| 怎么避免内存碎片？ | 用 jemalloc 优化分配，开启 active defrag，避免大量小对象零散分配。 |
