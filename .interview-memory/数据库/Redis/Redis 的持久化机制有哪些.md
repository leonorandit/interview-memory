# Redis 的持久化机制有哪些？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| **快** | 快AOF，两机制 | RDB 恢复快但可能丢数据，AOF 更安全但文件大 |

## 原始知识

- **RDB**：**快照** 持久化，Save / Bgsave 生成二进制文件
  - **紧凑** 二进制，恢复**速度快**
  - 可能**丢数据**（两次快照之间崩溃）
- **AOF**：**Append Only File**，记录**写命令**
  - 数据**更安全**，默认每秒 fsync
  - 文件**大**，恢复**慢**（重放命令）
- **混合持久化**：Redis 4.0+，**RDB + AOF**
  - 先 RDB **全量**，再 AOF **增量**
- **AOF 重写**：**bgrewriteaof** 压缩命令，减小体积

## 顺口溜

**快AOF，两机制**
> RDB 快照恢复快，AOF 命令更安全，混合两者最周全

## 单字锚点 + 防崩推导

- **锚点字**：**快**
- **扩展口诀**：快AOF，两机制
- **为什么选它**：RDB 最大的卖点就是"恢复速度快"，"快"字直接对应 RDB 的核心优势，看到"快"先想到 RDB，再想到和它互补的 AOF
- **推导链**：
  1. 快 → RDB 快照 → 二进制紧凑文件 → 恢复像拷贝文件一样快
  2. 快 → 但也有代价 → 两次快照之间宕机 = 数据丢失（丢最后一次快照后的数据）
  3. AOF →  Append Only File → 记写命令 → 像记流水账 → 更安全（每秒刷盘）
  4. AOF → 但文件大、恢复慢 → 要 bgrewriteaof 压缩
  5. 两机制 → Redis 4.0+ 混合持久化 → 开头 RDB 全量 + 后面 AOF 增量 → 又快又安全

## 一字一画面

- **快** —— 像**相机快门咔嚓一声**：瞬间拍下整个内存的全景照片，存成一张高清小图，恢复快得像翻开相册（本质：fork + bgsave 生成二进制快照）
- **A** —— 像**记账本逐笔记录**：每花一笔钱就写一行，翻账本很慢但一笔不漏，丢了还能按账本重做（本质：append 写命令日志）
- **混** —— 像**相册封面 + 后面补拍的照片**：先贴一张全家福（RDB），之后再拍的就贴小快照（AOF），翻开既快又全（本质：混合持久化 RDB-AOF）

## 面试话术

> "Redis 主要有两种持久化机制——RDB 和 AOF。RDB 是快照，二进制紧凑、恢复速度快，但可能丢数据；AOF 是记录写命令，数据更安全、最多丢一秒，但文件大、恢复慢。Redis 4.0 之后支持混合持久化，先用 RDB 做全量再用 AOF 做增量，兼顾速度和安全性。"

## 防混补丁

- **容易和什么搞混**：把 RDB 和 AOF 的优缺点记反，或以为只能二选一
- **补丁口诀**：RDB 快但丢，AOF 全但慢，混合两手抓

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Bgsave 的原理？ | fork 子进程，写时复制（COW），子进程写 RDB 文件，父进程继续服务 |
| AOF 重写期间数据怎么保证不丢？ | 重写缓冲区（aof_rewrite_buf），子进程写新 AOF 时父进程同时把新命令写进缓冲区和旧 AOF，最后追加 |
| 生产环境怎么配置？ | 通常两者都开，RDB 做备份，AOF 做主恢复；或开混合持久化 |
| RDB 和 AOF 同时存在时怎么恢复？ | 优先用 AOF，因为数据更完整；Redis 启动时自动判断 |
