## 单字锚点 + 防崩推导

**锚点字**：失

**扩展口诀**：失误可，三问题

**为什么选它**：失字概括了分布式锁的各种失效场景——误删、丢失、不可重入。

**推导链**：
失 → 锁失效 → 场景一：误删除 → 业务执行时间超过锁过期时间，锁自动释放，其他客户端获取锁并执行，原客户端业务完成后可能误删别人的锁 → 场景二：不可重入 → 同一线程已经持有锁，再次获取时会阻塞，导致死锁 → 场景三：锁丢失 → 主从架构下主节点加锁后宕机，锁信息未同步到从节点，从节点晋升为主后锁丢失 → 附加：程序异常退出未释放锁（靠过期时间兜底）、看门狗续期异常（续期失败导致锁过期）

## 一字一画面

**失** —— 像租的充电宝超时归还：你还没充完电（业务未完），充电宝自动解锁被别人借走（锁过期），你回来找不到自己的充电宝，可能把别人的拔了（误删）。不可重入就像你租了一个柜子，想再开第二层却发现系统不让你重复租用。主从延迟就像你在 A 店租的充电宝，A 店突然倒闭数据没同步到 B 店，B 店不知道你已经租了，又把同一个充电宝租给别人（锁丢失）。（本质：分布式环境下时间、状态同步、异常处理的多重挑战）

## 面试话术

Redis 分布式锁常见的问题有这么几个。第一是锁误删除，业务执行时间超过了锁的过期时间，锁自动释放了，这时候别的客户端拿到了锁，等原客户端执行完可能会把别人的锁删掉。第二是不可重入，同一个线程如果已经持有锁，再次获取会失败，可能导致死锁。第三是主从延迟导致锁丢失，主节点加锁后还没同步到从节点就挂了，从节点升主后锁就没了。另外还有程序异常退出没释放锁，以及看门狗续期失败的情况。

## 防混补丁

容易和死锁搞混？死锁是循环等待，锁失效是锁本身没了或错了，记住"死锁是互相掐，失效是绳子断"。

## 追问预演

1. **怎么防止误删？**
   答：给每个锁设置唯一 value（如 UUID+线程 id），解锁时先判断再删。
2. **怎么实现可重入？**
   答：Redisson 用 Hash 结构，key 是锁名，field 是线程标识，value 是重入次数。
3. **看门狗续期失败怎么办？**
   答：业务逻辑要做幂等，或者合理评估过期时间，不依赖无限续期。
