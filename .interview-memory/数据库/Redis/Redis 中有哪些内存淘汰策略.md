# Redis 中有哪些内存淘汰策略？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 淘 | 淘近LFU，八策略 | 淘 = 淘汰；近 = LRU/LFU；八 = 8 种策略 |

## 原始知识

- **noeviction**：默认不淘汰，直接报错
- **allkeys-lru**：所有 key 按 LRU 淘汰
- **allkeys-random**：随机淘汰
- **volatile-lru**：过期 key 中 LRU 淘汰
- **volatile-random**：过期 key 中随机淘汰
- **volatile-ttl**：过期 key 中快过期的淘汰
- **allkeys-lfu**：所有 key 按 LFU 淘汰
- **volatile-lfu**：过期 key 中 LFU 淘汰

## 顺口溜

```
全随机，全近全频；过随机，过近过频过TTL；不淘就报错
```

## 单字锚点 + 防崩推导

| 锚点字 | 扩展口诀 | 为什么选它 | 推导链 |
|--------|----------|------------|--------|
| 淘 | 淘近LFU，八策略 | "淘" = 淘汰，面试官问淘汰策略，第一个字就是"淘" | 淘 → 淘汰策略 → allkeys / volatile 两类 → lru / random / ttl / lfu 四种方式 → 加 noeviction 共 8 种 |

**防崩推导**：只记得"淘"字时，推导链如下：
1. "淘"就是淘汰，内存不够了要"淘"掉一些 key
2. 淘哪些？——**所有 key**（allkeys）还是**只淘带过期时间的**（volatile）？
3. 怎么淘？——按**最近最少用**（LRU）、**最少用**（LFU）、**随机**（random）、**快过期的**（TTL）
4. 不想淘？——**noeviction**，直接报错
5. 2 类 × 4 方式 = 8 种（注意 random 和 ttl 没有 LFU 的变种，但总体就是 8 个）

## 一字一画面

- **淘** —— 像周末跳蚤市场摊主：摊位（内存）满了，必须"淘"掉一些东西才能继续进货（本质：内存不足时的强制清理行为）
- **近** —— 像超市里刚上架就没人买的酸奶：按"最近最少被拿"的标准下架（本质：LRU，最近最少使用）
- **频** —— 像书店里的畅销书榜单：卖得少的（使用频率低）先撤柜（本质：LFU，使用频率最少）
- **随** —— 像抽奖箱里随机抽一个倒霉蛋：不分青红皂白，抽到谁就是谁（本质：random，完全随机淘汰）

## 面试话术

> Redis 的内存淘汰策略一共有 8 种，默认是 noeviction，就是不淘汰，内存满了直接报错。其他的可以分成两大类：一类是针对所有 key 的，一类只针对设置了过期时间的 key。每一类下面又有 LRU、LFU、random 这几种方式，还有 volatile-ttl 是专门淘汰快要过期的 key。实际生产中比较常用的是 allkeys-lru 和 volatile-lru。

## 防混补丁

- **容易和什么搞混？** 把 LRU 和 LFU 搞混，或者以为所有策略都会淘汰所有 key。
- **补丁口诀**：LRU 看"多久没摸"，LFU 看"摸过几次"；allkeys 全杀，volatile 只杀带过期时间的。

## 追问预演

| 面试官追问 | 简要答案 |
|-----------|----------|
| 默认的淘汰策略是什么？ | noeviction，不淘汰，内存满了直接返回错误 |
| allkeys-lru 和 volatile-lru 有什么区别？ | allkeys-lru 是在所有 key 里按 LRU 淘汰；volatile-lru 只在设置了过期时间的 key 里按 LRU 淘汰 |
| LRU 和 LFU 的区别是什么？ | LRU 看最后一次使用的时间，越久没用越容易被淘汰；LFU 看使用次数，使用次数越少越容易被淘汰 |
| 生产中推荐用哪种策略？ | 一般推荐 allkeys-lru，适用于大多数缓存场景 |
