# Redis 在生成 RDB 文件时如何处理请求？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 写 | 写时复，bgsave | fork 子进程，COW 不阻塞 |

## 原始知识

- **bgsave** 后台保存
- **fork 子进程** 生成 RDB
- **主进程继续** 处理请求
- **Copy-on-Write** 写时复制
- **fork 时共享** 父进程内存
- **写操作复制页** 只拷修改的页
- **不阻塞主进程** 用户无感知
- **save 命令** 阻塞主进程/不推荐
- **AOF 重写** 同样 fork 子进程

## 顺口溜

bgsave 后台跑，fork 子进程快照；
写时复制页，主进程不挡道。

## 单字锚点 + 防崩推导

- **锚点字**：写
- **扩展口诀**：写时复，bgsave
- **为什么选它**：生成 RDB 时核心问题是“写”怎么办？写时复制就是答案。
- **推导链**：
  1. 写 → 写时复制（Copy-on-Write）
  2. 时 → 时机：bgsave 触发时
  3. 复 → 复制：写操作复制修改的内存页
  4. bgsave → 后台保存，fork 子进程
  5. 子进程读共享内存生成 RDB，主进程继续处理请求

## 一字一画面

- **写** —— 像[复写纸]：一张透明复写纸盖在 Redis 内存书上，子进程透过纸看书，主进程要写时，复写纸把那一页复制一份给主进程改（本质：写操作触发内存页复制，子进程仍读旧页）
- **时** —— 像[时钟分叉]：一个红色时钟突然分叉出两个指针，旧指针静止记录快照，新指针继续走动处理请求（本质：fork 瞬间内存状态冻结给子进程）
- **复** —— 像[复印机]：复印机只复印被修改的那几页纸，没被修改的继续共享原件（本质：只复制修改的页，节约内存）

## 面试话术

- “Redis 生成 RDB 用的是 bgsave，它会 fork 一个子进程，子进程读取共享内存生成快照，主进程继续处理请求。”
- “如果主进程发生写操作，会用 Copy-on-Write 机制，只复制被修改的内存页，这样既能保证快照一致性，又不阻塞服务。”

## 防混补丁

容易和 AOF 搞混？**RDB 是快照**，fork 子进程；AOF 是日志，也是 fork 子进程重写，但平时是追加写。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Copy-on-Write 有什么缺点？ | 如果写操作多，内存页复制多，内存占用会膨胀。 |
| 为什么不用 save？ | save 阻塞主进程，期间所有请求都卡住。 |
| AOF 重写也是 COW 吗？ | 是的，同样 fork 子进程，重写期间新命令写到 AOF 缓冲区和重写缓冲区。 |
