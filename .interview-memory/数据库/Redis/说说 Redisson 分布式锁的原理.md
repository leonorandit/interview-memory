## 原始知识

- 基于 **Redis Hash** 结构
- **Lua脚本** 原子加锁
- 命令是 **hincrby + pexpire**
- **看门狗** 自动续期
- 默认 **30秒** 过期，**10秒** 续一次
- **可重入锁** 同线程计数+1
- **RedLock** 多节点加锁（有争议）
- 解锁用 **Lua判断+del**
- 只有 **持有锁的线程** 才能释放

## 单字锚点 + 防崩推导

**锚点字：看**

- **扩展口诀**：看Lua，三原理
- **为什么选它**："看"就是"看门狗"，Redisson最有特色的就是自动续期机制，看到"看"就想起看门狗
- **推导链**：
  - 看 → 看门狗 → 看门狗干啥的？→ **自动续期**，防止业务没执行完锁就过期了
  - 多久续？→ 默认30秒过期，**1/3时间（10秒）** 续一次
  - 除了看门狗还有啥？→ **Lua原子脚本**（hincrby加锁+pexpire设过期）
  - 还有啥？→ **可重入**（同一个线程多次加锁，计数+1，释放时计数-1到0才删）
  - 三个原理：**Lua原子性、看门狗续期、可重入计数**
  - 多节点版本 → RedLock（Redis作者提出，但业界有争议，认为用Redisson单节点+主从/哨兵就够了）

## 一字一画面

**看** —— 像[院子里的狼狗看大门]：大狼狗（看门狗线程）趴在 Redisson 锁的大门口，每隔10秒站起来巡一圈（续期），看看院子里的人（业务线程）干完活没有。只要人还在干活，狼狗就拍拍门把锁的时间再延长30秒。人走了（释放锁），狼狗也回去睡觉。（本质：后台定时任务自动延长锁过期时间）

**Lua** —— 像[一气呵成写毛笔字]：蘸墨、起笔、运笔、落款，中间不能被打断。Redis执行Lua脚本就像写一幅一气呵成的大字，hincrby加锁和pexpire设过期两个命令在一个脚本里，要么一起成功，要么一起不执行，谁也插不进来。（本质：利用Redis单线程特性保证原子性）

## 面试话术

"Redisson分布式锁原理我总结为三点：第一用Lua脚本原子加锁，hincrby加锁同时pexpire设过期时间；第二看门狗机制自动续期，默认锁30秒过期，看门狗每10秒检查续一次；第三是可重入，同一个线程多次加锁计数会累加，释放时减到零才真正删锁。"

## 防混补丁

容易和 **"SET NX EX"** 搞混？
- SET NX EX是**最基础的Redis分布式锁**，不能续期、不能重入
- Redisson在此基础上加了**看门狗+可重入+阻塞等待**
- **补丁口诀**：SET NX EX是基础款，Redisson是豪华版，多了看门狗和重入

容易和 **"ZooKeeper分布式锁"** 搞混？
- ZK用**临时顺序节点+Watcher监听**，是 CP 模型
- Redisson用**Redis+Lua**，是 AP 模型，性能更高
- **补丁口诀**：ZK靠节点和监听，Redisson靠Lua和看门狗

## 追问预演

**Q1：看门狗续期如果业务线程挂了怎么办？**
A：线程挂了看门狗也停止续期，锁到期自动释放，不会死锁。

**Q2：Redisson锁是阻塞还是非阻塞的？**
A：默认阻塞，可以设置等待时间和锁租期；拿不到锁会自旋重试。

**Q3：RedLock到底靠不靠谱？**
A：有争议。Redis作者提出，但批评者认为时钟偏移、网络分区会导致问题。实际生产一般用单节点+哨兵/集群，配合业务幂等。

**Q4：可重入锁怎么实现的？**
A：Hash结构，key是锁名，field是线程标识（UUID+线程ID），value是重入计数。加锁时hincrby计数+1，解锁时-1，为0时del。
