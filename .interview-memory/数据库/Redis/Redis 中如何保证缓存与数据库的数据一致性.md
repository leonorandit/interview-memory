## 原始知识

- **Cache Aside**：旁路缓存，常用
- 先更新DB **再删缓存**（推荐）
- 先删缓存 **再更新DB**（有脏读风险）
- **Read Through**：读穿透，缓存无则加载
- **Write Through**：写穿透，同步更新缓存
- **Write Behind**：异步写回，先写缓存再异步刷DB
- **延迟双删**：删缓存→更新DB→休眠→再删缓存
- **Canal** 监听binlog异步删缓存
- 没有 **绝对一致**，只有 **最终一致**

## 单字锚点 + 防崩推导

**锚点字：一**

- **扩展口诀**：一删延，四策略
- **为什么选它**："一"就是"一致性"，缓存一致性问题，看到"一"就想起"删缓存"和"延迟"
- **推导链**：
  - 一 → 一致性 → 缓存和DB不一致怎么办？→ 核心思路是**删缓存**而不是更新缓存
  - 为什么删而不是更新？→ 并发时**更新缓存容易覆盖**，删缓存下次读时重新加载更安全
  - 但删了还有问题？→ 延迟双删：先删缓存 → 更新DB → **休眠几百毫秒** → 再删一次缓存，把中间读到的旧缓存清掉
  - 还有更省心的？→ **Canal监听MySQL binlog**，DB变了自动异步删缓存
  - 四种策略是哪四个？→ **Cache Aside（旁路）、Read Through（读穿透）、Write Through（写穿透）、Write Behind（异步写回）**
  - Cache Aside最常用：读时先查缓存，没有则查DB写入缓存；写时先更新DB再删缓存

## 一字一画面

**一** —— 像[两个人对表对不上]：左边的人（缓存）戴着手表，右边的人（数据库）也戴着手表。两个人表走快了或走慢了，时间就对不上（不一致）。最简单的办法是——把左边人的表摘掉（删缓存），让他直接看右边人的表（查DB）。但摘表一瞬间左边人可能还在看旧时间，所以等几秒再摘一次（延迟双删）。最聪明的办法是雇一个监工（Canal）盯着右边人的表，只要右边人调了时间，监工立刻跑过去把左边人的表摘掉。（本质：以数据库为准，通过删缓存或异步通知保证最终一致）

**延** —— 像[红绿灯的黄灯缓冲]：绿灯变红灯前亮3秒黄灯，给正在过马路的人一个缓冲。延迟双删里的"休眠"就像这黄灯，给数据库更新和旧缓存读取之间留一个时间差，把漏网之鱼清掉。（本质：用时间窗口覆盖并发读写的竞态条件）

## 面试话术

"缓存一致性没有银弹，一般来说最常用的是Cache Aside模式，也就是更新数据库之后删缓存。为了降低并发下的脏读概率，可以用延迟双删——更新完数据库后休眠几百毫秒再删一次缓存。如果要更自动化，可以用Canal监听MySQL的binlog，数据库有变更就异步去删对应的缓存键。"

## 防混补丁

容易和 **"先删缓存再更新DB"** 搞混？
- 先删缓存再更新DB，如果更新DB之前另一个线程来读，会**把旧数据又写回缓存**，导致长期不一致
- **推荐是先更新DB再删缓存**，即使删失败，也只是多一次DB查询，不会脏
- **补丁口诀**：先DB后删稳，先删后DB脏

容易和 **"更新缓存"** 搞混？
- 写频繁时**更新缓存**会被频繁覆盖，且缓存计算可能复杂
- **删缓存**更简单，下次读时自然加载最新值
- **补丁口诀**：写时删缓存，读时懒加载

## 追问预演

**Q1：先更新DB再删缓存，如果删缓存失败怎么办？**
A：可以用消息队列重试，或者Canal监听binlog异步删，保证最终删除成功。

**Q2：延迟双删的休眠时间怎么定？**
A：一般几百毫秒，根据主从延迟或业务读耗时来估算，也可以异步线程池代替同步休眠。

**Q3：Write Behind 有什么优缺点？**
A：写性能最高，因为只写缓存。但可能丢数据（缓存挂了没刷回DB），适合写极多、容忍短暂不一致的场景。

**Q4：Canal方案有什么缺点？**
A：引入额外组件，有延迟（秒级），且需要维护Canal服务的高可用。

**Q5：为什么不用分布式事务保证强一致？**
A：2PC性能太差，Redis不支持XA，且缓存是加速层，通常接受**最终一致性**。
