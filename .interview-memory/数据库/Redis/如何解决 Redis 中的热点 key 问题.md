# 如何解决 Redis 中的热点 key 问题？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| **热** | 热分本，五解法 | 热点 key 用拆分、本地缓存、读写分离、限流、预热来治 |

## 原始知识

- **热点 key**：某个 key **被大量请求访问**，如秒杀商品信息、明星微博
- **本地缓存**：Guava / Caffeine + Redis **二级缓存**，挡一波流量
- **key 拆分**：一个热点 key 拆成 **key_1 / key_2 / key_n**，分散到不同槽
- **读写分离**：**主从架构** 分散读压力，从节点扛查询
- **限流降级**：**熔断** 保护 Redis，防止打崩
- **缓存预热**：**提前加载**，分散访问时间，避免集中爆发

## 单字锚点 + 防崩推导

- **锚点字**：**热**
- **扩展口诀**：热分本，五解法
- **为什么选它**："热"就是热点 key 本身，看到"热"字立刻想到"某个 key 被万人围观的滚烫场景"
- **推导链**：
  1. 热 → 热点 key → 某个 key 被大量请求集中访问
  2. 热 → 分 → key 拆分（key_1, key_2...），把万人排队拆成五个入口
  3. 热 → 本 → 本地缓存（Guava/Caffeine），家门口先挡一波
  4. 热 → 读写分离 → 主写从读，多个从节点分担查询压力
  5. 热 → 限流降级 → 请求太多就熔断，保护 Redis 不崩
  6. 热 → 预热 → 提前加载，别让流量集中在同一秒

## 一字一画面

- **热** —— 像**明星演唱会入口**：几万人挤一个门，检票口（Redis）被踩扁（本质：单节点 QPS 有上限，单 key 集中访问）
- **分** —— 像**把演唱会拆成五个场馆同时开**：每个场馆人数骤减，检票口压力分散（本质：key 拆分 + hash tag 分散槽位）
- **本** —— 像**小区保安先查业主卡**：不用每次都去派出所（Redis）查户口，门口先挡住 80% 的查询（本质：本地二级缓存降低 Redis 访问频次）

## 面试话术

> "热点 key 问题是某个 key 被高频访问导致 Redis 单节点压力过大。我的解决思路是'热分本'三个字——热就是识别热点，分是 key 拆分或读写分离，本是本地缓存。再加上限流降级做保护、缓存预热分散流量，总共五种手段组合拳。"

## 防混补丁

- **容易和什么搞混**：把热点 key 和缓存击穿搞混，热点 key 是"某个 key 访问量太大"，击穿是"key 过期了 DB 被打穿"
- **补丁口诀**：热点是太忙，击穿是刚好过期，病因不同药方不同

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 如何发现热点 key？ | Redis 4.0+ 有 hotkeys 参数，或借助 monitor / 业务日志 / 客户端统计 |
| 本地缓存和 Redis 怎么保持一致性？ | 设置较短 TTL + 主动失效通知 / 消息队列同步更新 |
| key 拆分后怎么保证读取到完整数据？ | 拆分的是读热点，比如商品信息不变，随机读 key_1~key_n 内容相同 |
| 限流怎么实现？ | 单机用 Guava RateLimiter，分布式用 Redis + Lua 脚本做令牌桶 |
