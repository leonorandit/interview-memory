## 原始知识

- **单线程原因**：
  - 避免锁竞争和上下文切换开销
  - 内存操作 CPU 不是瓶颈，多线程收益低
  - 顺序执行天然原子性，逻辑简单
- **6.0 多线程**：
  - 仅用于网络 IO 的读写解析
  - 命令执行仍保持单线程
  - 配置项：`io-threads-do-reads yes`
  - 线程数建议：CPU 核数，默认 4 线程
- **对比核心**：单线程保命令安全，多线程解网络瓶颈

## 顺口溜

> 单线无锁逻辑简，六版 IO 多线添，执行仍是一线牵。

## 单字锚点 + 防崩推导

- **锚点字**：线
- **扩展口诀**：线令简，6加IO
- **为什么选它**：「线」直接对应 Redis 最核心的架构特征——单线程
- **推导链**：
  1. 「线」→ 单线程 → Redis 经典设计
  2. 「令」→ 命令执行 → 单线程顺序执行，天然原子
  3. 「简」→ 逻辑简单 → 无锁、无并发 Bug
  4. 「6」→ Redis 6.0 → 引入多线程
  5. 「加 IO」→ 只加在网络 IO，命令执行还是单线程
  6. 为啥不改执行层？→ 保证命令原子性，避免锁和竞态

## 一字一画面

**线** —— 像一根红色的单根棉线穿珠子：一颗一颗按顺序穿（单线程命令执行），不会打结（无锁竞争），穿珠人不用换来换手（无上下文切换）。后来珠子太多了，穿珠人只负责穿（命令执行），叫了几个助手专门从筐里拿珠子和放好成品（6.0 IO 多线程），但穿珠的那根红线始终只有一根。（本质：命令执行串行保安全，IO 并行提吞吐）

## 面试话术

「Redis 设计为单线程，主要是因为它的操作都在内存上，CPU 本来就不是瓶颈，单线程可以避免锁竞争和上下文切换的开销，而且顺序执行逻辑非常简单。到了 6.0 版本，引入多线程主要是为了解决网络 IO 的瓶颈，多线程只负责读写和协议解析，真正的命令执行还是单线程，这样既能提升吞吐量，又不会影响命令的原子性。」

## 防混补丁

- **容易和 Memcached 多线程搞混**：Memcached 是多线程处理命令，Redis 命令执行始终单线程。
- **补丁口诀**：Memcached 多线跑命令，Redis 多线只搬数据，命令一线统到底。

## 追问预演

1. **Q：单线程怎么利用多核 CPU？**
   A：部署多个 Redis 实例，每个实例绑定一个核，或者使用 6.0+ 的 IO 多线程。
2. **Q：io-threads 配置要注意什么？**
   A：建议设置为 CPU 核数，默认 4；只建议在有大量短连接或高网络带宽场景开启。
3. **Q：多线程 IO 会有线程安全问题吗？**
   A：不会，因为命令执行阶段仍然是单线程串行，多线程只负责读写缓冲区。
4. **Q：为什么不用多线程执行命令？**
   A：内存操作已经很快，多线程加锁开销可能反而降低性能；且会破坏原子性保证，增加复杂度。
