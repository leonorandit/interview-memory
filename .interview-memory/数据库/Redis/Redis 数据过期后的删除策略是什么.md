# Redis 数据过期后的删除策略是什么？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 懒 | 懒定随，三策略 | 惰性省 CPU，定期省内存，淘汰是兜底 |

## 原始知识

- **惰性删除**：访问时检查过期并删除，节省 CPU 但浪费内存
- **定期删除**：每隔一段时间随机抽查部分 key 删除，平衡 CPU 和内存
- **内存淘汰策略**：当内存不足时按策略淘汰，如 LRU/LFU/allkeys-lru 等
- Redis 实际采用**惰性 + 定期**结合

## 顺口溜

```
懒猫查岗，定时抽查，内存不够就淘汰
```

## 单字锚点 + 防崩推导

| 锚点字 | 扩展口诀 | 为什么选它 | 推导链 |
|--------|----------|------------|--------|
| 懒 | 懒猫查岗才动手 | "懒" = 惰性删除，不主动干活 | 懒 → 惰性删除（被动触发）→ 定期删除（主动抽查）→ 淘汰策略（最后兜底） |

**防崩推导**：只记得"懒"字时，推导链如下：
1. Redis 数据过期了，key 不会立刻消失——肯定有个"偷懒"的机制
2. 什么时候才删？——你去查它（访问）的时候——这叫**惰性删除**
3. 光惰性不行，过期 key 堆积怎么办？——Redis 每隔一段时间随机抽查——这叫**定期删除**
4. 如果内存满了，过期 key 也扛不住——还有**内存淘汰策略**兜底

## 一字一画面

- **懒** —— 像一只橘猫趴在 Redis 上：只有你去摸它（访问 key）的时候，它才懒洋洋地爬起来把过期的 key 叼走扔掉（本质：被动触发，节省 CPU，但过期 key 可能长期占内存）
- **定** —— 像保洁阿姨定时推清洁车：每隔固定时间随机抽查几个房间（随机抽查部分 key），发现垃圾就清掉（本质：主动抽查，CPU 和内存的平衡）
- **随** —— 像超市临期食品专柜：内存不够时按规则（LRU/LFU）把"卖不出去"的 key 下架（本质：兜底机制，内存告急时强制清理）

## 面试话术

> Redis 处理过期 key 主要是两种策略配合：一个是惰性删除，就是你访问这个 key 的时候，Redis 才会检查它有没有过期，过期就删掉；另一个是定期删除，就是 Redis 每隔一段时间随机抽查一部分 key，把过期的清理掉。这两个结合起来，既不会一下子把所有过期 key 删掉卡死 CPU，也不会让过期 key 一直占着内存。如果内存实在不够了，还会触发内存淘汰策略。

## 防混补丁

- **容易和什么搞混？** 很多人以为 key 一到期 Redis 就马上删掉（主动删除），实际上 Redis 是"懒"的，不会立刻删。
- **补丁口诀**：过期≠立删，Redis 是懒猫不是闹钟。

## 追问预演

| 面试官追问 | 简要答案 |
|-----------|----------|
| 定期删除多久执行一次？ | 默认每 100ms 执行一次，每次随机抽查 20 个 key |
| 惰性删除的缺点是什么？ | 过期 key 如果一直不被访问，会一直占用内存 |
| 如果定期删除没删完，内存满了怎么办？ | 触发内存淘汰策略，按配置的规则淘汰 key |
| 为什么不是定时删除所有过期 key？ | CPU 开销太大，Redis 是单线程，会影响正常请求 |
