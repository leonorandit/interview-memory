# Redis 主从复制的实现原理是什么？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 同 | 同命部，三阶段 | 全量同步+部分重同步+命令传播 |

## 原始知识

- **全量同步** RDB 快照
- **缓冲区命令** 同步期间写入
- **首次连接** 必须全量
- **部分重同步** PSYNC
- **replication backlog** 复制积压缓冲区
- **offset 偏移量** 断线比对
- **命令传播** 主写异步发从
- **读写分离** 主写从读
- **心跳检测** repl-ping-replica-period

## 顺口溜

同全量，部重连；
命传播，读写分。

## 单字锚点 + 防崩推导

- **锚点字**：同
- **扩展口诀**：同命部，三阶段
- **为什么选它**：主从复制的核心就是“同”步数据。
- **推导链**：
  1. 同 → 同步：主从之间同步数据
  2. 全量同步（同）：首次连接或无法部分同步时，RDB 快照+缓冲区命令
  3. 部分重同步（部）：断线重连后，如果 offset 在 backlog 范围内，只补发缺失命令
  4. 命令传播（命）：主节点每执行一个写命令，异步发送给从节点
  5. 三阶段 → 全量同步 → 部分重同步（或直接进入命令传播）→ 持续命令传播

## 一字一画面

- **同** —— 像[复印机同步复印]：主节点是一本红皮书，复印机一次性复印整本给新从节点（全量），之后主书每改一字，复印机就同步改从书上的字（本质：全量初始化+持续同步）
- **部** —— 像[补作业]：从节点作业本掉了几页，老师 backlog 缓冲区还有备份，只对缺的几页进行补印，不用重做整本（本质：断线重连后只补发缺失命令）
- **命** —— 像[命令接力棒]：主节点每写一条命令，就像传接力棒一样异步传给从节点，从节点接过棒执行相同命令（本质：命令传播保证主从最终一致）

## 面试话术

- “Redis 主从复制分三个阶段：第一次连接做全量同步，主节点生成 RDB 并发送给从节点，同时把期间的写命令存到缓冲区；”
- “断线重连时会尝试部分重同步 PSYNC，如果复制偏移量在 backlog 范围内就直接补发命令；正常状态下主节点把每个写命令异步传播给从节点，实现读写分离。”

## 防混补丁

容易和 MySQL 主从搞混？**Redis 是全内存复制**，用 RDB 快照+命令传播，没有 binlog 概念，backlog 是环形缓冲区。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| backlog 太小会怎样？ | 断线时间长，offset 被覆盖，只能重新全量同步。 |
| 主从延迟怎么解决？ | 监控 offset 差值，业务上避免读写分离后立即读从库。 |
| 从节点能级联吗？ | 可以，从节点还可以有自己的从节点，减轻主节点压力。 |
