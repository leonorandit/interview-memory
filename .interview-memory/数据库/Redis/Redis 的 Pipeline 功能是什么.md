## 速记卡片

批 | 批一发，三优势 | 客户端缓冲多条命令，一次性收发

## 原始知识

- Pipeline 是**批量发送命令**
- 客户端**缓冲多条命令**
- **一次性发送**，**一次性接收结果**
- **减少网络 RTT** 往返时间
- n 条命令从 **n 次 RTT** 降到 **1 次 RTT**
- **不保证原子性**
- 与 **MULTI-EXEC 事务** 不同
- 适合**大量独立命令**场景
- 场景：**批量写入**
- 场景：**批量查询**

## 顺口溜

客户端里攒命令，一次发送一次收。
RTT 从 n 变到一，不保证原子是软钩。

## 单字锚点 + 防崩推导

- **锚点字**：批（批量）
- **扩展口诀**：批一发，三优势
- **为什么选它**：Pipeline 的本质就是批量处理，"批"字直接点题
- **推导链**：
  1. 批 → 批什么？→ 批的是命令，客户端攒一堆命令一起发
  2. 批 → 怎么发？→ 一次发送，一次接收所有结果
  3. 批 → 好处是什么？→ 省 RTT，n 次网络往返变成 1 次
  4. 批 → 有什么限制？→ 不保证原子性，中间可能被别人插命令
  5. 批 → 和事务区别？→ Pipeline 只是打包发送，MULTI-EXEC 才是原子提交
  6. 批 → 适合什么场景？→ 大量独立命令，比如批量写日志、批量查缓存
  7. 批 → 不适合什么？→ 命令之间有依赖关系或需要原子性保证的

## 一字一画面

**批** —— 像快递员把十个包裹塞进一个大纸箱：一趟车拉走，不用跑十趟（本质：客户端缓冲合并命令，减少网络传输次数）

**发** —— 像机关枪连射子弹：扳机扣住，子弹（命令）连续出膛，最后一起报靶（本质：命令连续发送，结果批量返回）

**优** —— 像高铁代替绿皮火车：同样跑一千公里，时间从 20 小时缩到 4 小时（本质：n 次 RTT 压缩到 1 次，网络延迟大幅下降）

## 面试话术

"Pipeline 就是客户端把多条 Redis 命令攒起来一次性发过去，服务端处理完再一次性把结果返回来。核心好处是省网络 RTT，比如 100 条命令本来要 100 次往返，用了 Pipeline 只要 1 次。"

"但要注意 Pipeline 不保证原子性，它只是打包发送，中间别的客户端命令还是能插进来。如果需要原子性，得用 MULTI-EXEC 事务或者 Lua 脚本。"

## 防混补丁

**容易和 MULTI-EXEC 事务搞混**？

补丁口诀：**"Pipeline 是快递合箱发货快，事务是银行转账要么成都成"**

- Pipeline：只是减少网络往返，不保证原子隔离
- MULTI-EXEC：命令排队后原子提交，中间不会被别人插队

## 追问预演

**Q：Pipeline 和 Lua 脚本有什么区别？**
A：Pipeline 减少网络 RTT 但不保证原子性；Lua 脚本在服务端原子执行，也能减少 RTT，但可能阻塞 Redis。

**Q：Pipeline 适合多大的批量？**
A：一般几百到几千条，太大的话客户端内存和服务端响应缓冲都会承压，可以分批 Pipeline。

**Q：Pipeline 命令中有失败的怎么办？**
A：Pipeline 会返回每个命令的结果，失败的命令返回错误，但不会影响其他命令执行。

**Q：集群模式下 Pipeline 能用吗？**
A：需要所有命令落在同一个 slot 或同一个节点上，跨 slot 需要拆分多个 Pipeline。
