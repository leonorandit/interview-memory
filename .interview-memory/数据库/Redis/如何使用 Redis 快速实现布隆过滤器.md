## 速记卡片

## 原始知识

- **Bitmap 位数组**：用二进制位标记元素
- **多个哈希函数**：一个元素映射多个位，降低冲突
- **Redis4.0+ RedisBloom 模块**：直接提供 BF.ADD/BF.EXISTS
- **Redisson 实现**：Java 客户端封装布隆过滤器
- **False Positive**：判断「存在」可能误判；判断「不存在」一定正确
- **不能删除**：位可能被多个元素共享，删除会误伤
- **适用场景**：缓存穿透防护、URL 去重、海量数据判重

## 顺口溜

> 布隆哈希位，存在可能骗，不在肯定无。

## 单字锚点 + 防崩推导

- **锚点字**：布
- **扩展口诀**：布哈希，可错无
- **为什么选它**：「布」是布隆过滤器的首字，一听就知道在讲这个结构
- **推导链**：
  1. 「布」→ 布隆过滤器 → 空间效率极高的概率型数据结构
  2. 「哈希」→ 多个哈希函数 + Bitmap 位数组
  3. 「可错」→ 判断存在可能误报（False Positive）
  4. 「无」→ 判断不存在一定不存在
  5. 为什么快？→ Redis Bitmap 位操作是 O(1)，纯内存

## 一字一画面

**布** —— 像一块布满小灯泡的巨大灯牌：每个元素经过好几把不同颜色的哈希刷子（多哈希），在灯牌上点亮几盏灯。检查的时候看这几盏灯是否都亮着——如果有一盏没亮，肯定没来过（不在一定无）；如果全亮着，可能是真的，也可能是别人刚好点了同样的灯（存在可能骗）。（本质：用多个哈希降低冲突概率，用位数组节省空间）

## 面试话术

「用 Redis 实现布隆过滤器，最直接的是用 RedisBloom 模块，4.0 以上版本可以加载，提供 BF.ADD 和 BF.EXISTS 命令。原理就是 Bitmap 加多个哈希函数，一个元素映射到位数组的几个位置上。它的特点是判断不存在一定准确，判断存在可能误判，而且不能删除元素。很适合做缓存穿透的防护。」

## 防混补丁

- **容易和 HashSet 搞混**：HashSet 不会误判，但空间占用大；布隆过滤器省空间，但会误判。
- **补丁口诀**：HashSet 准但胖，布隆瘦但骗；要空间选布隆，要准确选 Set。

## 追问预演

1. **Q：布隆过滤器的误判率怎么降低？**
   A：增加位数组长度、增加哈希函数个数，但会牺牲空间和 CPU。
2. **Q：为什么不能删除？**
   A：一个位可能被多个元素共享，删除会把其他元素的标记也清掉。
3. **Q：除了 RedisBloom，还有什么实现方式？**
   A：可以用 Redisson 封装，或者自己用 String 类型的 Bitmap 位操作实现（SETBIT/GETBIT）。
4. **Q：怎么用于缓存穿透防护？**
   A：请求先过布隆过滤器，不存在直接返回，存在再走缓存/数据库。
