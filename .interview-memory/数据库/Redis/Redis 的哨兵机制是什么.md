# Redis 的哨兵机制是什么？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 监 | 监故选，三作用 | 监控+故障转移+选主 |

## 原始知识

- **Sentinel** 高可用方案
- **监控** Monitoring
- **通知** Notification
- **自动故障转移** Auto failover
- **主观下线** SDOWN
- **客观下线** ODOWN
- **多数 Sentinel** 达成一致
- **选举新主** 优先级/偏移量/runid
- **客户端连 Sentinel** 获取主节点

## 顺口溜

Sentinel 三作用，监故选；
主观变客观，选主按优先。

## 单字锚点 + 防崩推导

- **锚点字**：监
- **扩展口诀**：监故选，三作用
- **为什么选它**：哨兵就是“监”控兵，盯着主从节点。
- **推导链**：
  1. 监 → 监控 Monitoring：Sentinel 持续监控主从节点健康
  2. 故 → 故障转移 Auto failover：发现故障自动处理
  3. 选 → 选举新主节点：优先级最高 → 复制偏移量最大 → runid 最小
  4. 三作用 → 监控、通知、自动故障转移
  5. SDOWN（单个 Sentinel 主观认为下线）→ ODOWN（多数 Sentinel 客观确认下线）

## 一字一画面

- **监** —— 像[监控摄像头]：一个红色摄像头 360 度旋转，盯着 Redis 主从服务器，红灯闪烁表示报警（本质：Sentinel 持续监控节点状态）
- **故** —— 像[救护车]：主节点房子着火了，救护车 Sentinel 冲过来，把从节点提拔成新房主（本质：自动故障转移，替换失效主节点）
- **选** —— 像[选班长]：三个从节点举手，老师 Sentinel 按成绩优先级、作业完成度偏移量、学号 runid 来点名（本质：多维度选举新主节点）

## 面试话术

- “Redis 哨兵是官方的高可用方案，主要做三件事：监控主从节点、发送通知、以及自动故障转移。”
- “当 Sentinel 发现主节点不可用时，会先主观下线 SDOWN，然后多个 Sentinel 投票达成客观下线 ODOWN，再按优先级和复制偏移量选举新主节点。”

## 防混补丁

容易和 Cluster 搞混？**哨兵是一主多从+高可用**，Cluster 是多主多从+数据分片，哨兵不拆分数据。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Sentinel 怎么保证高可用？ | 通常部署奇数个 Sentinel，至少 3 个，防止脑裂。 |
| 选举新主的规则？ | 先 slave-priority，再复制偏移量 offset 最大，最后 runid 最小。 |
| 客户端怎么知道新主节点？ | 连 Sentinel 用 SENTINEL get-master-addr-by-name 获取。 |
