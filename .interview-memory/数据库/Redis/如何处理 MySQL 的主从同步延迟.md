## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 延 | 延强并，五解法 | 主从延迟 = 从库单线程/大事务/网络，五种手段应对 |

## 原始知识

- **延迟原因**：从库单线程重放、主库大事务、网络抖动、从库性能差
- **强制读主库**：对一致性要求高的关键读走主库
- **并行复制**：MySQL 5.6+ 基于库并行，5.7+ 基于组提交/写集并行
- **分库分表**：分散主库压力，降低单实例同步负担
- **降低主库压力**：减少大事务、优化写入、避免高峰期大量 DML
- **使用缓存**：读请求优先走缓存，减少对从库的即时读依赖
- **判断延迟**：Seconds_Behind_Master、心跳表比对

## 顺口溜

> 从库单线慢，强读主库来，并行切分缓。

## 单字锚点 + 防崩推导

- **锚点字**：延
- **扩展口诀**：延强并，五解法
- **为什么选它**：「延」直接点出问题——同步延迟
- **推导链**：
  1. 「延」→ 主从同步延迟 → 从库跟不上主库
  2. 「强」→ 强制读主库 → 关键读直接走主库
  3. 「并」→ 并行复制 → 让从库多线程重放
  4. 还有啥？→ 分库分表、降低主库压力、加缓存
  5. 怎么知道延迟？→ Seconds_Behind_Master

## 一字一画面

**延** —— 像一条传送带上的包裹：主库在快速打包（写操作），从库只有一个工人（单线程 SQL 线程）在慢慢拆包。包裹堆成山（延迟增大）。解决办法：关键客户直接到主库取货（强制读主库），或者给从库多加几个工人（并行复制），或者把传送带分成多条（分库分表），或者在从库前再加个临时货架（缓存）。（本质：从库重放能力跟不上主库写入速度）

## 面试话术

「主从延迟主要是从库单线程重放 binlog 导致的，尤其是主库有大事务或者网络不稳定的时候。应对方案有几种：一是关键读强制走主库；二是开启并行复制，让从库多线程回放；三是分库分表降低单库压力；四是优化主库，减少大事务；五是读多的时候加一层缓存，降低对从库的即时依赖。」

## 防混补丁

- **容易和主从切换搞混**：延迟是「从库追不上」，切换是「主库挂了换从库」。
- **补丁口诀**：延迟是慢不是挂，切换是换不是等；慢要加速，挂要换。

## 追问预演

1. **Q：Seconds_Behind_Master 为 0 就一定没延迟吗？**
   A：不一定，如果 IO 线程卡住，SQL 线程追上了也会显示 0，建议配合心跳表判断。
2. **Q：MySQL 5.7 并行复制原理？**
   A：基于组提交（LOGICAL_CLOCK）或写集（WRITESET），判断事务之间无冲突就可以并行回放。
3. **Q：强制读主库怎么实现？**
   A：可以在事务中读、通过注解/Hint 指定数据源，或者根据业务场景在框架层路由。
4. **Q：大事务为什么导致延迟？**
   A：大事务在主库执行时间长，binlog 传到从库后，SQL 线程也要一次性执行完，会阻塞后续事件。
