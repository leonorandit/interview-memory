## 单字锚点 + 防崩推导

- **锚点字**：缓（缓存）
- **扩展口诀**：缓锁分，六场景
- **为什么选它**：Redis 最广为人知的用途就是缓存，"缓"字是入口，由此发散其他场景
- **推导链**：
  1. 缓 → 缓存 → 热点数据、Session 存储，Redis 最基础用法
  2. 缓 → 锁 → 分布式锁，setnx 或 Redisson，解决并发互斥
  3. 缓 → 分 → 分布式计数器，incr/decr，原子加减
  4. 缓 → 排行榜 → ZSet 排序集合，按分数排名次
  5. 缓 → 限流 → 令牌桶或计数器窗口，控接口频率
  6. 缓 → 消 → 消息队列，List 做队列、Stream 做流处理
  7. 缓 → 还有？→ 布隆过滤器（判存在）、Geo（附近的人）、会话存储、秒杀库存预扣
  8. 缓 → 为什么都能做？→ Redis 数据结构丰富（String/Hash/List/Set/ZSet/Bitmap/Geo/Stream）

## 一字一画面

**缓** —— 像冰箱保鲜层放着今晚要吃的肉：比去菜市场快，但不如冷冻层持久（本质：内存读取速度极快，但容量有限，存热点数据）

**锁** —— 像厕所门上"有人"的红灯：一个人进去亮起，别人看见红灯就等着（本质：setnx 设置标记，互斥访问共享资源）

**分** —— 像比赛计分牌翻数字：翻一下加一分，全场都看得见最新分（本质：incr 原子操作，高并发计数不丢失）

**排** —— 像高考金榜从状元往下排：第一名 730 分，第二名 725 分，ZSet 按 score 有序（本质：有序集合天然适合做排行榜）

**限** —— 像地铁闸机刷卡：一秒只能通过那么多人，超了就得等（本质：控制请求速率，保护后端服务）

**消** —— 像快餐店叫号器：做好一个喊一个，先来后到排队取餐（本质：List 做队列，生产者放、消费者取）

## 面试话术

"Redis 应用场景非常多，最核心的是缓存，比如热点数据和 Session。然后还有分布式锁，用 setnx 或 Redisson 实现；计数器用 incr 做原子加减；排行榜用 ZSet 按分数排序。"

"另外还有限流，可以用令牌桶或计数器窗口实现；消息队列用 List 或 Stream 做异步解耦；布隆过滤器判断元素可能存在或一定不存在；Geo 做附近的人；秒杀场景里还可以做库存预扣。"

## 防混补丁

**容易和 Memcached 搞混**？

补丁口诀：**"Memcached 是纯缓存纯 String，Redis 是瑞士军刀数据结构多"**

- Memcached：只支持简单 KV，适合做纯缓存
- Redis：支持 String/Hash/List/Set/ZSet/Bitmap/Geo/Stream，能做缓存之外更多事

## 追问预演

**Q：Redis 做分布式锁和 ZooKeeper 有什么区别？**
A：Redis 性能高但可靠性依赖过期时间，ZK 可靠性高（临时顺序节点）但性能略低。Redisson 的看门狗机制可以自动续期。

**Q：ZSet 做排行榜怎么翻页？**
A：用 ZREVRANGE 按分数倒排，指定 start 和 stop 做分页；要查某个用户的排名用 ZREVRANK。

**Q：Redis 限流和网关限流怎么选？**
A：网关限流在入口统一做，Redis 限流可以在业务层做更细粒度控制，比如按用户 ID 限流。

**Q：Redis Stream 和 Kafka 有什么区别？**
A：Stream 是 Redis 内置轻量级消息流，适合中小规模；Kafka 是专业消息中间件，适合大数据量高吞吐。
