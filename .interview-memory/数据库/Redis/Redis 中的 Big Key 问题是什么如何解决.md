## 速记卡片

大 | 大拆删，四解法 | 单 key 过大，阻塞 Redis 主线程

## 原始知识

- Big Key 指**单个 key 的 value 过大**
- String > **10KB**
- Hash / List / ZSet / Set **元素过多**
- 危害：**阻塞 Redis**（单线程处理慢）
- 危害：**内存不均衡**（集群节点倾斜）
- 危害：**网络拥塞**（传输耗时）
- 危害：**主从同步延迟**
- 发现：**redis-cli --bigkeys**
- 发现：**rdb-tools** 分析
- 解决：**拆分大 Hash** 为多个小 Hash
- 解决：**删除用 unlink**（异步删除）
- 解决：**定期清理**
- 解决：**限制单个集合大小**

## 顺口溜

String 超十 K，集合元素堆成山。
拆小 unlink 删，定期清理设上限。

## 单字锚点 + 防崩推导

- **锚点字**：大（大 Key）
- **扩展口诀**：大拆删，四解法
- **为什么选它**：问题名字就叫 Big Key，"大"字是核心特征
- **推导链**：
  1. 大 → 什么大？→ 单个 key 的 value 太大，String 超 10KB，集合元素过多
  2. 大 → 为什么是大问题？→ Redis 单线程，处理大 key 时其他命令排队等
  3. 大 → 还有什么危害？→ 内存不均衡（集群里一个节点撑爆）、网络拥塞（传半天）、主从同步卡死
  4. 大 → 怎么发现？→ redis-cli --bigkeys 扫描，rdb-tools 离线分析 RDB 文件
  5. 大 → 怎么解决？→ 拆（大 Hash 拆成小 Hash）、删（unlink 异步删，不阻塞）、清（定期清理）、限（限制单个集合大小）
  6. 大 → 为什么不用 del？→ del 删大 key 会阻塞，unlink 丢给后台线程异步删
  7. 大 → 怎么预防？→ 设计时控制 key 大小，Hash 元素不超过 512，List 做分片

## 一字一画面

**大** —— 像一个人扛着冰箱上楼梯：堵在楼道中间，后面所有人都上不去（本质：单线程 Redis 处理大 key 耗时，阻塞后续所有命令）

**拆** —— 像把一摞书分到几个书包里：原来一个包装不下，现在三个包各背几本，轻松上路（本质：大 Hash 拆成多个小 Hash，分散压力）

**删** —— 像请搬家公司夜里悄悄搬走大件家具：不占用白天通道，住户正常进出（本质：unlink 异步删除，主线程不阻塞）

**限** —— 像超市购物车限高线：超过的不让进，从源头控制大小（本质：业务层限制单个集合元素数量）

## 面试话术

"Big Key 就是 Redis 里单个 key 的 value 太大，比如 String 超过 10KB，或者 Hash、List 这些集合元素过多。因为 Redis 是单线程的，处理这种大 key 的时候会把其他命令都卡住。"

"发现 Big Key 可以用 redis-cli --bigkeys 在线扫描，或者用 rdb-tools 离线分析 RDB 文件。解决的话主要是拆、删、清、限四招：大 Hash 拆成小 Hash；删除用 unlink 异步删避免阻塞；定期清理过期数据；设计时就限制单个集合的大小。"

## 防混补丁

**容易和 Hot Key 搞混**？

补丁口诀：**"Big Key 是胖子占通道，Hot Key 是明星被围堵"**

- Big Key：单个 key 的 value 体积大，处理慢阻塞线程
- Hot Key：某个 key 被频繁访问，QPS 高打挂节点，需要本地缓存或拆分 key 解决

## 追问预演

**Q：Big Key 具体多大算大？**
A：经验值 String > 10KB，集合元素数 > 5000 或总大小 > 1MB，具体还要看业务 RT 要求。

**Q：unlink 和 del 删除有什么区别？**
A：del 是同步删除，大 key 会阻塞主线程；unlink 是异步删除，先断开 key 引用，后台线程慢慢释放内存。

**Q：集群模式下 Big Key 有什么额外影响？**
A：会导致某个 slot 所在节点内存倾斜，甚至触发 rehash 和迁移时更慢；主从复制时大 key 传输造成延迟。

**Q：怎么拆分大 Hash？**
A：按字段 hash 取模拆成多个小 Hash key，比如 user:1000:profile_0、user:1000:profile_1，查询时根据规则路由。
