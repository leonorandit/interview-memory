# Redis 中跳表的实现原理是什么

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 跳 | 跳多速，概率升 | 多级索引 / 随机升层 / O(logn) / 空间换时间 |

## 原始知识

- **有序链表 + 多级索引**：底层是排序链表，上层建快速通道
- **节点随机升层**：概率 **1/2**，期望层数 **O(logn)**
- **查询 O(logn)**：从顶层跳着走，像地铁快车
- **插入删除 O(logn)**：先查位置，再局部调整索引
- **空间换时间**：多存指针，换查询速度
- **Redis ZSET 底层**：跳表 + **哈希表**（跳表保序，哈希表保 O(1) 查分）
- **与平衡树对比**：实现简单 / **范围查询友好**（中序遍历麻烦，跳表区间直接顺序走）

## 单字锚点 + 防崩推导

- **锚点字**：跳
- **扩展口诀**：跳多速，概率升
- **为什么选它**：跳表的核心就是"跳"，从上层索引往下跳，形象好记
- **推导链**：
  1. 跳 → 跳表（Skip List）
  2. 多 → 多级索引（不止一层快速通道）
  3. 速 → 速度快（O(logn) 查询）
  4. 概率升 → 节点怎么升到上层？随机抛硬币，概率 1/2
  5. 期望层数 → O(logn)，所以查询也是 O(logn)
  6. 底层是什么 → 有序链表
  7. Redis 哪里用 → ZSET = 跳表 + 哈希表
  8. 为什么不选平衡树 → 实现简单、范围查询友好

## 一字一画面

- **跳** —— 像[地铁快慢线]：慢车每站都停（底层链表），快车只停大站（上层索引），要去某站先看快车到哪，再换慢车，省了大量停靠时间。（本质：多级索引跳过无关节点）
- **多** —— 像[三层高架桥]：最底层是所有车辆（完整链表），第二层是公交车专用（跳一部分），第三层是高铁（跳更多），层数越多跳得越快。（本质：索引层数决定跳跃跨度）
- **概** —— 像[抛硬币爬楼]：每来一个节点抛硬币，正面就上楼，反面停住，爬到第几层全看运气，但整体期望是 logn 层。（本质：随机化保证期望复杂度，避免复杂再平衡）
- **速** —— 像[查字典先翻目录]：不从头逐字读，先看目录定位到章节，再细读，查询从 O(n) 变成 O(logn)。（本质：索引结构的空间换时间）

## 面试话术

"跳表本质是一个有序链表加上多级索引，每个节点通过随机抛硬币决定升到第几层，概率是 1/2，期望层数是 O(logn)，所以查询和插入都是 O(logn)。Redis 的 ZSet 底层就是跳表配合哈希表，跳表负责排序和范围查询，哈希表负责按成员查分数，做到 O(1)。相比平衡树，跳表实现简单多了，而且范围查询特别友好，直接顺着底层链表走就行。"

## 防混补丁

容易和 B+ 树搞混？记住：B+ 树是"块状分层磁盘友好"，跳表是"链表随机升层内存友好"；B+ 树要分裂合并，跳表抛硬币就行，实现简单是最大区别。

## 追问预演

- **Q：跳表时间复杂度为什么是 O(logn)？**
  - A：每层索引节点数是下层的 1/2，最多 logn 层，每层最多遍历常数个节点，所以查询 O(logn)。
- **Q：随机升层会不会退化到 O(n)？**
  - A：理论上极端情况会，但概率极低；Redis 限制了最大层数 64，保证最坏情况可控。
- **Q：Redis ZSet 为什么不用平衡树？**
  - A：跳表实现简单、代码量少、范围查询方便；配合哈希表已经能做到 O(1) 查分数。
- **Q：跳表和哈希表在 ZSet 里怎么配合？**
  - A：哈希表存 member→score 的映射，O(1) 查分数；跳表按 score 排序存 member，支持范围查询和排序。
- **Q：插入一个节点具体怎么做？**
  - A：先逐层查到插入位置，记录每层前驱；然后随机决定层数，在对应层插入并修改前驱指针。
