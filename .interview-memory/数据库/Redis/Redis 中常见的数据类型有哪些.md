# Redis 中常见的数据类型有哪些

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 五 | 五字列，五类型 | String/Hash/List/Set/ZSet + 扩展四宝 |

## 原始知识

- **String**：字符串 / 计数器 / 缓存（最基础）
- **Hash**：对象存储（**字段散列**，省空间）
- **List**：队列 / 栈（**双端操作**，LPush/RPop）
- **Set**：去重 / 交集并集（**无序唯一**）
- **Sorted Set**：ZSET 排行榜（**按分排序**）
- **Bitmap**：位图（**按位统计**，省内存）
- **HyperLogLog**：基数统计（**估算去重**，误差 0.81%）
- **Geo**：地理位置（**半径查询**，附近的人）
- **Stream**：消息流（**日志队列**，消费者组）

## 单字锚点 + 防崩推导

- **锚点字**：五
- **扩展口诀**：五字列，五类型
- **为什么选它**：Redis 最经典的答案是"五种基本数据类型"，"五"字一出口，基础框架就有了
- **推导链**：
  1. 五 → 五种基础类型
  2. 字 → String（字符串）
  3. 哈 → Hash（哈希，对象存储）
  4. 列 → List（列表，队列栈）
  5. 集 → Set（集合，去重交并）
  6. 排 → Sorted Set / ZSet（排序集合，排行榜）
  7. 延伸四宝 → Bitmap、HyperLogLog、Geo、Stream（位图超地流）

## 一字一画面

- **五** —— 像[五根彩色手指]：大拇指 String（最粗最常用），食指 Hash（指哪儿打哪儿的字段），中指 List（居中排队），无名指 Set（去重唯一），小拇指 ZSet（虽小但能排序）。（本质：五种基础数据结构的直观分类）
- **字** —— 像[红色计数器]：一按数字就跳，缓存 token、计数 incr，最简单直接。（本质：String 是 Redis 最原子的键值单元）
- **哈** —— 像[抽屉柜]：每个抽屉存一个对象的字段，拿 ID 取整行比 JSON 省空间。（本质：Hash 是 field-value 的嵌套映射）
- **位** —— 像[签到墙]：365 个灯泡，亮了就表示签到，1 亿用户只占 12MB。（本质：Bitmap 用 bit 位表示布尔状态）
- **超** —— 像[估算人数的眼睛]：不用点名，看一眼操场就估出大约多少人，快但不精确。（本质：HyperLogLog 用概率算法换极低内存）

## 面试话术

"Redis 主要有五种基础类型：String 做缓存和计数器，Hash 存对象，List 做队列，Set 做去重和交集，Sorted Set 做排行榜。另外还有四种扩展类型：Bitmap 做位统计，HyperLogLog 做基数估算，Geo 做地理位置，Stream 做消息流。"

## 防混补丁

容易把 Set 和 ZSet 搞混？记住：Set 是"一群人乱站"，ZSet 是"按身高排队"，有没有分数是核心区别。

## 追问预演

- **Q：String 最大能存多少？**
  - A：单个 value 最大 512MB，但生产环境建议不要超过 10KB。
- **Q：Hash 和 String 存对象有什么区别？**
  - A：Hash 可以单独取改某个字段，省内存；String 存整个 JSON，序列化反序列化开销大。
- **Q：List 和 Stream 做队列的区别？**
  - A：List 是简单队列无持久化消费组，Stream 支持消费者组、ACK 确认、消息持久化，更像 Kafka。
- **Q：Bitmap 和 Set 去重哪个省内存？**
  - A：Bitmap 在数据密集时极省（1 亿用户占 12MB），但稀疏时浪费；Set 按实际元素占内存，灵活但单元素开销大。
