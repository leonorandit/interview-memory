# 什么是观察者模式？一般用在什么场景？

## 速记卡片
`观 | 观订阅，一多通 | 观vs发布订阅`

## 原始知识
- Subject**维护Observer列表**
- 状态变化**通知所有观察者**
- 场景：事件监听、消息订阅、GUI update
- **易混点**：与发布订阅模式区别
- **追问点**：推模型vs拉模型、内存泄漏风险

## 顺口溜
**观订阅，一多通**

## 单字锚点 + 防崩推导
- **锚点字**：**观**
- **扩展口诀**：**观订阅，一多通**
- **为什么选它**：观察者模式的灵魂就是"观察"，一方观察另一方的变化。
- **防崩推导**：
  - 观 → 观察 → 有人盯着Subject → Subject维护一个观察者列表
  - 观 → 变化 → Subject状态变了 → 轮询通知所有观察者
  - 观 → 一对多 → 一个主题，多个观察者 → 事件驱动的基础
  - 观 vs 发布订阅 → 观是"直接盯着"，发布订阅是"通过中间商"

## 一字一画面
- **观** —— 像**粉丝关注博主**：博主（Subject）发新视频，所有粉丝（Observer）手机同时弹通知。（本质：被观察者维护观察者列表，变化时广播）
- **订** —— 像**订阅报纸**：报社有一份订户名单，新报纸印出来按名单一家家送。（本质：观察者先注册，变化时主动推送）
- **通** —— 像**课堂点名**：老师（Subject）说"下课"，全班学生（Observers）同时收拾书包。（本质：一对多的联动更新）

## 面试话术
- "观察者模式就是**观订阅，一多通**。一个主题对象维护一堆观察者，自己状态一变就通知大家更新。像微信公众号，号主发文，所有关注的人都能收到。Java里的Observable和Listener机制都是这个。"
- "它是事件驱动编程的基础，解耦了事件源和事件处理者，双方只依赖抽象接口。"

## 防混补丁
- 容易和**发布订阅模式**搞混。观察者模式Subject和Observer直接打交道；发布订阅模式中间有Broker解耦。补丁：**观察直接盯，发布订阅走中介**。

## 追问预演
- **Q：观察者和发布订阅的区别？**
  - A：观察者模式是紧耦合的一对多，观察者直接订阅主题；发布订阅模式引入了事件中心/消息中间件，发布者和订阅者互相不知道对方存在。
- **Q：推模型和拉模型的区别？**
  - A：推模型是Subject把变化数据一起推送给观察者；拉模型是Subject只通知有变化，观察者主动来拉数据。推简单但可能推送冗余数据，拉更灵活但增加交互。
- **Q：观察者模式有什么风险？**
  - A：如果观察者忘记取消订阅会造成内存泄漏；通知顺序如果依赖具体观察者可能出问题；循环依赖会导致死循环。
