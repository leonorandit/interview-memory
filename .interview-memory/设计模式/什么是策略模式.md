## 原始知识

- **定义**：算法族**分别封装**，可互相替换
- **核心**：把**变化**的算法独立出来
- **Context**：持有 Strategy **接口**
- **Strategy**：算法**抽象接口**
- **ConcreteStrategy**：具体算法**实现**
- **支付渠道**：微信、支付宝、银行卡**切换**
- **排序算法**：快速排序、归并排序**替换**
- **促销规则**：满减、打折、积分**规则切换**
- **与状态模式区别**：策略由**客户端**选择，状态由**对象内部**流转
- **消除 if-else**：把分支逻辑拆成**独立类**
- **开闭原则**：新增算法**不修改**原有代码

## 单字锚点 + 防崩推导

**锚点字**：策

**扩展口诀**：策族替，三场景

**为什么选它**："策"就是策略，策略模式的核心就是"策略"二字，算法族 = 策略族，替换 = 策略切换。

**推导链**：
1. 锚点字"策" → 策略模式
2. 口诀"策族替" → 策略 = 算法族 + 可替换
3. 算法族 → 一族算法，完成同一件事的不同方式
4. 封装 → 每个算法独立成一个类，实现同一接口
5. 可替换 → Context 持有 Strategy 接口，运行时注入不同实现
6. 口诀"三场景" → 支付渠道、排序算法、促销规则
7. 支付渠道 → 微信支付、支付宝、银联，统一下单接口
8. 排序算法 → 数据量小用插入，大数据用快排，动态切换
9. 促销规则 → 双十一是满减，618 是打折，策略配置化
10. 与状态模式区别 → 策略是客户端主动选，状态是对象内部条件触发变

## 一字一画面

**策** —— 像**武将腰间的三把刀**：一把朴刀、一把大刀、一把匕首，出任务前选一把插上（本质：Context 根据场景注入不同 Strategy 实现）

**族** —— 像**同一家出的三兄弟**：长相骨架一样（接口相同），但擅长功夫不同（实现不同）（本质：算法族都实现同一 Strategy 接口，行为各异）

**替** —— 像**赛车换轮胎**：进站前用雨胎，进站后换干胎，车还是那辆车（本质：Context 不变，运行时替换 Strategy 对象）

## 面试话术

"策略模式就是把一族算法分别封装起来，让它们可以互相替换。核心结构是 Context 持有一个 Strategy 接口，具体算法由不同的实现类提供。常见场景比如支付系统里切换微信支付宝，排序框架里切换算法，或者电商促销里换不同的优惠规则。它和状态模式的区别是，策略由客户端主动选择，状态是对象内部自己流转的。"

## 防混补丁

容易把**策略模式和状态模式**搞混？补丁口诀：**策是选刀出门战，态是自动变嘴脸，策由外定态由内，一个替换一个演**。

## 追问预演

- **Q**：策略模式怎么避免 context 里堆满 if-else？
  **A**：可以用工厂或 Map 把策略名和实现类映射起来，根据参数直接取对应策略，context 里只剩一行 get。
- **Q**：策略模式和多态有什么区别？
  **A**：多态是语言特性，策略模式是用多态解决"算法切换"这一特定问题的设计模式，包含上下文和策略的协作结构。
- **Q**：如果策略很多，类爆炸怎么办？
  **A**：可以用函数式接口或 lambda 减少类数量，或者在策略内部用简单工厂再做一层聚合。
