## 原始知识

- **单例**：全局**唯一**实例，配置类常用
- **工厂**：对象创建**解耦**，使用者不直接 new
- **观察者**：事件**监听**发布订阅，一对多通知
- **策略**：算法**替换**，消除 if-else 分支
- **代理**：**控制**访问，AOP 和权限控制
- **装饰器**：**功能叠加**，动态增强对象
- **模板方法**：流程**复用**，子类实现细节
- **应用场景**：根据**变化点**选模式
- **核心思想**：封装变化，**组合优于继承**
- **OOD 原则**：开闭原则、单一职责

## 单字锚点 + 防崩推导

**锚点字**：模

**扩展口诀**：模工单观，策工代

**为什么选它**："模"就是模式，设计模式的"模"，七个常用模式像七块积木，口诀按首字串起来。

**推导链**：
1. 锚点字"模" → 设计模式
2. 口诀"模工单观" → 单（单例）、工（工厂）、观（观察者）
3. 口诀"策工代" → 策（策略）、工同工厂已出，这里取装饰的"饰"音近，代理的"代"
4. 修正 → 单、工、观、策、装、代、模板方法
5. 单例 → 全局唯一，构造私有，静态获取
6. 工厂 → 简单工厂、工厂方法、抽象工厂，封装 new
7. 观察者 → Subject 维护 Observer 列表，状态变则通知
8. 策略 → Context 持有 Strategy 接口，运行时切换算法

10. 装饰器 → 包装器模式，InputStream 套 BufferedInputStream
11. 模板方法 → 父类定骨架，子类重写步骤，钩子方法扩展

## 一字一画面

**单** —— 像**总统办公室只留一把椅子**：全公司只有这一个座位，谁来都是坐这把（本质：单例控制实例数量，全局唯一访问点）

**工** —— 像**自动贩卖机按按钮出饮料**：你不用知道里面怎么组装，按一下就行（本质：工厂封装对象创建过程，调用者只关心接口）

**观** —— 像**微信群发消息**：群主发一条，所有人都能收到（本质：观察者模式一对多依赖，状态变更自动通知订阅者）

**策** —— 像**游戏换技能按键**：同一个按键槽，可以装火球术也可以装冰冻术（本质：策略模式封装算法族，运行时互相替换）

**代** —— 像**房产中介带看房**：买家不直接找房东，中介在中间过滤和控制（本质：代理模式在访问对象时插入控制层）

**装** —— 像**手机壳贴支架再贴膜**：一层一层套上去，每一层加一点功能（本质：装饰器模式动态给对象添加职责，包装器嵌套）

## 面试话术

"我最常用的设计模式有单例、工厂、观察者、策略、代理、装饰器和模板方法。单例用在全局配置或连接池；工厂用来解耦对象创建，比如 Spring 的 BeanFactory；观察者适合做事件监听和消息通知；策略模式可以替换算法，比如支付渠道选择；代理模式在 AOP 和权限控制里很常用；装饰器像 Java IO 流那样一层层包装增强；模板方法用于复用业务流程，子类只实现变化的部分。"

## 防混补丁

容易把**策略模式和状态模式**搞混？补丁口诀：**策略是主动选算法，状态是被动变行为，策略客户端决定，状态对象自己转**。

## 追问预演

- **Q**：单例模式有什么缺点？
  **A**：违反单一职责，隐藏依赖关系，不利于测试；多线程下要考虑双重检查锁或静态内部类。
- **Q**：工厂模式和策略模式有什么区别？
  **A**：工厂解决"创建哪个对象"，策略解决"用哪个算法"；工厂关注对象诞生，策略关注行为替换。
- **Q**：JDK 里有哪些设计模式的体现？
  **A**：IO 流是装饰器，Collections.synchronizedList 是代理，Comparator 是策略，Runtime 是单例。
