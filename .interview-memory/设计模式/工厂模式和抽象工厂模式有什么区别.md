# 工厂模式和抽象工厂模式有什么区别？

## 速记卡片
`族 | 族单多，两工厂 | 单产品vs产品族`

## 原始知识
- **工厂模式**：一个工厂一个**产品等级**
- **抽象工厂**：一个工厂多个**相关产品族**
- 抽象工厂**复杂度高**但扩展产品族方便
- 场景：UI组件跨平台、数据库访问
- **易混点**：三个工厂模式层级关系
- **追问点**：抽象工厂怎么扩展新产品等级

## 顺口溜
**族单多，两工厂**

## 单字锚点 + 防崩推导
- **锚点字**：**族**
- **扩展口诀**：**族单多，两工厂**
- **为什么选它**：抽象工厂的灵魂是"产品族"，一族产品一起切换。记住"族"就记住核心差异。
- **防崩推导**：
  - 族 → 家族 → 一族相关产品 → 比如Windows家族有Windows按钮+Windows文本框
  - 单 → 工厂方法 → 一个工厂只造**一种**产品的不同变体（按钮工厂造各种按钮）
  - 多 → 抽象工厂 → 一个工厂造**一族**产品（Windows工厂造按钮+文本框+滚动条）
  - 族 → 切换 → 把Windows工厂换成Mac工厂 → 整族UI一起换
  - 族 → 代价 → 新增产品等级（如新增下拉框）→ 所有工厂都要改

## 一字一画面
- **族** —— 像**全家装套餐**：现代风套餐（抽象工厂）包含现代沙发+现代茶几+现代灯，一键全换；换成北欧风套餐，全部一起变。（本质：一族产品整体切换）
- **单** —— 像**专门店**：只卖沙发（一个产品等级），但沙发有现代款、北欧款、中式款。（本质：工厂方法模式，一个工厂造一类产品的不同变体）
- **多** —— 像**宜家展厅**：一个展厅（具体工厂）里，同风格的床、柜、桌配好了整套卖。（本质：抽象工厂，一个工厂产出多个相关产品的组合）

## 面试话术
- "工厂模式和抽象工厂的核心区别是**族单多，两工厂**。工厂方法模式是一个工厂只造一类产品，比如按钮工厂造各种按钮；抽象工厂是一个工厂造一整族相关产品，比如Windows UI工厂同时造Windows按钮、Windows文本框、Windows滚动条。换平台时，把Windows工厂换成Mac工厂，整族UI一起切换。"
- "抽象工厂适合跨平台场景，比如同一套系统要支持Windows和Linux两套UI风格。缺点是新增一个产品等级（比如新增下拉框），所有具体工厂都要改。"

## 防混补丁
- 容易和**简单工厂、工厂方法**搞混。三层关系：简单工厂（一个工厂造所有）→ 工厂方法（一个工厂造一类）→ 抽象工厂（一个工厂造一族）。补丁：**简单一锅煮，方法分锅炒，抽象整桌席**。

## 追问预演
- **Q：抽象工厂怎么扩展新产品等级？**
  - A：这是抽象工厂的痛点。新增产品等级（如新增下拉框）需要在抽象工厂接口新增方法，所有具体工厂类都要实现，违反开闭原则。可以通过简单工厂+反射等方式变通。
- **Q：三个工厂模式的演进关系？**
  - A：简单工厂（参数判断，违反开闭）→ 工厂方法（每个产品一个工厂，符合开闭）→ 抽象工厂（处理多产品族）。
- **Q：实际项目中怎么用？**
  - A：Spring的BeanFactory是工厂方法；JDBC的Connection在不同数据库（MySQL/Oracle）返回不同Statement实现，有抽象工厂的影子。
