# 请描述简单工厂模式的工作原理

## 速记卡片
`造 | 造参产，一分支 | 造vs策略`

## 原始知识
- 工厂类**根据参数创建对应产品**
- **解耦**客户端与具体产品
- **违反开闭原则**（新增产品需改工厂）
- 场景：对象创建逻辑复杂、需要统一管理创建
- **易混点**：与策略模式区别
- **追问点**：工厂方法模式怎么改进

## 顺口溜
**造参产，一分支**

## 单字锚点 + 防崩推导
- **锚点字**：**造**
- **扩展口诀**：**造参产，一分支**
- **为什么选它**：工厂模式的本质就是"造"对象，把new操作集中管理。
- **防崩推导**：
  - 造 → 造对象 → 不用new直接用 → 交给工厂类造
  - 造 → 参数 → 给工厂传一个标识 → 工厂根据参数判断造哪个产品
  - 造 → 一分支 → 工厂里一堆if-else或switch → 新增产品要改工厂代码
  - 造 → 缺点 → 违反开闭原则 → 所以进化出工厂方法模式（每个产品一个工厂）

## 一字一画面
- **造** —— 像**自动售货机**：按按钮（传参数）→ 机器内部判断 → 掉出对应饮料（产品）。你不用管饮料从哪来，只管按按钮。（本质：封装对象创建过程）
- **参** —— 像**餐厅菜单**：报菜名（参数），后厨根据菜名做对应菜品，客人不用进厨房。（本质：参数决定创建哪种具体产品）
- **支** —— 像**老式电话交换机**：插线员（工厂）看你要接哪条线（参数），手动插到对应孔。新增线路要换交换机板子。（本质：扩展要修改工厂内部）

## 面试话术
- "简单工厂就是**造参产，一分支**。一个工厂类接收参数，根据参数判断new哪个具体产品返回给客户端。好处是把创建逻辑集中了，客户端不用到处new；坏处是新增产品要改工厂里的if-else，违反开闭原则。"
- "所以它叫'简单'工厂，只是入门级用法。正式项目里一般用工厂方法模式或者抽象工厂模式来规避这个缺点。"

## 防混补丁
- 容易和**策略模式**搞混。简单工厂是造对象，策略是换行为。工厂返回的是不同类的实例，策略是把不同算法封装成可互换的策略对象。补丁：**工厂造人，策略换招**。

## 追问预演
- **Q：简单工厂违反开闭原则，怎么改进？**
  - A：升级为工厂方法模式，每个具体产品对应一个具体工厂类，新增产品只需新增工厂类，不用改原有代码。
- **Q：简单工厂和工厂方法模式的区别？**
  - A：简单工厂是一个工厂类造所有产品；工厂方法模式是抽象工厂定义接口，每个具体产品由自己的具体工厂来造。
- **Q：什么时候用简单工厂？**
  - A：产品类型不多、创建逻辑简单、不会频繁扩展的场景。或者作为内部工具，不对外暴露扩展点。
