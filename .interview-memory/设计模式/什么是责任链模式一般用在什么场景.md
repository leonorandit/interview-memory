# 什么是责任链模式？一般用在什么场景？

## 速记卡片
`链 | 链传递，审审批 | 链vs装饰`

## 原始知识
- 请求沿**处理链**传递
- 每个节点**决定处理或转交**
- 节点间**解耦**，互不知晓全貌
- 场景：**审批流**、过滤器链、异常处理链
- **易混点**：与装饰器模式区别
- **追问点**：链过长性能问题、如何终止传递

## 顺口溜
**链传递，审审批**

## 单字锚点 + 防崩推导
- **锚点字**：**链**
- **扩展口诀**：**链传递，审审批**
- **为什么选它**：责任链的本质就是"链"状传递，像击鼓传花。
- **防崩推导**：
  - 链 → 多个节点串成链 → 请求从链头进
  - 链 → 每个节点像安检口 → 能处理就处理，不能就往后传
  - 链 → 审批场景最典型 → 员工→经理→总监→CEO
  - 链 vs 装饰器 → 链是"选一个人干活"，装饰器是"层层包装一起干"

## 一字一画面
- **链** —— 像**工厂流水线**：零件（请求）从第一个工位流到最后一个，每个工人看一眼，该我装轮子就装，不该我就推给下一位。（本质：请求在链上流动，找到合适的处理者）
- **传** —— 像**击鼓传花**：花在谁手里谁决定是表演还是继续传。（本质：每个节点有处理权或转发权）
- **审** —— 像**多级签字**：报销单先找直属领导，领导批不了再往上交。（本质：审批流是责任链最经典场景）

## 面试话术
- "责任链模式就是**链传递，审审批**。请求进来后沿着一条处理链往下走，每个节点判断自己能不能处理，能就处理，不能就交给下一个。像审批报销单，员工→经理→总监，各管各的额度。"
- "它解决了把请求和多个处理者解耦的问题，客户端只需要把请求发给链头，不用关心谁最终处理。"

## 防混补丁
- 容易和**装饰器模式**搞混。装饰器是层层包装，每一层都参与；责任链是选一个处理，其他不参与。补丁：**装饰全上阵，链上只一人**。

## 追问预演
- **Q：责任链和装饰器的区别？**
  - A：装饰器是洋葱模型，每层都执行，层层增强；责任链是链表模型，只有一个节点处理，要么处理要么传递。
- **Q：链太长怎么办？**
  - A：可以结合缓存，或者改用其他模式如策略模式直接路由到处理者。
- **Q：如何终止传递？**
  - A：节点处理完后不调用 next 即可终止。
