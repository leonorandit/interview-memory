# 什么是设计模式？请简述其作用。

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 经 | 经复扩，三作用 | 经 = 经验复用；复 = 复用性；扩 = 可扩展性 |

## 原始知识

- 设计模式是软件开发中反复出现的问题的解决方案（GoF 23 种）
- 三大类型：**创建型 / 结构型 / 行为型**
- 作用：**提高代码复用性 / 可扩展性 / 可维护性 / 降低耦合**
- 不是教条是指导原则，与实际业务结合使用

## 顺口溜

```
前人栽树后人乘凉，创建结构行为三帮忙；复用扩展易维护，灵活运用别死板
```

## 单字锚点 + 防崩推导

| 锚点字 | 扩展口诀 | 为什么选它 | 推导链 |
|--------|----------|------------|--------|
| 经 | 经验复用可扩展 | "经" = 经验，设计模式就是前人总结的"经验" | 经 → 经验总结（GoF 23 种）→ 三大类型（创建/结构/行为）→ 三作用（复用/扩展/维护）→ 注意：是指导原则，不是教条 |

**防崩推导**：只记得"经"字时，推导链如下：
1. "经" = 经验，设计模式就是软件界的"经书"，前人踩坑后总结出来的最佳实践
2. 谁总结的？——GoF（四人帮），一共 23 种经典模式
3. 分哪几类？——**创建型**（怎么创建对象）、**结构型**（怎么组装类和对象）、**行为型**（对象之间怎么交互协作）
4. 有什么用？——**复用**（代码不用重复写）、**扩展**（加功能不用改老代码）、**维护**（代码清晰好读懂）
5. 注意事项？——不是死规矩，要结合业务灵活运用

## 一字一画面

- **经** —— 像一本翻烂的武功秘籍：封皮上写着"23 式"，每一页都是前辈高手被 bug 暴打后总结出的招式（本质：经验的系统化总结，不是凭空创造）
- **创** —— 像工厂的流水线模具：造对象不用每次手搓，用模具一套就是一个标准产品（本质：创建型模式，封装对象创建过程）
- **构** —— 像乐高积木的拼接说明书：小零件怎么搭成大楼，结构清晰才能稳（本质：结构型模式，组合类和对象形成更大结构）
- **行** —— 像交响乐团的指挥棒：谁什么时候发声，怎么配合，指挥一棒定乾坤（本质：行为型模式，对象之间的职责分配和算法封装）

## 面试话术

> 设计模式其实就是软件行业的前人在开发过程中反复遇到一些问题，然后总结出来的通用解决方案。最经典的就是 GoF 的 23 种设计模式，分为创建型、结构型和行为型三大类。它的主要作用就是让代码更容易复用、更容易扩展、也更容易维护，能降低模块之间的耦合度。当然设计模式不是死规矩，是指导原则，具体用哪种模式还是要结合实际业务场景来选。

## 防混补丁

- **容易和什么搞混？** 把设计模式当成万能公式，什么场景都硬套，或者以为用了设计模式就一定好。
- **补丁口诀**：模式是经验不是法律，过度设计反遭殃；简单问题简单做，复杂问题才请模式帮。

## 追问预演

| 面试官追问 | 简要答案 |
|-----------|----------|
| GoF 是谁？ | Gang of Four，Erich Gamma、Richard Helm、Ralph Johnson、John Vlissides，四人合著《设计模式》一书 |
| 三大类型分别是什么？ | 创建型（5 种）：单例、工厂、抽象工厂、建造者、原型；结构型（7 种）：适配器、桥接、组合、装饰、外观、享元、代理；行为型（11 种）：策略、模板方法、观察者等 |
| 你最常用的是哪个模式？ | 根据个人实际项目经验回答，如单例、工厂、策略、观察者等 |
| 设计模式有什么缺点？ | 可能引入额外复杂度，过度设计会让代码难懂；需要结合场景，不能硬套 |
| 除了 GoF 23 种，还了解哪些？ | Spring 里的 IOC/AOP 思想、MVC 模式、MVVM 等 |
