# TCP 的粘包和拆包能说说吗？

## 速记卡片
`流 | 流无定，三原因 | 无边界字节流`

## 原始知识
- TCP是**字节流无边界**
- 发送100+100，收可能200
- **Nagle算法**合并小包致粘包
- **MSS分片**大包拆分致拆包
- 解决：**固定长度**/分隔符/Length+Content
- Netty解码器处理
- **易混**：UDP无粘包

## 顺口溜
**流无定，三原因**

## 单字锚点 + 防崩推导
- **锚点字**：**流**
- **扩展口诀**：**流无定，三原因**
- **为什么选它**："流"是TCP的本质特征，也是粘包拆包的根因——没有消息边界。
- **防崩推导**：
  - 流 → TCP字节流 → 没有消息边界
  - 发送方send两次100字节，接收方可能一次收到200（粘包），也可能收到50+50+100（拆包）
  - 原因1：**Nagle算法**为了减少网络开销，把小包攒一起发 → 粘包
  - 原因2：**MSS限制**，IP层分片或TCP拆成多个段 → 拆包
  - 原因3：接收方缓冲区处理速度不同，应用层读取时机不一 → 粘或拆
  - 解决思路：给"流"加边界 → 固定长度（截断填充）、分隔符（找标志位）、Length+Content（先读长度再读内容）

## 一字一画面
- **流** —— 像**水龙头流出的水**：没有天然的"一包水"概念，你接一次可能接到半杯，也可能接到一杯半，水本身不告诉你边界在哪。（本质：TCP传输的是连续字节流，不保留应用层消息边界）
- **无** —— 像**撕掉页码的书**：字都在，但不知道哪到哪是一页，读着读着就串行了。（本质：协议层不维护消息边界）
- **定** —— 像**用固定杯子量水**：每杯250ml，多了倒掉少了加满，总有明确的一杯。（本质：固定长度协议，给流加边界）
- Nagle —— 像**快递员攒件**：小包裹不单独送，攒满一车再出发，导致多个包裹粘在一起送。（本质：减少小包数量，提高网络利用率）
- MSS —— 像**大车限高2米**：货物3米高必须拆成两趟运，一趟1.5米一趟1.5米。（本质：最大报文段长度限制导致分段）

## 面试话术
- "TCP粘包拆包的根本原因，我用五个字概括：**流无定，三原因**。TCP是字节流协议，本身没有消息边界。比如应用层send两次100字节，接收方可能一次收到200（粘包），也可能收到50+150（拆包）。三个原因：一是Nagle算法把小数据包攒一起发导致粘包；二是MSS限制把大包拆分导致拆包；三是接收端缓冲区处理不及时。解决方法就是给流加边界，常见三种：固定长度、分隔符、Length+Content，Netty里也有对应的解码器。"
- "记住UDP不存在粘包，因为它是有边界的"数据报"，send一次recv一次，一个包就是一个完整消息。"

## 防混补丁
- 容易和"TCP丢包"搞混。粘包不是丢数据，是数据边界乱了，数据一个字不少。
- 补丁：**粘包不少字，只是分不清**

## 追问预演
1. **UDP会有粘包问题吗？**
   - UDP是数据报，有明确边界，send一次recv一次，一个包就是一个完整消息，不存在粘包。这是TCP和UDP在传输模型上的本质区别。

2. **Nagle算法一定导致粘包吗？怎么关？**
   - Nagle是延迟发送算法，攒小包会增大粘包概率，但不是必然。可以通过Socket的TCP_NODELAY选项关闭，常用于实时性要求高的场景如游戏。

3. **Length+Content具体怎么实现？**
   - 先读4字节int得到长度L，再精确读L字节，这就是一帧完整消息。Netty的LengthFieldBasedFrameDecoder就是这个原理，最常用也最灵活。

4. **如果固定长度，数据不够怎么办？**
   - 需要填充（padding），比如固定128字节，实际数据100字节就补28字节0，接收方按128字节截取。缺点是浪费带宽。

5. **拆包和IP分片有什么关系？**
   - IP分片是网络层因为MTU限制对IP包的分割；TCP拆包是传输层因为MSS限制对TCP段的分割。两者可能同时发生，但解决粘包拆包通常在应用层做帧定界。
