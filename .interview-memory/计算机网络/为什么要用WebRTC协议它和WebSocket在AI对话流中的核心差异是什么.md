# 为什么要用WebRTC协议它和WebSocket在AI对话流中的核心差异是什么

## 速记卡片
`卡 | 等则卡，补则畅，音内置 | WebRTC vs WS`

## 原始知识
- **等则卡（WebSocket/TCP）**：丢包强制等待重传，后续数据全部阻塞，延迟越堆越高（50-500ms）
- **补则畅（WebRTC/UDP）**：丢包不等待，用前后帧插值填补（PLC），播放不中断，延迟稳定（50-150ms）
- **音内置（WebRTC）**：原生自带回声消除（AEC）、噪声抑制（NS）、自动增益控制（AGC）、自适应码率（ABR），WebSocket 需自建
- **P2P 直连**：WebRTC 通过 ICE/STUN/TURN 实现 NAT 穿透，优先 P2P 降低延迟；失败时退化为 TURN 中转
- **信令配合**：WebRTC 建连仍需 WebSocket/HTTP 做 SDP 信令交换，两者不是替代关系
- **易混点**：WebRTC 完全替代 WebSocket、WebRTC 一定更快
- **追问点**：ICE/STUN/TURN 区别、PLC 原理、SDP 信令

## 顺口溜
**等则卡，补则畅，音内置**

## 单字锚点 + 防崩推导
- **锚点字**：**卡**
- **扩展口诀**：**等则卡，补则畅**
- **为什么选它**：AI 实时语音最大的敌人是"卡"。WebSocket 的卡顿根因是 TCP 重传等待，WebRTC 的畅快来自 UDP 插值补偿。
- **防崩推导**：
  - 卡 → WebSocket 基于 TCP → 丢包必须等重传 → 后续音频全阻塞 → 延迟堆积 → 50-500ms 不可控
  - 补 → WebRTC 基于 UDP → 丢包不等待 → 前后帧插值（PLC）→ 播放不中断
  - 畅 → 延迟稳定在 50-150ms → 实时对话流畅自然
  - 音 → 音频工程能力 → AEC + NS + AGC + ABR → WebRTC 原生内置，WebSocket 全需自建

## 一字一画面
- **等** —— 像**排队等红灯**：绿灯坏了（丢包），所有车停下来等交警（重传），越等越长。（TCP 阻塞）
- **卡** —— 像**高速大堵车**：完全动不了，通话体验崩掉。（延迟堆积导致卡顿）
- **插** —— 像**急救包扎**：伤口不大，先贴创可贴（插值），不用等救护车（重传），行动不停。（UDP 快速处理）
- **畅** —— 像**畅通无阻的 ETC 高速**：不停车直接过，稳定低延迟。（50-150ms 流畅体验）
- **音** —— 像**专业录音棚设备**：自带降噪、音量平衡、回声消除，一站式搞定。（AEC/NS/AGC/ABR）

## 面试话术
- 问差异 → "核心差异我总结为**等则卡，补则畅，音内置**。**等则卡**是说 WebSocket 基于 TCP，丢包必须等重传，后续音频全堵住，延迟越堆越高；**补则畅**是说 WebRTC 基于 UDP，丢了包不等待，用前后帧插值填补，播放不中断，延迟稳定在 50-150ms；**音内置**是指 WebRTC 自带回声消除、噪声抑制、自适应码率等全套音频工程能力，用 WebSocket 全得自己造轮子。"
- 问关系 → "而且两者不是替代关系——WebRTC 建连时还需要 WebSocket 做 SDP 信令交换，WebSocket 跑信令，WebRTC 跑数据，各司其职。"

## 防混补丁
- 容易以为"WebRTC 完全不需要 WebSocket"。实际上 SDP 信令交换通常走 WebSocket。**补丁：WebRTC 跑数据，WebSocket 换信令，各司其职。**
- 容易以为"WebRTC 一定比 WebSocket 快"。NAT 打洞失败退化为 TURN 中转时，延迟可能反而更高。**补丁：P2P 通则快，TURN fallback 则慢。**

## 追问预演
1. **ICE/STUN/TURN 什么区别？** 答：ICE 是框架，先尝试本地直连，再用 STUN 打洞建立 P2P，最后用 TURN 服务器中转。
