# 分布式锁一般都怎样实现？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 红 | 红数ZK，三方案 | Redis 性能高，ZK 可靠性高，DB 最简单 |

## 原始知识

- **Redis**：SETNX + 过期时间，Redisson 看门狗自动续期，主从一致性风险
- **Zookeeper**：临时顺序节点，监听前一个节点，强一致性但性能较低
- **数据库**：唯一索引，乐观锁 version 字段
- **对比**：Redis 性能高，ZK 可靠性高，DB 最简单

## 顺口溜

Redis 快，ZK 稳，数据库最省心。

## 单字锚点 + 防崩推导

- **锚点字**：红
- **扩展口诀**：红数 ZK，三方案
- **为什么选它**：Redis 是分布式锁最热门的实现，一提到分布式锁先想到 Redis，"红"字最醒目
- **推导链**：
  1. 红 → Redis → 缓存层实现锁 → SETNX + 过期时间 → 主从风险 → Redisson 看门狗续期
  2. 红 → 联想到数字编号 → 数 → 数据库 → 唯一索引 / 乐观锁 version
  3. 红 → ZK（Zookeeper）→ 临时顺序节点 → 监听前一个节点释放 → 强一致

## 一字一画面

- **红** —— 像红灯：Redis 亮红灯停（获取锁），绿灯行（释放锁），灯有定时器（过期时间），突然断电灯还亮（主从一致性问题）
- **数** —— 像排队取号：数据库唯一索引就像窗口叫号，一号一人，重复号码直接拒绝
- **Z** —— 像拉链：Zookeeper 临时顺序节点像拉链齿，一个扣一个，前一个解开下一个才能动

## 面试话术

1. 分布式锁常见的有三种实现方式：基于 Redis、基于 Zookeeper、基于数据库，实际工作中 Redis 最常用，Redisson 封装得比较好。
2. 如果对一致性要求特别高，比如金融场景，建议用 Zookeeper，它是强一致的；如果追求性能，Redis 更合适，但要考虑主从切换时的锁丢失问题。

## 防混补丁

- **容易和本地锁搞混**：本地锁（synchronized、ReentrantLock）只能锁单个 JVM，分布式锁要跨进程跨机器，千万别说"用 synchronized 实现分布式锁"。
- **补丁口诀**：单机锁 JVM，分布式要联网。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Redis 主从锁丢失怎么解决？ | RedLock 方案，多个独立 Redis 节点半数以上加锁成功才算获取 |
| Redisson 看门狗机制是什么？ | 后台线程默认每 10 秒检查，锁快过期时自动续期 |
| Zookeeper 为什么性能低？ | 每次写操作都要半数以上节点确认，强一致性牺牲了吞吐量 |
| 数据库实现锁的缺陷？ | 无过期机制，宕机容易死锁；性能差，不适合高并发 |
