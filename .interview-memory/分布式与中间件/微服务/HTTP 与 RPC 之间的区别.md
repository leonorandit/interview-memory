# HTTP 与 RPC 之间的区别？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 通 | 通协效，五区别 | HTTP 通用性强，RPC 性能高，IDL 定契约 |

## 原始知识

- **HTTP**：通用协议，面向资源，文本传输，RESTful 风格
- **RPC**：远程过程调用，面向方法，二进制传输，高效
- **通用性**：HTTP 跨语言跨平台，RPC 需IDL（Protobuf / Thrift）
- **性能**：RPC 序列化体积小，连接复用，传输更快
- **调试**：HTTP 易调试（curl / Postman），RPC 需专用工具
- **场景**：HTTP 对外暴露 API，RPC 内部服务间调用

## 顺口溜

HTTP 通天下，RPC 效率高，对外用通用，对内要轻巧。

## 单字锚点 + 防崩推导

- **锚点字**：通
- **扩展口诀**：通协效，五区别
- **为什么选它**：HTTP 最大的标签就是"通用"，浏览器、手机、服务器全能用，"通"字一语道破
- **推导链**：
  1. 通 → 通用协议 → HTTP → 文本传输 / RESTful / 跨平台 / curl 调试
  2. 通 → 协 → 协议 → RPC 也是协议 → 但面向方法 / 二进制 / Protobuf IDL
  3. 通 → 效 → 效率 / 效果 → 性能对比 → RPC 序列化小、连接复用、效率高
  4. 通协效 → 五区别 → 通用性、协议层次、传输效率、调试难度、应用场景

## 一字一画面

- **通** —— 像通用插座：HTTP 像三孔万能插座，手机电脑电风扇都能插，谁都能用，但线粗传输慢（文本报文大）
- **协** —— 像合同协议：RPC 像双方签好的商业合同，必须先约定好格式（IDL），外人看不懂，但内部执行极快
- **效** —— 像高铁 vs 绿皮：RPC 像高铁专线，点对点直达；HTTP 像绿皮火车，站站停，兼容所有站台

## 面试话术

1. HTTP 和 RPC 最大的区别在于 HTTP 是通用协议，面向资源，适合对外暴露服务；RPC 是面向方法的远程调用，二进制传输，适合内部微服务之间通信。
2. 实际项目里，对外给前端或第三方用 HTTP + RESTful，内部服务之间用 Dubbo 或 gRPC 这种 RPC 框架，性能更好。

## 防混补丁

- **容易和"HTTP 也是 RPC 的一种"搞混**：严格来说 HTTP 可以作为 RPC 的传输载体（如 HTTP/JSON RPC），但通常说的 RPC 指基于 TCP 的二进制高效调用，千万别说"HTTP 就是 RPC"。
- **补丁口诀**：HTTP 可做 RPC 底，通常 RPC 是 TCP。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| gRPC 为什么性能高？ | 基于 HTTP/2 + Protobuf 二进制序列化，支持流式传输和多路复用 |
| RPC 需要服务治理吗？ | 需要，注册中心、负载均衡、熔断限流、链路追踪都要配 |
| HTTP/2 能替代 RPC 吗？ | 不能，HTTP/2 解决了队头阻塞和多路复用，但序列化和接口契约仍需 RPC 框架补齐 |
| 为什么内部不用 HTTP？ | HTTP 文本传输开销大，序列化慢，无长连接复用，QPS 比 RPC 低一个数量级 |
