# 让你设计一个 RPC 框架，怎么设计？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 代 | 代编网传，五模块 | 代理调用、序列化、网络传输、注册中心、负载容错 |

## 原始知识

- **服务暴露**：Provider 启动时把接口注册到注册中心
- **服务代理**：Consumer 通过动态代理像调本地方法一样调远程
- **序列化**：Protobuf / JSON / Hessian，对象转字节流
- **网络传输**：Netty / NIO / TCP，建立长连接通信
- **注册中心**：Zookeeper / Nacos / Consul，服务发现与治理
- **负载均衡**：随机 / 轮询 / 权重，分摊请求压力
- **容错机制**：重试 / 熔断 / 限流，保障系统稳定性

## 顺口溜

代理像本地，编码传过去，注册来发现，负载加容错。

## 单字锚点 + 防崩推导

- **锚点字**：代
- **扩展口诀**：代编网传，五模块
- **为什么选它**：RPC 的核心体验就是"像调用本地方法一样调用远程"，这个魔法全靠动态代理实现，"代"字是整个框架的灵魂
- **推导链**：
  1. 代 → 动态代理 → Consumer 侧生成代理对象 → 拦截方法调用 → 封装请求参数
  2. 代 → 编 → 编码 / 序列化 → Protobuf / JSON / Hessian → 对象 → 字节流
  3. 代 → 网 → 网络传输 → Netty / TCP / NIO → 长连接 → 发送到 Provider
  4. 代 → 传 → 传输完成后 → 服务发现 → 注册中心（Zookeeper / Nacos / Consul）
  5. 代编网传 → 五模块 → 代理层、序列化层、网络层、注册中心层、负载均衡与容错层

## 一字一画面

- **代** —— 像替身演员：Consumer 调用的是本地替身（动态代理），替身实际飞到外地（远程机器）拍完戏再回来交片（返回结果），用户完全看不出是替身
- **编** —— 像打包快递：把各种形状的对象（衣服、书籍、电器）统一塞进纸箱（序列化），贴上条形码（协议头），方便运输
- **网** —— 像高速公路：Netty 像双向八车道高速，TCP 长连接像ETC，不用每次停车取卡，一直畅通

## 面试话术

1. 如果让我设计一个 RPC 框架，我会从五个核心模块入手：代理层、序列化层、网络传输层、注册中心层、负载均衡与容错层。
2. 最核心的 trick 是动态代理，让调用方无感知地像调本地方法一样调远程；网络层我会选 Netty 做 NIO 长连接，序列化用 Protobuf 兼顾性能和跨语言。

## 防混补丁

- **容易和 HTTP 框架搞混**：RPC 框架不是简单的 HTTP 客户端封装，它要解决服务发现、负载均衡、熔断降级一整套治理问题，不只是发个请求。
- **补丁口诀**：RPC 不只是调接口，治理全家桶要带齐。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 动态代理用 JDK 还是 CGLIB？ | Consumer 接口用 JDK Proxy，无接口用 CGLIB；框架里一般优先 JDK |
| 为什么序列化选 Protobuf？ | 二进制体积小、序列化快、跨语言、有 .proto 强契约 |
| Netty 为什么适合 RPC？ | 基于 NIO 事件驱动，支持百万级长连接，内存零拷贝，吞吐高 |
| 注册中心挂了怎么办？ | Consumer 本地缓存服务列表，注册中心挂掉不影响已有调用；Provider 心跳检测自动剔除 |
| 负载均衡怎么做到无状态？ | 负载策略在 Consumer 端执行，每个 Consumer 独立计算，不依赖中心节点 |

## 关联知识点

- 动态代理（JDK Proxy / CGLIB / ByteBuddy）
- Netty 线程模型与零拷贝
- Protobuf / JSON / Hessian 序列化对比
- Zookeeper / Nacos / Consul 注册中心原理
- 熔断（Circuit Breaker）与限流（Rate Limiting）
- 一致性哈希负载均衡
