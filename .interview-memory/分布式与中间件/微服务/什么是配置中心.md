## 原始知识

- **集中管理**：配置与代码分离，统一管理
- **动态热更新**：不改代码不重启，配置实时生效
- **版本回滚**：支持历史版本查看与回退
- **灰度发布**：按机器/区域逐步推送新配置
- **常见中心**：Nacos、Apollo、Spring Cloud Config
- **推送方式**：推模式（服务端主动推）、拉模式（客户端定时拉）

## 顺口溜

> 配置集中管，热更新不重启，灰度回滚全。

## 一字一画面

**配** —— 像餐厅中央厨房配菜：所有配料（配置）统一放在厨房（配置中心），各分店（服务实例）按需取菜，厨房换菜谱（更新配置）不用关店（重启），还能先试卖三天（灰度），不行就换回老菜单（回滚）。（本质：配置与实例解耦，集中管控）

## 面试话术

「配置中心主要是把配置和代码解耦，做到集中管理、动态热更新。比如线上要改个超时时间，不用改代码发版，直接在配置中心改就能实时生效。常见的有 Nacos、Apollo、Spring Cloud Config，推送方式有推模式和拉模式两种。」

## 防混补丁

- **容易和注册中心搞混**：注册中心管「服务在哪」，配置中心管「服务怎么配」。
- **补丁口诀**：注册中心找人，配置中心发料；一个管地址，一个管参数。

## 追问预演

1. **Q：Apollo 和 Nacos 的区别？**
   A：Apollo 功能更全面（灰度、审计、版本管理），Nacos 兼顾注册中心和配置中心，部署更简单。
2. **Q：推模式和拉模式各有什么优缺点？**
   A：推模式实时性好，但服务端压力大；拉模式实现简单，有延迟但稳定性高。
3. **Q：配置热更新怎么保证一致性？**
   A：通常结合版本号 + 本地缓存 + 异步刷新，部分场景需配合熔断/降级策略。
