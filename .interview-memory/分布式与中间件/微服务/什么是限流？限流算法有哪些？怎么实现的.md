# 什么是限流？限流算法有哪些？怎么实现的？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 桶 | 桶漏窗，三算法 | 令牌桶可突发，漏桶强制平滑，滑动窗口更精确 |

## 原始知识

- **限流定义**：控制单位时间内的请求数量，保护系统不被打爆
- **令牌桶**：匀速产生令牌，桶满丢弃，突发流量可处理（Guava RateLimiter）
- **漏桶**：请求入桶，匀速流出，强制平滑，无法应对突发
- **滑动窗口**：固定窗口改进版，分段统计更精确（Sentinel）
- **计数器**：简单直观，但边界突刺问题严重
- **实现**：Guava RateLimiter、Sentinel、Nginx limit_req

## 顺口溜

令牌桶能突发，漏桶流水匀速，滑动窗口分段，计数器别独舞。

## 单字锚点 + 防崩推导

- **锚点字**：桶
- **扩展口诀**：桶漏窗，三算法
- **为什么选它**：限流算法中最经典、面试最常考的就是令牌桶，"桶"字形象且贯穿多种算法
- **推导链**：
  1. 桶 → 令牌桶 → 匀速产令牌 → 请求拿令牌通行 → 桶有容量 → 突发流量可缓冲 → Guava RateLimiter
  2. 桶 → 漏桶 → 请求像水倒入桶 → 底部匀速漏水 → 强制平滑 → 无法应对突发峰值
  3. 桶 → 窗 → 滑动窗口 → 固定窗口切多段 → 每段单独计数 → 汇总判断 → 更精确无突刺
  4. 桶漏窗 → 计数器作为补充 → 简单但边界问题大

## 一字一画面

- **桶** —— 像地铁闸机口的 token 桶：工作人员每秒往桶里扔 10 个筹码（令牌），乘客拿筹码才能进站，桶最多装 100 个，突然来一大波人桶里有存货就能快速放行（突发），用完了就得等下一批筹码
- **漏** —— 像输液吊瓶：药水一滴一滴匀速下落，不管上面倒得多快，下面只能按滴速走，强制平滑，想快也快不了
- **窗** —— 像百叶窗：把大窗户切成很多小叶片（时间分段），每片单独看透光量，汇总才知道整体亮度，比整块玻璃（固定窗口）判断准得多

## 面试话术

1. 限流就是控制单位时间内系统能处理的请求量，防止流量突增把服务打挂，常见的算法有令牌桶、漏桶和滑动窗口。
2. 实际工作中我用 Guava RateLimiter 做单机限流，用 Sentinel 做分布式限流和熔断，令牌桶适合允许一定突发的场景，漏桶适合要求严格匀速的场景。

## 防混补丁

- **容易和熔断搞混**：限流是"太多人排队，门口拦住"，熔断是"后端已炸，直接拒绝"；限流防的是流量大，熔断防的是故障扩散。
- **补丁口诀**：限流看流量，熔断看健康。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| 令牌桶和漏桶的本质区别？ | 令牌桶允许突发（桶里有存货就能快速消费），漏桶强制平滑（流出速率恒定） |
| 固定窗口有什么问题？ | 窗口边界突刺，如 1:59 和 2:01 各来一波，实际 2 秒内超量 |
| Sentinel 滑动窗口怎么实现？ | 将时间划分为多个小窗口（如 500ms），每个窗口独立计数，滑动时汇总近 N 个窗口 |
| 分布式限流怎么实现？ | Redis + Lua 原子脚本，或 Sentinel 集群模式，全局共享计数器 |
| 限流满了返回什么？ | 通常返回 429 Too Many Requests，或自定义降级页面 / 兜底数据 |
