## 速记卡片

降 | 降替默，三策略 | 主动牺牲非核心，保核心功能运行

## 原始知识

- **系统负载过高**时主动牺牲非核心功能
- 保核心功能继续运行
- 策略：**返回默认值**
- 策略：**缓存旧数据**
- 策略：**简化流程**
- 策略：**关闭非核心功能**
- **熔断是被动触发**，降级可主动配置
- Hystrix/Sentinel 实现
- 触发条件：**QPS** / **响应时间** / **错误率**

## 顺口溜

负载高了先降级，非核心功能让路行。
替默简闭四把刀，Hystrix 里配得清。

## 单字锚点 + 防崩推导

- **锚点字**：降（降级）
- **扩展口诀**：降替默，三策略
- **为什么选它**："降"字直接对应"服务降级"，一听就懂是系统扛不住时往下压
- **推导链**：
  1. 降 → 系统负载过高 → 扛不住了
  2. 降 → 主动降下来 → 不是等崩了才动
  3. 降 → 降什么？→ 非核心功能
  4. 降 → 保什么？→ 核心功能必须稳
  5. 降 → 怎么降？→ 替（替换默认值）、默（静默/缓存旧数据）、简（简化流程）、闭（关闭非核心）
  6. 降 → 和熔断区别？→ 熔断是被动的保险丝烧断，降级是主动的预先裁员
  7. 降 → 谁来执行？→ Hystrix、Sentinel 这些组件
  8. 降 → 什么时候触发？→ QPS 太高、响应时间太长、错误率飙升

## 一字一画面

**降** —— 像商场限流时拉下黄色卷帘门：保安主动把侧门锁了，只留正门让顾客进（本质：主动收缩服务边界，保主通道畅通）

**替** —— 像餐厅高峰期用预制菜代替现炒：味道差点但能快速上桌（本质：返回默认值，牺牲质量保响应）

**默** —— 像电话占线时播放录音："您拨打的用户正在通话中"（本质：用缓存旧数据或静默处理，不给后端加压）

## 面试话术

"服务降级就是系统扛不住的时候，主动把非核心功能砍掉，保核心链路还能跑。比如电商大促时，商品推荐可以降级成走缓存甚至不展示，但下单支付绝对不能挂。"

"降级和熔断不一样，熔断是保险丝烧断了被动保护，降级是我们提前配置好的主动策略，像 Hystrix 和 Sentinel 都支持按 QPS、响应时间、错误率这些指标来触发。"

## 防混补丁

**容易和熔断搞混**？

补丁口诀：**"熔断是被动的保险丝烧断，降级是主动的人才优化"**

- 熔断：下游服务挂了，我被动断开保护自己
- 降级：我自己扛不住了，主动放弃非核心业务

## 追问预演

**Q：降级和熔断可以同时用吗？**
A：可以。熔断保护调用方不被拖垮，降级保护系统整体资源，两者是互补关系。

**Q：降级数据不一致怎么办？**
A：降级一般返回缓存或默认值，业务上能接受短暂不一致；核心数据不会降级。

**Q：Sentinel 降级策略有哪些？**
A：平均响应时间、异常比例、异常数三种熔断降级策略，还有系统负载保护。

**Q：怎么判断哪些功能可以降级？**
A：按业务优先级划分，核心交易链路不可降级，边缘功能如推荐、评论可以降级。
