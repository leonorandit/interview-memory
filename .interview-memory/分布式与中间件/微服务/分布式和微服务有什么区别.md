# 分布式和微服务有什么区别？

## 速记卡片
`拆 | 拆独治，四区别 | 微服务是分布式子集`

## 原始知识
- 分布式：**系统部署方式**
- 分布式：多机协同/解决单点瓶颈
- 微服务：**架构风格**
- 微服务：按业务能力拆分
- 微服务：**小团队独立运维**
- 微服务：**去中心化治理**
- 微服务**一定**是分布式
- 分布式**不一定**是微服务
- 微服务强调**业务拆分和独立部署**
- 分布式强调**系统协作和数据一致性**

## 顺口溜
**拆独治，四区别**

## 单字锚点 + 防崩推导
- **锚点字**：**拆**
- **扩展口诀**：**拆独治，四区别**
- **为什么选它**："拆"是微服务最显性的特征，也是区分它和一般分布式的核心——按业务拆、拆得细、独立治理。
- **防崩推导**：
  - 拆 → 按业务能力拆分服务（订单服务、用户服务）→ 拆得细
  - 独 → 每个服务**独立开发、独立部署、独立数据库** → 团队自治
  - 治 → 去中心化治理，服务间通过轻量协议通信（HTTP/gRPC）→ 而非集中式ESB总线
  - 四区别 → ①概念范畴不同（部署方式vs架构风格）②拆分维度不同（物理部署vs业务逻辑）③团队组织不同（共享vs独立小团队）④治理方式不同（集中式vs去中心化）
  - 核心关系：**微服务一定是分布式的**（独立部署必然跨进程/跨机器），**但分布式不一定是微服务**（比如传统SOA集中式治理，或单纯的数据库主从集群）

## 一字一画面
- **拆** —— 像**乐高积木按颜色拆分**：把混在一起的红黄蓝积木分开装盒，每盒只一种颜色，但盒与盒之间可以组合搭城堡。（本质：按业务能力垂直拆分，高内聚低耦合）
- **独** —— 像**独立门牌号的联排别墅**：每户有自己的水电表（独立数据库），自己装修自己住（独立部署），一家漏水不影响邻居。（本质：服务自治，故障隔离）
- **治** —— 像**小区自治业委会**：没有中心物业总管一切（去中心化），各家有问题直接敲门沟通（点对点通信），而不是先去物业前台排队。（本质：去中心化治理，轻量通信）
- **区** —— 像**拼图vs积木**：分布式是把一幅大图切成几块拼（注重拼起来完整），微服务是把积木按类型分盒（注重每盒独立又能组合）。（本质：分布式重协作，微服务重自治）

## 面试话术
- "分布式和微服务，我用六个字概括：**拆独治，四区别**。分布式是一种系统部署方式，强调多台机器协同工作，解决单点性能和可靠性问题；微服务是一种架构风格，核心是按业务能力拆分，每个服务由小团队独立开发、独立部署、独立运维。关键关系是：**微服务一定是分布式的，但分布式不一定是微服务**。比如一个单体应用做成分布式部署（多机负载均衡），它是分布式但不是微服务；而一个微服务系统必然部署在多台机器上。四个核心区别：概念范畴、拆分维度、团队组织、治理方式。"
- "实际选型上，小团队别盲目上微服务，运维复杂度会拖死你。演进路径一般是：单体 → 分布式单体 → 微服务。"

## 防混补丁
- 容易和"SOA"搞混。SOA也是服务化，但是集中式治理（ESB总线），拆分粒度更粗。
- 补丁：**微服务是轻量SOA，去掉了ESB大管家**

## 追问预演
1. **微服务一定比分布式单体好吗？**
   - 不是。团队小、业务简单时，微服务带来的运维复杂度（服务发现、链路追踪、分布式事务）可能得不偿失。演进路径一般是：单体 → 分布式单体 → 微服务。

2. **微服务拆分粒度的标准是什么？**
   - 围绕业务能力（DDD限界上下文），一个服务对应一个业务边界，团队能"两个披萨"吃完（Amazon两个披萨团队原则，8-10人）。太细了网络开销大，太粗了失去微服务意义。

3. **分布式系统的核心挑战是什么？**
   - 网络分区、数据一致性、CAP权衡、服务间通信延迟和容错。微服务在此基础上还要解决服务治理、链路追踪、配置中心、分布式事务等问题。

4. **单体应用做成分布式部署算微服务吗？**
   - 不算。只是同一套代码部署多份做负载均衡，没有按业务拆分数据库和代码，属于分布式但不是微服务。微服务必须按业务能力垂直拆分。

5. **分布式和集群有什么区别？**
   - 集群是多台机器做同一件事（如MySQL主从、Nginx负载均衡），强调"同构复制"；分布式是多台机器做不同的事协同完成一个业务，强调"异构协作"。
