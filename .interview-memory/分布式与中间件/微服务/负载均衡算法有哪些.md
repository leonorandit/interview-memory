# 负载均衡算法有哪些？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 均 | 均权最，六算法 | 轮询依次分，加权按比重，最小连接看负载，IP Hash 保会话 |

## 原始知识

- **轮询（Round Robin）**：依次轮流分配，公平简单
- **随机（Random）**：随机选一台，实现简单
- **加权轮询（Weighted Round Robin）**：按权重比例分配，性能好就多分
- **加权随机（Weighted Random）**：按权重随机，实现更简单
- **最小连接数（Least Connections）**：当前连接最少的优先分，适合长连接
- **IP Hash**：同 IP 固定分配到某台机器，保持会话（Session）
- **一致性哈希（Consistent Hashing）**：减少节点变动影响，虚拟节点防倾斜

## 顺口溜

轮询排队来，加权看能耐，最小连接选清闲，IP Hash 不挪窝。

## 单字锚点 + 防崩推导

- **锚点字**：均
- **扩展口诀**：均权最，六算法
- **为什么选它**：负载均衡的核心目标就是"均匀"分配流量，"均"字直接点题，最易联想
- **推导链**：
  1. 均 → 均匀 → 轮询（Round Robin）→ 依次排队 → 最公平
  2. 均 → 权 → 权重 → 机器性能不同 → 加权轮询 / 加权随机 → 能者多劳
  3. 均 → 最 → 最小连接数（Least Connections）→ 谁最闲分给谁 → 动态负载感知
  4. 均权最 → 六算法 → 加上随机、IP Hash、一致性哈希（共 7 种，口诀六算法取最常见的六类）

## 一字一画面

- **均** —— 像发扑克牌：庄家一张一张轮流发给每个玩家，不偏不向，人手一张（轮询）
- **权** —— 像分蛋糕按饭量：大个子（高性能机器）多分一块，小朋友（低配机器）少分一块，按能力比例切（加权轮询）
- **最** —— 像机场选安检口：旅客抬头看哪个口排队最短，直奔最闲那个窗口（最小连接数）
- **I** —— 像固定座位：同一个手机号（IP）每次来都坐同一个座位（机器），东西不用挪（会话保持）

## 面试话术

1. 负载均衡算法常见的有六种：轮询、随机、加权轮询、加权随机、最小连接数和 IP Hash，另外一致性哈希在缓存场景也很常用。
2. 实际选型要看场景：无状态服务用轮询或加权轮询，有长连接或会话保持需求用 IP Hash 或最小连接数，缓存分片用一致性哈希减少节点变动影响。

## 防混补丁

- **容易和一致性哈希搞混**：一致性哈希不是负载均衡的通用算法，它主要是解决缓存节点增减时大量 key 失效的问题，普通 HTTP 请求负载均衡一般不用它。
- **补丁口诀**：负载轮询权最 IP，缓存场景一致哈希。

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| Nginx 默认用什么算法？ | 轮询（Round Robin），可配 weight 变加权轮询 |
| 最小连接数怎么获取连接数？ | 负载均衡器维护后端连接计数，每次分配选计数最小的 |
| IP Hash 有什么问题？ | 某 IP 流量极大时单台机器被打爆，且 NAT 出口多用户共享一个 IP |
| 一致性哈希虚拟节点作用？ | 防止真实节点少时哈希环分布不均，虚拟节点让负载更平滑 |
| 加权轮询和加权随机哪个好？ | 加权轮询更平滑均匀；加权随机实现更简单，但可能短期不均匀 |
