# 说一下消息队列的模型有哪些？

## 速记卡片
`模 | 模点对，两模型 | P2P单消费，Pub/Sub广播`

## 原始知识
- **点对点模型（P2P）**：队列每条消息只被**一个消费者消费**
- **发布订阅模型（Pub/Sub）**：Topic **广播**给多个订阅者
- **Kafka**：分区 + **消费者组**，组内每条消息只被一个消费者消费（P2P），多组订阅则广播（Pub/Sub）
- **RabbitMQ**：通过 **Exchange** 交换机类型（direct/topic/fanout/headers）实现不同路由策略
- **易混点**：P2P 的"一个消费者"是逻辑概念，物理上可以是集群；Pub/Sub 的订阅者是独立的消费组
- **追问点**：Kafka 消费者组怎么 rebalance？RabbitMQ fanout 和 direct 区别？

## 顺口溜
**模点对，两模型**

## 单字锚点 + 防崩推导
- **锚点字**：**模**
- **扩展口诀**：**模点对，两模型**
- **为什么选它**：问题问"模型"，锚点字"模"天然挂钩，"点对"谐音"队点"，一听就想到队列和点对点。
- **防崩推导**：
  - 模 → 模型 → 两种模型
  - 点 → **点对点（P2P）** → 队列模式 → 一条消息进队 → 只能被**一个**消费者拿走
  - 对 → ** Pub/Sub** → 发布订阅 → 一对多 → 一条消息上 Topic → **多个**订阅者都能收到
  - 两模型之外 → Kafka 分区消费者组（P2P + Pub/Sub 混合）、RabbitMQ 交换机类型（路由策略）

## 一字一画面
- **点** —— 像**银行取号排队**：一个号码只能被**一个**窗口叫走，叫完就作废。（点对点，单消费）
- **对** —— 像**小区业主群@所有人**：物业发一条通知，群里每个业主手机上都能收到。（发布订阅，广播）
- **模** —— 像**乐高两种基础底板**：一种底板只能插一个系列（P2P），另一种底板能同时兼容多个系列堆叠（Pub/Sub）。（模型分类）
- **队** —— 像**食堂打饭窗口排一队**：饭一份份打，一个人领一份，不能一份饭两个人分。（队列的单消费特性）

## 面试话术
- "消息队列主要有**两种模型**，口诀是**模点对**：**点对点（P2P）** 和 **发布订阅（Pub/Sub）**。P2P 就像银行取号，一条消息只能被一个消费者消费；Pub/Sub 就像微信群广播，Topic 上的每个订阅者都能收到同一条消息。"
- "实际产品中，**Kafka** 用**消费者组**巧妙融合了两种模型：组内是 P2P（一条消息只被组内一个消费者消费），不同组之间是 Pub/Sub（多组都能订阅同一个 Topic）。**RabbitMQ** 则通过 **Exchange** 的四种类型（direct/topic/fanout/headers）来灵活控制路由策略。"

## 防混补丁
- 容易把 P2P 的"一个消费者"理解成物理上的一个进程。补丁：**P2P 的一个消费者是逻辑消费组，物理上可以是一个集群**。Kafka 消费者组就是典型例子。

## 追问预演
1. **问**：Kafka 消费者组 rebalance 是什么？
   **答**：当消费者加入或退出组时，Coordinator 重新分配分区给消费者，确保分区均匀负载，但 rebalance 期间消费会暂停。
2. **问**：RabbitMQ fanout 和 direct 有什么区别？
   **答**：fanout 是广播，无视 routing key，所有绑定队列都收到；direct 是按精确 routing key 匹配投递。
3. **问**：什么场景用 P2P，什么场景用 Pub/Sub？
   **答**：任务分发、负载均衡用 P2P；事件通知、多系统数据同步用 Pub/Sub。
4. **问**：消息队列怎么保证顺序性？
   **答**：单分区/单队列 + 单消费者，或者通过消息 key 哈希到固定分区来保证局部有序。
