# RabbitMQ 怎么实现延迟队列？

## 速记卡片

| 锚点字 | 口诀 | 核心差异 |
|--------|------|----------|
| 延 | 延T死，两法延 | TTL+DLX原生组合 vs 插件直接支持 |

## 原始知识

- **TTL**：消息或队列设置过期时间
- **DLX**：死信交换机，过期消息的"回收站"
- **插件法**：`rabbitmq_delayed_message_exchange` 直接声明 x-delayed-message 类型交换机
- **轮询法**：定时任务扫描数据库/Redis，再发消息（兜底方案）
- **核心链路**：消息 → 带TTL的队列 → 过期 → DLX → 死信队列 → 消费

## 顺口溜

> 延T死，两法延，
> 插件直接最省盐，
> 原生TTL配死信，
> 轮询兜底别嫌烦。

## 单字锚点 + 防崩推导

**锚点字：延**

- **扩展口诀**：延T死，两法延
- **为什么选它**：延迟队列的核心就是"延"——延长时间、延后处理
- **推导链**：
  1. 延 → 延迟队列 → 需要"让消息等一会儿"
  2. 等一会儿 → 设置 **T**TL（Time To Live）过期时间
  3. TTL到了 → 消息变"死信" → 进 **死**信交换机（DLX）
  4. 怎么延？→ **两法**延：
     - 法一：原生 TTL + DLX（队列级/消息级TTL）
     - 法二：插件 `rabbitmq_delayed_message_exchange`（交换机级延迟）
     - 法三（兜底）：定时任务轮询

## 一字一画面

**延** —— 像【红灯倒计时】：数字从60跳到0（红色数字闪烁，滴答声），0变绿灯瞬间车辆启动（本质：时间一到，触发动作）

**T** —— 像【沙漏】：上半部沙子缓缓流下，最后一粒落下时底部亮起红灯（本质：TTL是时间容器，漏完即过期）

**死** —— 像【医院急诊绿色通道】：普通病房（原队列）病人心跳停止，自动滑入绿色推床（死信队列），推床直奔手术室（DLX路由）（本质：过期/异常消息的转移通道）

## 面试话术

> "RabbitMQ 实现延迟队列主要有两种方式。第一种是原生的 TTL 加死信交换机，消息在队列里待到过期，自动被转发到死信队列消费；第二种是安装 `rabbitmq_delayed_message_exchange` 插件，直接声明延迟交换机，更简洁。如果要求不高，也可以用定时任务轮询兜底。"

## 防混补丁

**容易和什么搞混？** 把 TTL 设置成队列级别还是消息级别——队列级 TTL 会让整个队列的消息统一过期，消息级 TTL 是每条消息单独算时间。补丁口诀：**"队列统一消息单，混用先死谁短算。"**

## 追问预演

| 追问 | 简要答案 |
|------|----------|
| TTL 设置在队列和消息上有什么区别？ | 队列级：整个队列统一过期时间；消息级：单条消息独立计算，混用时以较短者为准 |
| 插件法和原生法各有什么优缺点？ | 插件法简洁、精确，但需安装插件、社区维护；原生法无依赖，但配置复杂、不易管理大量延迟级别 |
| 如果消息还没过期，消费者挂了怎么办？ | 消息仍在队列中等待，不影响TTL计时，过期后正常进入死信队列 |
| 大量延迟消息会造成什么问题？ | 消息堆积在内存，可能导致内存耗尽；插件法将消息存在Mnesia，也有上限 |
