# RocketMQ 的事务消息有什么缺点？你还了解过别的事务消息实现吗？

## 速记卡片
`缺 | 缺耦回，三缺点 | 侵入强、负载高、吞吐低`

## 原始知识
- **侵入性强**：业务代码需改，实现**本地事务** + **回查接口**
- **系统负载高**：Broker **定时回查**，增加网络压力和业务查询量
- **单条吞吐量低**：一条消息走**两阶段提交**，延迟高于普通消息
- **其他实现**：**本地消息表**（DB + MQ）、**Saga 模式**（长事务编排）、**TCC**（Try-Confirm-Cancel）、**Seata AT 模式**（自动补偿）
- **易混点**：TCC 和 Saga 都属于分布式事务，不是消息队列事务
- **追问点**：Seata AT 如何实现无侵入？Saga 适合什么场景？

## 顺口溜
**缺耦回，三缺点**

## 单字锚点 + 防崩推导
- **锚点字**：**缺**
- **扩展口诀**：**缺耦回，三缺点**
- **为什么选它**：问题直接问"缺点"，锚点字"缺"天然挂钩，三缺点压成一句话。
- **防崩推导**：
  - 缺 → 缺点 → 三个缺点
  - 耦 → **侵入性强（耦合高）**：业务代码里要嵌本地事务和回查逻辑
  - 回 → **回查负载高**：Broker 定时回查，像催债电话不断打
  - 缺（第三个）→ **吞吐量低缺速度**：两阶段提交，单条消息延迟大
  - 其他实现 → 本地消息表、Saga、TCC、Seata AT → 四种替代方案

## 一字一画面
- **缺** —— 像**拼图上缺一块**：RocketMQ 事务消息很好，但缺了"无侵入"这块拼图。（缺点总起）
- **耦** —— 像**手机充电线缠在主机上**：拔不下来，手机和主机强绑在一起。（业务侵入性强）
- **回** —— 像**催债公司每天打电话**："钱到账了没？" 烦人且占线。（回查增加系统负载）
- **低** —— 像**高速收费站只开一个口**：车都能过，但一辆辆排队，整体速度慢。（单条吞吐量低）

## 面试话术
- "RocketMQ 事务消息主要有三个缺点，口诀是**缺耦回**：一是业务**侵入性强**，需要修改代码实现本地事务和回查接口；二是**回查机制**会增加系统负载；三是单条消息要走两阶段提交，**吞吐量相对较低**。"
- "除此之外，我还了解过其他实现方式：比如**本地消息表**，把消息持久化到业务 DB 再异步投递；**Saga 模式**适合长事务编排；**TCC** 侵入性也比较大，但一致性更强；还有 **Seata AT 模式**，基于 UndoLog 自动补偿，业务基本无侵入。"

## 防混补丁
- 容易把 TCC/Saga 当成消息队列的替代方案。补丁：**MQ事务是异步最终一致，TCC/Saga 是同步分布式事务，场景不同别乱套**。

## 追问预演
1. **问**：本地消息表怎么实现？
   **答**：业务表旁边加消息表，本地事务一起写，后台定时扫描未发送消息投递到 MQ，成功后改状态。
2. **问**：Seata AT 如何实现无侵入？
   **答**：Seata 代理数据源，自动解析 SQL 生成前后镜像（UndoLog），事务失败自动回滚，业务代码无感知。
3. **问**：Saga 适合什么场景？
   **答**：适合业务流程长、需要人工干预或外部系统多的场景，比如旅游预订（机票+酒店+租车）。
4. **问**：什么时候选 RocketMQ 事务消息，什么时候选 TCC？
   **答**：异步解耦、容忍最终一致选 RocketMQ；短事务、强一致、资源可预留选 TCC。
