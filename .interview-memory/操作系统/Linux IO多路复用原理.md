# Linux IO多路复用原理

## 速记卡片
`复 | 一复多，红黑触 | 一线监控多路`

## 原始知识
- **问题背景**：C10K问题，一个连接一个线程，上下文切换开销大
- **核心思想**：一个线程**复用**监控多个fd，哪个就绪处理哪个
- **Select**：bitmap数组，**1024限制**，每次拷贝到内核，轮询O(n)
- **Poll**：链表，**无数量限制**，但仍需轮询O(n)
- **Epoll**：**红黑树**存fd + **就绪链表** + 回调机制，事件驱动O(1)
- **LT水平触发**：fd就绪后**持续通知**，直到处理完
- **ET边沿触发**：状态变化时**只通知一次**，必须一次处理完
- **易混点**：多路复用=异步？不是！多路复用是同步非阻塞
- **追问点**：epoll为什么高效？ET和LT区别？

## 顺口溜
**一复多，红黑触** —— *一服多，红黑触*

## 单字锚点 + 防崩推导
- **锚点字**：**复**
- **扩展口诀**：**一复多，红黑触**
- **为什么选它**：IO多路复用的灵魂就是"复用"——复用一个线程监控多路连接。
- **防崩推导**：
  - 复 → **复用** → 一个线程**复用**监控**多**个fd → 解决C10K问题
  - 一 → **一个线程** → 避免多线程上下文切换开销
  - 多 → **多路** → 成千上万个连接同时监控
  - 黑 → **红黑树** → epoll用红黑树管理所有注册的fd，增删查O(log n)
  - 触 → **事件触发** → fd就绪时内核**回调**触发，加到就绪链表
  - 就绪 → 只处理**就绪的** → epoll_wait直接返回活跃fd，O(1)
  - 演进 → Select→Poll→Epoll，从轮询到事件驱动的三代演进

## 一字一画面
- **一** —— 像**一个全能服务员**：一个人同时服务全场十张桌子，不是一桌配一个服务员。（单线程复用）
- **复** —— 像**瑞士军刀**：一把刀展开多种功能，工具复用。（复用思想）
- **多** —— 像**监控室的十块屏幕**：一个保安同时盯着十个画面，哪块有动静看哪块。（多路监控）
- **黑** —— 像**机场航班排序屏**：按航班号有序排列（红黑树），查找、新增、取消都很快。（有序管理）
- **触** —— 像**门口红外感应器**：有人经过自动亮灯并记录（回调机制），不用保安一直盯着门。（事件触发）
- **就绪** —— 像**叫号系统的"请到X号窗口"**：只处理被叫到的号，不用遍历所有排队的人。（只处理活跃的）

## 面试话术
- 问原理 → "IO多路复用的核心是**一复多，红黑触**。传统BIO一个连接一个线程，C10K时上下文切换拖垮系统。多路复用让一个线程同时监控成千上万个fd。epoll内部用红黑树管理所有注册fd，fd就绪时内核通过回调把它加到就绪链表，epoll_wait直接返回活跃的连接，O(1)复杂度。"
- 问场景 → "高并发服务端都用epoll，比如Nginx、Redis。Nginx单worker进程就能处理几万并发连接，靠的就是epoll的事件驱动机制。"

## 防混补丁
- 容易把"IO多路复用"当成"异步IO"。补丁：**多路复用是同步非阻塞**。数据拷贝阶段用户进程还是要自己等，只是监控阶段不阻塞。真正的异步IO是内核拷贝完才通知你，Linux网络AIO很少见。

## 追问预演
1. **为什么需要IO多路复用？** → C10K问题。一个连接一个线程，线程数爆炸导致上下文切换开销巨大；多路复用一个线程监控所有连接，资源消耗低。
2. **epoll为什么比select/poll高效？** → 1）没有重复拷贝fd集合到内核；2）不需要遍历所有fd轮询，只处理就绪的；3）内核通过回调维护就绪链表，epoll_wait直接返回。
3. **ET和LT有什么区别？** → LT水平触发：fd就绪后持续通知，直到处理完，编程简单。ET边沿触发：状态变化时只通知一次，必须一次读完，效率高但编程复杂，容易漏事件。
4. **epoll为什么用红黑树？** → 增删查都是O(log n)，且是有序结构，方便内核管理大量fd；epoll_ctl操作fd时效率高。
5. **Redis为什么用epoll？** → Redis是单线程模型，必须高效监控大量客户端连接。epoll让单线程同时处理成千上万个socket的就绪事件，避免阻塞和轮询，实现高性能网络IO。
