# Select、Poll、Epoll 之间有什么区别？

## 速记卡片
`表 | 表大触，三演进 | 轮询到事件驱动`

## 原始知识
- **Select**：fd_set数组，**1024限制**
- **每次拷贝到内核**，轮询O(n)
- **Poll**：链表，**无数量限制**
- **仍需轮询**，遍历所有fd
- **Epoll**：红黑树+就绪链表
- **事件驱动O(1)**，只处理就绪的
- **支持ET和LT**：边沿触发和水平触发
- **适用场景**：Epoll高并发，Select/Poll少量连接
- **易混点**：Poll也有数量限制？没有
- **追问点**：ET和LT区别？epoll为什么用红黑树？

## 顺口溜
**表大触，三演进** —— *表大触，三演镜*

## 单字锚点 + 防崩推导
- **锚点字**：**表**
- **扩展口诀**：**表大触，三不同**
- **为什么选它**：三者都是"监视表"机制，"表"是共同内核，差异在表怎么管。
- **防崩推导**：
  - 表 → 都是监控fd的**表** → 但表的结构和查法不同
  - 表 → **Select**用固**定**数组 → **1024上限** → 每次拷贝到内核 → 轮询O(n)
  - 大 → **Poll**用链**表** → 没上限（但链表**大**了也慢） → 仍要轮询O(n)
  - 触 → **Epoll**事件**触**发 → 红黑树存所有fd + 就绪链表存活跃的 → 只处理**触**发的 → O(1)
  - 三演进 → Select→Poll→Epoll 是Linux IO多路复用的三代演进
  - 结论 → 高并发用Epoll，少量连接Select/Poll够用

## 一字一画面
- **表** —— 像**点名册**：老师拿本册子点名，看谁到了。（fd监控表）
- **表（Select）** —— 像**固定32页的点名册**：最多记1024人，每次上课都重新复印一份给班长（拷贝到内核），班长从头到尾念名字（轮询O(n)）。
- **大（Poll）** —— 像**活页夹点名册**：想加多少页加多少页（无数量限制），但班长还是要从头到尾念（轮询O(n)），人多了照样慢。
- **触（Epoll）** —— 像**教室门口装感应器**：学生进门自动亮灯加到"已到名单"（就绪链表），老师只看亮灯的名单（事件驱动O(1)），不用点名。感应器背后有个总控面板按学号排序（红黑树），方便管理所有学生。

## 面试话术
- 问区别 → "口诀是**表大触，三演进**。Select用固定数组，有1024限制，每次调用拷贝fd到内核，轮询O(n)；Poll用链表，无数量限制，但仍需轮询O(n)；Epoll用红黑树加就绪链表，事件驱动，只处理就绪的fd，O(1)。Epoll适合高并发连接，Select和Poll适合少量连接。"
- 问原理 → "Epoll内部用红黑树管理所有注册的fd，保证增删查是O(log n)；有fd就绪时内核通过回调把它加到就绪链表；epoll_wait直接返回就绪链表，不用遍历所有fd。"

## 防混补丁
- 容易觉得Poll比Epoll好因为Poll也"无限制"。补丁：**无限制≠高性能**。Poll还是轮询，连接数一上来O(n)拖死；Epoll是事件驱动，连接数再多也只处理活跃的。

## 追问预演
1. **ET和LT有什么区别？** → LT水平触发：fd就绪后一直通知，直到处理完。ET边沿触发：状态变化时只通知一次，必须一次处理完，效率更高但编程复杂。
2. **epoll为什么用红黑树？** → 增删查都是O(log n)，且是有序树，方便内核管理大量fd；epoll_ctl增删fd时效率高。
3. **select的1024限制能改吗？** → 可以重编译内核改FD_SETSIZE，但通常不建议，因为数组大了轮询更慢。
4. **epoll_wait返回的是什么？** → 返回就绪的fd数组，用户直接处理这些fd即可，不用自己遍历所有fd。
5. **为什么epoll比select快？** → 1）没有重复拷贝fd集；2）没有遍历所有fd的轮询开销；3）内核通过回调机制直接维护就绪列表。
